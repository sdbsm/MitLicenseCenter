# Infrastructure Integration & Adapters

This document defines how the platform interacts with the underlying hosting infrastructure (1C Cluster, IIS, Windows Server). All infrastructure logic MUST be abstracted behind interfaces (Adapter pattern).

## 1. 1C Cluster Integration

### Sole adapter: RAS via `rac.exe` (ADR-16)

The 1C cluster adapter is exclusively `rac.exe` driven against a local `ras.exe` (listening on TCP 1545 by default). See **ADR-16** for why this is the sole adapter.

- **Get sessions:** `rac.exe <endpoint> session list --cluster=<uuid>`. The cluster UUID is resolved by `rac.exe <endpoint> cluster list` and **cached across calls** in `IClusterUuidCache` (singleton, key = `(ExePath, Endpoint)`, MLC-041) — a warm cache skips the `cluster list` spawn entirely; it re-resolves only on a cold cache, an endpoint/path change, or a stale-UUID error.
- **Kill session:** `rac.exe <endpoint> session terminate --cluster=<uuid> --session=<session-uuid>`. The kill reason is recorded in `AuditLog` only — it is **not** passed to `rac.exe` (no `--error-message` flag is emitted).
- **Authentication:** `--cluster-user=<user>` / `--cluster-pwd=<password>` from `Settings.OneC.Cluster.AdminUser` and `Settings.OneC.Cluster.AdminPassword`. Both can be left empty for clusters with no registered administrators (anonymous mode). **Accepted risk (ADR-21):** the password is passed on the `rac.exe` command line — readable by other processes/local admins for the brief life of each invocation — because supported `rac.exe` versions accept it **only** via `--cluster-pwd` (no stdin/file/env channel). On this single-node, admin-only host the residual risk is dominated by the operator's existing privileges; the value is DPAPI-encrypted at rest (ADR-8) and decrypted only at spawn time. Revisit if `rac.exe` ever gains a non-cmdline password channel. See **ADR-21**.
- Implementation: `RacExecutableRasClusterClient` (`Infrastructure/Clusters/`) wraps the process via `IRacProcessRunner` → `SystemProcessRacRunner`. Output is parsed by the pure-static `RacOutputParser` (CP866 OEM code page on RU Windows — see ADR-3.3 for the encoding caveat). Spawn budget: ≤ 26 `rac.exe` processes per minute under sustained over-quota load (warm UUID cache halves the steady-state hot-polling and kill-path spawn rate; MLC-041) — well within OS / antivirus tolerances on a single-node deployment. See **ADR-3.3** for the exact CLI contract, output grammar, kill-idempotency markers, and error semantics.
- **Tool path:** `Settings.OneC.RAS.ExePath` (version-specific, no seeded default since 1C 8.5 moved `rac.exe` out of `1cv8\common\` into `1cv8\<version>\bin\`). Set from the «Платформа 1С» picker on the «Параметры» page — picking an installed platform writes both this path and `OneC.DefaultPlatformVersion` (version parsed from the path); see ADR-3.3.
- **RAS endpoint:** `Settings.OneC.RAS.Endpoint` (no seeded default; `localhost:1545` is the conventional value — named only in the setting's description, matching the catalog table below which shows "—"). The «Параметры» page edits only the **port** (host fixed to `localhost`, single-node topology — §3) and stores `localhost:<port>`; the wire `host:port` format is unchanged.

### Idempotent kill protocol
Before issuing a kill, the adapter pipeline (`KillEnforcer` → `IClusterClient.KillSessionAsync`):
0. Skips any candidate younger than the kill grace period (`Enforcement.KillGraceSeconds`, default 15 s) — 1C fills `user-name` only after authentication, so this avoids terminating a just-connected session before its user is known. Newest-first ordering means the youngest is first, so a within-grace candidate stops the tenant's cycle (wait), not a fall-through to an older session. See DECISIONS.md "Kill grace period".
1. Re-fetches the target session by `SessionId` from the cluster snapshot.
2. Verifies that `(InfobaseId, AppID, StartedAt)` match the snapshot that triggered the decision.
3. Issues the kill only on match. A mismatch causes the kill to be skipped — the next reconciliation cycle will re-evaluate.
4. Treats `rac.exe` stderr `«Сеанс с указанным идентификатором не найден»` as a successful (idempotent) kill — the session is no longer there, which is the desired end state.

Every kill is recorded in `AuditLog` with reason (`LimitExceeded` or `ManualByAdmin`) and snapshot context. A blank `user-name` renders as «без пользователя» in the audit description (`SessionDisplay`).

### RAS health probe (ADR-16)

`RasHealthProbingService : BackgroundService` calls `IClusterClient.PingAsync` every 30s and publishes the result to `IRasHealthReader` (a singleton consumed by the `/api/v1/dashboard/summary` endpoint). The Dashboard surfaces three states on the RAS health card: `OK` (success), `Сбой` (danger, with last-error tooltip + consecutive-failures count), and `Проверка…` (neutral, only during the first 30s after backend startup before the initial probe completes). Health transitions are **not** written to `AuditLog` — the operator sees state in real time on the card.

### Long-lived RAS TCP socket (Strategy B) — deferred
Replacing `rac.exe`-per-cycle with a long-lived socket on 1545 is a backlog item (see `ROADMAP.md`), gated on real-world latency measurement. Until then, the spawn budget above is the binding contract.

### Unassigned cluster bases — resolve list + ignore-list (MLC-092/095, ADR-29)
Cluster bases present in the cluster but not entered in the panel are surfaced for the operator to resolve (assign to a client or hide); symmetrically, panel rows whose base has vanished from the cluster are surfaced as a drift signal. `GET /api/v1/infobases/unassigned` (Admin-only — Viewer sees nothing new) injects `IClusterClient` directly (the `DiscoveryEndpoints` precedent, ADR-20 / ADR-16 — no `rac.exe` in Web) and returns `UnassignedInfobasesResponse { Items[], HiddenItems[], MissingItems[], Available, Error, CheckedAtUtc }`:
- **Diff.** `Items` = the live RAS base list **minus** the registered `Infobase.ClusterInfobaseId`s **minus** the `HiddenClusterInfobases` rows; each item is `{ ClusterInfobaseId, Name, Description }`. `HiddenItems` (the ignore-list, `{ …, HiddenAtUtc, HiddenBy }`) is served from the DB and rendered **always**, even when RAS is down.
- **Reverse diff (MLC-095, direction (b)).** `MissingItems` = panel rows whose `ClusterInfobaseId` is **absent** from the cached RAS snapshot — a base registered in the panel but deleted/re-created in the cluster (a «dead soul»); each item is `{ InfobaseId, TenantName, Name, ClusterInfobaseId }` (joined to the tenant name by the `ReconciliationJob.infobaseMap` pattern). Computed from the **same** cached snapshot as `Items` and **only when `Available:true`** — on `Available:false` it is empty (a RAS poll failure is not a missing base, so there are no false drift marks). Read-only signal: there is no mutation here; the operator removes a dead row through the ordinary infobase-delete flow. The list `GET /infobases` is untouched — `MissingItems` rides this endpoint instead (its envelope is a critical Zod boundary, ADR-10.1).
- **Server-side TTL cache (60 s, ADR-29).** A singleton (`UnassignedInfobasesCache`, the `IClusterUuidCache` shape) caches **only the RAS snapshot** together with `CheckedAtUtc` (the actual poll time, not the response time); `?refresh=true` bypasses it. The **diff** (registered/hidden) is recomputed on every request, so create/hide/unhide are visible immediately without waiting for the TTL. A failed poll is cached too (it smooths RAS load on failure). The TTL is a **code constant**, not a settings key (the whitelist catalog is not grown — ADR-29).
- **Honest degraded signal.** RAS unavailable (adapter `Available:false` or an exception) → `Available:false` + a sanitized Russian `Error`; the raw `rac.exe` stderr (may carry server names/paths) goes only to the server log (the discovery pattern, MLC-009). `OperationCanceledException` propagates. The frontend hides the banner on `!Available` (no false zero).
- **Ignore-list mutations.** `POST /api/v1/infobases/unassigned/{clusterInfobaseId}/hide` (body `{ Name }` — the snapshot; length 1–200 validated by hand, since minimal-API DataAnnotations don't run at runtime — `CLAUDE.md` gotcha) and `DELETE …/{clusterInfobaseId}/hide` (Admin). Guards: hide of an already-registered base → `409 UNASSIGNED_ALREADY_ASSIGNED`; repeat hide → `409 UNASSIGNED_ALREADY_HIDDEN` (with a `PK_HiddenClusterInfobases` uniqueness backstop on the race); un-hide of an absent row → `404`; empty/over-200 `Name` → `400`. Both write audit (`UnassignedInfobaseHidden=14` / `UnassignedInfobaseUnhidden=15`, server-scope `tenantId: null`) via `HttpContext.AuditAsync`. `InfobasesEndpoints.CreateAsync` deletes the matching ignore-list row in the same `SaveChanges` when a base with that `ClusterInfobaseId` is created.

## 2. IIS & Publication Management

The panel performs three publication operations (ADR-4); it never enforces a desired `default.vrd` state.

### Publishing via `webinst`
`POST /api/v1/publications/{id}/publish` (re)creates a standard 1C web publication by running the `webinst` CLI through `IWebinstPublisher` (`OneCWebinstPublisher`, ADR-20):
- **Executable** = `…\1cv8\<PlatformVersion>\bin\webinst.exe`, resolved from the publication's `PlatformVersion` via the same install-root scan as platform discovery (`WebinstExeResolver` + `OneCInstallRoots`). The version determines which `webinst.exe` runs, and therefore the `wsisapi.dll` version baked into the generated `web.config`.
- **Arguments** (`WebinstArgs`): `-publish -iis -wsdir <virtualPath-without-slash> -dir <physical folder> -connstr "Srvr=<cluster>;Ref=<infobase name>;"`. The cluster address is the **host of `Settings.OneC.RAS.Endpoint`** (`host:port` → `host`) — single-host topology (ADR-28): the 1C cluster and RAS share the host, so there is no separate cluster-address key. The infobase name is `Infobase.Name`.
- **webinst overwrites `default.vrd` and `web.config` wholesale.** OData / HTTP-services / OpenID are therefore not panel-managed (ADR-4); set them in Designer or a hand-prepared `-descriptor` template afterwards. The publication's `Source` flag is set to `Webinst` on success; re-publishing a non-`Webinst`, currently-published publication requires explicit operator confirmation (`Confirm=true`), audited as `PublicationPublished = 212`.
- **Encoding:** `webinst` writes its output as **UTF-16LE** (unlike `rac.exe`'s OEM/CP866). The adapter decodes accordingly. On non-zero exit the raw output is logged server-side and `409 PUBLISH_FAILED` is returned with a sanitized Russian `Detail`.
- **Concurrency cap (MLC-046):** the spawn of `webinst` is wrapped in `IWebinstConcurrencyGate` (singleton `SemaphoreSlim`, default 3) inside `OneCWebinstPublisher`. A batch publish (the panel publishing a multi-selection — see `05_FRONTEND.md`) calls this same per-publication endpoint N times, so this cap is the real spawn-budget boundary (family of [ADR-3.3](DECISIONS.md#33-rac-cli-spawn-contract)): it holds regardless of caller — multiple operators, a future second consumer — not just the client's own pool. A single publish takes a free slot instantly (no behaviour change). Change-platform is a lightweight `web.config` edit and is **not** gated.

### Changing platform via `web.config`
`POST /api/v1/publications/{id}/change-platform` rewrites **only** the `wsisapi.dll` version segment in `web.config` (fallback: `default.vrd`, for older builds that keep the ISAPI handler there) to the requested version (`WsisapiVersionRewriter`, atomic temp-then-`File.Replace`). `default.vrd` content is otherwise untouched. The target version must be installed (validated against platform discovery). Audited as `PublicationPlatformChanged = 213`. `IIS`/IO failures map to `409 IIS_RECONCILE_FAILED` / `IIS_ACCESS_DENIED` with a sanitized `Detail` + `correlationId`.

### Read-only status
`OneCIisPublishingService.ReadActualStateAsync` reads the live IIS facts (site exists, virtual path exists, `web.config` exists, platform version from the `wsisapi.dll` path) via `Microsoft.Web.Administration.ServerManager` + filesystem; `PublicationStatusEvaluator` maps them to `Published / NotPublished / Error / Unknown`. It never mutates IIS. The status is refreshed by a 5-minute Hangfire job (`PublicationStatusRefreshJob`, throttled to `Settings.Drift.IntervalMinutes`) and on demand by `POST /api/v1/publications/{id}/check`; results land in `Publication.LastCheckStatus` / `LastCheckAt` / `LastCheckDetails`. No audit is written for a status read.

### Path resolution
`VrdPathResolver` resolves the publication's physical folder (and the `default.vrd` / `web.config` paths under it): **override-first** (`PhysicalPathOverride`, prefilled by the add/edit form as `{Settings.IIS.DefaultVrdRoot}\{databaseName}`, editable), else **convention** `{Settings.IIS.DefaultVrdRoot}/{siteName}/{trimmedVirtualPath}`. If the path is wrong, status reads `NotPublished` even for a healthy publication — set `PhysicalPathOverride` to the exact physical folder shown in IIS Manager.

### IIS lifecycle management (MLC-047, ADR-24)
Beyond the three publication operations, the panel manages the IIS **hosting lifecycle** from a dedicated «Управление IIS» block on the Publications page. This is server-scope (not per-publication) and goes through a separate Application port `IIisLifecycleService` → `OneCIisLifecycleService` (ADR-20 unchanged):
- **Operations.** Application pool `recycle` / `start` / `stop`, site `start` / `stop` / `restart` (all via `ServerManager` runtime commands — no `CommitChanges`), and server-wide **`iisreset`** — restart (no args), `/stop`, `/start` (spawning `…\System32\iisreset.exe`, OEM/CP866 output like `rac.exe`, 90s timeout). Targets are discovered from the **live IIS** (`ServerManager.ApplicationPools` / `Sites` with current state via `GET /api/v1/iis/application-pools` · `/sites`), not from publication rows. The **overall IIS status** is read from the **`W3SVC` Windows service** via `ServiceController` (`GET /api/v1/iis/server`) — deliberately not from `ServerManager`, because once IIS is stopped `ServerManager` can no longer read pool/site state, whereas the service status stays readable (so the UI can show «Остановлен» and offer «Запустить IIS»).
- **Endpoints.** `/api/v1/iis/*` — discovery `GET` (Viewer): `server`, `application-pools`, `sites`; mutations `POST` (Admin): `application-pools/{recycle,start,stop}`, `sites/{start,stop,restart}`, `reset` / `stop` / `start` (pool/site bodies carry the target `Name`; `recycle`/`reset`/`stop` carry `Confirm`; `start` takes none).
- **Safety.** Destructive operations are serialized by `IIisResetConcurrencyGate` (singleton `SemaphoreSlim(1, 1)`) — two `recycle`/`iisreset` never overlap. `recycle`, `reset` and server `stop` require a server-side `Confirm=true` (else `409 IIS_CONFIRM_REQUIRED`); `start` is recovery (no gate). The UI shows a plain confirmation dialog for destructive actions (no typed token — 06_UI_DESIGN §7). `iisreset`/stop interrupts **all** sites on the server (and the panel itself if hosted there).
- **Audit.** Every mutation writes a server-scope row (`TenantId=null`): `IisApplicationPoolRecycled=220`, `…Started=221`, `…Stopped=222`, `IisSiteStarted=223`, `…Stopped=224`, `…Restarted=225`, `IisReset=226`, `IisStopped=227`, `IisStarted=228`. `KeyNotFoundException`→`404`; `UnauthorizedAccessException`→`409 IIS_ACCESS_DENIED`; COM/IO/timeout→`409 IIS_OPERATION_FAILED` (sanitized `Detail` + `correlationId`).

### IIS Administration
- The backend interacts with IIS using the `Microsoft.Web.Administration` library (`ServerManager`) — read-only for status; publication mutations are `webinst` + the `web.config` edit; the **hosting lifecycle** (pool/site/`iisreset`, MLC-047) is the third class of mutation (see above).
- **Discovery error contract (`GET /discovery/databases`, `/discovery/iis-sites`, `/discovery/platform-versions`, `/discovery/sql-instances`).** On any infrastructure failure the endpoint returns `200 OK` with `DiscoveryResponse { Available: false, Error: <short Russian message> }` so the form falls back to manual entry. The raw `ex.Message` is **never** placed in `Error`; the full exception is logged. `OperationCanceledException` (request abort) propagates. `GET /discovery/databases` takes **no server parameter** (single-host, ADR-28) — it reads the instance from `Settings.Sql.Server`; an empty setting yields `Available: false` with a message pointing to «Параметры» (the form then accepts a manually typed database name). `GET /discovery/sql-instances` enumerates local MSSQL instances from the registry (`Instance Names\SQL`, both 64- and 32-bit views) — server-side, localhost-only, no SQL Browser / UDP scan; `MSSQLSERVER` maps to `localhost`, named instances to `localhost\<instance>`; it feeds the «SQL Server» picker on «Параметры».

## 3. Windows Server & Security Context

### Deployment Topology — Single Node
All components run on the same Windows Server host: the .NET backend service, the IIS hosting the 1C publications, the MSSQL instance, and the 1C cluster itself. No remoting, no WinRM, no cross-host adapters. The IIS Adapter operates against a local IIS only, and `default.vrd` files live on the local filesystem. **If this topology assumption ever changes, every infrastructure adapter requires a re-review** (remote IIS administration via WinRM/PowerShell Remoting, distributed locks, etc.).

**The backend serves the panel page itself (ADR-30).** Kestrel hosts the built React SPA from its own `wwwroot` **same-origin with the API** (`UseStaticFiles` for the hashed `/assets/*` + a history-fallback that returns `index.html` for any non-reserved path, so deep-link / refresh of client routes work; `/api` and `/hangfire` stay reserved → an unknown `/api/*` honestly returns `404`, never the SPA shell). **IIS is therefore needed only for the managed 1C publications the panel administers — not for hosting the panel page** (two distinct IIS roles). The two hosting roles are decoupled: the panel can run on a host where IIS serves only 1C bases, or even before IIS is configured. `dotnet publish` produces a self-contained artefact (the SPA travels inside `wwwroot`); dev is unchanged (Vite on :5173 proxies `/api` and `/hangfire` to the backend). See `OPERATIONS.md` «Backend hosts the SPA» for the build/cache details and the `SkipSpaBuild` / `PrebuiltSpaDist` publish flags.

### Service Account & Permissions
- **Service Account:** The backend .NET service runs under a specific Windows Service Account (e.g., `NT AUTHORITY\NETWORK SERVICE` or a custom local/AD account).
- **Permissions:** This account requires:
  - Read/Write access to the physical folders where publication files (`default.vrd`, `web.config`) are stored (the platform-change edit writes there).
  - Permissions to interact with the IIS Metabase.
  - Network access to the 1C Cluster API/RAS ports (default 1545, etc.).
  - **Execute** permission on the resolved `Settings.OneC.RAS.ExePath` (typically `C:\Program Files\1cv8\<version>\bin\rac.exe`) **and** on `…\1cv8\<version>\bin\webinst.exe` for every platform version published from the panel, plus **read** access to the rest of `1cv8\<version>\bin\` so co-located DLLs load. Default ACLs grant this to `Users` on stock 1C installs; locked-down custom service accounts may need explicit Read+Execute grant.
  - Read/Write access to `%ProgramData%\MitLicenseCenter\keys\` (Data Protection key ring — see ADR-8).

### Secrets Storage
Secrets (1C cluster credentials, MSSQL connection strings, RAS credentials) are stored encrypted using the **ASP.NET Core Data Protection API**, which is DPAPI-backed on Windows. Keys live under `%ProgramData%\MitLicenseCenter\keys\` in production (machine-scoped, service-account-readable) and **must be included in the operator's backup set (see `OPERATIONS.md` and ADR-15)** — without them restored backups cannot decrypt their own secrets. See ADR-8 for the full policy.

**Development override:** when `ASPNETCORE_ENVIRONMENT=Development` the backend persists keys to `%LocalAppData%\MitLicenseCenter\keys\` instead. This avoids the need for elevation just to run the app locally. Local dev keys are NOT a substitute for the production key ring and must never be confused with the production key ring the operator backs up.

**Persistence layout:** runtime configuration lives in the `dbo.Settings` table. Each row has both a `ValueText NVARCHAR(MAX) NULL` column (plain payload) and a `Value VARBINARY(MAX) NULL` column (DPAPI-encrypted UTF-8 bytes). `IsSecret BIT` decides which column is authoritative on write. Operationally this means: **deleting `%ProgramData%\MitLicenseCenter\keys\` makes every `IsSecret=true` row unreadable** — the 1C cluster admin password and (later) RAS credentials would have to be re-entered through the «Параметры» UI. The Settings table is part of the operator's MSSQL backup; the DPAPI key ring is part of the operator's file-level backup. Both must be restored together — see `OPERATIONS.md` and ADR-15. The DPAPI protector purpose-string is `mlc.settings.v1` — a future scheme change bumps the suffix and migrates rows, leaving the key ring file format unchanged.

### Admin Authentication
Administrators authenticate against the panel using ASP.NET Core Identity with local accounts stored in the same MSSQL domain database (schema `auth`). Cookie-based session auth (HttpOnly, Secure, SameSite=Strict). Two roles: `Admin` and `Viewer`. See ADR-7.

`SettingValueKind ∈ {Text, Number, Url, HostPort, Path}`. A `null`/whitespace payload clears the value (`IsSet=false`); for a secret that means "remove the password". Validation: `Number` = `int.TryParse` + range; `HostPort` = `^[^\s:]+:\d+$` + port in [1024, 65535]; `Path`/`Text` have no format check — any value is accepted, and a blank one simply clears the setting (it is not rejected). The whitelist takes priority over DB rows — a key not in the catalog is never surfaced.

`Sql.Server` is the **single source of truth** for the one SQL instance hosting client databases (single-host, ADR-28): the backend reads it at runtime — `SqlDatabaseDiscovery` (database-name discovery) and `BackupsEndpoints` (backup enqueue) — and the «Добавить инфобазу» form prefills the database-name picker from it. It is **not** a per-base value: `Infobase` no longer carries a `DatabaseServer` column (dropped, MLC-088). The two remaining `*Default*` keys (`IIS.DefaultSiteName`, `OneC.DefaultPlatformVersion`) are **UI-only** form-prefill (no backend reader): they seed the form so the operator does not retype the same IIS site / platform version for every base, and the values are still persisted per-base on `Publication` (overridable in the «Дополнительно» disclosure). On the «Параметры» page `Sql.Server` is the **«SQL Server»** section (a picker over locally-discovered MSSQL instances, `GET /discovery/sql-instances`, with a manual fallback); `IIS.DefaultSiteName` lives in the «Публикации IIS» section and `OneC.DefaultPlatformVersion` is set by the «Платформа 1С» picker alongside the rac.exe path (MLC-055, ADR-3.3). There is no «Значения по умолчанию для новых баз» section.

## 5. Background Job Execution

Infrastructure operations (especially mass IIS updates or 1C cluster scans) can be blocking.
- The Reconciliation Loop (Session Monitor) runs asynchronously and on a two-tier cadence (hot 3–5s for at-risk tenants, cold 20–30s for everyone). Enforcement (session kill) runs on **both** tiers under a shared in-process lock (`IEnforcementGate`, singleton `SemaphoreSlim(1, 1)`) so exactly one enforcement path kills at a time (cold Hangfire job + hot `BackgroundService`; over-kill protection, MLC-001). The hot tier reuses its single `session list` fetch for both the UI overlay and the enforcement fresh-check, so it adds **no** extra `rac.exe` spawn; `session terminate` is spawned only when there are real kills and is transient (it stops once `Consumed == Limit`). The ADR-3.3 spawn budget (≤ 26 procs/min) is unchanged. See ADR-6 / ADR-6.1.
- Direct calls to the 1C Cluster Adapter MUST have strict timeouts to prevent the background worker from hanging if rac.exe / ras.exe become unresponsive. `RacExecutableRasClusterClient` uses a 30s per-invocation deadline (ADR-3.3).
- **`RasHealthProbingService`** (ADR-16) runs as a `BackgroundService` with a 30s `IClusterClient.PingAsync` cadence. Publishes `IRasHealthReader` snapshot for the Dashboard RAS health card. Audit-neutral. This snapshot is the **only** RAS source the readiness probe reads — the `GET /api/v1/health/ready` endpoint (MLC-040) reuses it and never spawns its own `rac.exe`, so health requests stay inside the spawn budget ([ADR-3.3](DECISIONS.md#33-rac-cli-spawn-contract)). Liveness `GET /api/v1/health` is unchanged (cheap, dependency-free). See `OPERATIONS.md` «Проверки готовности».
- A separate **publication status-refresh job** (`PublicationStatusRefreshJob`) runs every 5 minutes (throttled to `Settings.Drift.IntervalMinutes`), reading each `Publication`'s live IIS state (read-only) and writing `Publication.LastCheckStatus` / `LastCheckAt` / `LastCheckDetails`. It never mutates IIS and writes no audit; (re)publication and platform change are explicit admin actions.
- An **audit retention job** runs daily at 03:00 UTC and deletes rows from `dbo.AuditLogs` older than `Settings.Audit.RetentionDays` (default 365, range [30, 3650]). Implementation: batched `DELETE TOP (5000)` with commit-per-batch via `ExecuteSqlInterpolatedAsync` against `[Timestamp]`. Audit row `AuditLogsPurged=500` is written only when the total delete count is non-zero. CRON is fixed in code — the retention window is operator-tunable from the Settings page (section «Хранение журнала аудита»), the cadence is not. `dbo.AuditLogs` is the only application table that grows unbounded; every other table is reference data or `UPDATE`-only.
- **License usage time-series** (MLC-048, ADR-25): the cold `ReconciliationJob` samples instantaneous per-tenant consumption every cycle (≈25s, reusing `LicenseConsumption.CountByTenant` and the live limit — no extra `rac.exe` spawn; idle tenants sample `0`). A singleton `ILicenseUsageAccumulator` folds samples into the current 15-minute bucket (running min/max/sum/count per tenant) and on a bucket boundary the job persists one aggregate row per tenant (`min`/`max`/`avg(double)`/`Limit`) to the telemetry table `dbo.LicenseUsageSnapshots` — **outside** the enforcement lock. The current partial bucket lives only in the singleton's memory and is lost on restart (best-effort telemetry). A **license-usage retention job** (`LicenseUsageRetentionJob`) runs daily at 03:30 UTC (offset from audit retention) and deletes `LicenseUsageSnapshots` older than `Settings.LicenseUsage.RetentionDays` (default 365, range [30, 3650]) — batched (5000) provider-portable `ExecuteDelete` with commit-per-batch, **no audit row** (housekeeping). CRON is fixed in code; the retention window is operator-tunable. `dbo.LicenseUsageSnapshots` is telemetry (`Infrastructure/Reporting`), FK to `Tenant` `SetNull` so history survives tenant deletion.
- **Hangfire job-history retention** is deterministic: `JobRetentionStateFilter` (a global `IApplyStateFilter` wired into `AddHangfire(...)`) expires succeeded/deleted jobs after 2 days, so the high-frequency `cold-snapshot` (one job per minute) cannot accumulate history in the `hangfire` schema under Hangfire's implicit default. Failed jobs are left untouched so they stay visible for diagnosis. The window is a fixed in-code constant (internal knob, not operator-tunable).

## 6. Host Performance Metrics (раздел «Быстродействие», MLC-064 / ADR-26)

The «Быстродействие» section reads live host saturation metrics through the Application port `IHostMetricsProbe` → `OneCHostMetricsProbe` (Infrastructure, `[SupportedOSPlatform("windows")]`, registered under `#pragma CA1416` in `AddInfrastructure` — pattern `OneCIisPublishingService`; `StubHostMetricsProbe` serves unit tests). Web never touches WMI / `System.Diagnostics.Process` directly (ADR-20). The probe is a **singleton** holding the previous poll's snapshot to compute deltas; `GET /api/v1/performance/host` (role `Viewer`) returns a `HostMetricsSnapshot` and **persists nothing** (Live model, ADR-26).

- **Sources.** CPU % + `ProcessorQueueLength`, RAM `AvailableMBytes` + `PagesPerSec`, and disk queue come from WMI `Win32_PerfFormattedData_*`; disk **latency** (`Avg. Disk sec/Read|Write`) is cooked by hand from `Win32_PerfRawData_PerfDisk_PhysicalDisk` (`PERF_AVERAGE_TIMER` = `((N1−N0)/Frequency)/(Base1−Base0)`) because the formatted class truncates the sub-millisecond fraction to integer seconds. Total RAM is read once from `Win32_ComputerSystem`. Per-process CPU%/RAM come from `System.Diagnostics.Process` (`TotalProcessorTime` delta ÷ elapsed ÷ core count; `WorkingSet64`).
- **Why WMI, not `PerformanceCounter`.** Performance-counter category/counter **names are localized on RU Windows**, so the English-name `PerformanceCounter` lookup throws «category does not exist»; WQL perf-class **property names are invariant** (CLAUDE.md «Гочи Windows/1С»). The track's stand is RU Windows.
- **Delta & first poll.** Process CPU% and disk latency need two samples; the singleton keeps the prior snapshot (pattern `ColdThrottleState`/`IClusterUuidCache`). The first poll returns `Measuring=true` and zeroes for those metrics. The probe is **defensive** — a failed WQL query is logged and degrades that metric to `0`; a snapshot is always returned (best-effort live).
- **Attribution.** Processes are grouped into families (`OneC`/`Mssql`/`OsUpdate`/`Antivirus`/`Other`) by the pure `ProcessFamilyGrouping` over `ProcessFamilyMap`, parsed from `Settings.Performance.ProcessFamilyMap` (operator-tunable without redeploy — pattern `OneC.LicenseConsumingAppIds`; blank → default set; unmatched process names fall into `Other`).
- **Honest attribution signal (MLC-064a).** Reading a process's CPU/RAM throws `Win32Exception` when the backend lacks rights to it (another account's process); the probe **counts** these into `HostMetricsSnapshot.ProcessesInaccessible` instead of swallowing them, and exposes the derived `AttributionIncomplete` (`> 0`). An under-privileged backend cannot read another account's `rphost`/`sqlservr`, so their load drops out and the section would otherwise show a misleading «всё Прочее»; the count drives a warning banner (05_UI_REQUIREMENTS §3.8). The kernel pseudo-processes Idle (PID 0) / System (PID 4) are unreadable on every host regardless of privilege and belong to no family, so they are excluded from the count (the signal must not cry wolf under the normal admin-elevated backend). A process that exits mid-read (`InvalidOperationException`) is transient churn, not a rights gap, and is likewise not counted. Required permissions — `OPERATIONS.md` «Быстродействие — required permissions».

### 6.1. 1C load — live sessions & worker processes (MLC-066 / ADR-26)

The «Быстродействие» section's second drill-down («кто грузит внутри 1С») reads live cluster load through the existing `IClusterClient` (no new adapter — the `rac.exe` boundary already exists, ADR-16/20). `GET /api/v1/performance/onec-sessions` (role `Viewer`) composes a `OneCLoadSnapshot` from two Application ports — `ListSessionLoadsAsync` (perf-rich sessions, `OneCSessionLoad`) + `ListProcessesAsync` (worker processes, `OneCProcessLoad`) — and **persists nothing** (Live model). The endpoint is a vertical slice: Web calls the two `IClusterClient` methods and stamps `CapturedAtUtc`; it never touches `rac.exe`.

- **Sources & mapping.** Both come from `rac.exe` (`session list` and the new `process list`), parsed by the defensive `RacOutputParser` into the all-nullable `OneCSessionLoad` / `OneCProcessLoad` (full field list + numeric gotchas — ADR-3.3 «Performance field mapping»). The `MLC-063` reconnaissance confirmed the perf fields are present on 8.5.1.1302 (the empty PR-3.8 fixture was a fresh-session artefact, not a version limit). `ListSessionLoadsAsync` is the **same spawn** as the kill-path `ListActiveSessionsAsync` but a richer projection (`ClusterSession` stays lean for enforcement); `ListProcessesAsync` is the **+1 spawn per poll**.
- **Spawn budget.** 2 `rac.exe` per poll on a warm cache (session list + process list), reusing the cached cluster UUID (`IClusterUuidCache`, no extra `cluster list`). Pull-on-demand only — spawned while the tab is open (~5 s poll), never in the background (ADR-3.3 «Spawn cadence»).
- **Best-effort.** `rac` not configured / unreachable → empty lists (the established `ListActiveSessionsAsync` convention: absence of signal ≠ error), so the section degrades to «нет данных» rather than erroring.

### 6.2. SQL load — live DMV (MLC-068 / ADR-26)

The «Быстродействие» section's third drill-down («1С грузит SQL?») reads live MSSQL DMVs through the Application port `ISqlPerformanceProbe` → `SqlPerformanceProbe` (Infrastructure). Unlike the host probe this adapter is **plain ADO.NET** (`Microsoft.Data.SqlClient`, like `SqlDatabaseDiscovery`) — **not** Windows-only, so it carries no `[SupportedOSPlatform]`/`#pragma CA1416`. `GET /api/v1/performance/sql` (role `Viewer`) returns a `SqlPerformanceView` and **persists nothing** (Live model). Web never touches DMVs directly (ADR-20); it only stitches the database→tenant attribution from its **own** `AppDbContext` (a vertical slice over the panel's DB, not a 1C/IIS boundary).

- **Connection.** The probe inherits the auth parameters from `ConnectionStrings:Default` (same account as the panel) and overrides `InitialCatalog=master` (DMVs are server-scoped) + a short timeout — the `SqlDatabaseDiscovery` pattern. Valid because the single-node topology (ADR-26 / Locked Operational Constraints) co-locates the panel DB and the 1C infobase databases on one MSSQL instance, so `Default`'s `DataSource` **is** the server hosting the infobases. No new setting is introduced.
- **Sources.** Active requests + blocking chains — `sys.dm_exec_requests` ⨝ `sys.dm_exec_sessions` `OUTER APPLY sys.dm_exec_sql_text` (top 50 by `cpu_time`, user processes only, excluding the probe's own `@@SPID`; `blocking_session_id` carries the chain, a zero → `null`); per-database **IO-stall** — `sys.dm_io_virtual_file_stats` aggregated per `database_id`; **wait-stats** — `sys.dm_os_wait_stats`. IO-stall and wait-stats are **cumulative since SQL startup**, so they are reported as a **delta** between polls (the singleton holds the previous read; a counter reset / negative delta is dropped, benign idle wait types are filtered out). The first poll reports `Measuring=true` for the deltas; the active-requests list is instantaneous and always populated.
- **Attribution.** 1C-originated SQL is flagged by `program_name='1CV83 Server'` (`IsOneC`, MLC-063). The endpoint resolves every database name seen in the snapshot to a tenant via the pure `SqlAttributionResolver` over `Infobase.DatabaseName` (case-insensitive); a database with no registered infobase (`master`/`tempdb`/the panel's own DB) resolves to a null tenant — visible but «ничья». Granularity is the **database** — a SQL request is **not** traced to a 1C session/user (ADR-26; all 1C→SQL traffic shares one `program_name`/host).
- **Permissions & degraded signal.** The probe checks `HAS_PERMS_BY_NAME(NULL, NULL, 'VIEW SERVER STATE')` first; without the right it returns `Status=PermissionDenied` (honest degraded signal, pattern MLC-064a — not an empty «all clear»). An unreachable SQL / unset connection string → `Status=Unavailable`. Required grant — `OPERATIONS.md` «Быстродействие — required permissions».

### 6.3. Recording — on-demand investigation (MLC-070 / ADR-26)

The «Быстродействие» section's fourth capability is **Recording**: unlike the three live drill-downs (pull-on-demand, nothing persisted), a recording is turned on/off **manually** for an investigation and **persists** a time-series to the panel's DB. The Application port `IPerfRecordingService` → `PerfRecordingService` (Infrastructure, **singleton**) holds the one active recording in memory (id + start + sample count, serialized through a `SemaphoreSlim`) and persists via `IServiceScopeFactory` (scoped `AppDbContext` + `IClusterClient` per operation — pattern `HotTierPollingService`). A background `PerfRecordingSamplingService` (`BackgroundService`) drives the timer.

- **Entities (telemetry).** `PerfRecording` (`Id`, `StartedAtUtc`, `StoppedAtUtc?`, `Status` `Active`/`Stopped`/`Interrupted`, `StartedBy`, `StopReason?` `Manual`/`TimeLimit`/`SampleLimit`) + `PerfRecordingSample` (`RecordingId` FK **cascade**, `SampleUtc`, flat level-1 host columns, `ProcessesInaccessible`, and JSON columns `ProcessGroupsJson` / `OneCLoadJson` / `SqlLoadJson`). They live in `Infrastructure.Reporting` next to `LicenseUsageSnapshot` (telemetry entities Web reads via a vertical slice, ADR-20) — deliberately **not** in the adapter namespace `Infrastructure.Performance`, which the layer guard (`LayerBoundaryTests`) forbids Web from referencing. Enum columns are `int` (`HasConversion`), the wire form is the string (ADR-10.1). JSON columns reuse the same Application records the live views emit (`ProcessGroupUsage` / `OneCLoadSnapshot` / `SqlPerformanceSnapshot`), serialized through the shared `PerfSampleJson` options so the viewer deserializes one format.
- **Sampling.** Each tick captures the host snapshot (`IHostMetricsProbe`), the top-N 1С culprits (`IClusterClient.ListSessionLoadsAsync` + `ListProcessesAsync`) and the SQL snapshot (`ISqlPerformanceProbe`), trims the 1С sessions / SQL requests to the top-N by load, and writes one `PerfRecordingSample`. Cadence is `Settings.Performance.RecordingSampleIntervalSeconds` (default 15 s; the probes' delta state is shared with the live polls — both want «since last capture»). A best-effort source (rac not configured, SQL degraded with no rows) is stored as `null`, not a false zero.
- **Auto-stop & lifecycle.** Two independent limits, whichever fires first: `Performance.RecordingMaxDurationMinutes` (default 60) and `Performance.RecordingMaxSamples` (default 1000). The triggering sample is written **before** the stop, so the record is complete. Time flows through `TimeProvider` (deterministic auto-stop tests). On process restart the in-memory active state is lost, so `PerfRecordingSamplingService` reconciles any row left `Active` to `Interrupted` at startup (best-effort, like ADR-25's lost partial bucket). There is **no nightly retention job** — a recording is bounded by its own duration.
- **API.** `/api/v1/performance/recordings`: `POST` (start — one at a time, 409 `RECORDING_ACTIVE` otherwise) + `POST …/{id}/stop` + `DELETE …/{id}` (409 if still `Active` — stop first) are `Admin`; `GET` (list, newest first) + `GET …/{id}` (the sample series, JSON columns deserialized) are `Viewer`. Start/stop is **not audited** in v1 (operational tooling — a new `AuditActionType` would burn a frozen int; ADR-26).
- **Spawn budget.** Sampling spawns `rac.exe` + DMV reads on a background timer while a recording runs — the one perf path that spawns without an open tab. Operator-initiated and bounded by auto-stop at the sample interval (≫ the 5 s live poll), so the budget stays finite (ADR-3.3 «Spawn cadence»).

## 7. Backup strategy is operator responsibility

Per ADR-15, the deployment's backup **strategy** — scheduling, full/differential chains, restore — is **out of application scope**. The operator backs up the MSSQL database, the Data Protection key ring under `%ProgramData%\MitLicenseCenter\keys\`, and optionally `appsettings.Production.json`, using platform tooling (SQL Server Maintenance Plans, Veeam, Windows Server Backup, Robocopy + Scheduled Task). The key ring and the database form one logical backup unit — restoring one without the other yields an unreadable database. See `OPERATIONS.md` for the operator's checklist. The single carved-out exception is the **operator-initiated on-demand `COPY_ONLY` backup** of a client database (ADR-27) — its SQL mechanics live in the adapter below (§8); it adds no scheduling and no restore path.

## 8. On-demand SQL database backup (MLC-076/077 / ADR-27)

The on-demand backup feature (ADR-27) talks to MSSQL exclusively through the Application port `ISqlBackupService` → `SqlBackupAdapter` (Infrastructure). Like the SQL probe (§6.2) it is **plain ADO.NET** (`Microsoft.Data.SqlClient`) — **not** Windows-only, no `[SupportedOSPlatform]`/CA1416 — and inherits connection parameters from `ConnectionStrings:Default`, overriding `DataSource` to the requested server and `InitialCatalog=master` (valid under the single-node co-located topology). Web never issues `BACKUP`/`xp_*` itself (ADR-20); the adapter namespace `Infrastructure.Backups` is on the layer guard's forbidden list (`LayerBoundaryTests`). Stateless → singleton; in tests — the programmable `FakeSqlBackupService` (the real adapter is integration-only, like `SqlPerformanceProbe`).

- **One call = the whole safe cycle.** `BackupAsync(server, database, folderRoot, safetyMarginMb)` performs, in order: sysadmin check (`IS_SRVROLEMEMBER('sysadmin')`) → the database exists in `sys.databases` → size estimate = used ROWS pages (`SUM(FILEPROPERTY(name,'SpaceUsed'))·8 KB` in the database context — a deliberate over-estimate, `COMPRESSION` shrinks the real file) → disk-guard: `folderRoot` must be a **local** drive path (`X:\…`), free space from `xp_fixeddrives` must cover estimate + `Backup.DiskSafetyMarginMb` → `xp_create_subdir <root>\<database>` (idempotent) → `BACKUP DATABASE … TO DISK=@path WITH COPY_ONLY, COMPRESSION, CHECKSUM, FORMAT, INIT, STATS=10` → `RESTORE VERIFYONLY FROM DISK=@path WITH CHECKSUM` → **only after verify** `xp_delete_file` of the previous `.bak` in the database's subfolder (keep-latest-1: new-before-delete, never the reverse) → file size from `msdb.dbo.backupset.compressed_backup_size` (best-effort: absent → `null`, not a failure). The file is `<root>\<database>\<database>_yyyyMMdd_HHmmss.bak`.
- **Injection-safe by construction.** The database name is checked against `sys.databases` and enters dynamic SQL only through `QUOTENAME`; the file path is always a string parameter (`sp_executesql @path`).
- **Never throws → typed degraded results** (honest-signal pattern, §6.2): no sysadmin → `PermissionDenied`; `NULL` estimate → `EstimateUnavailable` (**the backup does not start without an estimate** — there is no fixed GB floor); not enough space → `InsufficientSpace` with the numbers in the message; database missing / UNC root / SQL error → `BackupFailed` + message. Cancellation propagates (not an infrastructure failure). `DeleteBackupsOlderThanAsync(server, folder, cutoffUtc)` exposes the same server-side `xp_delete_file` for retention and manual deletion — also never-throws (`SqlDeleteResult`).
- **Server-side file ops & time base.** All file operations run on the SQL host via `xp_*` — one account (the SQL service account) both writes and deletes, so there is no cross-identity ACL split. `xp_delete_file` compares against file times in the SQL host's **local** time, so the adapter converts the port's UTC cutoff through the local time zone — correct because the panel and SQL are co-located (single-node).
- **Timeouts.** `BACKUP`/`VERIFYONLY` run with an hours-scale `CommandTimeout` (large databases take tens of minutes and more) — deliberately unlike the 5 s probe timeouts; metadata steps stay short.
- **Bookkeeping entity.** `DatabaseBackup` (`Infrastructure.Reporting`, telemetry-style like `PerfRecording` — **no FK** to `Infobase`, the row outlives infobase deletion; enum columns are int via `HasConversion`, the wire form is the string) records every requested backup: status `Queued/Running/Succeeded/Failed`, the typed `FailureReason`, requester, timestamps, file path/size. Indexes: `RequestedAtUtc` (listing) and `(DatabaseServer, DatabaseName, Status)` (active-backup lookups). Frozen audit slots `BackupRequested=510 … BackupsPurged=514` are reserved in the 500 maintenance series.
- **Required grants** — `OPERATIONS.md` «Бэкап — required permissions» (sysadmin: `xp_*` file operations demand it).

### 8.1. Orchestration — queue, pump, recovery (MLC-077)

The Application port `IBackupOrchestrator` → `BackupOrchestrator` (Infrastructure, **singleton**) drives the queue. The queue **is** the `DatabaseBackups` table (`Queued` rows = FIFO queue, `Running` = in flight), so requests survive a panel restart; the in-memory piece is only the set of currently executing `(server, database)` pairs plus a wake signal. Bookkeeping (enqueue, start, finish, recovery) is serialized through one `SemaphoreSlim`; the long `BACKUP` itself runs **outside** the gate in its own task and fresh DI scope — backups of different databases are parallel by design (deliberate difference from `PerfRecordingService`). DB and the scoped `IAuditLogger` are resolved via `IServiceScopeFactory`; time flows through `TimeProvider`.

- **Enqueue.** `RequestAsync(infobaseId, server, database, requestedBy)`: if a `Queued`/`Running` row for that `(server, database)` already exists → `AlreadyActive` with the **existing** row's id (the endpoint maps it to 409 `BACKUP_ACTIVE`); otherwise insert a `Queued` row and wake the pump.
- **Pump tick.** `PumpOnceAsync` (the deterministic test seam) enforces all three constraints in one pass: re-reads `Backup.MaxParallel` from `ISettingsSnapshot` **every tick** (clamped to the whitelist range 1..8; a change takes effect on the next tick without a restart — lowering it never kills running backups, it just stops admitting new ones) and, while `running < max`, takes the **oldest** `Queued` row whose pair is not in the executing set (FIFO + per-database lock + ceiling), marks it `Running` (`StartedAtUtc`) and launches the backup task. `Backup.FolderPath` is read at start time; if it was cleared while the request waited, the row is closed as `Failed/BackupFailed` with an explicit message — the worker never throws.
- **Finish.** Under the gate: success → `Succeeded` + `CompletedAtUtc` + `FilePath`/`FileSizeBytes` from the adapter result, audit `BackupSucceeded=511`; failure → `Failed` + typed `FailureReason`/`ErrorMessage`, audit `BackupFailed=512`. The audit initiator is the row's `RequestedBy`; no `tenantId` (the telemetry row has no FK — the client linkage is carried by the endpoint's `BackupRequested` audit). The pair is removed from the executing set and the pump is woken so the next queued backup starts immediately. The orchestrator never deletes files itself — keep-latest runs **inside** `BackupAsync` (§8, after `VERIFYONLY`).
- **Pump driver.** `BackupPumpService` (`BackgroundService`, pattern `PerfRecordingSamplingService`): at startup calls `RecoverInterruptedAsync`, then loops on «wake signal OR a 5 s timeout» → `PumpOnceAsync`; tick exceptions are logged, the service never dies.
- **Restart recovery.** `RecoverInterruptedAsync` closes orphaned `Running` rows as `Failed/Interrupted` («файл может быть неполным» — the timestamped name + verify-before-delete keep a partial `.bak` from ever being trusted; the TTL job removes it). `Queued` rows are left untouched and get re-picked by the pump.

### 8.2. Retention job (MLC-077)

`IBackupRetentionJob` → `BackupRetentionJob` (Hangfire recurring `backup-retention`, **03:15 UTC** daily — offset from 03:00 audit / 03:30 license-usage; cadence fixed, the window is `Backup.TtlHours`, default 24, clamped to 1..8760). If `Backup.FolderPath` is not configured the job is a logged no-op. Otherwise: (а) **files** — for every `DISTINCT (DatabaseServer, DatabaseName)` of `DatabaseBackups` rows it calls `DeleteBackupsOlderThanAsync(server, <root>\<database>, now − TTL)` (server-side `xp_delete_file`, never-throws); (б) **rows** — `DatabaseBackup` rows older than the cutoff (by `RequestedAtUtc`, never touching `Queued`/`Running`) are deleted in provider-portable batches, each batch wrapped in `Database.CreateExecutionStrategy().ExecuteAsync` (the MLC-074 pattern — mandatory under `EnableRetryOnFailure`); (в) audit `BackupsPurged=514` (initiator `System`) is written **only** when at least one row was deleted.

### 8.3. API (MLC-077)

`/api/v1/backups` (vertical slice, ADR-20 — handlers read their own `AppDbContext`; SQL file operations only via `ISqlBackupService`, enqueueing only via `IBackupOrchestrator`):

| Route | Role | Behaviour |
|---|---|---|
| `GET /backups[?infobaseId=]` | Viewer | List, newest first; the FE polls while `Queued`/`Running` rows exist |
| `GET /backups/{id}` | Viewer | One row; 404 |
| `POST /backups` `{infobaseId}` | Viewer | Resolves the infobase (404) → its `server+db`; unset `Backup.FolderPath` → 409 `BACKUP_FOLDER_NOT_CONFIGURED`; duplicate of an active pair → 409 `BACKUP_ACTIVE`; else 201 + audit `BackupRequested=510` **with the infobase's `TenantId`** |
| `DELETE /backups/{id}` | Admin | 404 if missing; `Running` → 409 `BACKUP_ACTIVE` (finish first); if the row has a `FilePath`, the `.bak` is removed server-side first (`DeleteBackupsOlderThanAsync` over the file's subfolder with `cutoff = now` — keep-latest-1 means the subfolder holds exactly that file); a failed file removal keeps the row and returns 409 `BACKUP_DELETE_FAILED`; success deletes the row + audit `BackupDeleted=513` |

Starting a backup is deliberately **Viewer** (the operator button, ADR-27); deletion and the `Backup.*` settings (`PUT /settings/{key}`) stay Admin. Status/reason enums travel as strings (ADR-10.1).
