# Установка MitLicense Center

Документ описывает установку на хост заказчика: системные требования, пошаговое
прохождение мастера, учётные записи и права, обновление поверх существующей версии,
удаление.

---

## 1. Системные требования

### Операционная система и роли

| Компонент | Требование |
|---|---|
| ОС | Windows Server 2019 / 2022, 64-разрядная (x64) |
| IIS | Роль веб-сервера IIS установлена и работает |
| SQL Server | SQL Server 2019 / 2022 (любой редакции; Express достаточен для малых нагрузок) |
| Платформа 1С | Установлена платформа 1С:Предприятие с утилитами `rac.exe`, `ras.exe`, `webinst.exe` |
| Права установщика | Локальный администратор (требует `PrivilegesRequired=admin` в `.iss`) |

### .NET runtime

Артефакт собирается **self-contained single-file win-x64** (`--self-contained true
-p:PublishSingleFile=true` в `publish-release.ps1`): среда выполнения .NET вшита в
`MitLicenseCenter.Web.exe`. Устанавливать .NET на хост **не требуется**.

### Учётная запись для SQL (создаётся оператором заранее)

Установщик не создаёт учётные записи — он их только проверяет. До запуска мастера
оператор обязан подготовить одну из двух учёток:

- **Режим (A) Windows-аутентификация** — доменный или локальный Windows-аккаунт,
  которому выдана роль `sysadmin` на экземпляре SQL Server. Этот аккаунт будет
  учётной записью службы Windows.
- **Режим (B) SQL-аутентификация** — SQL-логин (например `sa`) с ролью `sysadmin`.
  Служба в этом режиме работает под `LocalSystem`.

## 2. Что устанавливается

### Служба Windows

| Параметр | Значение |
|---|---|
| Имя службы (`sc create`) | `MitLicenseCenter` |
| Отображаемое имя | `MitLicense Center` |
| Тип запуска | `auto` (автоматический) |
| Исполняемый файл | `{app}\MitLicenseCenter.Web.exe` |
| Kestrel, порт | `http://+:<порт>` — по умолчанию `8080`, задаётся в мастере |
| Учётная запись (режим A) | Windows-аккаунт, введённый оператором |
| Учётная запись (режим B) | `LocalSystem` |

Каталог установки по умолчанию: `%ProgramFiles%\MitLicense Center`
(`DefaultDirName={autopf}\MitLicense Center` в `.iss`).

### Каталог данных

`%ProgramData%\MitLicenseCenter` — создаётся секцией `[Dirs]` с флагом
`uninsneveruninstall` (не удаляется при деинсталляции автоматически).

Содержимое:

- **key ring** — файлы ASP.NET Data Protection (назначение `mlc.settings.v1`);
  ключи расшифровывают секретные настройки в `dbo.Settings` (см. ADR-8).
- **`initial-admin.secret`** — одноразовый файл с паролем первого администратора
  (только чистая установка; удаляется бэкендом при первом старте, см. §5).

### Конфигурационный файл

`{app}\appsettings.Production.json` — генерируется мастером из ввода оператора.
Содержит строки подключения (Connection Strings) к SQL Server и сетевые параметры
(`Urls`, `AllowedHosts`). При SQL-аутентификации содержит пароль в открытом виде —
NTFS ACL ограничивают чтение (см. §4).

### Правило брандмауэра

Входящее TCP-правило `MitLicense Center` для выбранного порта открывается через
`netsh advfirewall firewall add rule`.

### Запись меню «Пуск»

Интернет-ярлык `%ProgramData%\Microsoft\Windows\Start Menu\Programs\MitLicense Center\
MitLicense Center.url` — открывает панель в браузере по умолчанию.

### Запись IIS

Панель **не** регистрирует собственный сайт IIS. Kestrel-служба работает как
самостоятельный веб-сервер на выбранном TCP-порту. IIS используется панелью как
управляемый объект (публикации инфобаз через `webinst.exe`), но не как хост
самой панели.

## 3. Мастер установки: страницы по коду

Мастер реализован в секции `[Code]` файла `installer/MitLicenseCenter.iss`.
Страницы перечислены в порядке отображения.

### 3.1 Страница выбора каталога (стандартная Inno)

Стандартный экран выбора целевого каталога (`wpSelectDir`).
По умолчанию: `%ProgramFiles%\MitLicense Center`.

### 3.2 Резервная копия перед обновлением (только апгрейд)

Показывается **только** при обновлении поверх существующей установки
(`not ServiceExists` → страница пропускается на чистой установке).

Текст экрана требует подтвердить, что оператор сделал резервную копию:

1. Базы данных SQL Server панели.
2. Каталога ключей шифрования `%ProgramData%\MitLicenseCenter` (key ring).

Кнопка «Далее» заблокирована, пока не установлен чекбокс:
«Я создал резервную копию базы данных SQL Server и каталога ключей шифрования».

### 3.3 SQL Server

Два поля ввода:

| Поле | По умолчанию | Пример |
|---|---|---|
| Экземпляр SQL Server | `.` | `.\SQLEXPRESS`, `СЕРВЕР\ЭКЗЕМПЛЯР` |
| Имя базы данных | `MitLicenseCenter` | любое допустимое имя |

База создаётся автоматически при первом старте службы, если её ещё нет (EF Core
миграции, `fail-fast` при старте).

Оба поля обязательны — пустое значение блокирует переход «Далее».

### 3.4 Режим аутентификации в SQL Server

Радиокнопки (взаимоисключающие):

- **(A) Windows-аутентификация** — `Trusted_Connection=True` в строке подключения;
  служба запускается под указанным Windows-аккаунтом. Выбрать, если учётные данные
  — Windows-аккаунт (`ДОМЕН\Пользователь` или `.\Пользователь`).
- **(B) SQL-аутентификация** — `User Id=…;Password=…` в строке подключения;
  служба работает под `LocalSystem`. Выбрать, если учётные данные — SQL-логин
  (например `sa`).

По умолчанию выбран режим (A).

### 3.5 Учётные данные

Два поля (подписи меняются в зависимости от режима):

- **Режим (A):** «Windows-аккаунт службы (ДОМЕН\Пользователь или .\Пользователь)» /
  «Пароль».
- **Режим (B):** «SQL-логин (например sa)» / «Пароль».

Кнопка **«Проверить подключение»** запускает временный PowerShell-скрипт
(`System.Data.SqlClient`):

- Режим (A): проверяет достижимость инстанса под `Integrated Security` установщика-
  администратора (права сервис-аккаунта в SQL проверяются позже — при первом старте
  службы).
- Режим (B): полноценный тест введёнными SQL-кредами.

Строка подключения передаётся через временный `.ps1`-файл (не через командную строку),
файл удаляется сразу после проверки. Кнопка «Далее» заблокирована до успешного
прохождения теста.

### 3.6 Сеть

| Поле | По умолчанию | Описание |
|---|---|---|
| TCP-порт панели | `8080` | Порт Kestrel (`http://+:<порт>`) и входящего правила брандмауэра |
| AllowedHosts | `*` | Допустимые имена хостов в заголовке `Host`; `*` — любой |

Мастер проверяет, что порт — число в диапазоне 1–65535, и что поле `AllowedHosts`
не пустое.

При переходе «Далее» с этой страницы мастер проверяет целевую базу данных на наличие
существующих учётных записей панели (`auth.Users`). Если БД уже инициализирована —
страница задания пароля администратора (§3.7) пропускается, оператор получает
информационное предупреждение.

### 3.7 Учётная запись администратора (только чистая установка)

Показывается только при чистой установке на пустую базу данных. Пропускается:

- при обновлении (admin уже существует в БД);
- при чистой установке поверх уже инициализированной БД.

| Поле | Описание |
|---|---|
| Пароль администратора | Логин фиксированный: `admin` |
| Подтверждение пароля | Должно совпадать с первым полем |

Требования к паролю (проверяются в мастере и повторно сидером при первом старте):

- не менее **12 символов**;
- содержит **заглавную** и **строчную** буквы;
- содержит **цифру**;
- содержит **спецсимвол** (любой не-буквенно-цифровой символ).

### 3.8 Подтверждение и установка (стандартная Inno)

Стандартный экран «Готово к установке» с кратким итогом выбранных параметров.

## 4. Учётные записи и права

### Права в SQL Server

Учётная запись (Windows-аккаунт или SQL-логин) должна иметь роль **`sysadmin`**
на экземпляре SQL Server. Это необходимо для:

- создания базы данных при первом старте (если база не существует);
- применения EF Core миграций (DDL);
- создания схем `auth`, `dbo`, `hangfire`.

### Права Windows-аккаунта (режим A)

- Право **«Вход в качестве службы»** (`SeServiceLogonRight`) — SCM выдаёт его
  автоматически при `sc create obj=…/password=…` на валидном аккаунте. Если SCM
  не смог выдать право — служба не запустится; право выдаётся вручную через
  `secpol.msc` → «Локальные политики» → «Назначение прав пользователя» →
  «Вход в качестве службы».
- Членство в группе **`IIS_IUSRS`** или `Administrators` — требуется, если панель
  управляет IIS (публикации через `webinst.exe`, управление пулами). **(не проверено
  на стенде — выводится из того, что webinst и Microsoft.Web.Administration требуют
  привилегий IIS)**

### NTFS ACL: каталог `%ProgramData%\MitLicenseCenter`

Устанавливается через `icacls` в процедуре `HardenDataDirAcl` в `ssPostInstall`.

Что делает инсталлятор:
1. `/inheritance:r` — снимает наследование ACL от `%ProgramData%` (где по умолчанию
   `Users:RX`).
2. Добавляет явные ACE с наследованием на дочерние объекты `(OI)(CI)`:
   - `NT AUTHORITY\SYSTEM` — полный доступ (`F`);
   - `BUILTIN\Administrators` — полный доступ (`F`);
   - (только режим A) сервис-аккаунт Windows — `Modify` (`M`).

Команды используют SID-нотацию (`*S-1-5-18`, `*S-1-5-32-544`) — независимо от
локали Windows (локализованные имена «Администраторы» / «Administrators» не
используются).

ACL идемпотентны: накатываются и при чистой установке, и при апгрейде.

Сбой `icacls` выводит предупреждение с путём и кодом ошибки, **не прерывает
установку**: служба работоспособна, но защита секретов на диске не применена.
Оператор может выполнить ужесточение вручную.

### NTFS ACL: `{app}\appsettings.Production.json`

Устанавливается через `HardenConfigAcl` сразу после записи файла.

1. `/inheritance:r` — снимает наследование от `%ProgramFiles%` (где `Users:RX`).
2. `NT AUTHORITY\SYSTEM` и `BUILTIN\Administrators` — полный доступ.
3. (только режим A) сервис-аккаунт Windows — только чтение (`R`).

В режиме (B) файл содержит SQL-пароль в открытом виде; доступ к нему имеют только
SYSTEM и Administrators (ADR-8).

## 5. Первый вход: механизм `initial-admin.secret` (ADR-31)

### Как формируется пароль

При чистой установке на пустую базу мастер записывает заданный оператором пароль в
файл:

```
%ProgramData%\MitLicenseCenter\initial-admin.secret
```

Файл записывается **до** ужесточения ACL каталога (`HardenDataDirAcl`), чтобы
`(OI)(CI)` ACE накрыло и этот файл — в режиме (A) сервис-аккаунт получает `Modify`
и может его прочитать и удалить.

### Что делает бэкенд при первом старте

1. Читает `initial-admin.secret`.
2. Создаёт пользователя с логином `admin` и прочитанным паролем.
3. **Удаляет файл** `initial-admin.secret`.

После этого файл не существует.

### Если файл не был записан

Если `SaveStringToFile` завершился с ошибкой (сообщение в мастере), бэкенд создаёт
первого администратора со **случайным паролем** и записывает его в **Журнал событий
Windows** (источник `MitLicenseCenter`, журнал `Приложение`).

### Вход и смена пароля

После установки:

1. Откройте панель: `http://localhost:<порт>/` (или URL с нужным именем хоста).
2. Войдите как `admin` с паролем, заданным в мастере.
3. Смените пароль через меню профиля — раздел «Сменить пароль» (страница `/profile`).

**Первого администратора принудительная смена пароля НЕ затрагивает.** Механизм
форс-смены в продукте есть (`AppUser.MustChangePassword`; вход с этим флагом
блокируется экраном смены пароля до её выполнения), но `IdentitySeeder` не
выставляет флаг первому администратору: пароль ему задал сам оператор в мастере
(или он сгенерирован случайно и записан в Журнал событий — см. выше), поэтому
смена для `admin` остаётся добровольной.

Принудительная смена применяется только к учётным записям, созданным или сброшенным
позже через раздел «Администраторы» панели: им выдаётся временный пароль, и при
первом входе панель требует его сменить.

## 6. Обновление поверх существующей версии

Запустите новый файл `MitLicenseCenter-Setup-<версия>.exe`. Установщик автоматически
определяет наличие предыдущей установки по AppId Inno (`B7E9F3A2-…`).

### Порядок действий установщика при апгрейде

1. **Страница-предупреждение о бэкапе (§3.2)** — обязательное подтверждение чекбокса;
   без него кнопка «Далее» заблокирована.

2. **Остановка службы** (процедура `PrepareToInstall`, до замены файлов):
   установщик вызывает `ControlService(SERVICE_CONTROL_STOP)` и ждёт фактической
   остановки до ~30 секунд (60 итераций по 500 мс). Ожидание необходимо, чтобы
   снять блокировку с `MitLicenseCenter.Web.exe` перед копированием новой версии.

3. **Замена файлов** (секция `[Files]`, `ignoreversion`).
   `appsettings.Production.json` **не перезаписывается** (`skip-if-exists` на апгрейде —
   проверка `ServiceExists and FileExists`): правки оператора сохраняются.

4. **`ssPostInstall`** — последовательность после копирования:
   - `WriteProductionConfig` — пропускает запись конфига (файл уже существует).
   - `HardenConfigAcl` — накатывает ACL на конфиг (идемпотентно).
   - `WriteInitialAdminPassword` — пропускает (апгрейд: admin уже создан).
   - `HardenDataDirAcl` — накатывает ACL на каталог данных (идемпотентно).
   - `sc config` — выравнивает `binPath` и учётную запись под текущий выбор мастера;
     ошибка — предупреждение (не прерывает).
   - `netsh add rule` — обновляет правило брандмауэра под актуальный порт.
   - `sc start` — запускает службу; `rc=1056` (`ALREADY_RUNNING`) считается успехом.

5. **Применение миграций БД** — при первом старте новой версии бэкенд самостоятельно
   применяет новые EF Core миграции (`fail-fast`, ADR-18). Если миграции провалились
   и службу не удалось запустить — финальный экран мастера показывает **ошибку**
   (красный заголовок «Установка завершена с ОШИБКОЙ»), а не «успех».

### Что сохраняется при апгрейде

| Что | Результат |
|---|---|
| `appsettings.Production.json` | Сохраняется (не перезаписывается) |
| `%ProgramData%\MitLicenseCenter` (key ring) | Не трогается |
| База данных SQL Server | Не трогается |
| Учётные записи панели | Не трогаются |

### Если старт службы провалился

Финальный экран мастера показывает ошибку (красный заголовок). Причину смотреть
в **Журнале событий Windows** (журнал «Приложение», источник `MitLicenseCenter`):
бэкенд пишет туда причину отказа (`fail-fast`, ADR-18) — недоступность SQL,
ошибка миграций, неверные права учётной записи службы и т.п.

Для запуска службы вручную после устранения причины: `services.msc` или
`sc start MitLicenseCenter`.

Если миграции применились частично — восстановить базу данных и key ring из
резервной копии, сделанной на шаге §3.2.

## 7. Удаление

Удаление запускается через «Программы и компоненты» (appwiz.cpl) или повторным
запуском установщика.

### Что удаляется автоматически

Секция `[UninstallRun]`:
- Остановка службы: `sc stop MitLicenseCenter`.
- Удаление службы: `sc delete MitLicenseCenter`.
- Удаление правила брандмауэра: `netsh advfirewall firewall delete rule name="MitLicense Center"`.

Секция `[UninstallDelete]`:
- Каталог ярлыка меню «Пуск»: `%ProgramData%\Microsoft\Windows\Start Menu\Programs\MitLicense Center`.
- Файл `{app}\appsettings.Production.json` (сгенерирован мастером вне секции `[Files]`).
- Остальные файлы каталога установки (`{app}`) удаляются штатным механизмом Inno.

### Каталог данных — по выбору оператора

При удалении появляется диалог (кнопка «Нет» выбрана по умолчанию):

> «Удалить конфигурацию и КЛЮЧИ ШИФРОВАНИЯ из  
> `%ProgramData%\MitLicenseCenter` ?  
> Нет (рекомендуется) — сохранить для переустановки.  
> Да — удалить ключи. ВНИМАНИЕ: без них секреты в базе данных (dbo.Settings)
> расшифровать НЕЛЬЗЯ.»

- **Нет (по умолчанию)** — каталог `%ProgramData%\MitLicenseCenter` с key ring
  сохраняется. Key ring и БД — единый бэкап-юнит (ADR-8): без ключей секретные
  настройки в `dbo.Settings` не расшифровать.
- **Да** — каталог удаляется (`DelTree`). Выбирать только если БД тоже выводится
  из эксплуатации.

### Что не удаляется никогда

| Что | Почему |
|---|---|
| База данных SQL Server | Установщик не знает крединалов SQL на момент деинсталляции; удаляется вручную через SSMS или `sqlcmd` |
| `%ProgramData%\MitLicenseCenter` | По умолчанию сохраняется (дефолт диалога — «Нет») |

## Приложение: диагностика проблем запуска

Установщик записывает лог в `%TEMP%\Setup Log *.txt` (`SetupLogging=yes` в `.iss`).

Если служба не стартует после установки или обновления:

1. Открыть **Журнал событий Windows** → журнал «Приложение» → источник
   `MitLicenseCenter`. Бэкенд пишет причину отказа при старте.
2. Проверить подключение к SQL Server — инстанс, имя БД, права учётной записи
   службы (роль `sysadmin`).
3. Режим (A): проверить право «Вход в качестве службы» (`secpol.msc`).
4. Проверить лог установщика: `%TEMP%\Setup Log *.txt`.
5. Убедиться, что порт не занят другим процессом:
