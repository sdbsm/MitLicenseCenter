# Документация MitLicense Center — навигационный индекс

Этот файл — карта документации: где что лежит, кому читать и в каком порядке.
Начните отсюда, если не знаете, куда идти дальше.

---

## Маршруты по ролям читателя

### Администратор / ИТ заказчика
*Разворачивает панель и отвечает за её работоспособность.*

1. `README.md` — первое знакомство, что за продукт и что потребуется
2. `docs/INSTALL.md` — системные требования, пошаговая установка, обновление, удаление
3. `docs/OPERATIONS.md` — служба Windows, конфигурация, логи, бэкап/restore, диагностика
4. `SECURITY.md` — модель безопасности, принятые риски, права службы, транспорт
5. `docs/USER_GUIDE.md` — как работает панель, чтобы консультировать операторов

---

### Оператор панели
*Работает с интерфейсом каждый день: лицензии, публикации, сеансы.*

1. `docs/01_OVERVIEW.md` — что умеет продукт, ключевые понятия и глоссарий
2. `docs/USER_GUIDE.md` — вход, онбординг клиента, лимиты, разделы панели, бэкапы
3. `docs/06_UI_GUIDE.md` — дизайн-система и UX-паттерны (опционально, при вопросах к интерфейсу)

---

### Разработчик
*Вносит изменения в код, пишет тесты, разбирается в архитектуре.*

1. `docs/01_OVERVIEW.md` — продукт, границы, глоссарий (обязательно до чтения кода)
2. `docs/02_ARCHITECTURE.md` — слои, anti-corruption граница, интеграции, deployment
3. `docs/03_DOMAIN_MODEL.md` — сущности, инварианты, enum-ы, схема БД
4. `docs/04_BACKEND.md` — структура .NET-проектов, конвенции API, EF, фоновые задания
5. `docs/05_FRONTEND.md` — структура SPA, сетевой слой, TanStack Query, формы, i18n
6. `docs/06_UI_GUIDE.md` — shadcn/ui, Tailwind, layout, UX-паттерны, доступность
7. `docs/DEVELOPMENT.md` — пререквизиты, скрипты, тесты, CI, специфика Windows/1С
8. `docs/DECISIONS.md` — действующие ADR: почему принято именно такое решение
9. `SECURITY.md` — модель угроз и принятые риски (для задач, затрагивающих безопасность)

---

### AI-агент / куратор
*Планирует задачи, ставит постановочные PR, ревьюит, не пишет код напрямую.*

1. `CLAUDE.md` — регламент агента: правила работы, запреты, дисциплина
2. `docs/DECISIONS.md` — ADR: обязывающие архитектурные решения
3. `docs/PROJECT_BACKLOG.md` — активный реестр задач (читается первым при каждом старте)
4. `docs/01_OVERVIEW.md` → `docs/02_ARCHITECTURE.md` и профильные доки — по задаче

## Принципы канона

**Источник правды — код.** Документация описывает факт кода, а не намерение. Если
формулировка в docs и поведение кода расходятся, правда — в коде; документ нужно исправить.

**Present-tense.** Все утверждения в docs/ — о текущем состоянии системы, не о том,
что «будет» или «планируется» (для планов есть `ROADMAP.md`).

**Doc-driven изменения.** Изменение поведения системы сопровождается правкой
соответствующего doc-файла в том же PR — документация не отстаёт от кода.

**Без дублирования цифр.** Конкретные числа (количество эндпоинтов, настроек, миграций)
не фиксируются в docs — они дрейфуют с каждым PR. За актуальными цифрами — в профильные
файлы (`04_BACKEND.md`, `03_DOMAIN_MODEL.md`) и сам код.
