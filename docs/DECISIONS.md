# Архитектурные решения (ADR)

Действующие, обязывающие архитектурные решения MitLicense Center. Каждый ADR описан в формате
**Контекст / Решение / Последствия / Статус**. Номера ADR стабильны — код и другие документы
ссылаются на них по номеру, история нумерации неприкосновенна (отменённые решения сохраняются
как записи со статусом «Отменён»).

Терминология — по глоссарию `01_OVERVIEW.md`. Источник истины для каждого решения — код
репозитория; статус фиксирует, соответствует ли формулировка факту кода: **подтверждён** (код
соответствует дословно), **изменён** (формулировка приведена к факту кода) или **отменён**.

---

## ADR-1. Фреймворк фронтенда — React (SPA)

**Контекст.** Панель — отдельный веб-интерфейс над .NET-бэкендом, который инкапсулирует
инфраструктуру (1С, IIS, SQL). UI должен быть полностью отвязан от серверной логики через
строгую границу API.

**Решение.** Фронтенд — React (Single Page Application) на TypeScript. UI общается с бэкендом
только через относительные вызовы `/api/v1/...`.

**Последствия.** Строгая граница API; UI и бэкенд развиваются независимо за единым контрактом.
Отвергнуты Blazor WebAssembly, ASP.NET Core MVC, Vue.js.

**Статус:** подтверждён.

## ADR-2. Способ интеграции с кластером 1С

**Контекст.** Панель администрирует кластер 1С на целевых одноузловых развёртываниях 8.5,
где REST API кластера по умолчанию не опубликован.

**Решение.** Кластер 1С администрируется исключительно через RAS (`ras.exe`, по умолчанию
TCP 1545), управляемый утилитой `rac.exe`. Это **единственный** адаптер кластера — см. ADR-16
(обоснование консолидации) и ADR-3.3 (полный CLI-контракт).

**Последствия.** Нет лишней HTTP-поверхности. Отвергнуты REST API кластера (не опубликован
на целевых деплоях), COM-соединение (`V83.COMConnector` — медленно и течёт), прямое вмешательство
в БД.

**Статус:** подтверждён.

## ADR-3. Подход к контролю лицензий (enforcement)

**Контекст.** Платформа 1С не даёт надёжного нативного хука, чтобы отклонить вход по внешним
мультитенантным правилам.

**Решение.** Контроль квот выполняется фоновым циклом согласования на основе снимков сеансов
(каданс — ADR-6). Реального перехвата соединений / хуков веб-сервера нет.

**Последствия.** Фоновый воркер не блокирует кластер при входе; избыточные сеансы завершаются
постфактум. Отвергнут перехват соединений в реальном времени.

**Статус:** подтверждён.

## ADR-3.3 — CLI-контракт `rac.exe` (RAS)

**Контекст.** `RacExecutableRasClusterClient` (`Infrastructure/Clusters/`) — единственный
адаптер кластера; `RacOutputParser` и его тесты ссылаются на этот ADR. Нужен авторитетный
контракт вызова `rac.exe`.

**Решение.**

- **Путь к утилите.** Хранится в `Settings.OneC.RAS.ExePath` (без сидируемого дефолта — на 1С 8.5
  `rac.exe` лежит в версионном `C:\Program Files\1cv8\<версия>\bin\`). UI «Параметры» предлагает
  единый выбор «Платформа 1С» по `GET /discovery/rac-paths`; выбор пишет **обе** настройки —
  `OneC.RAS.ExePath` (путь) и `OneC.DefaultPlatformVersion` (версия из пути). В `dbo.Settings`
  ключи остаются раздельными.
- **Эндпоинт RAS.** `Settings.OneC.RAS.Endpoint` (вид `HostPort`, обычно `localhost:1545`),
  передаётся **первым позиционным аргументом** (`rac.exe <host:port> <режим> <команда> ...`).
  UI редактирует только порт (хост фиксирован `localhost` — single-host, ADR-28), но формат
  `host:port` на проводе неизменён.
- **Аутентификация.** `--cluster-user=<имя>` `--cluster-pwd=<пароль>` (два отдельных флага —
  `--auth=u:p` не существует). Если оба пусты, флаги опускаются (анонимное администрирование
  кластеров без зарегистрированных администраторов).
- **Кодировка.** `rac.exe` пишет stdout/stderr в активной OEM-кодовой странице процесса
  (CP866 на RU Windows). `SystemProcessRacRunner` читает сырой поток и декодирует через
  `Encoding.GetEncoding(OEMCodePage)` (`CodePagesEncodingProvider`), с откатом на UTF-8.
  `ProcessStartInfo.Standard*Encoding` **не** задаётся.
- **Формат вывода.** Вертикальные key-value блоки (`kebab-key : value`, CRLF, записи разделены
  пустыми строками).
- **Парсер (`RacOutputParser`, статический, чистый).** Регекс `^\s*(?<key>[a-z0-9-]+)\s*:\s?(?<value>.*?)\s*$`.
  Неизвестные ключи игнорируются; запись без обязательного ключа отбрасывается с Debug-логом.
  **Никогда не бросает** — кривая строка это Debug-skip.
- **Команды.**

  | Метод адаптера | Вызов |
  |---|---|
  | `PingAsync` | `rac.exe <ep> cluster list` — exit 0 + ≥1 кластер → `Ok=true`; иначе `Ok=false`. |
  | `ListActiveSessionsAsync` | `rac.exe <ep> session list --cluster=<uuid> [auth]` — `<uuid>` резолвится через `cluster list` и **кэшируется между вызовами** в `IClusterUuidCache` (singleton; ключ `(ExePath, Endpoint)`). |
  | `KillSessionAsync` | `rac.exe <ep> session terminate --cluster=<uuid> --session=<sid> [auth]` — команда именно `session terminate`. Флаг `--error-message` **не передаётся** (см. примечание ниже). |
  | `ListSessionLoadsAsync` | `rac.exe <ep> session list --cluster=<uuid> [auth]` — **тот же** спавн, маппится в перф-богатый `OneCSessionLoad` (раздел «Быстродействие»). |
  | `ListProcessesAsync` | `rac.exe <ep> process list --cluster=<uuid> [auth]` — рабочие процессы кластера → `OneCProcessLoad`. |

- **Маппинг сеанса (`rac` → `ClusterSession`).** `session → SessionId`, `infobase → ClusterInfobaseId`,
  `app-id → AppId`, `user-name → UserName`, `host → Host`, `started-at → StartedAtUtc`
  (ISO без оффсета, парсится `AssumeUniversal | AdjustToUniversal`). `ConsumesLicense` —
  **факт `rac session list --licenses`** (сеанс присутствует в выводе `--licenses` → потребляет
  лицензию; **ADR-48**), а не эвристика по `app-id`. `app-id` остаётся атрибутом сеанса (колонка/
  фильтр «Сеансы»), но в подсчёте лицензий не участвует; white-list `OneC.LicenseConsumingAppIds`
  удалён.
- **Маппинг перфа (`OneCSessionLoad` / `OneCProcessLoad`).** Все перф-поля nullable — парсер
  «никогда не бросает», отсутствующее поле даёт `null`, а не дроп. `memory-current` может быть
  отрицательным (середина GC) → знаковый `long`; `avg-call-time` дробное с научной нотацией
  → `NumberStyles.Float` + `InvariantCulture`.
- **Семантика ошибок (все сбои дают exit `255`).** Недоступный RAS / неизвестный UUID → пустой
  список + Warning или `Ok=false`. `session terminate` со stderr «Сеанс с указанным идентификатором
  не найден» → `Killed=false, AlreadyGone=true` (идемпотентность). Кривой UUID → `Killed=false, AlreadyGone=false` + Error.
- **Жизненный цикл процесса.** `System.Diagnostics.Process`, `UseShellExecute=false`, дедлайн
  **30 с** на вызов, `process.Kill(entireProcessTree: true)` по отмене. Спавн скрыт за
  `IRacProcessRunner` → `SystemProcessRacRunner`.
- **Бюджет спавнов (обязывающий).** При тёплом кэше UUID — **1 `rac.exe` на `ListActiveSessionsAsync`**
  (`session list`) и **1 на kill** (`session terminate`); лишний `cluster list` спавнится только на
  холодном кэше / смене endpoint / инвалидации (≤ 2). **Холодный тир дополнительно делает 1 вызов
  `session list --licenses`** (факт потребления лицензии, **ADR-48**) — итог: холодный цикл = обычный
  список + `--licenses`; горячий тир остаётся 1 вызовом (берёт классификацию из холодного снимка).
  Кэп 20 kill/цикл. Раздел «Быстродействие» (Live) даёт +1 спавн (`process list`) и работает
  **только** при открытой вкладке; «Запись» спавнит по таймеру, но ограничена авто-стопом записи.

**Последствия.** Контракт фиксирован и обязывающий. Альтернатива «долгоживущий TCP-сокет на порту
1545» (**Strategy B**) рассмотрена и **отклонена** (решение владельца, 2026-06-14): она означала бы
переписать закрытый бинарный протокол RAS на .NET (реверс-инжиниринг, ломкость по версиям 8.5.x,
потеря страховки официального `rac.exe`), а измеренной проблемы латентности нет — кросс-вызовой кэш
UUID кластера держит бюджет ≤ 26 proc/min с запасом. Вернуть Strategy B можно только при измеренной
проблеме латентности на боевой нагрузке. Это и есть канонический дом решения (ROADMAP его не несёт).
**Примечание (ревизия):** старая формулировка указывала вызов
`session terminate … --error-message="<reason>"` с показом причины завершаемому пользователю.
По факту кода (`RacExecutableRasClusterClient.KillSessionAsync`, `BuildArgsWithAuth`) флаг
`--error-message` **не эмитируется** (поиск `error-message` по `backend/src` — 0 совпадений);
команда строится только из `session terminate`, `--cluster=<uuid>`, `--session=<sid>` и опциональной
аутентификации.

**Статус:** изменён (формулировка `KillSessionAsync` приведена к факту: `--error-message` не передаётся).

## ADR-4. Обновление публикаций IIS — webinst + web.config

**Контекст.** Публикации на IIS создаёт штатная утилита 1С `webinst.exe`. Панель должна
управлять публикацией, не моделируя «желаемое состояние», которое операторы в панели не ведут.

**Решение.** Стандартные публикации (пере)создаются через `webinst` (`-publish -iis`,
из версионного `…\1cv8\<версия>\bin\webinst.exe`). Смена версии платформы переписывает **только**
сегмент пути `wsisapi.dll` в `web.config` (хирургическая правка; `default.vrd` не трогается).
Состояние публикации **только для чтения**: панель сообщает факт (`Published / NotPublished / Error`)
и не навязывает желаемое состояние. Флаг `Source` публикации (`Webinst` / `Configurator` / `Unknown`)
фиксирует происхождение и гейтит повторную публикацию: перезапись не-`Webinst`, опубликованной
публикации требует явного подтверждения оператора. Пакетные операции (публикация и смена платформы
над мультивыбором) исполняются как N **идемпотентных одиночных** вызовов тех же per-publication
эндпоинтов; одновременные спавны `webinst` ограничены серверным синглтоном `IWebinstConcurrencyGate`
(см. ADR-33). Прогон привязан к открытой вкладке.

**Последствия.** `webinst` перезаписывает `default.vrd`/`web.config` целиком, поэтому тонкая
настройка (OData / HTTP-сервисы / OpenID) ведётся вне панели. *Хостинговый* жизненный цикл IIS
(recycle/start/stop пула, сайта, `iisreset`) — отдельная возможность (ADR-24), инвариант
read-only публикации не нарушает. Отвергнуты: хирургический патч `default.vrd` + дрейф-реконсиляция
(прежняя модель, ADR-4.1, отменена); единый синхронный bulk-эндпоинт.

**Статус:** подтверждён.

## ADR-6. Каданс цикла согласования (двухуровневый)

**Контекст.** Единый 30-секундный цикл оставляет окно, в котором сверхлимитный пользователь
успевает войти и потерять данные при kill; глобальный 3–5-секундный цикл перегружает кластер.

**Решение.** Двухуровневый цикл. **Hot** — тенанты при ≥ 90% потребления опрашиваются каждые
3–5 с. **Cold** — полный снимок всех тенантов каждые 20–30 с. Тенант покидает hot-уровень после
двух подряд cold-циклов ниже порога. Enforcement идёт на **обоих** уровнях (ADR-6.1).

**Последствия.** Окно enforcement ограничено ≤ 5 с для тенантов из группы риска при низкой
базовой нагрузке. Отвергнуты единый 15–30 с цикл и посекундный опрос для всех.

**Статус:** подтверждён.

## ADR-6.1 — Hot-уровень в BackgroundService (не Hangfire) и тоже enforce'ит

**Контекст.** Минимум CRON Hangfire — 1 минута, а hot-уровню нужен каданс 3–5 с. Из-за того,
что kill теперь шлют оба пути (cold Hangfire-джоб и hot `BackgroundService`), `[DisableConcurrentExecution]`
(cold-vs-cold) недостаточен.

**Решение.** Оба уровня — `BackgroundService`: hot-опрос `HotTierPollingService`, cold-согласование
`ColdTierPollingService` (таймер читает `Polling.ColdIntervalSeconds` каждый цикл; на старте —
немедленный warm-up снимка). **Оба уровня enforce'ят:** kill в конце каждого cold-цикла **и** на
каждом hot-тике для hot-тенантов; hot переиспользует один `ListActiveSessionsAsync`, который и так
делает для UI, как fresh-check (без лишнего спавна `rac.exe`). Оба пути берут общий внутрипроцессный
замок `IEnforcementGate` (см. ADR-33) на всё окно «fetch + kill».

**Последствия.** Латентность kill для уже-hot тенанта ограничена одним hot-интервалом (≈ 4–5 с).
Single-node, один процесс → внутрипроцессного замка достаточно.

**Статус:** подтверждён.

**Cold-тир — `BackgroundService`, не Hangfire-джоб.** Cold-обход выполняет
`ColdTierPollingService : BackgroundService`, чей таймер реально соблюдает
`Polling.ColdIntervalSeconds` (дефолт 15с). Hangfire для cold не подходит: минимум 5-польного
CRON — 1 минута, поэтому интервал < 60с недостижим (троттл `< coldInterval` не срабатывал бы).
На single-host (ADR-28) последовательная cold-петля + `IEnforcementGate` делают лок избыточным —
`[DisableConcurrentExecution]`/`[AutomaticRetry(0)]` и отдельный троттл не нужны; warm-up снимка —
немедленный первый прогон сервиса.

## ADR-6.2 — `SessionKilled=200` + дифференциатор `AuditReason`

**Контекст.** Нужно различать автоматический и ручной kill в аудите.

**Решение.** Единый `AuditActionType.SessionKilled = 200` покрывает оба случая; различие живёт
в `AuditReason` (`LimitExceeded = 1`, `ManualByAdmin = 2`). Отдельное значение
`SessionKilledManual = 201` отвергнуто.

**Последствия.** `AuditReason` уже даёт фильтрацию; `LimitChanged = 201` зарезервирован.

**Статус:** подтверждён.

## ADR-7. Аутентификация администратора

**Контекст.** Локальная панель без внешних систем; нужен простой и безопасный механизм входа
для 5–20 пользователей на same-origin SPA.

**Решение.** ASP.NET Core Identity с локальными учётками в MSSQL (схема `auth`); cookie-аутентификация
(HttpOnly, `SameSite=Strict`, `Secure=Always` вне Development, скользящее истечение 8 ч). Две роли:
`Admin` (полный доступ, включая kill) и `Viewer` (только чтение). Identity сконфигурирована через
`AddIdentityCore<AppUser>` (не полный `AddIdentity`) + `AddRoles` + `AddSignInManager` +
`AddDefaultTokenProviders`. Аутентификация — ровно логин + пароль + кука; внутреннего 2FA нет
(ADR-15) — второй фактор делегирован сетевому периметру. Базовая колонка Identity
`AspNetUsers.TwoFactorEnabled` остаётся в схеме, но не читается и не пишется.

Первый администратор **сидируется `IdentitySeeder` в синхронном стартовом пайплайне (runtime,
после миграций)** — не миграцией. На пустой БД (`userManager.Users.AnyAsync()` ложно) сидер
создаёт учётку `admin`: либо с паролем оператора из одноразового файла `initial-admin.secret`
(установщик, ADR-31; пароль в лог не пишется, фиксируется только факт — `EventId 1002`), либо со
случайным паролем, записанным в лог (`EventId 1001`). Жизненный цикл учётной записи (force-смена
пароля при первом входе, отметка `LastLoginAt`, отзыв доступа) описан в ADR-37, ADR-38, ADR-39.

**Последствия.** Полностью локальная аутентификация без внешних зависимостей, общий с доменом MSSQL.
Сидинг гарантированно завершён до приёма первого запроса (ADR-18). Отвергнуты Windows Integrated Auth,
JWT-bearer (кука проще для same-origin SPA), внешние IdP.

**Статус:** изменён (первый админ сидируется `IdentitySeeder` в runtime-пайплайне, а не миграцией;
добавлена ссылка на жизненный цикл учётки — ADR-37/38/39).

## ADR-8. Управление секретами

**Контекст.** Секретные настройки (пароли SQL/кластера и др.) нельзя хранить открыто; при этом
восстановленный на новом железе бэкап должен читаться — значит, ключи шифрования должны быть
переносимы вместе с БД.

**Решение.** ASP.NET Core Data Protection. Значения секретных настроек шифруются в `dbo.Settings`
(`Value VARBINARY(MAX)`, протектор с purpose-строкой `mlc.settings.v1` — `SettingsStore`).
**Key ring** (мастер-ключи) хранится в **файловой системе в переносимом ОТКРЫТОМ виде** —
`%ProgramData%\MitLicenseCenter\keys\` (dev: `%LocalAppData%\…\keys\`), **без** шифрования самих
ключей. По факту кода `AddDataProtection` (`Infrastructure/DependencyInjection.cs`,
метод `AddDataProtection`) вызывает только `.SetApplicationName("MitLicenseCenter")` и
`.PersistKeysToFileSystem(...)` — **без** `ProtectKeysWithDpapi` / `ProtectKeysWithCertificate`.
Защита key ring на диске — **NTFS ACL** (установщик рвёт наследование и даёт доступ только
SYSTEM, Administrators и — в Windows-auth установке — служебной учётке). В среде `Test`
(интеграционный `MlcWebApplicationFactory`) метод уводит key ring в `%TEMP%\MitLicenseCenter\
test-keys` (всегда доступен тест-процессу) вместо `%ProgramData%` — тест не зависит от ACL
боевого каталога (MLC-189; раньше валило 8 middleware-тестов в worktree/CI).

**Последствия / компромисс.** Переносимость бэкапа поставлена выше шифрования ключей at-rest:
бэкап-юнит «key ring + БД» восстанавливается на любом новом железе/учётке, что и делает
восстановленный бэкап читаемым (ADR-15, `OPERATIONS.md`). Кто может прочитать key ring через
ACL — уже локальный администратор хоста, то есть уже контролирует всё на машине (та же логика,
что в ADR-21), поэтому шифрование ключей at-rest почти ничего не даёт против этой модели угроз,
ломая restore-on-new-hardware. Отвергнуты `ProtectKeysWithDpapi` (привязывает ключи к машине/учётке),
`ProtectKeysWithCertificate` (тянет управление `.pfx`), Vault/Key Vault (внешние системы), открытые
значения в `dbo.Settings`/конфиге.

**Статус:** подтверждён (по факту кода: `AddDataProtection` без `ProtectKeysWith*`).

## ADR-10.1 — TypeScript-типы API пишутся вручную (без OpenAPI-codegen)

**Контекст.** Поверхность API мала (~7 фич-файлов типов), фронт и бэк ведёт один автор; ставить
codegen-конвейер дороже, чем поддерживать типы руками.

**Решение.** Типы API фронтенда **пишутся вручную** в `frontend/src/features/*/types.ts` плюс
хелпер запросов `frontend/src/lib/api.ts`. Спек OpenAPI — справочный контракт, но генератор по нему
**не запускается** (в `frontend/package.json` нет шага codegen). Типы правятся вручную при изменении
формы эндпоинта, сверяясь с живым Swagger UI; бэкенд сериализует camelCase (см. ADR-32), что 1:1
совпадает с TS-интерфейсами. Runtime-валидация применяется **точечно**, не везде: Zod-схемы
охраняют ответы, где тихий дрейф контракта опасен — `GET /auth/me`, `POST /auth/login` (`roles`
управляет role-гейтингом), `GET /sessions/snapshot` и пагинированные конверты (`{items,total,page,pageSize}`)
для тенантов и инфобаз (через `pagedResponseSchema` в `frontend/src/lib/apiSchema.ts`). На этих
границах TS-тип выводится из схемы (`z.infer`), и несоответствие бросает контролируемый `ApiSchemaError`.
Nullable-поля, которые бэкенд опускает на проводе (ADR-32), моделируются хелпером `omittable(schema)`
(принимает и отсутствующий ключ, и `null`, нормализует в `null`), не `z.nullable()`.

**Последствия.** Hand-written типы дешевле codegen-конвейера при текущем масштабе; Swagger UI
служит браузерным контрактом для сверки. Принятие OpenAPI-codegen (`MLC-006(a)`) или сплошной
Zod-валидации (`MLC-016`) — осознанный будущий шаг в бэклоге, не подразумеваемая текущая гарантия;
он *вытесняет* этот ADR без церемонии отзыва. Отвергнуты: codegen-конвейер в сборке и Zod на каждой
границе ответа.

**Статус:** изменён (точечная Zod-валидация и omittable изложены как текущее состояние).

## ADR-14. CI/CD — GitHub Actions, только CI (без CD в v1)

**Контекст.** Деплой пока — ручная операторская процедура; автоматизация деплоя ждёт, пока
история развёртывания стабилизируется.

**Решение.** GitHub Actions на каждый push и PR в `main`. Бэкенд: `restore → build → test`.
Фронтенд: `install → lint → type-check → test → build`. PR блокируется, пока любая джоба красная.
Автоматизации деплоя нет: деплой — **ручная операторская процедура** (`OPERATIONS.md`). В `scripts/`
лежат: `build.ps1`, `build-installer.ps1`, `db-reset.ps1`, `dev.ps1`, `perf-counters.ps1`,
`perf-seed.ps1`, `publish-release.ps1`, `reset-admin.ps1`, `shadcn-add.ps1`. Из них **сборочно-упаковочные**:
`build.ps1` (гейт здоровья проекта), `publish-release.ps1` (тонкая обёртка над `dotnet publish` —
производит устанавливаемый артефакт, по умолчанию self-contained single-file `win-x64`),
`build-installer.ps1` (publish → ISCC, собирает GUI-установщик, ADR-31). Это **упаковка и локальная
установка/апгрейд на хосте**, а **не** CD: ничто в `.github/workflows/` или `scripts/` не связывает
зелёную сборку с работающим хостом. Удалённого/скриптового деплоя
(`scripts/Deploy-MitLicenseCenter.ps1`) нет — это пункт бэклога.

**Последствия.** Минимальный жизнеспособный страховочный слой; CD и любой скриптовый деплой ждут
стабилизации истории развёртывания. Fail-fast bootstrap (ADR-18) уже делает ручную процедуру
безопасной: новая версия синхронно применяет миграции на старте или отказывается обслуживать.
GUI-установщик (ADR-31) превращает шаги ручного копирования в мастер, но не делает деплой
пайплайном. Отвергнуты: отсутствие CI, полный CD по тегу, self-hosted runners / Jenkins / Azure DevOps,
поставка deploy-скрипта до стабилизации процедуры.

**Статус:** изменён (перечень `scripts/` приведён к фактическому составу из девяти файлов).

**Обновление (MLC-124, 2026-06).** Добавлен отдельный workflow `release.yml`
(`workflow_dispatch` + push тега `v*`): job на `windows-latest` прогоняет `build-installer.ps1`
(publish → ISCC → Setup.exe) и загружает артефакт. Это **CI-гейт упаковочного конвейера**
(REL-12): несобираемость `.iss` теперь выявляется автоматически, а не в день поставки.
`release.yml` ≠ CD: зелёная сборка не связывается с работающим хостом. `ci.yml` получил
paths-фильтры (не гоняет backend/frontend job на чисто-doc и перекрёстных изменениях),
шаг `format:check` для frontend (`pnpm format:check`) и информационные аудиты уязвимостей
(`dotnet list package --vulnerable`, `pnpm audit`, `continue-on-error`). Добавлен
`dependabot.yml` (NuGet/npm/Actions, weekly) — гигиена зависимостей (REL-14/REL-21).
Гарантия качества по-прежнему — локальный `.\scripts\build.ps1` (Actions красный по биллингу,
решение 2026-06-10).

## ADR-15. Граница сферы: бэкап и двухфакторная аутентификация

**Контекст.** Задача приложения — лицензирование сеансов 1С. Стратегия бэкапа и второй фактор
решаются на уровне платформы и сетевого периметра.

**Решение.** Внутреннего 2FA нет постоянно (вне сферы); **плановая/автоматизированная** оркестрация
бэкапа и **восстановление** — вне сферы (ответственность оператора), с одним узким
оператор-инициируемым исключением (ADR-27). Оператор владеет стратегией бэкапа через платформенный
тулинг (SQL Maintenance Plans, Veeam, Windows Server Backup, Robocopy + Task Scheduler). Приложение
**не** планирует бэкапы, не крутит backup-loop, не поставляет restore-CLI и ничего не восстанавливает.
ADR-27 добавляет **оператор-инициируемый on-demand single-database `COPY_ONLY`-бэкап** как кнопку-удобство;
он ручной, делает только отдельную копию и `COPY_ONLY` гарантированно не трогает внешнюю
дифференциальную цепочку. 2FA — концерн сетевого уровня (LAN/VPN, периметровый firewall, AD/SSO
на edge). `AspNetUsers.TwoFactorEnabled` остаётся в схеме, но операционно инертен. Бэкап-юнит:
БД MSSQL + key ring Data Protection (+ опционально `appsettings.Production.json`) — восстановление
одного без другого даёт нечитаемую БД.

**Последствия. Locked:** повторное введение **плановой** оркестрации бэкапа или любой возможности
**restore** требует свежего ADR; 2FA остаётся заблокированным. On-demand-бэкап ограничен ADR-27.
Отвергнуты: плановая/always-on оркестрация бэкапа внутри приложения, внутренний TOTP/WebAuthn/SMS.

**Статус:** подтверждён.

## ADR-16. RAS как единственный адаптер кластера 1С

**Контекст.** REST API кластера эмпирически отсутствует на целевых деплоях (`HttpClient` висит
весь 100-секундный таймаут), а двухадаптерный код тёк диагностически непонятными для оператора
режимами сбоя.

**Решение.** Слой адаптера кластера — исключительно `rac.exe`. Настройки оператора: `OneC.RAS.Endpoint`
(по умолчанию `localhost:1545`) и `OneC.RAS.ExePath` (без дефолта — ADR-3.3). `OneC.Cluster.AdminUser` /
`OneC.Cluster.AdminPassword` идут в `--cluster-user` / `--cluster-pwd` (оба могут быть пустыми).
Карточка кластера на дашборде — **карточка здоровья RAS** на базе `RasHealthProbingService : BackgroundService`
(30 с `PingAsync`) → `IRasHealthReader`. Три состояния: `OK`, `Сбой`, `Проверка…`. Переходы здоровья
не аудируются. `AuditActionType` `300`/`301` остаются историческими.

**Последствия. Locked:** повторное введение любого не-`rac.exe` адаптера кластера требует сначала
явно отозвать ADR-16. Отвергнуты: REST primary с RAS-fallback, RAS-only за фиче-флагом, обёртка
Polly-breaker вокруг RAS, удаление карточки дашборда.

**Статус:** подтверждён.

## ADR-17. Дефолты формы добавления инфобазы (только UI, не глобализация полей)

**Контекст.** Форма «Добавить инфобазу» содержала много повторяющихся в рамках развёртывания полей.
Нужно упростить её, не жертвуя per-base гибкостью и не затевая доменную миграцию ради UX.

**Решение.** Форма упрощена до трёх всегда-видимых per-base полей (клиент, кластерная инфобаза по
имени, БД SQL); остальные deployment-uniform поля убраны в свёрнутый блок «Дополнительно».
Повторяющиеся значения (IIS-сайт, версия платформы) получают **form-prefill настройки** —
`IIS.DefaultSiteName`, `OneC.DefaultPlatformVersion` — задаются один раз в «Параметрах» и
предзаполняют каждую новую базу. Эти ключи **только для UI** — ни один backend-сервис их не читает;
`Infobase`/`Publication` по-прежнему хранят значения per-base, оператор может переопределить в
«Дополнительно». Выбор кластерной инфобазы автозаполняет отображаемое имя; виртуальный путь по
умолчанию `"/" + slug(databaseName)`.

**Последствия. Locked:** form-prefill ключи не должны обрастать backend-читателями — продвижение
одного в runtime-resolved глобал требует сначала отозвать ADR-17. Поле SQL-сервера (`Defaults.DatabaseServer`)
было таким образом продвинуто и **переименовано в `Sql.Server`** (ADR-28 вытесняет ADR-17 для этого
поля); `IIS.DefaultSiteName` и `OneC.DefaultPlatformVersion` остаются UI-only под этим ADR (версия
платформы законно различается per-base, IIS-сайт — per-publication значение). Отвергнута глобализация
полей (удаление из сущностей и runtime-резолв из настроек).

**Статус:** изменён (отношение к ADR-28 изложено как текущее состояние).

## ADR-18. Fail-fast bootstrap

**Контекст.** Прежний `Task.Run` в `ApplicationStarted` глотал `throw`, и при сбое миграции/сидинга
хост слушал в полуинициализированном состоянии без видимого оператору сигнала.

**Решение.** Миграции и сидинг выполняются **синхронно в стартовом пайплайне, до `app.RunAsync()`**.
Порядок фиксирован: **создать БД, если её нет** (`DatabaseBootstrapper.EnsureDatabaseCreatedAsync`,
сырое подключение к `master`, `IF DB_ID(@name) IS NULL EXEC('CREATE DATABASE ...')`) → миграции EF
+ роли/первый админ (`IdentitySeeder`) → каталог настроек (`SettingsSeeder`). Любой сбой логируется
`Critical` и пробрасывается из `Main`, процесс падает с ненулевым кодом, **не приняв ни одного запроса**.
`IdentitySeeder` пропускает `MigrateAsync` на нереляционном провайдере (in-memory тесты через
`WebApplicationFactory<Program>`). Создание БД предшествует миграциям, потому что `MigrateAsync` под
`EnableRetryOnFailure` не создаёт отсутствующую БД (ошибка 4060 классифицируется как транзиентная).
Создание БД вызывается до регистрации Hangfire (он тоже коннектится к БД) и гейтится непустой
строкой подключения.

**Последствия.** Работающий слушающий процесс — это полностью смигрированный и засеянный процесс.
Требует прав `CREATE DATABASE` (`sysadmin`/`dbcreator`). **Locked:** инициализация, обязанная
завершиться до приёма трафика, живёт в синхронном pre-`Run` блоке. Отвергнуты: `Task.Run` с
unobserved `throw`; `StopApplication()` после приёма трафика.

**Статус:** подтверждён.

## ADR-19. Глобальный ProblemDetails + бэкстоп уникальности

**Контекст.** Непойманное исключение не должно отдавать голый 500 без тела или утекать стектрейс;
документированный 409 должен держаться даже при гонке записи.

**Решение.** Пайплайн регистрирует `AddProblemDetails()` и внешний `UseExceptionHandler()` — любое
непойманное исключение отдаётся как RFC 7807 `ProblemDetails` (с `traceId`). `CustomizeProblemDetails`
переписывает 5xx в нейтральный русский title/detail, никогда не эмитируя текст исключения или
стектрейс. Pre-flight `AnyAsync` в `Infobase`/`Tenant` остаётся happy-path, но `SaveChanges` обёрнут
**бэкстопом**: `DbUpdateException` от нарушения уникального индекса маппится в тот же 409 `ProblemCodes.*`,
что и pre-check. Нарушенный индекс различается по **имени** (стабильный идентификатор в ошибке SQL
Server), гейтится `SqlException.Number ∈ {2601, 2627}`. `DbUniqueViolation.Identify` — единый
распознаватель.

**Последствия.** Целостность контракта API под редкой гонкой записи 5–20 пользователей; деталь не
утекает на пути 500. EF InMemory не воспроизводит unique-violations — бэкстоп юнит-тестируется
интерсептором `SaveChanges`. **Locked:** новый конфликт по-прежнему добавляет `ProblemCodes.*`
(контракт 409 в `03_DOMAIN_MODEL.md`); бэкстоп переиспользует существующие коды. Отвергнуты: явная
транзакция с `UPDLOCK`; матчинг по локализованному тексту ошибки; новые 409-коды для гонки.

**Статус:** подтверждён.

## ADR-20. Вертикальные срезы доступа к данным в Web (minimal API + `AppDbContext`)

**Контекст.** Один HTTP-транспорт, один автор на обоих концах, правила используются ровно в одном
месте — дедуплицировать нечего.

**Решение.** Web-эндпоинты (minimal-API в `Web/Endpoints/*Endpoints.cs`) инжектят `AppDbContext` и
типы Identity **напрямую** и владеют бизнес-правилами (валидация, проверки уникальности, каскады,
запись `AuditLog`) **в хендлере**. Это **вертикальные срезы**, не строгая Clean/onion-слойка с
Application use-case прослойкой. Проект `MitLicenseCenter.Application` намеренно тонкий: интерфейсы
и контракты (`IClusterClient`, `ISettingsSnapshot`, `ISettingsStore`, DTO, `SettingDefinitions`…),
без per-endpoint use-case хендлеров. **Что НЕ ослаблено:** антикоррупционная граница к 1С/IIS
(ADR-5/ADR-16) стоит неизменной — Web **никогда** не трогает `rac.exe`, `ras.exe`, IIS,
`Microsoft.Web.Administration` напрямую; всё взаимодействие с инфраструктурой идёт через
Infrastructure-адаптер за Application-интерфейсом. Вертикальный стиль ограничен доступом к
собственной MSSQL-БД панели и Identity.

**Последствия.** Слой use-case'ов был бы церемонией без выигрыша переиспользования. **Условие
ревизии (не Locked):** если появится **второй потребитель** тех же правил (второй транспорт или
фоновый воркер с той же валидацией/уникальностью/аудитом вне HTTP) — извлечь правила в Application
use-cases тогда (`MLC-011(a)`); продвижение вытесняет этот ADR без церемонии отзыва. Отвергнут полный
onion с use-case-хендлером на эндпоинт.

**Статус:** подтверждён.

## ADR-21. Принятый риск — пароль кластера 1С в командной строке `rac.exe`

**Контекст.** В поддерживаемых версиях 1С 8.3–8.5 `rac.exe` принимает пароль кластера **только**
через флаг `--cluster-pwd` — нет stdin-промпта, файла пароля, env-канала.

**Решение.** Пароль администратора кластера (`Settings.OneC.Cluster.AdminPassword`) передаётся
`rac.exe` как флаг `--cluster-pwd=<password>` (`RacExecutableRasClusterClient.BuildArgsWithAuth`,
рядом с `--cluster-user`). На Windows командная строка процесса читаема другими процессами той же
(или более привилегированной) учётки и локальными администраторами, поэтому пароль кратко виден
инспектору процессов на время каждого вызова `rac.exe`. Это зафиксировано как **принятый
остаточный риск**, не устраняется.

**Последствия. Компенсирующие факторы:** (1) single-node — командная строка никогда не уходит в
сеть; (2) admin-only хост — кто читает чужую командную строку, уже локальный администратор с полным
контролем над кластером/MSSQL/key ring, и утёкший пароль не даёт ничего сверх того, чем он уже
владеет; (3) пароль шифрован at-rest в `dbo.Settings` (ADR-8), расшифровывается только на момент
спавна; (4) хост за сетевым периметром (второй фактор, ADR-15). Вызовы короткоживущие (дедлайн 30 с,
≤ 26 проц./мин). **Сигнал ревизии:** немедленно принять, если поддерживаемый `rac.exe` получит
stdin/файл/env-канал пароля (спавн уже изолирован за `IRacProcessRunner`); пересмотреть при изменении
single-node admin-only топологии. Отвергнуты: stdin/temp-файл/env-доставка (не поддержано `rac.exe`);
анонимный режим кластера; долгоживущий RAS-сокет-адаптер (аутентифицируется так же).

**Статус:** подтверждён.

## ADR-22. Транспортное усиление через конфиг-флаг

**Контекст.** Single-node может стоять за терминирующим TLS реверс-прокси (IIS ARR / Nginx) либо
без него; безусловное `UseHttpsRedirection`/`UseHsts` ломает dev и даёт двойной редирект/петли за
прокси.

**Решение.** Прод-усиление транспорта **опционально через конфиг**, не вшито безусловно. (1) HTTPS-redirect
+ HSTS (`app.UseHsts()` + `app.UseHttpsRedirection()`) включаются **только** при `Security:EnforceHttps=true`
**и** не-Development (решение — чистый хелпер `TransportSecurity.ShouldEnforceHttps`). (2) Swagger UI
отдаётся при `Development || Security:EnableSwagger` (`TransportSecurity.ShouldEnableSwagger`).
(3) Шифрование SQL-канала `Encrypt=True` только в `appsettings.Production.json`; базовый `appsettings.json`
держит `Encrypt=False` для dev. (4) `AllowedHosts` сужается с `*` до имени хоста панели в прод-профиле.
Cookie `Secure=Always` вне Development (ADR-7) независим от этих флагов.

**Последствия.** Конфиг-флаг сохраняет рабочий dev-дефолт и даёт оператору усилить прод без правки
кода. Панель остаётся LAN/VPN-периметровой (ADR-15). **Сигнал ревизии:** если развёртывание
стандартизируется на терминировании TLS самим сервисом, поднять `Security:EnforceHttps` в дефолт
прод-профиля (вытесняет без церемонии). Отвергнуты: безусловные `UseHttpsRedirection`/`UseHsts`;
`Encrypt=True` в базовом `appsettings.json`; удаление Swagger целиком.

**Статус:** подтверждён.

## ADR-23. Время на проводе — UTC с суффиксом `Z`

**Контекст.** SQL Server `datetime2` без зоны материализуется EF как `Kind=Unspecified`, и
`System.Text.Json` эмитит время **без** `Z` — относительные метки в UI «уезжали».

**Решение.** Каждый `DateTime`, который сериализует API, несёт явный UTC-обозначитель (`...T09:00:00Z`).
Время **хранится** в UTC (`TimeProvider.GetUtcNow().UtcDateTime` везде), **транспортируется** в UTC,
**отображается** в локальной зоне браузера. Настройки таймзоны нет — браузер знает зону ОС.
Механизм: глобальный read-only value-конвертер в `AppDbContext.ConfigureConventions`
(`Properties<DateTime>().HaveConversion<UtcDateTimeConverter>()`, `SpecifyKind(..., Utc)`) переставляет
каждый `DateTime`/`DateTime?` как UTC. Конвертер read-only (запись — identity), тип колонки не меняет,
**миграции не генерирует**. Гарантировано тестами `UtcDateTimeConventionTests` и `UtcDateTimeJsonTests`.

**Последствия.** Один UTC-инстант на проводе делает все метки в UI консистентными по построению;
браузер локализует бесплатно. **Сигнал ревизии:** если v1 когда-либо станет рендерить локализованное
время **на сервере** (плановый email/отчёт) — ввести явную display-зону там; хранение/транспорт
остаются UTC. Отвергнуты: per-property `SpecifyKind` на каждом контракте; глобальный JSON-конвертер,
всегда добавляющий `Z`; операторская настройка таймзоны.

**Уточнение (MLC-177).** Принцип «отображается в локальной зоне» распространяется и на **выбор
диапазона** в отчётах: date-range трактует выбранный день в браузерной зоне — локальная полночь и
конец суток сериализуются в UTC-границы запроса (`reportsUrlState`, через нативный `new Date(y,m,d,…)`,
DST-safe). Так «1 день» = 00:00–24:00 локально и совпадает встык с локально подписанной осью графика;
данные на проводе и в хранилище остаются UTC (бакеты `BucketStartUtc`).

**Статус:** подтверждён.

## ADR-24. Управление жизненным циклом IIS из панели

**Контекст.** Оператору нужно управлять хостингом IIS (пулы, сайты, `iisreset`) с сервера, не
заходя на него вручную; это отдельно от read-only состояния публикации (ADR-4).

**Решение.** Панель управляет **хостинговым жизненным циклом** IIS — `recycle`/`start`/`stop` пула,
`start`/`stop`/`restart` сайта и серверный `iisreset` — из блока «Управление IIS» над списком
публикаций. Цели (пулы, сайты) обнаруживаются из **живого IIS** (`ServerManager.ApplicationPools`/`Sites`
с текущим состоянием); общий статус IIS — из службы `W3SVC` (`ServiceController`, читается даже при
остановленном IIS). Все операции идут через Application-порт `IIisLifecycleService` → `OneCIisLifecycleService`
(Infrastructure): команды пула/сайта через `Microsoft.Web.Administration.ServerManager` (runtime,
без `CommitChanges`), `iisreset` — спавн `iisreset.exe` (OEM/CP866, 90 с). Граница ADR-20 неизменна.
Группа эндпоинтов `/api/v1/iis/*` (discovery `Viewer`; мутации `Admin`). Каждая мутация аудируется
(`IisApplicationPoolRecycled=220` … `IisStarted=228`, `TenantId=null`). Разрушительные операции
показывают подтверждение в UI + серверный гейт `Confirm=true` (409 `IIS_CONFIRM_REQUIRED` без него);
`start` — восстановление, без гейта. Разрушительные операции сериализуются синглтоном
`IIisResetConcurrencyGate` (ADR-33).

**Последствия.** Это первые **серверного масштаба** мутации IIS, поэтому служебной учётке нужны
права управлять пулами/сайтами и запускать `iisreset` (start/stop `W3SVC`/`WAS`). `iisreset`/stop
прерывает **все** сайты сервера, включая саму панель, если она хостится на этом IIS — это явно
показано в диалоге подтверждения. Операции синхронны и привязаны к открытой вкладке. ADR-4 управляет
*желаемым состоянием публикации* (read-only); ADR-24 — *хостинговым жизненным циклом*; они
ортогональны. Отвергнуты: расширение `IIisPublishingService`; per-row кнопки пула/сайта; PowerShell Remoting.

**Статус:** подтверждён.

## ADR-25. Сбор временного ряда потребления лицензий

**Контекст.** Система иначе не хранит историю потребления (активные сеансы живут только в
in-memory `ActiveSessionSnapshotStore`; персистят только события `AuditLogs`). Разделу «Отчёты»
нужен временной ряд.

**Решение.** Панель собирает **временной ряд потребления лицензий**. Метрика — мгновенный `Consumed`
(лицензионные конкурентные сеансы) плюс `Limit` тенанта на момент семпла. Семплы берутся **внутри
cold `ReconciliationJob`**, переиспользуя уже посчитанные `LicenseConsumption.CountByTenant` и живой
лимит — **без нового спавна `rac.exe`** (ADR-3.3/16); idle-тенанты семплируются `Consumed=0`.
Процессный **синглтон-аккумулятор** (`ILicenseUsageAccumulator`) сворачивает каждый семпл в текущий
**15-минутный бакет** (running min/max/sum/count per tenant) и на границе бакета эмитит одну
агрегатную строку на тенанта (`min`/`max`/`avg`, **`max` обязателен** — для конкурентных лицензий
пик и упирается в лимит). Строки персистят в телеметрическую таблицу `dbo.LicenseUsageSnapshots`
**вне enforcement-замка** (телеметрия не блокирует kill). **Один уровень, без rollup** — гранулярность
одна, чистится retention-настройкой + ночной джобой (`LicenseUsage.RetentionDays`, дефолт 365;
`license-usage-retention` в 03:30 по часам хоста — ADR-52). Сущность — **телеметрия в Infrastructure**
(`Infrastructure/Reporting/LicenseUsageSnapshot`, FK к `Tenant` `OnDelete(SetNull)`), по прецеденту
`AuditLog`, не доменный агрегат.

**Последствия.** Текущий открытый бакет живёт только в памяти синглтона — рестарт процесса теряет
его (приемлемо для best-effort телеметрии). Семплируются активные тенанты с ≥1 инфобазой. Отвергнуты:
доменный агрегат / EF-маппленный value-object; многоуровневый rollup; отдельный спавн `rac.exe`;
персист под enforcement-замком.

**Статус:** подтверждён.

## ADR-26. Подсистема «Быстродействие» — Live (pull, без персиста) + Запись (on-demand таблица)

**Контекст.** Живой вопрос «что насыщено прямо сейчас и виновата ли 1С?» истории не требует;
вопрос-расследование «зафиксировать что происходит N минут» требует ограниченной, переживающей
рестарт записи.

**Решение.** Раздел «Быстродействие» отвечает через светофор насыщения (CPU `Processor Queue Length`,
disk `Avg. Disk sec/Read|Write`, RAM `Available MBytes` + `Pages/sec` — сигналы насыщения, не голая
утилизация) + атрибуцию по семьям процессов (1С · MSSQL · обновления ОС · антивирус · прочее).
Два режима сбора:

- **Live** — всегда доступен, **pull-on-demand**: пока открыта вкладка, фронт опрашивает (~5 с),
  бэк читает источники на каждом poll. **Ничего не персистится** — закрытие вкладки останавливает
  сбор. Эндпоинты `GET /api/v1/performance/host` (хост-снимок), `/onec-sessions`
  (`OneCSessionLoad` + `OneCProcessLoad` через `IClusterClient`, 2 спавна `rac.exe`/poll),
  `/sql` (DMV: активные запросы + блокировки + IO-stall + дельта `wait_stats` через
  `ISqlPerformanceProbe`), все роль `Viewer`.
- **Запись** — включается/выключается **вручную** для расследования: процессный синглтон
  `IPerfRecordingService` держит одну активную запись, фоновый `PerfRecordingSamplingService`
  семплирует по таймеру (`Performance.RecordingSampleIntervalSeconds`, дефолт 15 с), пишет каждый
  тик (хост-снимок + top-N виновников 1С/SQL) в **таблицу БД** до ручного стопа или авто-стопа
  (`Performance.RecordingMaxDurationMinutes` / `…MaxSamples`). Одна запись за раз. Телеметрические
  сущности `PerfRecording`/`PerfRecordingSample` живут в `Infrastructure.Reporting`. На рестарте
  строки `Active` примиряются в `Interrupted`. Эндпоинты `/performance/recordings`: `POST` (start),
  `POST …/{id}/stop`, `DELETE …/{id}` — `Admin`; `GET` — `Viewer`. Always-on сбора и ночного
  retention нет.

**Последствия. Адаптеры (ADR-20):** хост через `IHostMetricsProbe` → `OneCHostMetricsProbe`
(Windows-only, WMI — perf-классы локализованы только по property-name, поэтому WQL, не
`PerformanceCounter`); 1С через `IClusterClient`; SQL через `ISqlPerformanceProbe` → `SqlPerformanceProbe`
(ADO.NET, не Windows-only; нужно `VIEW SERVER STATE`, иначе `Status=PermissionDenied`). Атрибуция
SQL **гранулярна по базе** (`DB_NAME → Infobase.DatabaseName → tenant`), не до сеанса (всё 1С→SQL
делит `program_name='1CV83 Server'`). Пробы — синглтоны, держат предыдущий снимок для дельты;
первый poll → `Measuring=true`. Недоступные процессы **считаются** (`AttributionIncomplete`), не
глотаются. Перф-сэмплы — измерения, не действия → своя таблица, не `AuditLog`. **Уточнение (MLC-179):**
операторские действия с записью (старт/стоп/удаление) — изменения состояния и аудируются новой
700-серией (`PerfRecordingStarted=700`/`Stopped=701`/`Deleted=702`, host-уровень, `TenantId=null`,
только при фактическом успехе); сами сэмплы по-прежнему вне `AuditLog`. Отвергнуты: always-on
«чёрный ящик» с retention; трассировка SQL → сеанс → пользователь;
удалённый сбор с других узлов кластера.

**Статус:** подтверждён.

## ADR-27. On-demand бэкап базы SQL (оператор-инициируемый, `COPY_ONLY`)

**Контекст.** Операторам нужна кнопка одно-кликового полного бэкапа клиентской базы, не нарушающая
их внешнюю (часто дифференциальную) стратегию.

**Решение.** Панель предлагает **per-database on-demand полный бэкап** с карточки инфобазы — тонкая
обёртка над `BACKUP DATABASE … WITH COPY_ONLY, COMPRESSION, CHECKSUM, FORMAT, INIT`. Это **дополняет
ADR-15**: on-demand — внутри; плановая оркестрация и restore — вне. `COPY_ONLY` несущий: обычный
FULL сбросил бы дифф-базу (DCM) и тихо сломал внешнюю цепочку; `COPY_ONLY` полностью восстановим, но
DCM не трогает — поэтому один универсальный путь, без флага «diff vs full». Адаптер (ADR-20):
`ISqlBackupService` → `SqlBackupAdapter` (ADO.NET). **Никогда не бросает** на инфраструктурном сбое —
возвращает типизированный degraded-результат (`InsufficientSpace`/`EstimateUnavailable`/`PermissionDenied`/`BackupFailed`).
Оркестрация: процессный синглтон `IBackupOrchestrator` + фоновый `BackupPumpService` гонят очередь,
которая **есть** таблица `DatabaseBackups` (в `Infrastructure.Reporting`); каждый тик стартует старейшую
`Queued` строку с не-выполняющейся парой `(server, database)` — per-database исключение + per-server
кэп параллелизма (`Backup.MaxParallel`) + FIFO. Длинный `BACKUP` идёт *вне* гейта. На рестарте
осиротевшие `Running` примиряются в `Failed`/`Interrupted`. Хранение: keep-latest-one per database,
замена через *новый → `RESTORE VERIFYONLY` → удалить прежний* (никогда delete-then-backup). Ночной
`BackupRetentionJob` (03:15 по часам хоста — ADR-52) удаляет `.bak` старше `Backup.TtlHours` (дефолт 24). Файловые операции
серверные (`xp_create_subdir`/`xp_fixeddrives`/`xp_delete_file`). Disk-guard: оценка ≥ used ROWS pages
+ `Backup.DiskSafetyMarginMb`; нет оценки → не стартуем (`EstimateUnavailable`). Роли: запуск бэкапа —
`Viewer`; удаление и смена `Backup.*` — `Admin`. Аудит: `BackupRequested=510` … `BackupsPurged=514`.

**Последствия.** SQL-логин панели — **sysadmin** (`xp_*` требуют sysadmin); нет прав → `PermissionDenied`,
не 500. `COPY_ONLY` + on-demand-only + no-restore держат поверхность узкой, дисциплину ADR-15 целой.
Отвергнуты: планирование/always-on; обычный FULL; restore из панели; per-server diff/full флаг;
фиксированный пол свободного места.

**Уточнение (MLC-178) — список сверяется с фактом на диске.** Строки `DatabaseBackups` расходились
с реальным `.bak` (ручное удаление файла; keep-latest вытеснил файл, но строка осталась; TTL-чистка
удалила файл раньше строки), и панель показывала бэкап, которого нет. Две правки в границах ADR:
(1) при успешном бэкапе keep-latest теперь обнуляет `FilePath`/`FileSizeBytes` предыдущих
`Succeeded`-строк той же `(server, db)` (`BackupOrchestrator.CompleteAsync`) — строки сохраняются как
история, но не указывают на вытесненный файл; (2) список выдаёт живой флаг `fileAvailable`
(server-side `xp_fileexist` тем же каналом `ISqlBackupService`, батч на выдачу, «never throws» →
`null` = «не проверили»; FE различает (MLC-182): `Succeeded`+`false` → «файл отсутствует» (строка
ссылалась на файл, а его нет — неожиданно пропал, заметный сигнал); `Succeeded` без пути → «вытеснен»
(keep-latest обнулил путь — норма, спокойная пометка), вместо немого «—»). Read-only по
семантике: строки не удаляются, аудит-enum не трогается, планировщик/restore не вводятся. Полная
реконсиляция с удалением сиротских строк отвергнута (теряет историю операций).

**Уточнение (MLC-183) — предпоказ disk-guard.** Расчёт disk-guard (оценка размера базы по занятым
страницам `FILEPROPERTY` + свободное место `xp_fixeddrives` + запас `Backup.DiskSafetyMarginMb`)
вынесен из `BackupAsync` в метод порта `ISqlBackupService.EstimateAsync` (единственная реализация:
её зовёт и `BackupAsync`-стоп-кран, и предпоказ). Read-only `GET /api/v1/backups/estimate` (Viewer)
отдаёт свободно/оценку/достаточно ДО запуска (ненастроенная папка/нет sysadmin/SQL недоступен →
200 degraded, не ошибка). UI окна бэкапов показывает «свободно · оценка ~… (до сжатия)» и при нехватке
— предупреждение, **но кнопку не блокирует**: оценка — верхняя граница до сжатия, серверный guard
остаётся единственным стоп-краном. Аудит не пишем (read-only), границы ADR-27/15 не меняются.

**Статус:** подтверждён.

## ADR-28. Single-host топология — один хост, один кластер 1С, один инстанс SQL

**Контекст.** Панель несла обобщённый мультисерверный «на всякий случай» каркас (per-base
`DatabaseServer`, параметр `server=`, ключ `OneC.Cluster.Server`), который моделировал никогда не
существовавшее в поле мультихост-развёртывание и стоил UX-трения.

**Решение.** Развёртывание **single-host по дизайну и подтверждено финальным**: один Windows Server
хост держит вместе .NET-бэкенд, IIS, инстанс MSSQL и кластер 1С; ровно **один кластер 1С** и **один
инстанс SQL** для всех клиентских БД. Это формализует давнее Locked-ограничение «single-node». Убраны
рудименты: per-base `Infobase.DatabaseServer` **удалён**, инстанс SQL живёт только в `Settings.Sql.Server`
(переименован из `Defaults.DatabaseServer`) и теперь **единственный runtime-resolved источник**
(`SqlDatabaseDiscovery` и `BackupsEndpoints` его читают; пусто → `Available:false` на пикере БД и
`409 SQL_SERVER_NOT_CONFIGURED` на бэкапе); `GET /discovery/databases` **не** принимает параметр
`server=`; ключ `OneC.Cluster.Server` **удалён** — адрес кластера для `webinst -connstr "Srvr=…"`
выводится из хоста `OneC.RAS.Endpoint`.

**Последствия.** Одна настройка на концерн (`Sql.Server`, `OneC.RAS.Endpoint`) проще в эксплуатации.
ADR-28 **вытесняет ADR-17** для поля SQL-сервера (стало runtime-resolved глобалом); два других
prefill-ключа (`IIS.DefaultSiteName`, `OneC.DefaultPlatformVersion`) остаются UI-only под ADR-17.
ADR-4 (webinst) неизменен — сменился только источник `Srvr=`. **Locked:** повторное введение любой
мультихост-возможности (SQL отдельно от хоста панели, второй инстанс SQL для клиентских БД,
мультиузловой кластер 1С) требует **сначала отозвать ADR-28** и запускает полный re-review
инфраструктурных адаптеров. Отвергнуты: сохранение мультисерверного каркаса; реизвлечение SQL-сервера
из `ConnectionStrings:Default` вместо настройки.

**Статус:** подтверждён.

## ADR-29. Discovery-first добавление инфобазы + ignore-list кластерных баз

**Контекст.** База в кластере 1С, но не заведённая в панель, — слепое пятно: `ReconciliationJob`
маппит сеансы по `ClusterInfobaseId` и тихо дропает несовпавшие (их сеансы не считаются ни в чьём
лимите). Симметрично: база в панели, но удалённая из кластера, — «мёртвая душа».

**Решение.** Кластерные базы, не заведённые в панель, делаются **видимыми и actionable** (предупреждающий
баннер + диалог разрешения на «Базах»), «Добавить базу» становится **discovery-first**. Состояния
«база без клиента» нет — `Infobase.TenantId` остаётся **NOT NULL**; «Назначить» — обычный create-поток,
предзаполненный выбранной кластерной базой; список нераспределённых вычисляется on-demand. Read-эндпоинт
`GET /api/v1/infobases/unassigned` (Admin) диффит живой снимок RAS (`IClusterClient.ListInfobasesAsync()`,
ADR-20) минус зарегистрированные минус ignore-list; RAS недоступен → `Available:false` с
санитизированным русским сообщением (не пустой список). Снимок RAS кэшируется (синглтон, форма
`IClusterUuidCache`, TTL 60 с — константа, не настройка; `?refresh=true` мимо кэша); дифф
пересчитывается каждый запрос. Ignore-list — таблица `HiddenClusterInfobases` (PK `ClusterInfobaseId`,
снимок `Name`) через `POST/DELETE …/unassigned/{id}/hide` (Admin). Аудит: `UnassignedInfobaseHidden=14`,
`UnassignedInfobaseUnhidden=15`. Обратное направление (база удалена из кластера) — **третий список**
`MissingItems[]` в **том же** ответе `GET /infobases/unassigned`: строки панели, чей `ClusterInfobaseId`
отсутствует в кэшированном снимке RAS; вычисляется **только при `Available:true`** (RAS-сбой не есть
пропавшая база). Новой таблицы/миграции/джобы/мутации/аудита/настройки для этого нет — чистое read-only
наблюдение; удаление «мёртвой» строки идёт обычным infobase-delete-потоком. UI — только Admin; Viewer
ничего нового не видит.

**Последствия.** Это **read-only наблюдение, без авто-фикса** — точно инвариант ADR-4, в **обоих**
направлениях: панель показывает дрейф (нераспределённые и пропавшие-из-кластера) и предлагает
оператор-driven разрешения (assign / hide / delete), но никогда не авто-регистрирует, не авто-удаляет
и не мутирует кластер. Оба списка читают кластер только через единый RAS-адаптер (ADR-16), без нового
пути доступа. Отвергнуты: nullable `Infobase.TenantId`; фоновая snapshot-джоба с персистом списка
кластерных баз; настройка для TTL; серверный фильтр списка по «не найдено»; новые коды аудита для
сигнала «мёртвой» строки.

**Статус:** изменён (обратное направление `MissingItems` изложено как часть текущего состояния).

## ADR-30. Бэкенд хостит SPA (same-origin); IIS не нужен для страницы панели

**Контекст.** Прежняя модель ставила IIS+ARR перед панелью, чтобы отдавать статику — самый хрупкий
шаг любого установщика. На single-host (ADR-28) Kestrel может отдавать страницу сам.

**Решение.** .NET-бэкенд (Kestrel) **сам отдаёт собранный React SPA, same-origin с API**. В `Program.cs`:
`app.UseStaticFiles()` (после блока HSTS/HTTPS-redirect и **до** `UseAuthentication`, чтобы логин-страница
и бандлы грузились анониму) отдаёт хэшированные `/assets/*`; терминальный `app.MapFallback(...).AllowAnonymous()`
(после маппинга Hangfire-дашборда, **до** регистрации recurring-джоб) возвращает `wwwroot/index.html`
для запросов, не подхваченных эндпоинтом/статикой (требование HTML5-history `createBrowserRouter`,
чтобы deep-link/refresh клиентских маршрутов работали). Предикат fallback — чистый, тестируемый
`SpaFallback.IsReservedPath`: `/api` и `/hangfire` **зарезервированы** (никогда не переписываются в
`index.html`, неизвестный `/api/*` честно отдаёт 404). `index.html` отдаётся `no-cache, no-store,
must-revalidate`; `/assets/*` кэшируются надолго. Если `wwwroot` отсутствует (обычный `dotnet build`/dev),
fallback отдаёт 404 (страницу даёт Vite :5173). Сборка/публикация: MSBuild-таргет `CopySpaToPublish`
(`AfterTargets="Publish"`) собирает фронт (`pnpm install --frozen-lockfile` + `pnpm build`) и копирует
`frontend/dist` в `wwwroot` — **только на publish**. Escape-hatches: `-p:SkipSpaBuild=true`,
`-p:PrebuiltSpaDist=<path>`. `wwwroot/` git-ignored. Рантайм-таргет не требует node.

**Последствия.** Строгая граница ADR-1 неизменна — SPA по-прежнему общается с бэкендом только через
относительные `/api/v1/...`; same-origin **убирает** нужду в CORS (CORS-middleware не добавляется).
Флаг `Security:EnforceHttps` (ADR-12) нетронут. Один Kestrel-процесс отдаёт и API, и страницу, поэтому
установщик (ADR-31) ставит **одну Windows-службу** без IIS-сайта для панели — IIS остаётся только для
управляемых 1С-публикаций. Граница ADR-20 не затронута. Отвергнуты: IIS+ARR перед статикой; отдельный
статик-сервер (nginx/вторая служба); `UseDefaultFiles` + in-tree SWA (связал бы каждый `dotnet build`
с фронт-сборкой).

**Статус:** подтверждён.

## ADR-31. GUI-установщик (Inno Setup) для единственного хоста

**Контекст.** Панель должна ставиться и обновляться на single-host без ручного провязывания IIS;
скелет с LocalSystem по умолчанию падал на хостах, где у SYSTEM нет прав SQL.

**Решение.** Панель поставляется с **GUI-установщиком** (`.exe`-мастер, **Inno Setup 6**, Unicode),
который ставит и обновляет single-host-развёртывание. Он ставит self-contained артефакт публикации
(ADR-30 / ADR-14 — `MitLicenseCenter.Web.exe` с встроенным .NET 10 + `wwwroot`), регистрирует и
запускает **одну Windows-службу** (`MitLicenseCenter`), открывает выбранный LAN-порт в firewall и
пишет рабочий `appsettings.Production.json` из ввода мастера. Источник — `installer/MitLicenseCenter.iss`,
сборка — `scripts/build-installer.ps1` (publish → ISCC). Inno Setup — предусловие сборочной машины,
не рантайм-зависимость. Ключевые элементы:

- **Интерактивный мастер** собирает инстанс SQL, БД, **учётную запись службы** (виртуальная
  `NT SERVICE\MitLicenseCenter` по умолчанию либо именованная Windows-учётка / gMSA — ADR-49), сетевой
  порт (дефолт `8080`) и `AllowedHosts`. Подключение к локальному SQL — всегда `Trusted_Connection=True`
  (SQL-логин/пароль в конфиг не пишутся). **SQL-логин учётки службы создаёт сам установщик** (ADR-49)
  под Integrated Security запускающего администратора — предусловие: он `sysadmin` на локальном SQL.
  В мастере прогоняется тест достижимости инстанса до установки.
- **Пароль администратора в мастере.** На **чистой установке** страница «Учётная запись администратора»
  собирает пароль первого админа (валидируется против Identity-политики). На `ssPostInstall` он пишется
  в одноразовый файл `%ProgramData%\MitLicenseCenter\initial-admin.secret` под ACL каталога key ring;
  `IdentitySeeder` читает файл на первом старте, создаёт `admin` и **удаляет файл**, не логируя пароль
  (`EventId 1002`). Если файла нет (dev, `db-reset`, ручной деплой) — fallback на случайный пароль в лог
  (`EventId 1001`, ADR-7). На апгрейде страница пропускается, файл не пишется.
- **Распознавание существующей БД.** «Чистая установка» ≠ «пустая БД» (uninstall не трогает БД). Мастер
  **зондирует** целевую БД при уходе со страницы «Сеть» (`IF OBJECT_ID('auth.Users','U') IS NULL …`):
  fail-open; найдя существующих пользователей — пропускает страницу пароля, не пишет `.secret`, показывает
  информер. Зонд keyed на таблицу `auth.Users` — при смене маппинга Identity обновить запрос в lockstep.
- **Регистрация службы — в `[Code]` с проверкой кодов возврата.** `sc create`/`config`/`description`/`start`,
  авто-создание SQL-логина (`ProvisionSqlLogin`, ADR-49) и `netsh` идут через `Exec(...)` с проверкой rc.
  На чистой установке ненулевой `sc create` или провал провижининга SQL-логина → `MsgBox` + **откат
  установки** (нет «успешной» установки без работоспособной службы). Ошибка **1057**
  (`ERROR_INVALID_SERVICE_ACCOUNT`) получает выделенное сообщение (для именованной учётки — проверить имя
  и пароль; для gMSA — отметить чекбокс). Неудачный `sc start` предупреждает и указывает на Event Log
  (fail-fast bootstrap, ADR-18), не прерывая (служба уже создана). Пароли не появляются в сообщениях.
- **Апгрейд на месте.** Фиксированный `AppId` детектит существующую установку. `PrepareToInstall`
  останавливает службу и ждёт реальной остановки до `[Files]`. Три вещи переживают апгрейд:
  `appsettings.Production.json` (на апгрейде writer пропускает существующий файл; на чистой установке
  всегда (пере)пишется), key ring под `%ProgramData%\MitLicenseCenter` (`uninsneveruninstall`), БД 1С
  (никогда не трогается). После замены файлов служба стартует, миграции применяются fail-fast (ADR-18).
- **Бэкенд service-aware.** `Program.cs` зовёт `builder.Host.UseWindowsService(o => o.ServiceName = "MitLicenseCenter")`
  (это и делает `sc create` рабочим — под SCM хост сигналит `SERVICE_RUNNING`, content root = каталог exe,
  логи → Event Log). No-op в консоли/dev.
- **Консервативный uninstall.** Останавливает + удаляет службу, убирает firewall-правило, ярлык и
  сгенерированный `appsettings.Production.json`. **Никогда не трогает БД SQL.** Для конфига + key ring
  показывает keep-data промпт (дефолт No = сохранить).
- **Без подписи кода.** Установщик неподписан (решение оператора) — SmartScreen предупредит на первом
  запуске; принято для LAN-инструмента.

**Последствия.** Это следствие ADR-30 на single-host (ADR-28): один Kestrel → одна служба, без IIS-сайта
для страницы (IIS остаётся только для управляемых 1С-публикаций). Fail-fast bootstrap (ADR-18) делает
«stop → replace → start» безопасным апгрейдом. Это **упаковка + локальная установка/апгрейд**, не CD
(ADR-14). Открытый порт предполагает LAN/VPN-периметр (ADR-7/15). Connection-test — PowerShell-сниппет
`System.Data.SqlClient` (без нового CLI-verb в exe). Отвергнуты: MSI/WiX; IIS+ARR перед страницей
(отозвано ADR-30); подпись кода в v1; новый exe-verb для теста. (Прежде отвергнутое «установщик,
создающий SQL-принципал» **пересмотрено ADR-49**: для виртуальной учётки SID появляется только при
регистрации службы, поэтому логин создаёт установщик.)

> **Изменён ADR-49 (трек MLC-170).** Идентичность службы переведена на виртуальную учётную запись
> `NT SERVICE\MitLicenseCenter` (Windows-аутентификация как стандарт), режим B (LocalSystem + SQL-логин)
> убран, установщик сам создаёт SQL-логин. Текст этого ADR приведён к факту (выше): мастер собирает
> учётную запись службы вместо «режима аутентификации», провижининг логина — часть регистрации службы.

> **Изменён MLC-175 (Трек 2, редизайн GUI).** Выбор личности провижининга стал **явным**: страница
> режима (радиокнопки Integrated Security / SQL-логин) + условная страница учётных данных SQL-логина —
> вместо неявного «пусто = Integrated» (MLC-172). Отдельная кнопка «Проверить подключение» убрана:
> проверка достижимости SQL авто-выполняется при «Далее» (для Integrated — со страницы режима, для
> SQL-логина — со страницы кредов). Экран «Готово к установке» получил сводку выбранных параметров
> (`UpdateReadyMemo`). Порядок `ssPostInstall` (ADR-49) и провижининг (`FROM WINDOWS` + `sysadmin`) не
> менялись. Реализация — родные контролы Inno на отдельных страницах (radio+edit на одной странице не
> отрисовывались — баг MLC-172).

**Статус:** изменён (ADR-49 — идентичность службы; UX мастера — MLC-175; остальное в силе).

## ADR-32. Wire-контракт JSON — camelCase + опускание null-полей

**Контекст.** Hand-written TS-типы (ADR-10.1) должны 1:1 совпадать с формой JSON на проводе без
codegen. Нужен предсказуемый, стабильный формат сериализации, а nullable-поля не должны замусоривать
ответы явными `null`.

**Решение.** Глобальная конфигурация `System.Text.Json` в `Program.cs` (`builder.Services.Configure<JsonOptions>`):
`PropertyNamingPolicy = JsonNamingPolicy.CamelCase` (свойства DTO на проводе — camelCase) и
`DefaultIgnoreCondition = JsonIgnoreCondition.WhenWritingNull` (свойство со значением `null` сериализуется
как **отсутствующий ключ**, никогда как явный `null`). Доменные enum'ы (`AuditActionType`, `AuditReason`
и др.) сериализуются **по имени** через `JsonStringEnumConverter` — машинно-читаемый контракт без хрупкой
привязки к числам.

**Последствия.** TS-интерфейсы пишутся под camelCase verbatim. Поскольку null-поля опускаются, Zod-границы
(ADR-10.1) моделируют их хелпером `omittable(schema)` (принимает и отсутствующий ключ, и `null`,
нормализует в `null`), а **не** `z.nullable()` (тот требует присутствия ключа и отверг бы wire-форму,
как только значение становится null). Урок зафиксирован для каждой новой nullable-границы FE.

**Статус:** подтверждён (новый ADR — теневое решение, зафиксированное по факту `Program.cs`).

## ADR-33. Внутрипроцессные синглтон-гейты конкурентности

**Контекст.** Несколько операций на single-host нельзя выполнять одновременно (over-kill при двух путях
enforcement, параллельные спавны `webinst`, два `iisreset`), а часть состояния (UUID кластера) должна
переживать scoped-инвокации фоновых джоб. Распределённый замок не нужен — один процесс, один хост.

**Решение.** Единый паттерн: **процессные синглтоны** в `Infrastructure/DependencyInjection.cs`,
координирующие конкурентность/состояние без распределённого механизма.

- `IEnforcementGate` (`EnforcementGate`, `SemaphoreSlim(1,1)`) — общий замок обоих путей enforcement
  (cold Hangfire-джоб и hot `BackgroundService`); берётся на всё окно «fetch fresh list + kill», чтобы
  kill исполнял ровно один путь за раз (защита от over-kill, ADR-6.1).
- `IWebinstConcurrencyGate` (`WebinstConcurrencyGate`) — ограничитель одновременных спавнов `webinst`
  при пакетной публикации (N одиночных вызовов); кэп общий на процесс, независим от клиента (семья
  спавн-бюджета ADR-3.3, ADR-4).
- `IIisResetConcurrencyGate` (`IisResetConcurrencyGate`, `SemaphoreSlim(1,1)`) — сериализатор
  разрушительных IIS-операций; два `recycle`/`iisreset` никогда не перекрываются (ADR-24).
- `IClusterUuidCache` (`ClusterUuidCache`) — кросс-вызовный кэш UUID кластера; переживает scope'ы
  hot/cold-джоб, снимает лишний `cluster list` перед каждой командой (ADR-3.3). Той же формы — кэш
  снимка RAS для `/infobases/unassigned` (ADR-29) и singleton-пробы перфа (ADR-26), держащие предыдущий
  снимок для дельты.

**Последствия.** Один хост, один процесс → внутрипроцессного синглтона достаточно; распределённый
замок не нужен и не вводится. **Сигнал ревизии:** при отказе от single-host (отзыв ADR-28) каждый такой
гейт/кэш требует пересмотра (распределённый замок, общий кэш) — это часть мандата re-review
инфраструктурных адаптеров. Отвергнуты: распределённые замки; фиксированный `SemaphoreSlim` там, где
нужен динамический кэп (бэкап использует оркестратор-таблицу, ADR-27); Hangfire-`[DisableConcurrentExecution]`
как единственный механизм (не покрывает hot-путь вне Hangfire).

**Статус:** подтверждён (новый ADR — единое оформление паттерна, существующего в коде).

## ADR-34. CSRF — antiforgery нет, защита cookie `SameSite=Strict`

**Контекст.** Панель — same-origin SPA с cookie-аутентификацией. Нужна защита от CSRF при минимуме
церемонии.

**Решение.** Отдельного antiforgery-токена нет — в `backend/src` отсутствуют `AddAntiforgery`,
`ValidateAntiForgeryToken` и любой XSRF-механизм (поиск даёт 0 совпадений). Защита от CSRF —
аутентификационная кука с `SameSite=Strict` (`Program.cs`, `AddCookie`: `Cookie.SameSite =
SameSiteMode.Strict`, `HttpOnly = true`, `Secure=Always` вне Development). При `SameSite=Strict`
браузер не прикладывает куку к кросс-сайтовым запросам, поэтому сторонний сайт не может выполнить
аутентифицированный запрос от имени пользователя.

**Последствия.** Это **осознанное решение** для same-origin SPA: SPA и API на одном origin, кросс-сайтовых
форм-постов нет, поэтому `SameSite=Strict` закрывает вектор CSRF без накладных расходов на токен-поток
(который при hand-written API-клиенте ADR-10.1 пришлось бы вести вручную). **Сигнал ревизии:** если
появится кросс-origin потребитель API или встраивание панели в стороннюю страницу — пересмотреть
(antiforgery-токен или CORS-политика с явной защитой). Отвергнут: antiforgery-токен на каждый мутирующий
запрос (избыточен для same-origin при `SameSite=Strict`, усложняет hand-written клиент).

**Статус:** подтверждён (новый ADR — решение зафиксировано по факту кода: antiforgery отсутствует,
`SameSite=Strict` присутствует).

## ADR-35. Устойчивость Hangfire-джоб — хранение, retry-политика, graceful shutdown

**Контекст.** Hangfire-история растёт от рекуррентных джоб (самая частая —
`publication-status-refresh`, раз в 5 мин; cold-цикл с MLC-154 вынесен в
`ColdTierPollingService` вне Hangfire). Hangfire истекает завершённые джобы по `ExpireAt`,
но дефолт (1 день) задаётся неявно в библиотеке. Также при эволюции набора джоб
старые recurring-джобы остаются в схеме, если их не снять явно. Без явной политики все джобы получают
дефолтные **10 ретраев** Hangfire — лишний красный шум при стойком сбое БД; а упавшие джобы по дефолту
не истекают вовсе — копились бы бесконечно.

**Решение — хранение.** Срок хранения задаёт фильтр `JobRetentionStateFilter` (`IApplyStateFilter`,
подключён `cfg.UseFilter(...)` в `Program.cs`): `Succeeded`/`Deleted` живут **2 дня**; `Failed` —
**30 дней** (отдельное, более долгое окно: упавшие джобы остаются видимыми для разбора, но не копятся
вечно). Устаревшие recurring-джобы снимаются явно `RecurringJob.RemoveIfExists(...)` — в текущем коде
`RemoveIfExists("drift-check")` (наследие удалённого enforcement публикаций, ADR-4).

**Решение — retry-политика.** У каждой recurring-джобы — осознанный `[AutomaticRetry]` на методе
**интерфейса** (job'ы зарегистрированы через интерфейс, Hangfire берёт фильтры с зарегистрированного
метода — тот же инвариант, что у `[DisableConcurrentExecution]`):

- идемпотентные ночные housekeeping-джобы `audit-retention` (03:00), `license-usage-retention`
  (03:30), `backup-retention` (03:15) — времена по часам хоста (ADR-52):
  `Attempts = 3, OnAttemptsExceeded = Fail` — несколько
  попыток на транзиентные блипы, затем видимый Fail; следующий суточный запуск самоисцеляется;
- `publication-status-refresh` (`*/5 * * * *`, read-only): `Attempts = 0` — упавший тик безвреден,
  следующий тик восстановит.

Cold-цикл (`ColdTierPollingService`) больше не Hangfire-джоб (MLC-154, ADR-6.1): его устойчивость
обеспечивает сам `BackgroundService` — упавший прогон логируется и самоисцеляется следующим тиком
таймера, без retry-фильтров.

**Решение — graceful shutdown.** Recurring-джобы регистрируются с плейсхолдером `CancellationToken.None`
в выражении `RecurringJob.AddOrUpdate<T>(j => j.RunAsync(CancellationToken.None), …)`. Hangfire при
выполнении ПОДМЕНЯЕТ его реальным токеном, сигналящимся при остановке сервера / abort'е. Все тела джоб
прокидывают полученный `ct` в свои EF/IO-вызовы, поэтому при остановке службы завершаются кооперативно.
`CancellationToken.None` здесь — идиоматический плейсхолдер, а НЕ «неотменяемый» токен; замена его на
захваченный токен сломала бы подстановку. Контракт «тело уважает `ct`» зафиксирован тестом.

**Последствия.** Пара дней истории успешных + месяц упавших в дашборде; схема ограничена. Шум от
ретраев на стойком сбое устранён, упавшая джоба видна и самоисцеляется следующим запуском. Все пороги —
внутренние knob'ы, оператору не tuneable (как фиксированный CRON у retention-джоб), поэтому константы в
коде, а не Setting — без раздувания каталога настроек и миграций.

**Статус:** подтверждён (хранение/снятие — теневое решение по факту `JobRetentionStateFilter`/`Program.cs`;
retry-политика, 30-дневное окно `Failed` и фиксация контракта graceful shutdown добавлены в MLC-123).

**Update (MLC-191).** Cron-времена ночных суточных джоб теперь считаются по **местному поясу хоста**
(`TimeZoneInfo.Local`), а не по UTC — см. ADR-52. Указанные в этой записи метки (03:00/03:15/03:30) —
по часам хоста; retry-политика и хранение от этого не меняются.

## ADR-36. Политика lockout входа — 5 попыток / 15 минут

**Контекст.** Локальная парольная аутентификация (ADR-7) нуждается в защите от перебора пароля без
внешних систем.

**Решение.** Identity сконфигурирована (`AddIdentityCore<AppUser>` в `Infrastructure/DependencyInjection.cs`)
с lockout: `options.Lockout.MaxFailedAccessAttempts = 5`, `options.Lockout.DefaultLockoutTimeSpan =
TimeSpan.FromMinutes(15)`. Вход (`AuthEndpoints.LoginAsync`) вызывает `PasswordSignInAsync(...,
lockoutOnFailure: true)`, так что неудачные попытки инкрементируют счётчик и при 5-й учётка блокируется
на 15 минут. Парольная политика рядом: длина ≥ 12, обязательны цифра, строчная, заглавная и спецсимвол.

**Последствия.** После 5 неудачных попыток учётка временно недоступна для входа — `PasswordSignInAsync`
вернёт `IsLockedOut`, эндпоинт отдаёт `401` и пишет аудит `LoginFailed` с причиной «учётная запись
заблокирована». Lockout автоматически снимается через 15 минут; админ может снять его раньше через
`POST /users/{id}/enable` (`ResetAccessFailedCountAsync` + снятие `LockoutEnd`). Значения — код-константы,
оператору не tuneable.

**Статус:** подтверждён (новый ADR — теневое решение, зафиксированное по факту `DependencyInjection.cs`).

## ADR-37. «Отключение пользователя» = Identity-lockout до `DateTimeOffset.MaxValue`

**Контекст.** Нужно отключать учётку панели, не вводя отдельный флаг `IsActive` и доменную миграцию,
переиспользуя уже имеющийся Identity-механизм lockout.

**Решение.** «Отключение» учётной записи (`POST /api/v1/users/{id}/disable`, Admin) реализовано как
**Identity-lockout с `LockoutEnd = DateTimeOffset.MaxValue`** (`UsersEndpoints.DisableAsync` →
`userManager.SetLockoutEndDateAsync(user, DateTimeOffset.MaxValue)`), **не** отдельным флагом. «Включение»
(`POST …/enable`) снимает lockout (`SetLockoutEndDateAsync(user, null)`) и сбрасывает счётчик неудачных
попыток (`ResetAccessFailedCountAsync`). «Активна» вычисляется как «не под действующим lockout»
(`IsActive`: `LockoutEnd is null` или `<= now`); это же используется в списке `/users` и в guard'ах.
Guard'ы: нельзя отключить собственную учётку (`409 UserCannotDisableSelf`); нельзя отключить последнего
активного администратора (`409 UserLastActiveAdmin`, счёт идёт именно по роли Admin). Отключение ротирует
security-stamp (ADR-39), чтобы уже выданная кука жертвы умерла.

**Последствия.** Нет лишней колонки/миграции — отключение ложится на готовый Identity-механизм:
действующий `LockoutEnd` в будущем блокирует `PasswordSignInAsync`. Поскольку «активность» = отсутствие
lockout, обычный временный lockout от перебора (ADR-36) и постоянное «отключение» — одна модель с разным
`LockoutEnd`. Отвергнут отдельный флаг `IsActive` (дублировал бы lockout и требовал миграцию).

**Уточнение (MLC-180) — жёсткое удаление учётки.** «Отключение» (lockout) — мягкий вариант; рядом
добавлено необратимое удаление `DELETE /api/v1/users/{id}` (Admin, `UserManager.DeleteAsync`,
каскад Identity-таблиц ролей/токенов). Переиспользует **те же два guard'а** (нельзя себя —
`409 UserCannotDisableSelf`; нельзя последнего активного администратора — `409 UserLastActiveAdmin`).
История переживает удаление: аудит/бэкапы/perf ссылаются на инициатора **строкой**, не FK.
Новое аудит-событие `UserDeleted=109` (frozen-int, server-scope, пишется только при фактическом
успехе). Подтверждение — FE-барьер по ADR-45 (диалог «да/нет» с пометкой необратимости, без ввода
имени; серверного `Confirm`-гейта нет, паритет с удалением клиента). «Отключить» остаётся.

**Статус:** подтверждён (новый ADR — теневое решение, зафиксированное по факту `UsersEndpoints`).

## ADR-38. Force-password-change + `LastLoginAt` в жизненном цикле учётки

**Контекст.** Админ создаёт/сбрасывает учётку с временным паролем, которого не должен знать постоянно;
оператору нужна видимость последнего входа.

**Решение.** Сущность `AppUser` несёт `MustChangePassword` и `LastLoginAt`. При создании учётки
(`POST /users`, Admin) и сбросе пароля (`POST /users/{id}/reset-password`, Admin) выдаётся сгенерированный
**временный** пароль и ставится `MustChangePassword = true` (`UsersEndpoints`). Контракт `POST /auth/login`
и `GET /auth/me` возвращают `CurrentUserResponse(UserName, Roles, MustChangePassword)` — фронт держит
пользователя на блокирующем экране смены пароля, пока флаг стоит. Смена пароля (`POST /auth/change-password`)
при успехе снимает `MustChangePassword` и переиздаёт куку (`RefreshSignInAsync`); тот же эндпоинт
обслуживает и обычную смену, и блокирующий экран форс-смены. Успешный вход пишет `user.LastLoginAt =
clock.GetUtcNow().UtcDateTime` (`AuthEndpoints.LoginAsync`); список `/users` отдаёт `LastLoginAt` (колонка
«Последний вход»).

**Последствия.** Админ не знает чужой постоянный пароль — пользователь обязан сменить временный при первом
входе. `LastLoginAt` даёт оператору видимость активности учёток. Поля — часть Identity-сущности, схема
расширена под них (миграция Identity). Связано с ADR-7 (аутентификация) и ADR-39 (отзыв доступа).

**Статус:** подтверждён (новый ADR — теневое решение, зафиксированное по факту `AuthEndpoints`/`UsersEndpoints`/`AuthContracts`).

## ADR-39. Немедленный отзыв доступа — `SecurityStampValidator`

**Контекст.** Кука скользящая (8 ч, ADR-7); при активности жертвы доступ был фактически бессрочным —
отзыв роли/пароля/учётки не убивал уже выданную куку. Сам по себе lockout (ADR-37) закрывает только
будущий вход (`PasswordSignInAsync`), не отзывает выданную скользящую куку.

**Решение.** Доступ отзывается через **security-stamp**: при каждой операции отзыва соответствующий
эндпоинт ротирует `AspNetUsers.SecurityStamp` через `userManager.UpdateSecurityStampAsync(user)` —
это `DisableAsync` (отключение, ADR-37), `ResetPasswordAsync` (сброс пароля) и `ChangeRoleAsync` (смена
роли) в `UsersEndpoints`. Кука валидируется `SecurityStampValidator`: в `Program.cs`
`options.Events.OnValidatePrincipal = SecurityStampValidator.ValidatePrincipalAsync` сверяет stamp из
куки со stamp'ом в БД на каждой ревалидации; расхождение → кука отвергается (принудительный SignOut).
Поскольку `AddIdentityCore` (в отличие от полного `AddIdentity`) **не** регистрирует
`ISecurityStampValidator`/`ITwoFactorSecurityStampValidator`, они регистрируются явно в
`Infrastructure/DependencyInjection.cs` (`ISecurityStampValidator` → `SecurityStampValidator<AppUser>`,
`ITwoFactorSecurityStampValidator` → `TwoFactorSecurityStampValidator<AppUser>` — без него валидатор
падает на резолве зависимости, хотя 2FA не используется). Рядом в Web подключена служебная no-op-кука
`TwoFactorRememberMeScheme` — чтобы «sign-out everywhere» валидатора не падал «No sign-out handler».
Интервал ревалидации — `SecurityStampValidatorOptions.ValidationInterval = TimeSpan.FromMinutes(2)`
(`DependencyInjection.cs`): на горячем пути stamp не перечитывается на каждом запросе, только раз в интервал.

**Последствия.** Отозванная кука перестаёт работать в пределах ≤ 2 минут (компромисс «свежесть отзыва ↔
нагрузка на БД»; требование SEC-01 — 1–5 мин). Self-смена пароля переиздаёт куку свежим stamp'ом
(`RefreshSignInAsync`), своя сессия не рвётся. Без этого механизма разжалованный Admin продолжал бы
ходить с Admin-claims до естественного истечения куки (8 ч). Отвергнуты: меньший интервал (лишние
round-trip'ы в БД); больший интервал (расширяет окно, в котором уволенный/разжалованный ходит со старой
кукой).

**Статус:** подтверждён (новый ADR — теневое решение, зафиксированное по факту `Program.cs`/`DependencyInjection.cs`/`UsersEndpoints`).

## ADR-40. Операционная модель устойчивости службы — recovery-политика

**Контекст.** Служба `MitLicenseCenter` стартует fail-fast (ADR-18): при недоступном SQL bootstrap роняет
процесс. После перезагрузки хоста SQL Server может подниматься медленнее панели — тогда служба падает и без
авто-перезапуска остаётся выключенной до ручного вмешательства. Напрашивающееся «решение в лоб» —
`depend= MSSQLSERVER` в SCM (служба не стартует, пока не поднимется SQL) — НЕВЕРНО: имя `MSSQLSERVER`
верно только для дефолтного инстанса, у именованного служба зовётся `MSSQL$NAME`, а при **удалённом** SQL
локальной службы, от которой можно зависеть, нет вовсе → панель никогда не стартовала бы.

**Решение.** Установщик (`installer/MitLicenseCenter.iss`, процедура `StartServiceAndFinalize`, ADR-31/ADR-49)
задаёт устойчивость одним механизмом:

1. **Recovery-политика SCM — единственный и основной механизм**, не зависит от расположения SQL. После
   создания/конфига службы выполняется
   `sc failure {#MyServiceName} reset= 86400 actions= restart/30000/restart/30000/restart/30000`
   (три перезапуска с задержкой 30 с, счётчик сбоев сбрасывается раз в сутки). Если при загрузке хоста SQL
   ещё не готов, служба падает (fail-fast) и SCM перезапускает её с задержкой — к этому моменту SQL поднят.
   Покрывает ВСЕ конфигурации, включая удалённый SQL.

2. **SCM-зависимость (`depend=`) — производная SQL-зависимость ОТОЗВАНА (MLC-227).** Жёсткий `depend=`
   имел критический побочный эффект: ручная остановка SQL (штатное обслуживание) каскадно гасила панель,
   а Windows не поднимает зависимые службы при обратном старте SQL → панель оставалась выключенной
   до ручного вмешательства. При этом БД панели лежит на том же SQL, и панель умеет переподключаться
   сама (EF `EnableRetryOnFailure`, см. OPERATIONS «SQL недоступен в рантайме»), поэтому принудительный
   останов лишён смысла. Начиная с 1.1.4 установщик **активно снимает** уже выставленную зависимость
   (процедура `ClearServiceDependencies`) и на чистой установке (заведомо без неё — no-op), и на апгрейде:
   `sc config {#MyServiceName} depend= /` (`/` = нет зависимостей). Это лечит стенды, поставленные до
   1.1.4 с прежним `depend= MSSQLSERVER`. Best-effort: rc<>0 → предупреждение + Log, без Abort.

`sc`-вызовы best-effort: при ненулевом коде возврата установка продолжается с `MsgBox`-предупреждением
и записью в лог (как обработка `icacls`/firewall, MLC-107) — устойчивость это hardening, а не жёсткий гейт.
Оператор проверяет/правит политику через `sc qfailure` или `services.msc → Восстановление`.

**Последствия.** После перезагрузки хоста панель самовосстанавливается без ручного старта при любом
размещении SQL (recovery-политика). Служба **не зависит** от службы SQL в SCM: ручная остановка SQL
не останавливает панель; панель продолжает работать (отдаёт 503 на операции с БД) и сама отвечает
после возврата SQL без ручного старта. Связано с ADR-18 (fail-fast bootstrap), ADR-31 (GUI-установщик),
ADR-28 (single-host).

**Статус:** подтверждён (MLC-123); производная SQL-зависимость отозвана MLC-227.

## ADR-41. Security response headers — CSP, X-Frame-Options, X-Content-Type-Options, Referrer-Policy

**Контекст.** Панель — same-origin SPA (ADR-1, ADR-30): React-фронтенд и API живут на одном
origin. Браузерные атаки (XSS, clickjacking, MIME-sniffing) требуют дефенс-ин-депт слоя на
уровне ответа — вне зависимости от прикладной защиты. Swagger UI (`/api/docs`, ADR-12) исторически
использует inline-скрипты и inline-стили, несовместимые со строгим `script-src`.

**Решение.** Ранний middleware `SecurityHeaders.UseSecurityHeaders` (MLC-125, SEC-07) ставит
четыре заголовка на **все** ответы (SPA, статика, API, fallback — до `UseStaticFiles`):

- `X-Content-Type-Options: nosniff` — запрет MIME-sniffing.
- `Referrer-Policy: no-referrer` — не утекать Referer на внешние ресурсы.
- `Content-Security-Policy` — same-origin политика (см. ниже). **Исключается** на путях `/api/docs/*`.
- `X-Frame-Options: DENY` — запрет встраивания в iframe. **Исключается** на `/api/docs/*`.

Content Security Policy (жёстко прошита — single-host, SPA-бандл известен):
```
default-src 'self'; base-uri 'self'; object-src 'none'; frame-ancestors 'none';
img-src 'self' data:; font-src 'self' data:; style-src 'self' 'unsafe-inline';
script-src 'self'; connect-src 'self'
```

Обоснование отдельных директив:
- `script-src 'self'` — без `unsafe-inline`: Vite-бандл в проде не содержит inline-script'ов.
- `style-src 'self' 'unsafe-inline'` — нужен для React/CSS-in-JS инлайновых стилей компонентов.
- `frame-ancestors 'none'` — дублирует `X-Frame-Options DENY` для современных браузеров.
- `object-src 'none'` — запрет Flash/embed.
- `img-src / font-src` + `data:` — иконки и шрифты могут быть base64 data URI.
- `connect-src 'self'` — XHR/fetch только к своему origin (API same-origin).

Исключение `/api/docs*`: SwaggerUI использует inline-скрипты — строгий `script-src 'self'` его ломает.
`nosniff` и `Referrer-Policy` остаются даже на Swagger-путях. HSTS/HTTPS-redirect управляются
отдельно через `EnforceHttps` (ADR-12) — не дублируются.
Значения зашиты жёстко — не tuneable оператором (single-host, SPA-бандл предсказуем).

**Последствия.** Браузеры применяют политику ко всем ответам панели. Swagger `/api/docs`
не сломан: CSP и XFO исключены. Прод-SPA не сломан: `script-src 'self'` совместим с
Vite-бандлом, `style-src 'unsafe-inline'` — с React CSS-in-JS. Покрыт интеграционными
тестами через `WebApplicationFactory<Program>` (`SecurityHeadersIntegrationTests`) и
юнит-тестами на `ApplyHeaders`/`IsSwaggerPath` (`SecurityHeadersTests`).

**Статус:** принят (MLC-125).

## ADR-42. Rate limiting на POST /auth/login — per-IP fixed window поверх Identity-lockout

**Контекст.** Identity-lockout (ADR-36) защищает учётную запись: после 5 неудачных попыток за
15 минут учётка блокируется. Это создаёт риск lockout-DoS: атакующий может намеренно заблокировать
учётку легитимного пользователя, слая 5 запросов с разных адресов. Кроме того, перебор паролей
через одного клиента не ограничивается — lock'ается только учётка, а не источник запросов.

**Решение.** Встроенный `Microsoft.AspNetCore.RateLimiting` (.NET 10) с политикой `"login"`:
per-IP fixed window, `PermitLimit=10`, `Window=1 min`, `QueueLimit=0` (MLC-125, SEC-08).
Партиционирование по `RemoteIpAddress` — троттлинг источника, не учётки.
Middleware `UseRateLimiter` ставится после `UseAuthorization` и перед маппингом эндпоинтов;
политика применяется только к `POST /api/v1/auth/login` через `RequireRateLimiting("login")`.
При превышении → `429 Too Many Requests` ДО тела `LoginAsync` (без входа в SignInManager →
`AuditActionType.LoginFailed` не пишется — это не «неудачный вход», это rate-limit reject).
Значения (10/мин) зашиты — не tuneable оператором.

Слои защиты входа (дефенс-ин-депт):
1. Rate-limit: **троттлит источник** (до хендлера, аудит не пишется) — ADR-42.
2. Identity-lockout: **защищает учётку** (внутри хендлера, аудит `LoginFailed` пишется) — ADR-36.
3. Cookie SameSite=Strict + SecurePolicy=Always — защита сессии.

**Последствия.** Автоматический перебор с одного IP блокируется на 10-й попытке за минуту.
Легитимный пользователь с нормальной активностью (несколько попыток войти) лимит не замечает.
Lockout-DoS через флуд не масштабируется — источник режется до бизнес-логики.
Покрыт интеграционными тестами: флуд → 429 (`LoginRateLimitFloodTests`), штатный запрос не
429 (`LoginRateLimitSingleTests`).

**Статус:** принят (MLC-125).

## ADR-43. Лицензия Inno Setup — свободна для коммерческого использования

**Контекст.** Finding REL-17 (audit/2026-06) отметил «баннер Non-commercial use only при сборке»

**Статус:** принят (MLC-126).

## ADR-44. Привилегии службы — повышенные по необходимости; low-priv аккаунт отложен

**Контекст.** Finding SEC-05 (audit/2026-06): служба `MitLicenseCenter` работает под
Windows-учётной записью, входящей в локальную группу Administrators, с ролью `sysadmin` на
SQL — привилегии выше реальной потребности. Компрометация web-процесса (парсит внешний
CP866-вывод `rac.exe`, взаимодействует с IIS через `ServerManager` и с SQL Server) означает
полный контроль над хостом (admin-эквивалент) и роль sysadmin на SQL. *(Идентичность учётки —
виртуальная `NT SERVICE\MitLicenseCenter` либо именованная/gMSA, ADR-49; на момент находки была
LocalSystem (режим B) либо именованный admin-аккаунт (режим A). Уровень привилегий этим не меняется.)*

**Решение.** Привилегии оставлены повышенными осознанно. Функции продукта — публикация и
recycle/iisreset IIS через `Microsoft.Web.Administration.ServerManager`, бэкап SQL через
`BACKUP DATABASE`, чтение DMV производительности — требуют высоких прав по своей природе.
Управление хостом и есть назначение продукта (§1 SECURITY.md).

**Результат runtime-трассы (MLC-152, стенд 2026-06-14, Procmon, 417 730 событий).** Триггер
наступил, трасса проведена. Вывод: **low-priv аккаунт недостижим без потери функций.** (1) Windows:
собственный процесс службы IIS-конфиг только читает, но привилегированные записи делегированы
детям под тем же токеном — публикация `webinst.exe → APPCMD.EXE` правит `applicationHost.config`
(`/commit:apphost`, admin), `iisreset.exe`/recycle управляют `W3SVC`/`WAS` (admin), запись в
`C:\inetpub` (admin); «узкий набор» = воссоздать администратора. (2) SQL: единственное, что
требовало `sysadmin` сверх `db_owner`, — бэкап через `xp_*`; даже сняв их (MLC-152), бэкап
**клиентских** баз требует `db_owner`/`db_backupoperator` на **каждой** базе, а серверного
«backup any database» короче `sysadmin` в SQL Server нет — для мультитенантного хостинга с
**динамически появляющимися** базами это непрактично (грант на каждую новую базу). Детали трассы —
`mlc-152-stand-trace.md`.

**Решение по итогу трассы:** low-priv (и Windows, и SQL) **не предпринимаем** — `sysadmin` на SQL и
admin-эквивалент Windows-аккаунта остаются **штатным профилем** продукта (управление хостом +
динамичный мультитенант — его природа). Кодовая попытка снять `sysadmin` с бэкапа (MLC-152) дала
лишь новую зависимость (каталог бэкапов на две учётки) без практической выгоды и **откачена**
(MLC-153, `xp_*` восстановлены).

**Последствия.** SEC-05 закрыт как принятый риск. Смягчения:
- single-host в LAN/VPN (ADR-28) — доступ к процессам узла только у администраторов хоста;
- §6 и §9 п.1 `SECURITY.md` фиксируют риск явно с ссылкой на этот ADR.

Учётка службы несёт эти привилегии явно: установщик добавляет её в локальную группу
Administrators (по well-known SID `S-1-5-32-544`) и создаёт SQL-логин с ролью `sysadmin`
(ADR-49) — см. `installer/MitLicenseCenter.iss`, процедуры `AddServiceAccountToAdministrators`
и `ProvisionSqlLogin`. Мастер (REL-06/MLC-127) предупреждает оператора о требованиях к
именованной учётке до установки (`CurPageChanged`).

> **Изменён ADR-49 (трек MLC-170).** Идентичность службы переведена на виртуальную учётную запись
> `NT SERVICE\MitLicenseCenter` (вместо LocalSystem / именованного admin-аккаунта), но **уровень
> привилегий этим ADR не затрагивается**: `sysadmin` в SQL + admin-эквивалент на хосте остаются
> штатным профилем. ADR-49 меняет только, *под какой* учёткой служба их несёт. Текст ADR-44 приведён
> к этому факту (идентичность — виртуальная учётка; привилегии — без изменений).

**Статус:** изменён (MLC-127; follow-up закрыт по итогам трассы MLC-152/153 — low-priv признан
недостижимым без потери функций, `sysadmin`/admin остаются штатным профилем; идентичность учётки
приведена к ADR-49 — виртуальная учётная запись, уровень привилегий без изменений).

## Заблокированные операционные ограничения (binding)

- **Приоритет kill:** при `Consumed > Limit` завершаются сеансы по `StartedAt DESC` (newest-first)
  до `Consumed == Limit`. Проще всего объяснить пользователям и не прерывает устоявшиеся сеансы.
- **Грейс kill:** сеанс моложе грейса (`Enforcement.KillGraceSeconds`, дефолт 15 с, диапазон 5–120,
  относительно `StartedAt` снимка) **не** подлежит limit-kill. 1С заполняет `user-name` только после
  аутентификации, а newest-first целит в только что подключившийся сеанс — без грейса мы бы системно
  завершали сеансы в окне «подключён, держит лицензию, ещё не вошёл» и писали пустого пользователя в
  аудит. Так как кандидаты newest-first, попадание младшего в грейс **останавливает** цикл этого тенанта
  (ждём), а не проваливается к kill старшего. Сверхлимит терпится ≤ грейса; следующий hot-тик (~4–5 с)
  переоценивает. Пустой `user-name`, переживший грейс, рендерится как «без пользователя» (`SessionDisplay`).
- **Идемпотентный протокол kill:** перед kill сеанс перечитывается по ID и сверяется `(InfobaseId, AppID,
  StartedAt)` со снимком; несовпадение → пропуск и ждать следующий цикл; «session not found» → успешный kill.
  **Аудит только по реальному исходу:** строка `AuditLog` пишется **тогда и только тогда**, когда
  `KillSessionResult` = `Killed` **или** `AlreadyGone`. При недоступном RAS / ошибке `rac.exe` (оба флага
  `false`) **ничего не завершается и ничего не аудируется**. Автоматический `KillEnforcer` (reason
  `LimitExceeded`, initiator `System`) логирует-и-пропускает; ручной `POST /api/v1/sessions/{id}/kill`
  (reason `ManualByAdmin`) возвращает ошибку вместо ложного `204` — `409 SESSION_STALE` при несовпадении
  дескриптора, `502 CLUSTER_UNAVAILABLE` при недоступном RAS. `404` означает, что сеанса не было в снимке.
- **Статус публикации:** Hangfire-джоба обновляет статус публикаций каждые 5 минут (троттл до
  `Drift.IntervalMinutes`, **read-only**); `POST /api/v1/publications/{id}/check` обновляет один on-demand.
  Авто-коррекции и enforcement желаемого состояния нет — (пере)публикация через `webinst`
  (`POST /publications/{id}/publish`) и смена платформы через `web.config`
  (`POST /publications/{id}/change-platform`) — явные аудируемые действия оператора.
- **Топология развёртывания:** single-node — бэкенд, IIS, MSSQL и кластер 1С на одном Windows Server
  хосте. Нет remoting, WinRM, кросс-хост адаптеров. Изменение этого предположения требует re-review каждого
  инфраструктурного адаптера (см. ADR-28, ADR-33).

## ADR-45. UX подтверждения опасных действий — дифференцированный барьер вместо type-to-confirm

**Контекст.** Исторически четыре диалога требовали **ручного ввода токена** для подтверждения
(type-to-confirm): публикация и снятие публикации (`PublishPublicationDialog`,
`UnpublishPublicationDialog` — ввод `siteName + virtualPath`), удаление базы и удаление клиента
(`DeleteInfobaseDialog`, `DeleteTenantDialog` — ввод имени). Паттерн задумывался как защита от
случайного разрушительного клика, но для **обратимых** действий он избыточен и тормозит штатную
работу оператора, а единый ручной ввод не различает обратимость действий.

**Решение.** Барьер дифференцируется по **обратимости** действия:

- **Обратимые** (публикация / снятие публикации — результат восстановим переопубликацией) →
  простое подтверждение «да/нет», ручной ввод токена **убран**.
- **Необратимые** (удаление базы, удаление клиента — каскадное, восстановления нет) →
  подтверждение «да/нет» с **явным предупреждением** о необратимости; ручной ввод убран, но
  **диалог-барьер сохранён** (нельзя удалить одним кликом).

Контракт API не меняется: `confirm: true` отправляется как раньше — это правка только UX-слоя
(frontend + i18n + тесты диалогов). Снятие намеренного type-to-confirm барьера на части действий
зафиксировано здесь явно (см. инвариант «не снимать барьеры молча»). Чекбокс «снять публикацию из
IIS» в диалоге удаления базы (MLC-113) сохраняется.

**Исключение — завершение сеанса (`KillSessionDialog`).** Принудительное завершение живого сеанса
1С отключает работающего пользователя (возможна потеря несохранённой работы) — это самое «живое»
из деструктивных действий, опаснее удаления простаивающей записи. Поэтому здесь type-to-confirm
**сохранён намеренно** (ввод `appId` сеанса для разблокировки кнопки) — усиленный барьер сверх
минимума «AlertDialog + предупреждение + фокус на Отмене». Это осознанное исключение из снятия
ручного ввода выше, а не недосмотр (зафиксировано при ревизии диалогов MLC-205).

## ADR-46. Движок таблиц — `@tanstack/react-table` для 6 списков управления; perf-виджеты на shadcn Table

**Контекст.** Списки фронтенда рендерились прямым проходом по строкам над shadcn `Table` без
табличного движка: серверная пагинация, точечная клиентская сортировка (сеансы) и фильтры в URL
писались вручную в каждой странице. `@tanstack/react-table` числился в зависимостях (`^8.21.3`), но
в коде не использовался. По мере роста списков появилась потребность в общих UX-фичах поверх таблиц
(видимость колонок, плотность строк, шаримый link на отфильтрованный вид) — дублировать их в каждой
странице дорого и расходится в деталях.

**Решение.** Общий компонент `DataTable` на `@tanstack/react-table` (`useReactTable`) — единый движок
для **6 списков управления**: Клиенты, Базы, инфобазы в карточке клиента (TenantDetail), Сеансы,
Аудит, Пользователи. `DataTable` рендерит через **существующие** shadcn-примитивы (`Table/TableHeader/
TableBody/TableRow/TableCell`) — примитивы не заменяются. Три общие фичи в тулбаре таблицы:

- **Видимость колонок** — dropdown с чекбоксами по `column.getCanHide()`.
- **Density-toggle** (комфортно/компактно) — единый хук `useTableDensity`, значение персистится в
  localStorage (`mlc-table-density`, дефолт `comfortable`); влияет на вертикальные паддинги.
- **URL-сериализация фильтров колонок** — хук `useUrlTableFilters` поверх `useSearchParams`
  (`?f_<colId>=…`), чтобы отфильтрованный вид шарился ссылкой; чужие параметры не трогаются.

Поддержаны клиентская и серверная пагинация (`getPaginationRowModel` / `manualPagination`) и
сортировка (`getSortedRowModel`). **Живые виджеты раздела `/performance` остаются на shadcn `Table`**:
это авто-обновляемые ридауты без фильтров/пагинации с кастомной отрисовкой — конверсия дала бы churn
и риск регрессии без выгоды от трёх фич.

**Инвариант сохранён:** статусы в колонках рендерятся **только через `StatusBadge`**.

**Последствия.** Новые списки управления строятся на `DataTable`; пилоты перевода — Клиенты
(серверная пагинация) и Сеансы (клиентская сортировка/пагинация) в MLC-144a. Компонент и хуки —
`frontend/src/components/ui/data-table/`.

## ADR-47. Управление службой RAS из панели — обнаружение, регистрация и перерегистрация под платформу/порт (single-host)

**Контекст.** `ras.exe` — серверная часть удалённого администрирования кластера 1С; единственный
адаптер кластера `rac.exe` (ADR-3.3/ADR-16) говорит с **локальным** `ras.exe` по TCP (дефолт
`localhost:1545`). Если на хосте `ras.exe` **не зарегистрирован службой Windows** и не запущен,
панель лишь видит «RAS недоступна» (`RasHealthProbingService`) и сделать ничего не может — оператор
регистрирует службу вручную из командной строки. При **обновлении платформы 1С** путь к `ras.exe`
меняется (новая папка версии), и существующая служба указывает на устаревший бинарь — снова ручная
операция. Это типичная «забытая» настройка при разворачивании на новом сервере и при апгрейде
платформы. ROADMAP отмечал регистрацию `ras.exe` службой как **ортогональную** серверную сторону
(не Strategy B). Прав хватает: служба панели работает от SYSTEM / admin-эквивалента — штатный
профиль (ADR-44, low-priv отклонён); установщик уже создаёт службы Windows через `sc.exe`
(ADR-31, MLC-107/116) — прецедент управления службами в проекте есть.

**Решение.** Панель управляет жизненным циклом **локальной** службы RAS — в границах single-host
(ADR-28: только свой узел, хост фиксирован `localhost`). Новый порт `IRasServiceManager`
(Infrastructure-адаптер) поверх `sc.exe` (тот же приём, что в установщике; OEM-декодирование вывода
как у `rac.exe`/`iisreset.exe`):

- **Обнаружение** — служба ищется по **`binPath`, содержащему `ras.exe`** (имя службы у операторов
  не стандартизировано), не по имени; читается состояние (running/stopped), версия платформы из пути
  и порт из аргументов. `ras.exe` обнаруживается в тех же bin-каталогах, что и `rac.exe`
  (`OneCInstallRoots` / `PlatformVersionDiscovery`).
- **Диагностика — 4 состояния:** (1) служба есть, запущена, `binPath` на актуальной платформе и порт
  совпадает с `OneC.RAS.Endpoint` → ОК; (2) службы нет → предложить регистрацию; (3) служба есть, но
  `binPath` на устаревшей версии **или** порт расходится с endpoint → предложить перерегистрацию;
  (4) служба есть, но остановлена → предложить запуск.
- **Регистрация** — `sc create` с `binPath` на `ras.exe` выбранной платформы (`PlatformPicker` /
  `OneC.DefaultPlatformVersion`), порт из `OneC.RAS.Endpoint` (единый источник правды), тип `auto`,
  локальный агент кластера (ragent) как цель — `localhost:<OneC.RAS.AgentPort>` (порт по умолчанию
  `1540`, MLC-194). **Перерегистрация** — stop → `sc config` (новый
  `binPath`/порт) → start, идемпотентно. **Запуск** — `sc start` остановленной службы.

API: `GET /api/v1/ras-service/status` (диагностика, Admin), `POST /api/v1/ras-service/{register|update|start}`
(Admin, аудит). Parity: Zod-схемы фронтенда отражают контракт в том же PR.

**UX.** Дом действия — секция «Подключение к 1С/RAS» в `/settings` (Admin-only), рядом с портом и
выбором платформы. Дашборд только **сигналит** недоступность RAS (`RasHealthProbingService`) со
ссылкой в настройки — кнопку на дашборд не выносим: раздел видят и Viewer, а действие меняет систему.
Диалог-подтверждения показывает **и** человекочитаемое описание (служба / порт / платформа), **и**
точную команду `sc …` (прозрачность + воспроизводимость оператором вручную). Барьер — простой
«да/нет» (действие обратимо, стиль ADR-45).

**Аудит.** Новая **600-серия** (frozen-int): `RasServiceRegistered=600`, `RasServiceUpdated=601`
(перерегистрация под платформу/порт), `RasServiceStarted=602`; 603+ — резерв. Server-scope
(`TenantId` не пишется); секреты в описание не кладутся.

**Последствия / границы.**

- `rac.exe` остаётся **единственным** адаптером кластера (ADR-16/3.3) — управление службой
  **ортогонально**: протокол RAS не реверсится, долгоживущий сокет не вводится (Strategy B остаётся
  отклонённой — ADR-3.3).
- Single-host (ADR-28) сохранён: одна локальная служба RAS, хост `localhost`; служба RAS слушает
  только loopback → firewall-правило не требуется.
- **Удаление и останов** службы RAS из панели — **вне объёма** (только register/update/start); резерв
  аудита 603+ под будущие действия.

**Статус:** принят (MLC-159, релиз `0.5.4-beta`).

**Update (MLC-162, релиз `0.5.5-beta`).** Обнаружение службы переведено с перебора
`sc query state= all` + `sc qc` на **чтение реестра** (`HKLM\SYSTEM\CurrentControlSet\Services\*\
ImagePath`, фильтр по `ras.exe`) + состояние через `ServiceController`: вызов `sc.exe` из-под
`Process.ArgumentList` для `state=`-энумерации на реальной ОС не отрабатывал (служба не находилась
→ ложное «не зарегистрирована»), а перебор сотен служб через `sc qc` был дорог. Реестровое чтение —
один проход, без спавнов. Принцип неизменен: служба ищется по `binPath`, содержащему `ras.exe`.
Команды-мутации (`sc create/config/start/stop`) строятся под нестандартный парсер `sc.exe` и
проверяются против реального `sc.exe`. Урок: адаптеры внешних CLI верифицируются на реальном
бинаре, не только на моках.

**Update (MLC-194, трек `v1.1.1`).** Порт локального агента кластера (ragent) — цель `ras.exe` —
вынесен из захардкоженной константы `localhost:1540` в настройку панели `OneC.RAS.AgentPort`
(Number, дефолт `1540`, диапазон 1024–65535, не секрет), читается тем же путём, что порт службы
RAS (`OneC.RAS.Endpoint`). Хост остаётся фиксированным `localhost` (single-host, ADR-28) —
настраивается **только порт**; пустое/некорректное значение → дефолт `1540`. На апгрейде пустую
существующую строку ключа лечит сидер (`HealRasAgentPortAsync`, паттерн MLC-117). Причина: на
нестандартном порту агента 1С прежняя константа собирала неверный адрес → служба RAS не цеплялась
к кластеру. Объём — только агент-порт → настройка; установщик `.iss` и сбор прочих параметров 1С
не затронуты (отложено).

## ADR-48. Источник истины потребления лицензии — факт `rac --licenses` (не эвристика по `app-id`)

**Контекст.** Главная функция панели — мониторинг и контроль лицензий; договорённость с клиентами:
**сеанс, потребляющий клиентскую лицензию 1С, оплачивается как место**. До этого решения
`ConsumesLicense` выводился **эвристикой** — членством `app-id` сеанса в white-list
`OneC.LicenseConsumingAppIds` (ADR-3.3, дефолт `{1CV8,1CV8C,WebClient,Designer,COMConnection}`).
Эвристика ошибается на неоднозначных типах подключений (`WSConnection`/`HTTPServiceConnection`/
`COMConnection`): держат ли они лицензию — зависит от конфигурации, и whitelist даёт неверный счёт
(риск недо-/переучёта мест → расхождение с биллингом). `rac session list --licenses` сообщает
**факт** per-session: сеанс, получивший клиентскую лицензию, присутствует в выводе с блоком лицензии
(`license-type`, `series`, `short-presentation`); сеанс без лицензии в вывод **не попадает вовсе**.
Подтверждено на стенде 8.5: при 4 сеансах (`3×1CV8C` + `BackgroundJob`) `--licenses` вернул ровно
3 строки `1CV8C` (`license-type: HASP`), `BackgroundJob` отсутствует.

**Решение.**

- **`ConsumesLicense` = факт `rac --licenses`.** Сеанс потребляет лицензию ⟺ его `session`-GUID
  присутствует в выводе `session list --licenses` (с непустым блоком лицензии — пояс-плюс-подтяжки
  к правилу членства). White-list `app-id` и настройка `OneC.LicenseConsumingAppIds` **удаляются**
  из механизма подсчёта. Счётчик «лицензия/не лицензия» в «Сеансах» — на том же факте.
- **`app-id` остаётся атрибутом сеанса** (колонка и фильтр отображения в «Сеансах»), но в подсчёте
  лицензий **не участвует**.
- **Два вызова на холодном тире.** `--licenses` — отдельная проекция rac **без поля `infobase`**
  (а оно нужно для привязки сеанса к тенанту). Поэтому холодный цикл (`ReconciliationJob` /
  `ColdTierPollingService`, каданс `Polling.ColdIntervalSeconds`, дефолт 15 с) делает **два** вызова:
  обычный `session list` (даёт `infobase`/`app-id`/`started-at` + весь инвентарь сеансов) и
  `session list --licenses` (множество `session`-GUID с лицензией), и сшивает их **по `session`-GUID**.
  Горячий тир (`HotTierPollingService`, дефолт 4 с) делает **один** вызов (обычный список) и берёт
  классификацию лицензий из последнего холодного снимка.
- **Инвариант «grace ≥ холодного интервала».** Сеанс моложе `Enforcement.KillGraceSeconds` (дефолт 15 с)
  не завершается (KillEnforcer). Так как факт обновляется каждый холодный цикл (15 с), любой сеанс,
  доросший до права на завершение (≥ grace), уже классифицирован фактом. Поэтому **каждое реальное
  завершение опирается на факт rac**, никогда — на иной источник. `KillGraceSeconds` нельзя опустить
  ниже `ColdIntervalSeconds` (guard) — иначе откроется окно завершения по неподтверждённому факту.
- **Свежий сеанс (ещё не в факт-снимке).** В подсчёт лимита не входит; в UI помечается
  **«определяется…»**, не «не потребляет»; в кандидаты на завершение не попадает (grace держит).
- **Недоступность `--licenses` (сбой/таймаут/права).** Факта нет → enforcement **приостанавливается**
  (сеансы не завершаются), панель честно показывает «данные о лицензиях недоступны», а не ложный 0.
  Эвристического отката на `app-id` **нет** — принцип «не рубить вслепую». Отчёт «Использование
  лицензий» в окне сбоя **не сэмплирует** (холодный цикл пропускает запись бакета, MLC-168) — честный
  пробел в истории, а не ложный 0 потребления (все сеансы `Pending` дали бы `consumed=0`).

**Последствия.** Потребление лицензии = контрактная истина 1С, точная и на неоднозначных типах
подключений. Цена: **+1 спавн `rac.exe` на холодный цикл** (≈ +4/мин при дефолте 15 с) — бюджет
ADR-3.3 обновлён («холодный = обычный список + `--licenses`»); при сбое rac лимиты на время не
принуждаются (сознательный размен на «не завершать сеанс без подтверждённого факта»). Альтернатива
«`--licenses` и на горячем тире» (факт каждые 4 с) **отклонена** — удваивает спавны на горячем 4-сек
пути ради косметической свежести счётчика для сеансов < 15 с, которых grace всё равно не даёт
завершить. ADR-3.3 (маппинг сеанса, бюджет спавнов) приведён к этому факту.

**Статус:** принят (реализация — трек «Релиз 0.6.0-beta», MLC-166).

## ADR-49. Идентичность службы — виртуальная учётная запись `NT SERVICE\MitLicenseCenter`; Windows-аутентификация как стандарт

**Контекст.** В режиме B (ADR-31) служба `MitLicenseCenter` работала под **LocalSystem** и
подключалась к SQL по **SQL-логину и паролю**, который установщик писал **открытым текстом** в
`appsettings.Production.json` (под NTFS ACL, SECURITY.md §9 п.7). Этот секрет — учётка уровня
`sysadmin` — оседал на диске и **попадал в бэкап конфига**; на ручном апгрейде оператор каждый раз
вводил креды. Поскольку SQL по топологии **всегда локальный** (ADR-28), доверенное подключение
Windows (`Trusted_Connection`) доступно на **любом** хосте фермы — и в домене, и в рабочей группе —
а значит SQL-пароль на диске можно устранить полностью.

**Решение.**

- **Служба работает под виртуальной учётной записью `NT SERVICE\MitLicenseCenter`** (по умолчанию)
  либо под **именованной Windows-учёткой / gMSA** (доменный сценарий, выбор в мастере). У виртуальной
  учётки **нет пароля** (им управляет Windows); `sc create … obj= "NT SERVICE\MitLicenseCenter"`
  одним вызовом регистрирует службу, материализует учётку + её SID и получает `SeServiceLogonRight`.
- **Подключение к локальному SQL — `Trusted_Connection=True`.** SQL-логин и пароль в
  `appsettings.Production.json` **больше не пишутся** (`BuildConnString` — единственная ветка, обе
  строки `Default`/`Hangfire` доверенные). Это устраняет единственный секрет SQL на диске.
- **SQL-аутентификация (прежний режим B / LocalSystem + SQL-логин) убрана** из мастера установки.
- **Установщик авто-создаёт SQL-логин** для учётки (`CREATE LOGIN [NT SERVICE\MitLicenseCenter]
  FROM WINDOWS` + членство `sysadmin`, идемпотентно) — раньше принципал создавал оператор заранее,
  но для виртуальной учётки это невозможно (её SID появляется только при регистрации службы).
  **Создаётся всегда Windows-логин** (`FROM WINDOWS`); SQL-логинов с паролем не создаётся ни для
  службы, ни иначе. Назначить роль `sysadmin` может только тот, кто сам `sysadmin` (правило SQL),
  поэтому провижининг идёт под sysadmin-личностью — **на выбор (MLC-171):**
    - **по умолчанию — Integrated Security установщика-админа** (без ввода пароля; работает и на
      инстансах «только Windows-аутентификация»);
    - **запасной — введённый SQL-логин с ролью `sysadmin`** (`sa` или иной; требует mixed-mode
      инстанса). Этот пароль используется **только разово** для создания Windows-логина, **не
      пишется в конфиг и нигде не сохраняется** (тем же приёмом, что тест подключения — временный
      `.ps1`, не командная строка) — инвариант «секрета SQL на диске нет» сохраняется.
  Предусловие смягчается до «**установщику доступна какая-либо `sysadmin`-личность**» (Windows
  или SQL-логин). На провале провижининга — `MsgBox(mbCriticalError)` + `RaiseException` (откат):
  иначе служба упадёт на первом старте с 18456 (молчаливый провал, который закрывал MLC-107).
- **Учётка добавляется в локальную группу Администраторов** (по well-known SID `S-1-5-32-544`,
  `Add-LocalGroupMember`, локаленезависимо) — нужно для IIS/`iisreset` (ADR-44). Best-effort:
  без него панель и SQL работают, деградируют только IIS-функции.
- **NTFS ACL** на каталог key ring (`Modify`) и `appsettings.Production.json` (`Read`) гранятся
  **этой учётке** (`icacls` по имени `NT SERVICE\…`, не локализуется) — **после** `sc create`,
  иначе имя не резолвится и Data Protection не прочитает ключи на первом старте.
- **Порядок `ssPostInstall`:** запись конфига → `initial-admin.secret` → регистрация службы
  (`sc create`/`config`) → провижининг SQL-логина (**жёсткий fail**) → добавление в Администраторы
  (best-effort) → ACL (грант учётке) → `sc start` **последним** (fail-fast bootstrap коннектится уже
  как эта учётка). На апгрейде шаги идемпотентны.
- **Уровень привилегий НЕ снижается** (ADR-44 остаётся в силе): `sysadmin` в SQL и admin-эквивалент
  на хосте сохраняются — меняется только **идентичность** службы, не объём прав.
- **Миграция уже установленных в режиме B** — **ручная документированная процедура** (OPERATIONS),
  установщик их автоматически не переводит (смена идентичности живой службы + прав в SQL рискованна).
- **gMSA-путь** (учётка без пароля) включается **явным чекбоксом** на странице именной учётки —
  не выводится из пустого пароля (иначе тихий провал `sc create` 1057).

**Последствия.** SQL-пароль убран с диска и из бэкапа конфига — **остаточный риск №7 SECURITY.md §9
закрыт**. Предусловие: установщику доступна `sysadmin`-личность — либо запускающий админ сам
`sysadmin` (Integrated Security, дефолт), либо вводится SQL-`sysadmin`-логин (MLC-171; раньше
DDL-прав установщику не требовалось вовсе); смягчение — ранний пробник `IS_SRVROLEMEMBER('sysadmin')`
+ внятное сообщение. Бэкенд **без изменений** (потребляет строку подключения как есть; EF и Hangfire прозрачны
к `Trusted_Connection`). Решение — следствие ADR-28 (локальный SQL) и опирается на ADR-8 (ACL key ring
распространяется на учётку). **Отвергнуто:** авто-миграция режима B (риск для живой инсталляции);
сохранение LocalSystem + SQL-логин; определение gMSA по пустому паролю; снижение привилегий (ADR-44).
ADR-31 и ADR-44 приводятся к этому факту в треке реализации (статус «изменён»).

**Статус:** принят (MLC-170 — реализация влита; провижининг-креды на выбор Integrated Security /
SQL-`sysadmin`-логин — MLC-171; **явный радио-выбор личности провижининга в мастере — MLC-175**,
UX-детали в ADR-31).

## ADR-50. Канал обновлений — проверка через GitHub Releases `latest`; уведомление, а не авто-установка

**Контекст.** Панель устанавливается локально на узел клиента (ADR-30/ADR-31, single-host ADR-28).
Релизы публикуются на GitHub. Оператору нужно ненавязчиво узнавать о новой версии, не отслеживая
репозиторий вручную. При этом панель — LAN-инструмент без CD-конвейера (ADR-14): автоматический
апгрейд боевого узла рискован, а сертификата подписи кода в проекте нет (ADR-31 — подпись отложена).

**Решение.**

- **Источник версии — `GET https://api.github.com/repos/{owner}/{repo}/releases/latest`** публичного
  репозитория, анонимно (без токена). Репозиторий — настройка `Updates.Repository` (default
  `sdbsm/MitLicenseCenter`). Сверка тега релиза с текущей версией панели (informational-версия сборки,
  как в `/health`) — чистый semver-компаратор `AppVersion`/`UpdateComparison` (Domain/Updates): сравнение
  по тройке major.minor.patch, при равной тройке release > prerelease; суффикс предрелиза сам по себе
  не ранжируется. Непарсимая сторона → «обновление недоступно» (баннер не показываем).
- **Анти-коррупционный адаптер `IGitHubReleaseClient`** (Application/Updates) изолирует контракт GitHub;
  реализация (Infrastructure) — **первый типизированный `HttpClient` в проекте** (`AddHttpClient`,
  таймаут 10 с, обязательный `User-Agent` — без него GitHub отвечает 403). Любой сбой (сеть, HTTP-не-2xx,
  rate-limit, битый JSON) ловится внутри и наружу отдаётся `null` = «проверка недоступна».
- **Ленивая проверка под `IMemoryCache`** (паттерн dashboard-summary), **без фонового hosted-service**:
  результат `/updates/status` кэшируется на `Updates.CheckIntervalHours` (default 6 ч; при недоступной
  проверке — короткий TTL 5 мин). Рубильник `Updates.Enabled` (1/0): 0 → в GitHub не ходим.
- **Эндпоинты.** `GET /updates/status` (Viewer) — статус для баннера; `POST /updates/check-now` (Admin) —
  сброс кэша + свежая проверка. Контракт `UpdateStatusResponse` (`CurrentVersion` всегда заполнен;
  `LatestVersion/ReleaseUrl/DownloadUrl` опускаются при `CheckAvailable=false`).
- **UX = уведомление, не действие.** Глобальный баннер «Доступна версия X.Y.Z» виден всем ролям; ссылка
  «Открыть релиз» (`html_url`) — всем; «Скачать установщик» (`browser_download_url` первого ассета `.exe`)
  и «Проверить сейчас» — только Admin. **Установщик запускается вручную админом под UAC** — бэкенд только
  отдаёт URL, ничего не скачивает и не запускает.

**Последствия.** Целостность скачанного установщика обеспечивается **HTTPS GitHub** (сертификата подписи
нет — ADR-31). Без хеш-проверки в панели, без авто-скачивания/авто-запуска, без рендера changelog (только
ссылка на релиз). **Аудит проверки обновлений не пишем** (enum `AuditActionType` заморожен — проверка
read-only и не меняет систему). Ортогонально ADR-28 (single-host): панель проверяет версию **самой себя**,
а не управляемых узлов. Чтобы панель увидела релиз, он должен иметь тег `vX.Y.Z` и прикреплённый ассет
`*.exe` (см. `docs/DEVELOPMENT.md`). Отвергнуто: фоновый поллинг hosted-service'ом (лишний для LAN-узла —
ленивого кэша достаточно); авто-апгрейд; проверка подписи/хеша в v1.

**Статус:** подтверждён.

## ADR-51. Релизный поезд — батч-релизы; мердж в `main` отвязан от выпуска

**Контекст.** Ранняя практика выпускала новую версию почти на каждую закрытую задачу (история тегов
`v0.4.0-beta`…`v1.0.0`) — много мелких релизов, и живой прогон на стенде требовался после каждого.
Владелец хочет копить изменения и выпускать осознанным батчем.

**Решение.** Мердж в `main` и выпуск релиза **разъединены**. Задачи вливаются в `main` без бампа
`<Version>` (`backend/Directory.Build.props`) и без тега — `main` всегда «следующий релиз в работе»
(vNext), копит изменения. Релиз режется отдельным шагом куратора **по готовности** (когда накопился
стоящий батч и стенд зелёный), без календарного ритма: поднять `<Version>` один раз → тег `vX.Y.Z`
(`release.yml` собирает `Setup.exe` и публикует GitHub Release) → **обычный релиз** = клиенты видят
баннер «доступна версия» (ADR-50). Semver: батч фич → minor, чистые фиксы → patch. Состав релиза
выводится из git (`git log <предыдущий-тег>..main`) и индекса закрытых задач
(`PROJECT_BACKLOG_ARCHIVE.md`); отдельный changelog в `docs/` не ведётся (present-tense канон).

**Обкатка.** Pre-release — клапан стейджинга: накопленную сборку можно опубликовать как pre-release
(`vX.Y.Z-rc1`), проверить на стенде и лишь затем выпустить обычный релиз. Проверка обновлений
игнорирует pre-release (`/releases/latest`, ADR-50) — клиенты pre-release не видят.

**Последствия.** Снят навык «задача = релиз»: закрытие задачи версию **не** бампает. Живой прогон на
стенде — один раз на релиз, не на задачу. Номер версии в подвале панели между релизами совпадает с
последним выпущенным (на `main` при желании можно держать суффикс `-dev`, но не обязательно). Релизный
конвейер (`release.yml`, ADR-14) и контракт обновлений (ADR-50) не меняются — меняется только каданс
и кто решает «пора».

**Статус:** принят (2026-06-17, решение владельца; каданс — по готовности).

## ADR-52. Время — хранение/транспорт в UTC; планировщик ночных джоб — по местному поясу хоста

**Контекст.** Backend хранит и транспортирует каждый `DateTime` в UTC (`TimeProvider.GetUtcNow()` при
записи, явный суффикс `Z` на проводе — ADR-23). Это осознанный выбор, а не случайность: UTC даёт
**единый инстант без неоднозначностей** (нет «двойного» или «несуществующего» часа при переходах
зима/лето), что критично для меток аудита, бакетов телеметрии и сравнения времён; **локализация —
ответственность фронтенда** (браузер знает пояс пользователя и форматирует UTC в местное время,
ADR-23). Хранить локальное время в БД означало бы прятать в каждой метке скрытый пояс и ломаться на
DST — поэтому хранилище и транспорт остаются UTC.

Отдельный вопрос — **планировщик** рекуррентных джоб. Hangfire по умолчанию считает cron-выражения в
**UTC** (явный `TimeZone` нигде не задавался). «Ночные» housekeeping-джобы задумывались как ночное окно
по часам сервера, но на хосте UTC+3 фактически уезжали в утро: `database-size-collection` `0 2 * * *`
= 02:00 UTC = 05:00 МСК, ретеншены 03:00–04:00 UTC = 06:00–07:00 МСК. Это противоречит замыслу
(«ночной housekeeping») и поведению прочих планировщиков на том же Windows-сервере (Task Scheduler,
SQL Agent — местное время).

**Решение.** Планировщик ночных рекуррентных джоб привязан к **местному поясу хоста**
(`TimeZoneInfo.Local`) через единый общий объект `NightlyJobSchedule.LocalTimeZoneOptions`
(`RecurringJobOptions { TimeZone = TimeZoneInfo.Local }`), переданный в перегрузку
`RecurringJob.AddOrUpdate<T>(id, methodCall, cron, options)` для всех шести суточных джоб:
`database-size-collection` (`0 2`), `audit-retention` (`0 3`), `backup-retention` (`15 3`),
`license-usage-retention` (`30 3`), `perf-recording-retention` (`45 3`), `database-size-retention`
(`0 4`). Указанные времена — **по часам хоста**; их относительные смещения (сбор раньше ретеншенов;
ретеншены разнесены, чтобы не пересекаться) сохранены. `publication-status-refresh` (`*/5 * * * *`) к
поясу нечувствителен — оставлен на дефолте.

Обоснование привязки к местному поясу: single-host (ADR-28) → пояс хоста = пояс оператора; совпадает с
поведением прочих планировщиков на Windows-сервере (Task Scheduler, SQL Agent); РФ без переходов DST →
риск нулевой; для DST-поясов раз в год один прогон может сдвинуться или повториться — безвредно для
**идемпотентных** ночных housekeeping-джоб.

**ВАЖНО:** меняется только *момент срабатывания планировщика*. **Хранение и транспорт времени остаются
UTC** — ADR-23 в силе; `TimeZoneInfo.Local` здесь не затрагивает метки данных в БД и на проводе.

**Последствия.** «Ночные» джобы реально идут ночью по местному времени сервера (на стенде — в
Hangfire-дашборде «Next execution» приходится на местную ночь). Выбор пояса вынесен в один общий
объект (без дубля `RecurringJobOptions` на каждой регистрации) и закреплён юнит-тестом
(`NightlyJobScheduleTests` — `LocalTimeZoneOptions.TimeZone == TimeZoneInfo.Local`), чтобы откат к
дефолтному UTC не прошёл молча. UTC-инвариант хранения (тест `UtcDateTimeConventionTests`, ADR-23) не
тронут.

**Статус:** подтверждён (2026-06-17, MLC-191; решение владельца — роадмап #1).

## ADR-53. Навигация и информационная архитектура single-host — меню 7 разделов; «Сеансы» = дом темы лицензий; «Отчёты» растворены

**Контекст.** Контроль лицензий — коммерческое ядро продукта (хостер продаёт сеансы 1С) — был
**размазан по трём разделам меню**: «Сеансы» (живой снимок потребления), «Клиенты» (лимиты и проценты
по клиентам) и «Отчёты» (история использования за период + размер баз). Оператору, чтобы ответить на
вопрос «сколько лицензий съедено и кем», приходилось прыгать между разделами. Сам раздел «Отчёты» был
сборным: license-часть (использование лицензий за период) и size-часть (размер баз за период) логически
тяготели к разным домам. Меню насчитывало 8 разделов; «Отчёты» как отдельная навигационная сущность
дублировал тему, у которой уже есть естественный дом.

**Решение.** Информационная архитектура single-host (ADR-28) приведена к **7 разделам меню**, а тема
лицензий собрана в один дом:

- **«Сеансы» = дом темы лицензий** — коммерческая единица панели, с тремя видами (переключатель
  `?view=` в URL): **«По клиентам»** (агрегат потребления, дефолт), **«Живые сеансы»** (текущий
  снимок) и **«Использование за период»** (`view=usage`) — это license-часть бывших «Отчётов»
  (`SessionsUsageView`).
- **Size-часть «Отчётов»** растворена во вкладку **«Размер баз»** дома «Базы» (`InfobasesSizeTab`).
- **Раздел «Отчёты» удалён** как навигационная сущность: нет пункта меню, нет маршрута `/reports`.
  KPI-карточки «Обзора» «Использовано/Свободно лицензий» ведут в `/sessions?view=usage`.
- Итоговое меню — **7 разделов**: Обзор · [Мониторинг] Сеансы, Быстродействие · [Управление] Клиенты,
  Базы · [Система] Аудит, Пользователи (Admin), Параметры (Admin).

**ВАЖНО — растворение чисто навигационное.** Backend-эндпоинты `/reports/license-usage[/:tenantId]`
и `/reports/database-size[/:tenantId]`, их Zod-контракты и building-blocks `features/reports/*`
(графики/сводки/хуки/`reportsUrlState`) **сохранены** и переиспользуются «Сеансами», «Базами» и
«Обзором». Parity BE↔FE не затронут — менялось только FE-потребление (какой дом монтирует компонент).

**Последствия.** Тема лицензий перестала быть размазанной: оператор отвечает на лицензионные вопросы в
одном доме «Сеансы». Меню короче на один пункт (8→7). `features/reports/` остаётся библиотекой
переиспользуемых компонентов без собственного маршрута. Удаление навигации не трогает данные и API —
откат к отдельному разделу при необходимости тривиален (вернуть маршрут + пункт меню поверх живых
эндпоинтов).

**Статус:** принят (2026-06-18, MLC-196a/b/c — Фаза 1 редизайн-трека; license-вид → «Сеансы» (196b),
size-вкладка → «Базы» (196b), удаление раздела «Отчёты» (196c)).

## ADR-54. Экран «Сервер» — единый дом управления службами узла и мониторинга обслуживания

**Контекст.** Управление узлом размазано и неполно: управление IIS живёт вкладкой внутри «Базы»,
регистрация службы RAS — в «Параметрах», а целостного статуса служб стека (сервер 1С, SQL) и
мониторинга штатного обслуживания SQL (резервные копии, планы обслуживания) нет вовсе. Оператору
негде одним взглядом понять «здоров ли узел» и управлять стеком из одного места.

**Решение.** Вводится новый раздел меню **«Сервер»** (группа [Мониторинг], меню 7→8) — единый дом
эксплуатации узла с тремя вкладками:

- **«Службы»** — статус всех служб стека (сервер 1С / `ragent`, при нескольких версиях платформы —
  списком; RAS; SQL Server; IIS) + жизненный цикл (старт/стоп/рестарт) для **сервера 1С, RAS, IIS**.
  **SQL — только наблюдение** (управление сознательно не даётся: панель и Hangfire-хранилище живут в
  том же инстансе SQL, ADR-28 — остановка SQL = самоликвидация панели). Здесь же: рабочие процессы 1С
  (`rphost`) — список + рестарт через RAC; расписание авто-рестартов сервера 1С. Сверху — общий
  индикатор здоровья узла, его же плашка выносится на «Обзор» со ссылкой сюда.
- **«IIS»** — детальное управление (пулы / сайты / `iisreset`) **переезжает сюда** из вкладки «IIS»
  дома «Базы»; прежняя вкладка удаляется.
- **«Обслуживание»** — **только чтение**: свежесть резервных копий (`msdb.dbo.backupset`) и планы
  обслуживания (`msdb.dbo.sysmaintplan_*` + задания SQL Agent) **с детализацией по задачам**
  (видно, что именно упало: проверка целостности / бэкап / реиндекс) и разделением «по расписанию /
  по запросу» (ручные под-планы не алертятся как просроченные).

Доступ: **Viewer видит статусы, Admin управляет** (паттерн «Параметров»/«Пользователей»).
Регистрация/обновление RAS пока **остаётся в «Параметрах»** — переезд (эксплуатация vs настройка)
решается отдельно позже.

**Последствия.** Меню 7→8; «Базы» теряет вкладку IIS. Капкан самоликвидации устранён по дизайну
(SQL без управления). Мониторинг обслуживания — чтение `msdb` тем же доверенным подключением, **без
новых таблиц**. Рестарт `rphost` — через RAC, контракт уточняется на стенде (исследовательская часть).
Существующий раздел «Резервные копии» (on-demand `COPY_ONLY`, ADR-27) **не трогается** — это «сделать
бэкап руками», тогда как «Обслуживание» — «следить, что штатные бэкапы идут». Откат тривиален
(раздел аддитивен поверх живых эндпоинтов).

**Статус:** принят (постановка 2026-06-19; реализуется в `MLC-212..219`, релиз 1.1.3).

## ADR-55. Надёжность операций управления службами — команда + верификация состояния с таймаутом

**Контекст.** Старт/стоп/рестарт службы асинхронны по природе: служба приходит в целевое состояние
за секунды–минуты. Наивная реализация «отправили команду → отрапортовали успех» **врёт** (команда
могла не поднять службу), а долгая операция не должна валить панель ошибкой или подвешивать запрос.

**Решение.** Каждая мутирующая операция службы подчиняется контракту надёжности:

1. **Команда + верификация.** После команды (`sc`/`iisreset`/ServerManager) опрашиваем **фактическое**
   состояние службы (`ServiceController`) до целевого (запущена / остановлена) в пределах таймаута;
   возвращаем достоверный результат, а не факт отправки команды.
2. **Ограниченное ожидание.** Два РАЗНЫХ таймаута: (а) таймаут самой команды (`sc start/stop` 30 с,
   `iisreset` 90 с — команда возвращается мгновенно) и (б) **бюджет верификации целевого СОСТОЯНИЯ**
   службы. Для `WindowsServiceController` бюджет верификации — **300 с** (MLC-224): холодный старт/стоп
   кластера 1С (ragent + rmngr + rphost, освобождение портов, слив сеансов) идёт минутами, и тесный
   30-секундный бюджет давал ложную ошибку «не успела запуститься/остановиться» при фактически
   поднимающейся службе. Полл (`ServiceController`) возвращается сразу при достижении состояния, поэтому
   высокий потолок не замедляет нормальный случай. По истечении бюджета — управляемая ошибка (409 +
   понятный русский текст), не падение и не немое зависание.
3. **Идемпотентность.** Старт уже запущенной / стоп уже остановленной службы = успех (коды `sc`
   1056/1062 уже обрабатываются).
4. **Сериализация.** Мутации одной службы — под единым замком (обобщение `IIisResetConcurrencyGate`),
   параллельные операции не наезжают.
5. **Особый случай «панель рубит себя»** (стоп/рестарт IIS, на котором работает сама панель):
   HTTP-ответ уже не придёт — намерение пишется в аудит **до** исполнения (fire-and-forget), а UI
   заранее предупреждает «панель станет недоступна, проверьте через …».
6. **UI отражает три состояния:** идёт (с оценкой «может занять до N с») → успех (обновлённый статус)
   → ошибка (что именно и что делать).

Защита от ошибочного клика на разрушительных операциях — **диалог подтверждения да/нет** (без
type-to-confirm; решение владельца 2026-06-19).

**Последствия.** Операции честны (нет ложных «успехов»), долгие рестарты не валят панель. Полинг
состояния с таймаутом — **новый код** (в текущем нет паттерна «опросить до целевого состояния»).
Контракт переиспользуется всеми службами (1С/RAS/IIS); read-only мониторинг «Обслуживания»
надёжен за счёт таймаутов запросов и деградации проб.

**Статус:** принят (постановка 2026-06-19; реализуется в `MLC-212..`, релиз 1.1.3).

## ADR-56. Рестарт рабочего процесса 1С (`rphost`) — завершение ОС-процесса по `Pid` из whitelist

**Контекст.** Оператору нужен мягкий рестарт зависшего/деградировавшего рабочего процесса 1С
(`rphost`) из вкладки «Службы». Но у утилиты `rac` **нет** команды «restart process» (есть только
`session terminate`). `rphost` — не служба Windows, поэтому контроллер службы (`ServiceController`,
ADR-55) к нему неприменим. Завершение процесса разрушительно — рвёт активные сеансы на нём.

**Решение.** Рестарт `rphost` = **завершение ОС-процесса по `Pid`**, после чего кластер 1С
(`ragent`) сам авто-поднимает новый рабочий процесс (с другим `Pid`). Операция обставлена
максимальной страховкой:

1. **Whitelist.** Завершать можно только `Pid`, присутствующий в текущем `rac process list`
   (`IClusterClient.ListProcessesAsync`). `Pid` не из списка → `404` (нельзя завершить произвольный
   ОС-процесс узла).
2. **Guard от переиспользования `Pid`.** Перед завершением проверяем, что ОС-процесс с этим `Pid` —
   действительно `rphost` (имя процесса без расширения, `ILocalProcessTerminator.GetProcessName`).
   Имя не совпало (ОС переназначила номер другому процессу) → `409`, не убиваем. Процесса уже нет
   (имя `null`) → идемпотентный успех.
3. **Верификация исчезновения (дух ADR-55).** После завершения опрашиваем `rac process list`, пока
   старый `Pid` не исчезнет, в пределах таймаута (~30 с, `OneCProcessRestartOptions`, через
   `TimeProvider`). Исчез → успех (правдивый результат: кластер заменил процесс). Не исчез за
   таймаут → управляемая ошибка `409`, не `500` и не зависание.
4. **Идемпотентность.** Если `Pid` уже отсутствует (whitelist не нашёл / ОС-процесс ушёл) — успех
   (рестартить нечего).
5. **Confirm-гейт + роль.** Только **Admin**; обязателен серверный **confirm да/нет**
   (`409 PROCESS_CONFIRM_REQUIRED`), как у stop/restart сервера 1С (ADR-55); FE предупреждает, что
   операция разорвёт активные сеансы на процессе.

Граница ADR-20: завершение ОС-процесса инкапсулировано в `ILocalProcessTerminator` (Application) +
реализация поверх `System.Diagnostics.Process` (Infrastructure) — `Process` не течёт в Web/эндпоинт.
Логика рестарта (whitelist + guard + kill + верификация) — `IOneCProcessRestartService`; чистая
классификация исхода вынесена под юнит (`OneCProcessRestartPolicy`).

**Последствия.** Аудит-код `OneCProcessRestarted = 805` (enum-серия 800 заморожена — append). Эндпоинт
`POST /api/v1/server/onec/processes/restart`. Откат тривиален (кнопка/эндпоинт аддитивны поверх
готового списка `MLC-219`). Контракт надёжности отличается от ADR-55 источником истины (опрос
`rac process list` вместо `ServiceController`), но следует тому же духу: правдивый результат,
ограниченное ожидание, управляемая ошибка вместо зависания.

**Статус:** принят (`MLC-220`, релиз 1.1.3).

## ADR-57. Производительность = Наблюдение + Расследование (двухрежимный раздел)

**Контекст.** По методике 1С расследование проблем производительности — воронка из трёх стадий:
*что медленно* (для пользователя) → *где узкое место* (ресурсы хост/1С/SQL) → *почему медленно*
(доказательно, по технологическому журналу 1С). Сейчас раздел «Производительность» (маршрут
`/performance`, в UI «Быстродействие») — только **live-приборная панель**, покрывающая стадию «где
узкое место»: probe WMI/Process, метрики кластера через `rac`, DMV SQL Server и узкая «Запись по
требованию» (`PerfRecording*`). Стадия «почему» — разбор технологического журнала 1С (ТЖ) — не
покрыта; нет ни сбора ТЖ, ни его парсинга, ни объекта-расследования.

**Решение.** Раздел **двухрежимный**. Режим **«Наблюдение»** — существующий live-режим
(переиспользует probe, источники метрик и атрибуцию база→клиент; описывается present-tense как есть).
Режим **«Расследование»** (трек 1.2) — объект «Дело» (`Investigation`) с окном времени поверх
**сбора и парсинга технологического журнала 1С** (целевая платформа 8.5, формат NDJSON). Переход
между режимами бесшовный: из «Наблюдения» можно начать расследование по текущему симптому. Проектная
основа, методические источники (ИТС) и целевой дизайн — база знаний `../research/perf-investigation/`
(точка входа [`00_INDEX.md`](../research/perf-investigation/00_INDEX.md), целевой дизайн —
[`30_TARGET_DESIGN.md`](../research/perf-investigation/30_TARGET_DESIGN.md)).

**Последствия.** «Быстродействие» — режим «Наблюдение»; слой ТЖ (сбор/парсинг/анализ) и доменная
сущность «Дело» расследования (`Investigation`/`Finding`/`CollectionConfig`, `03_DOMAIN_MODEL.md`)
реализованы. Single-host (ADR-28): ТЖ и метрики лежат на том же узле, где стоит панель; внешних
коллекторов/агентов нет. Безопасность сбора ТЖ — отдельный ADR-58. **Границы:** это НЕ клон ЦУП/ЦКК
и НЕ внешний агент; расследование ограничено **серверной** частью (клиентский ТЖ рабочих станций
арендаторов не собираем); замеры/Apdex (стадия «что медленно для пользователя») — поздний
опциональный этап.

**Статус:** реализован (трек 1.2, `MLC-228`…`MLC-249`; выпущен `v1.2.0-beta`).

## ADR-58. Политика безопасного сбора технологического журнала 1С

**Контекст.** `logcfg.xml` глобален на весь кластер/агента 1С (ragent, rmngr, все rphost всех
инфобаз), а не на одну ИБ. Неосторожный «полный ТЖ» (`<property name="all"/>` без фильтров) на проде
сам становится инцидентом производительности и забивает диск: на стенде 8.5 полный ТЖ ≈ **260 МБ на
`rphost` за ~100 с** (гигабайты в час с одного процесса). Каталог `conf`
(`C:\Program Files\1cv8\conf\`) доступен на запись только `SYSTEM`/`Администраторы`/`TrustedInstaller`;
сервисный аккаунт панели `NT SERVICE\MitLicenseCenter` права записи в `logcfg.xml` не имеет.

**Решение.** Инварианты безопасного сбора ТЖ (реализованы в треке 1.2):

1. **Только целевой сбор** — события сценария + фильтр длительности; панель **никогда** не ставит
   `<property name="all"/>`.
2. **Изоляция арендатора** — при привязке к `InfobaseId` шаблон **обязан** содержать фильтр
   `<eq property="p:processName" value="<имя ИБ>"/>`; без него собирается ТЖ всех арендаторов (лишняя
   нагрузка + утечка чужих данных).
3. **Бокс по времени** — окно сбора + авто-снятие `logcfg.xml` по таймауту.
4. **Лимит места + сторож диска** — авто-стоп при достижении порога (`StopReason=DiskLimit`).
5. **Один активный сбор** за раз.
6. **Гарантированное снятие + сторож на старте** — на старте панель сверяет фактический `logcfg.xml`
   с ожидаемым и принудительно восстанавливает «снятый» вид при отсутствии активного дела (защита от
   краха ОС, при котором конфиг остался бы в `conf`).
7. **Идемпотентность** постановки/снятия.
8. **Только Admin** запускает сбор; Viewer читает результаты.
9. **Аудит** каждой постановки/снятия (новый `AuditActionType` = новое int-значение — enum аудита
   заморожен, существующие не переназначаются).

Доступ к `logcfg.xml` решён грантом ACL `Modify` на каталог `conf` сервисному аккаунту панели
(установщиком); runtime-проба сбора при недостатке прав отдаёт оператору точную команду `icacls`
(исход старта `NoWriteAccess`/`AgentNoCollectionAccess`, эндпоинт `POST /investigations`). ТЖ
чувствителен (тексты запросов, имена, фрагменты данных ИБ) — обращение как с чувствительными данными;
фикстуры/примеры для тестов обезличены (репозиторий публичный).

**Последствия.** Сбор ТЖ безопасен по дизайну: оператор не может оставить «полный ТЖ», переполнить
диск или собрать данные чужого арендатора. Детальная политика, чек-листы и стендовые факты —
[`60_SAFETY.md`](../research/perf-investigation/60_SAFETY.md) и
[`40_TECHLOG.md`](../research/perf-investigation/40_TECHLOG.md). Связь с моделью ACL — ADR-8;
single-host локальность — ADR-28.

**Статус:** реализован (трек 1.2, этапы A–C; `MLC-230`…`MLC-239`; выпущен `v1.2.0-beta`).

## ADR-59. Секьюрность auth-куки следует за транспортом (отзыв жёсткого `Secure=Always` в проде)

**Контекст.** Кука сессии `mlc.auth` в проде помечалась `SecurePolicy = Always` безусловно (не
завися от `Security:EnforceHttps`) — браузер хранит и шлёт такую куку только по HTTPS. Но штатный
деплой мастера установщика — **plaintext HTTP на все интерфейсы** (`"Urls":"http://+:<порт>"`,
`EnforceHttps=false`, угроза-модель SECURITY §8.5 — «Принят»). На таком деплое:

- с самого сервера вход работал — браузеры считают `http://localhost` защищённым контекстом и
  принимают `Secure`-куку;
- **из локальной сети** (`http://<имя-хоста>` / по IP) вход был **физически невозможен**: браузер
  молча выбрасывает `Secure`-куку на не-localhost http. Логин на сервере проходил (`200`), но клиент
  оставался неавторизованным и его выкидывало на форму — внешне «пароль не подходит» у всех,
  включая `admin` (баг `MLC-240`). Это противоречило заявленному в OPERATIONS LAN-http-сценарию.

На http-деплое без TLS флаг `Secure` защиты и не даёт (шифровать нечего) — он только ломал вход.

**Решение.** Секьюрность куки следует за фактическим транспортом
(`TransportSecurity.AuthCookieSecurePolicy`, чистое решение под юнит-таблицу):

- **Development** → `SameAsRequest` (dev по http);
- **`Always`** — если приложение само терминирует TLS (`EnforceHttps=true`) **или** оператор за
  терминирующим TLS-реверс-прокси явно задал новый флаг **`Security:RequireSecureCookie=true`**
  (Kestrel видит http от прокси, поэтому нужен явный опт-ин);
- **иначе** (штатный http-LAN) → `SameAsRequest` — кука работает по http из сети, а на запросах по
  https автоматически получает `Secure`.

Это **явный отзыв инварианта** «cookie `Secure=Always` в проде» (SECURITY §8.5) — безопасный-по-
умолчанию замысел сохранён там, где есть TLS, и снят там, где TLS нет и он лишь ломал вход.

**Последствия.** Дефолтный http-LAN-деплой чинится сам, без правки конфига на стенде. Развёртывания
с TLS не меняются (`EnforceHttps=true` → `Always` как и раньше). **Изменение поведения только для
TLS-реверс-прокси:** ранее `Secure` ставился безусловно, теперь за прокси нужен
`RequireSecureCookie=true` — задокументировано в OPERATIONS и шаблоне `appsettings.Production.json`.
`SameSite=Strict` и `HttpOnly` не затронуты. Будущее усиление (вернуть авто-`Secure` за прокси без
флага) — через `ForwardedHeaders` с доверенными прокси.

**Статус:** принят (`MLC-240`, релиз 1.1.5).
