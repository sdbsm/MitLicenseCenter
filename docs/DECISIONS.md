# Architecture Decision Records (ADR)

Current, binding architectural decisions for MitLicense Center v1. Each ADR states the decision, the reason, and the key rejected alternative. ADR numbers are stable ŌĆö code and other docs reference them by number.

## 1. Frontend Framework
- **Decision:** React (Single Page Application) with TypeScript.
- **Rejected:** Blazor WebAssembly, ASP.NET Core MVC, Vue.js.
- **Reason:** Strict API boundary between the frontend and the .NET infrastructure backend; the UI is fully decoupled from server logic.

## 2. 1C Cluster Integration Method
- **Decision:** Administer the 1C cluster exclusively through RAS (`ras.exe`, default TCP 1545) driven by the `rac.exe` CLI. This is the **sole** cluster adapter ŌĆö see **ADR-16** for the consolidation rationale and **ADR-3.3** for the full CLI contract.
- **Rejected:** 1C Cluster REST API (not published by default on the target single-node 8.5 deployments ŌĆö see ADR-16); COM-connection (`V83.COMConnector`, slow and leaky); direct database tampering.
- **Reason:** RAS via `rac.exe` answers reliably on every target deployment; no extra HTTP surface to expose or maintain.

## 3. License Enforcement Approach
- **Decision:** Snapshot-based background reconciliation loop (see ADR-6 for cadence).
- **Rejected:** Real-time connection interception / web-server hooks.
- **Reason:** The 1C platform offers no reliable native hook to reject a login based on external multi-tenant rules. The background worker keeps the cluster unblocked during login.

## 4. IIS Publication Updates ŌĆö webinst + web.config
- **Decision:** Standard publications are (re)created via the `webinst` CLI (`-publish -iis`, run from the version-specific `ŌĆ”\1cv8\<version>\bin\webinst.exe`). A platform-version change rewrites **only** the `wsisapi.dll` path segment in `web.config` (surgical edit; `default.vrd` is left untouched). Publication state is **read-only**: the panel reports whether a publication is live (`Published / NotPublished / Error`) and never enforces a desired state. (The *hosting lifecycle* ŌĆö app-pool recycle/start/stop, site start/stop/restart, full `iisreset` ŌĆö is a separate, explicitly-operator-driven capability, see ADR-24; it does not make publication state enforced.)
- **Accepted trade-off:** `webinst` overwrites `default.vrd` and `web.config` wholesale, so OData / HTTP-services / OpenID and other fine-tuning are **not** managed by the panel ŌĆö they are configured outside it (Designer, or a hand-prepared `-descriptor` template). The publication's `Source` flag (`Webinst` / `Configurator` / `Unknown`) records provenance and gates re-publication: re-publishing a non-`Webinst`, currently-published publication requires explicit operator confirmation (it would discard manual config).
- **Batch operations (MLC-046):** the panel also drives publish and change-platform over a multi-select of publications. A batch is executed as N **idempotent single operations** against the same per-publication endpoints ŌĆö there is no bulk endpoint and no new audit event (each publication writes its own `PublicationPublished=212` / `PublicationPlatformChanged=213`). The overwrite gate is presented once for the whole selection (a single confirmation listing the affected non-`Webinst`, currently-published publications), then every item runs with `Confirm=true`. Concurrent `webinst` spawns are capped server-side by `IWebinstConcurrencyGate` (a singleton `SemaphoreSlim`, spawn-budget family of ADR-3.3) regardless of the caller; the frontend additionally runs the batch through a small pool. The run is driven from the browser and is therefore bound to the open tab (an observed deploy); an interrupted batch is safely resumed by re-selecting the failed items (re-publish is idempotent, and a now-`Webinst` publication no longer trips the gate). **Deferred option:** tab-independent execution (a Hangfire job + progress polling) ŌĆö taken only if unattended / scheduled mass-publish becomes a need.
- **Rejected:** surgical in-place XML patching of `default.vrd` to enforce a desired state + a periodic drift-reconcile loop (the prior model, ADR-4.1 ŌĆö now revoked). It modelled a desired state the operators do not actually maintain in the panel, and on modern 8.x builds the platform-version segment lives in `web.config` (so the vrd patch was a no-op there anyway). A single synchronous bulk endpoint was also rejected (~100 publications ├Ś ~60s `webinst` each does not fit one HTTP request, and progress would be invisible until completion).

## 5. Architectural Style
- **Decision:** Modular Monolith (C# / .NET) with strict logical boundaries between modules.
- **Rejected:** Distributed microservices.
- **Reason:** The system runs tightly against a specific single-node Windows stack (IIS, 1C Server, MSSQL). Microservices add network and operational overhead for no benefit at this scale.

## 6. Reconciliation Loop Cadence (Two-Tier)
- **Decision:** Two-tier loop. **Hot** ŌĆö tenants at Ōēź 90% consumption polled every 3ŌĆō5s. **Cold** ŌĆö full snapshot of all tenants every 20ŌĆō30s. A tenant exits the hot tier after two consecutive cold cycles below the threshold.
- **Rejected:** A single uniform 15ŌĆō30s loop; per-second polling for everyone.
- **Reason:** A single 30s loop leaves a window for an over-quota user to log in and lose unsaved work when killed; a global 3ŌĆō5s loop overloads the cluster as bases grow. The hot/cold split bounds the enforcement window to Ōēż 5s for at-risk tenants while keeping baseline load low ŌĆö enforcement runs on **both** tiers (kill on every hot tick for hot tenants, and on each cold cycle), see ADR-6.1.

### ADR-6.1 ŌĆö Hot-tier polling lives in a BackgroundService, not Hangfire ŌĆö and enforces
- **Decision:** Hot-tier polling runs as `HotTierPollingService : BackgroundService`. Cold reconciliation stays in Hangfire (`* * * * *` CRON with an internal throttle to `Polling.ColdIntervalSeconds`). **Both tiers enforce:** kill runs at the end of each cold cycle **and** on every hot tick for hot-tier tenants ŌĆö the hot tier reuses the single `ListActiveSessionsAsync` fetch it already makes for the UI overlay as the enforcement fresh-check, so it adds no extra `rac.exe` spawn (ADR-3.3). This is what actually delivers ADR-6's Ōēż 5s enforcement window for at-risk tenants (MLC-044).
- **Concurrency / over-kill (MLC-001):** because both a Hangfire job (cold) and a `BackgroundService` (hot) now issue kills, the Hangfire-only `[DisableConcurrentExecution]` (cold-vs-cold) is no longer sufficient. Both paths acquire a shared in-process lock ŌĆö `IEnforcementGate` (singleton `SemaphoreSlim(1, 1)`) ŌĆö for the whole "fetch fresh list + kill" window, so exactly one enforcement path runs at a time. The hot tier takes the lock **before** its fetch, so its single session list is read under the lock and a cold kill cannot slip in between fetch and kill (otherwise the reused list would be stale ŌåÆ possible over-kill). Single-node, one process ŌåÆ an in-process lock is sufficient (no distributed lock needed across the two paths).
- **Reason:** Hangfire's CRON minimum is 1 minute; the hot tier needs a 3ŌĆō5s cadence. Driving enforcement from the hot tick (not just the cold cycle) is what bounds the kill latency for an already-hot tenant to one hot interval (Ōēł 4ŌĆō5s) instead of one cold interval (Ōēł 25s).

### ADR-6.2 ŌĆö `SessionKilled=200` + `AuditReason` differentiator
- **Decision:** A single `AuditActionType.SessionKilled = 200` covers both automatic and manual kills; the distinction lives in `AuditReason` (`LimitExceeded = 1`, `ManualByAdmin = 2`).
- **Rejected:** A separate `SessionKilledManual = 201` value.
- **Reason:** `AuditReason` already provides the filtering; a second enum value doubles the surface for nothing. `LimitChanged = 201` is reserved for future tenant-limit-change audit.

## 7. Admin Authentication
- **Decision:** ASP.NET Core Identity with local accounts in MSSQL (schema `auth`); cookie auth (HttpOnly, Secure, SameSite=Strict). Two roles: `Admin` (full access incl. kill and reconcile) and `Viewer` (read-only). The first admin is seeded by migration with a random password written to the service log. Authentication is exactly username + password + cookie ŌĆö **in-app 2FA is out of scope** (ADR-15); the secondary factor is delegated to the network edge.
- **Rejected:** Windows Integrated Auth (fragile for non-Windows-account staff); JWT bearer (cookie is simpler/safer for a same-origin SPA); external IdPs (violates "no external systems").
- **Reason:** Fully local, no external dependencies, native to .NET, shares the domain MSSQL. The Identity base-class column `AspNetUsers.TwoFactorEnabled` stays in the schema (removing it needs a custom user-class refactor) but is never read or written.

## 8. Secret Management
- **Decision:** ASP.NET Core Data Protection. Secret-setting values are encrypted at rest in `dbo.Settings` (`Value VARBINARY(MAX)`, protector purpose-string `mlc.settings.v1`). The Data Protection **key ring** (the master keys) is stored as **plaintext XML** in `%ProgramData%\MitLicenseCenter\keys\` (dev: `%LocalAppData%\...\keys\`) ŌĆö deliberately **without** at-rest encryption of the keys themselves. The keys are intentionally portable: the backup unit "key ring + database" restores on any new hardware or account, which is what makes a recovered backup readable (ADR-15, `OPERATIONS.md`). The key ring is protected on disk by **NTFS ACL**: the installer breaks inheritance and grants access only to SYSTEM, Administrators, and ŌĆö in Windows-auth installs ŌĆö the service account (in **both** installation modes; see MLC-110, `installer/MitLicenseCenter.iss`). `appsettings.Production.json` (which carries the plaintext SQL password) is hardened the same way.
- **Rejected:** `ProtectKeysWithDpapi` (encrypting the key ring with DPAPI binds it to the machine/account ŌĆö a backup restored on new hardware or under a different account can no longer be decrypted, defeating the portable-backup requirement); `ProtectKeysWithCertificate` (drags in management of a `.pfx` and its password ŌĆö a secret to protect the secret); HashiCorp Vault / Azure Key Vault (external systems, forbidden); plaintext secret values directly in `dbo.Settings` or config files.
- **Reason / trade-off:** backup portability is valued above at-rest encryption of the keys. Anyone able to read the key ring through the ACL is already a local administrator of the host (SYSTEM, a member of Administrators, or the service account) and therefore already controls everything on the box ŌĆö the same reasoning as ADR-21. So at-rest key encryption would buy little against that threat model while breaking restore-on-new-hardware. The keys can be read only by SYSTEM / Administrators / the service account; everyone else (including interactive `Users`) is denied by the NTFS ACL.

## 10. REST API Versioning
- **Decision:** URI versioning ŌĆö all endpoints under `/api/v1/...` (`Asp.Versioning.Mvc`). Breaking changes introduce `/api/v2/...`. OpenAPI/Swagger UI at `/api/docs` (Swashbuckle; raw spec at `/api/docs/v1/swagger.json`) is the browsable **reference contract**. The TypeScript API types are hand-written, **not** generated from the spec ŌĆö see ADR-10.1.
- **Rejected:** Header-only versioning (harder to discover/test); no versioning.
- **Reason:** URI versioning is the most discoverable and tooling-friendly; frontend and backend deploy together so complex negotiation is unnecessary.

### ADR-10.1 ŌĆö Hand-written TypeScript API types (no OpenAPI codegen)
- **Decision:** The frontend's API types are **hand-written** in `frontend/src/features/*/types.ts` (one `types.ts` per feature) plus the request helper in `frontend/src/lib/api.ts`. The OpenAPI spec at `/api/docs/v1/swagger.json` is the reference contract, but **no code generator runs against it** ŌĆö there is no codegen step in `frontend/package.json`. The "TypeScript client is generated from the OpenAPI spec" wording in earlier drafts of ADR-10/ADR-13 was aspirational and never implemented; this ADR records the actual, deliberate state.
- **How the contract is kept in sync:** types are updated by hand whenever an endpoint's shape changes, cross-checked against the live Swagger UI at `/api/docs`. Backend DTOs serialize camelCase (System.Text.Json default), matching the TS interfaces verbatim. `api<T>()` casts the parsed JSON to the declared `T` with **no runtime validation** ŌĆö a mismatch surfaces as a downstream type error, not a guarded boundary failure.
- **Rejected:** wiring `openapi-typescript` / NSwag codegen into the build (extra toolchain plus either a running backend or a checked-in spec snapshot to generate against, for a surface of ~7 small feature type files owned by the same author who owns both ends); Zod schemas at every response boundary (heavier than the drift risk warrants at this scale). The cost outweighs the benefit for a 5ŌĆō20-user single-node panel.
- **Reason:** Monorepo + same author on both ends + a small, stable API surface ŌĆö hand-written types are cheaper to maintain than a codegen pipeline, and the Swagger UI already provides a browsable contract for cross-checking. **Backlog:** adopting OpenAPI codegen (`MLC-006(a)`) or Zod runtime validation at the response boundary (`MLC-016`) is a deliberate future step in `PROJECT_BACKLOG.md`, not an implied current guarantee. Promoting either does not require revoking this ADR ŌĆö it supersedes it.
- **Update (MLC-016, 2026-06-03) ŌĆö runtime validation adopted on a few critical boundaries, not everywhere:** the blanket "no runtime validation" above now has a deliberate, narrow exception. Because `zod` is already a direct dependency (paired with `react-hook-form`, ADR-11), adding runtime checks introduces **no new toolchain** ŌĆö the *Rejected* cost objection applied to validating *every* boundary, not a handful. Zod schemas now guard the responses where a silent contract drift is genuinely dangerous: `GET /api/v1/auth/me` + `POST /api/v1/auth/login` (`roles` drives `ProtectedRoute` role gating ŌĆö an authorization concern), `GET /api/v1/sessions/snapshot` (the over-limit / kill operational view), and the paginated list envelopes (`{items,total,page,pageSize}`) for tenants and infobases (via a shared `pagedResponseSchema(item)` factory in `frontend/src/lib/apiSchema.ts`). On those, the TS type is **derived from the schema** (`z.infer`, schemas live next to the feature's `types.ts`) so there is a single source of truth. `api<T>()` is unchanged for every other endpoint; it gained an optional `schema` parameter, and a schema mismatch throws a controlled `ApiSchemaError` (carrying `path` + the underlying issues) instead of returning a silently mistyped object. Validation runs only on 2xx success ŌĆö error envelopes keep the existing `ApiError` path. **Not done deliberately:** Zod on every response (the drift risk on the remaining stable, author-owned endpoints does not warrant the schema-maintenance overhead) and OpenAPI codegen (still `MLC-006(a)`). **Nullable fields the backend omits** (global `JsonIgnoreCondition.WhenWritingNull`, `Program.cs` ŌĆö a `null` value is sent as an *absent key*, never explicit `null`) are modeled on these guarded boundaries with the `omittable(schema)` helper (`frontend/src/lib/apiSchema.ts`), not `z.nullable()`: it accepts both the missing key and `null` (`.nullish()`) and normalizes to `null`, so the consumer type stays `T | null`. Plain `z.nullable()` requires the key to be present and would reject the wire shape as soon as the value is null.

## 11. UI Component Library
- **Decision:** shadcn/ui (copy-paste components owned in the repo) on top of Radix UI, styled with Tailwind CSS. Auxiliary: `lucide-react`, `@tanstack/react-table`, `react-hook-form` + `zod`, `sonner`, `recharts`, `date-fns`/`ru`. No other component libraries mixed in.
- **Rejected:** Material UI, Ant Design (heavier, opinionated, farther from the target look); a fully custom design system (overkill for a 5ŌĆō20 user panel).
- **Reason:** Owned in-repo (no vendor lock-in), accessible Radix primitives, CSS-variable theming (light + dark). Visual language, status semantics, table patterns and Russian microcopy are codified in `06_UI_DESIGN.md`.

## 12. Backend Runtime = .NET 10 (LTS)
- **Decision:** Target `net10.0` (LTS until Nov 2028).
- **Rejected:** `net8.0` (less runway); `net9.0` (STS, short window); .NET Framework 4.8 (legacy).
- **Reason:** Maximum LTS runway, latest C# 14 / EF Core 10 / ASP.NET Core 10; all planned libraries are compatible.

## 13. Repository Layout = Monorepo
- **Decision:** Single private GitHub repo ŌĆö `backend/` (.NET solution), `frontend/` (Vite/React/TS), `docs/`, `scripts/`, `.github/`.
- **Rejected:** Separate frontend/backend repos.
- **Reason:** Atomic cross-cutting commits, single CI pipeline, one source of truth. The hand-written TS API types (ADR-10.1) live next to the API that produces them, so a contract change and its type update land in the same commit.

## 14. CI/CD = GitHub Actions, CI Only (No CD in v1)
- **Decision:** GitHub Actions on every push and PR to `main`. Backend: `restore ŌåÆ build ŌåÆ test`. Frontend: `install ŌåÆ lint ŌåÆ type-check ŌåÆ test ŌåÆ build`. PRs are blocked while either job is red. No deployment automation: deploy is a **manual operator procedure** documented in `OPERATIONS.md` ("Deployment is manual") ŌĆö build with `scripts/build.ps1`, `dotnet publish` the backend, `pnpm build` the SPA, then stop/replace/restart the host. There is **no deploy script** in `scripts/` (only `build.ps1` / `db-reset.ps1` / `dev.ps1` / `shadcn-add.ps1`); a future `scripts/Deploy-MitLicenseCenter.ps1` is a backlog item, not a current artefact.
- **Rejected:** No CI (unacceptable for a system that auto-kills sessions); full CD on tag (premature); self-hosted runners / Jenkins / Azure DevOps (extra infra); shipping a deploy script before the manual procedure has stabilized (it would encode an unsettled story and drift from reality).
- **Reason:** Minimum viable safety net; CD ŌĆö and any scripted deploy ŌĆö waits until the deployment story stabilizes. Fail-fast bootstrap (ADR-18) already makes the manual procedure safe: a new version applies its migrations synchronously on start or refuses to serve, so the operator needs only build ŌåÆ publish ŌåÆ restart, not a migration runner.
- **Update (MLC-099, 2026-06-11) ŌĆö `scripts/publish-release.ps1` added as build/packaging tooling, not CD:** the script list above now also names `publish-release.ps1`. It is a thin wrapper over `dotnet publish` that **produces an installable artefact** (by default self-contained single-file `win-x64` so the host needs no .NET runtime; `-FrameworkDependent` keeps the framework-dependent option) ŌĆö it does **not** deploy. Deploy stays the manual operator procedure in `OPERATIONS.md`, and the deferred `scripts/Deploy-MitLicenseCenter.ps1` is still a backlog item: producing a release artefact is *packaging*, not *deployment automation*, so this ADR's "no CD / no deploy script" stance is unchanged. (This closes a codeŌåödoc divergence: the enumerated `scripts/` set above had drifted from the tree.)
- **Update (MLC-100, 2026-06-11) ŌĆö a GUI installer packs install/upgrade, but CD is still deferred:** the GUI installer (Inno Setup, **ADR-31**) and its build script `scripts/build-installer.ps1` are **packaging that also performs a local install/upgrade on the target host** (register + start a Windows service, open the firewall port, replace files in place while preserving config/key ring/DB) ŌĆö they are **operator-run on the host**, not a push from CI. There is still **no CIŌåÆhost deployment automation**: nothing in `.github/workflows/` or `scripts/` connects a green build to a running host; the operator builds `Setup.exe` and runs it. So the deferred `scripts/Deploy-MitLicenseCenter.ps1` (a *remote/scripted* deploy) and the "no CD" stance both stand ŌĆö the installer turns the *manual file-copy* steps of the "Deployment is manual" procedure into a wizard, but does not turn deployment into a pipeline.

## 15. Backup and Two-Factor Authentication Scope Boundary
- **Decision:** In-app 2FA is permanently **out of scope**; **scheduled/automated** backup orchestration and **restore** remain **out of scope** (operator responsibility) ŌĆö with one narrow, operator-initiated exception carved out by **ADR-27**.
  - **Backup.** The operator owns the deployment's backup *strategy* via platform tooling (SQL Maintenance Plans, Veeam, Windows Server Backup, Robocopy + Task Scheduler) ŌĆö including any scheduled full/differential chains. The app does **not** schedule backups, run a backup loop, ship a restore CLI, or restore anything. **Amended by ADR-27:** the panel *does* offer an **operator-initiated, on-demand, single-database `COPY_ONLY` full backup** as a convenience button ŌĆö it is manual (no scheduling), produces only a standalone copy, and `COPY_ONLY` guarantees it never disturbs the operator's external differential chain. It never restores. The backup-unit caveat is unchanged: MSSQL database + Data Protection key ring (+ optional `appsettings.Production.json`); restoring one without the other yields an unreadable database. See `OPERATIONS.md`.
  - **2FA** is a network-level concern ŌĆö LAN/VPN access, perimeter firewall, AD/SSO at the edge, physical access control. `AspNetUsers.TwoFactorEnabled` stays in the schema but is operationally inert.
- **Rejected:** **scheduled / always-on** backup orchestration inside the app (a Hangfire backup loop, desired-state backup management ŌĆö doubles the failure surface; the backup *strategy* is solved at the platform layer); in-app TOTP/WebAuthn/SMS (duplicates or weakens the network-edge protection every deployment already has). The on-demand button (ADR-27) is **not** this ŌĆö it adds no schedule and no restore path.
- **Reason:** Scope discipline ŌĆö the app's job is 1C session licensing; an on-demand one-click backup is a thin operator convenience that does not pull backup *strategy* into the app. **Locked:** re-introducing **scheduled** backup orchestration or any **restore** capability requires a fresh ADR; 2FA remains locked out. The on-demand backup exception is bounded by **ADR-27**.

## 16. RAS as Sole 1C Cluster Adapter
- **Decision:** The 1C cluster adapter layer is exclusively `rac.exe`. Operator-tunable inputs: `OneC.RAS.Endpoint` (default `localhost:1545`) and `OneC.RAS.ExePath` (no default ŌĆö see ADR-3.3). `OneC.Cluster.AdminUser` / `OneC.Cluster.AdminPassword` feed `rac.exe`'s `--cluster-user` / `--cluster-pwd` flags (both may be empty for clusters with no registered administrators).
- **Dashboard surface:** the cluster card is a **RAS health card** backed by `RasHealthProbingService : BackgroundService` (30s `PingAsync`) writing to `IRasHealthReader`. Three states: `OK`, `ąĪą▒ąŠą╣` (danger, last-error + consecutive-failures), `ą¤čĆąŠą▓ąĄčĆą║ą░ŌĆ”` (neutral, first 30s after startup). ## 18. Fail-fast Bootstrap
- **Decision:** Database migrations and seeding run **synchronously in the startup pipeline, before `app.Run()` opens the listener** ŌĆö not in a fire-and-forget `Task.Run` from `ApplicationStarted`. Order is fixed: **create the database if it is missing** (`DatabaseBootstrapper`, see below) ŌåÆ EF Core migrations + roles/first-admin (`IdentitySeeder`) ŌåÆ settings catalog (`SettingsSeeder`, which requires the `dbo.Settings` table the migration creates). Any failure logs `Critical` and lets the exception propagate out of `Main`, so the process exits non-zero **without ever accepting a request**. `IdentitySeeder` skips `MigrateAsync` on a non-relational provider (`Database.IsRelational()` is false) so `WebApplicationFactory<Program>` integration tests on the EF in-memory provider boot without a relational migration call.
- **Database creation precedes migrations (MLC-106).** On a truly empty SQL instance (the panel database does not yet exist) the bootstrap **creates it** before anything connects. The root cause it fixes: `db.Database.MigrateAsync()` does **not** create a missing database **under `EnableRetryOnFailure`** ŌĆö SQL error 4060 ("cannot open database") is classified as transient and retried, then `MigrateAsync` throws ŌåÆ fail-fast ŌåÆ the service crashes and the database is never created (reproduced on the stand; `db-reset` masked it in dev by pre-creating the DB). So `DatabaseBootstrapper.EnsureDatabaseCreatedAsync` runs **first**, in `Program.cs` immediately after `builder.Build()` and **before** the Hangfire dashboard / `RecurringJob.AddOrUpdate` registration (Hangfire's `SqlServerStorage` also connects to the DB) and before `IdentitySeeder`. It opens **one raw `Microsoft.Data.SqlClient` connection to `master`** (same credentials/`Encrypt`/`TrustServerCertificate` as `ConnectionStrings:Default`, only `InitialCatalog` swapped ŌĆö the `SqlDatabaseDiscovery` technique) and runs `IF DB_ID(@name) IS NULL EXEC('CREATE DATABASE [<escaped name>]')` ŌĆö the database name is checked as a parameter in `DB_ID(@name)` and the escaped name (`]`ŌåÆ`]]`) is interpolated into the DDL (DDL takes no object-name parameter). A single statement to `master` sidesteps the retry trap entirely. An **existing database is never touched** (`IF DB_ID IS NULL`), so in dev/ops where `db-reset`/the installer already created it the step is a no-op. The call is **gated** on a non-empty `ConnectionStrings:Default` (the EF in-memory `WebApplicationFactory<Program>` tests do not set it ŌåÆ the step is skipped) and wrapped in the same `LogCritical` + `throw` fail-fast contract as the seeder. The raw `SqlClient` lives in `Infrastructure.Persistence` (the DB-access layer, allowed there per ADR-20 ŌĆö not a 1C/IIS adapter); Web calls the static method as it already calls the seeders. **Requires `CREATE DATABASE` rights** (`sysadmin`/`dbcreator`), already required by the canon for discovery/backups.
- **Rejected:** The previous `_ = Task.Run(async () => { ŌĆ” throw; })` inside `ApplicationStarted` ŌĆö the `throw` was **unobserved**, so a failed migration or seed left the host listening in a half-initialized state (no admin account, or an unmigrated database) with no operator-visible signal. Calling `IHostApplicationLifetime.StopApplication()` after traffic is already accepted was also rejected as strictly weaker than never opening the port.
- **Reason:** Operability ŌĆö a running, listening process must be a fully migrated and seeded one (see `OPERATIONS.md`). The first admin is still seeded with a random password written to the log (ADR-7), now guaranteed before the first request can arrive. **Locked:** initialization that must complete before serving traffic belongs in the synchronous pre-`Run` block, not a detached task.

## 19. Global ProblemDetails + uniqueness backstop
- **Decision:** The Web pipeline registers `AddProblemDetails()` and an outermost `UseExceptionHandler()`, so every unhandled exception is returned as an RFC 7807 `ProblemDetails` (with a `traceId` extension) instead of a bodiless 500. `CustomizeProblemDetails` rewrites 5xx responses to a neutral Russian title/detail and never emits the exception message or stack trace (the exception is logged server-side; Development keeps the auto developer exception page). The check-then-insert guards in `Infobase`/`Tenant` endpoints keep their pre-flight `AnyAsync` as the friendly happy-path, but `SaveChanges` is now wrapped in a **backstop**: a `DbUpdateException` from a unique-index violation is mapped to the same documented 409 `ProblemCodes.*` as the pre-check. The violated index is distinguished by its **name** ŌĆö a stable schema identifier present verbatim in the SQL Server error ŌĆö gated by `SqlException.Number Ōłł {2601, 2627}`; the same index (`IX_Infobases_TenantId_Name`) maps to `NAME_DUPLICATE_IN_TENANT` on create/update but to `INFOBASE_NAME_TAKEN_IN_TARGET` on reassign, so the mapping is decided per-endpoint, not by a global table. `DbUniqueViolation.Identify` is the single recognizer.
- **Rejected:** Wrapping every insert in an explicit transaction with a `SELECT ŌĆ” WITH (UPDLOCK)` (heavier, and the unique index already enforces the invariant); matching on the localized error `detail` text (fragile across culture/EF versions); minting new 409 codes for the race path (the contract already has the right codes ŌĆö the backstop must reuse them so the frontend behaves identically whether the conflict is caught early or late).
- **Reason:** API-contract integrity under the rare 5ŌĆō20-user write race ŌĆö the documented 409 holds even when two requests pass the pre-check simultaneously, and no internal detail leaks on the 500 path. **Note:** EF InMemory does not reproduce unique-violations (tracked as MLC-008); the backstop is unit-tested with a `SaveChanges` interceptor that throws a SQL-Server-shaped `DbUpdateException`. **Locked:** a new conflict still adds a `ProblemCodes.*` constant (per the 409 contract in `03_DOMAIN_MODEL.md`); the backstop reuses existing codes, never invents its own.

## 20. Vertical-slice data access in Web (minimal API + `AppDbContext`)
- **Decision:** Web endpoints (minimal-API handlers in `Web/Endpoints/*Endpoints.cs`) inject `AppDbContext` and ASP.NET Core Identity types **directly** and own their business rules ŌĆö input validation, uniqueness checks, cascade handling, and `AuditLog` writes ŌĆö **in the handler**. These are **vertical slices**, not a strict Clean/onion layering with an Application use-case (CQRS/MediatR) tier between transport and persistence. The `MitLicenseCenter.Application` project is intentionally thin: it holds **interfaces and contracts** (`IClusterClient`, `ISettingsSnapshot`, `ISettingsStore`, `IReconciliationJob`, DTOs, `SettingDefinitions`ŌĆ”) and has **no** per-endpoint use-case handlers.
- **Scope of the relaxation (what is NOT loosened):** the anti-corruption boundary to 1C/IIS from `02_ARCHITECTURE_REQUIREMENTS.md` (core principle #1) and **ADR-5/ADR-16** stands **unchanged**. Web **never** touches `rac.exe`, `ras.exe`, IIS, or `Microsoft.Web.Administration` directly ŌĆö every infrastructure interaction goes through an Infrastructure adapter behind an Application interface. The vertical-slice style is confined to data access against the panel's **own** MSSQL domain database (`AppDbContext`) and Identity, where there is no external system to corrupt and no adapter contract at stake. The main prohibition of `02` is therefore intact.
- **Rejected:** a full onion with one Application use-case handler per endpoint (request/response DTO + handler + mapping for every read and write). For a single HTTP transport with one author on both ends, the rules are used in exactly one place ŌĆö there is nothing to deduplicate, so the handler tier is ceremony with no reuse payoff. `AppDbContext` + minimal API already provide a testable seam (endpoints are unit-tested on EF InMemory and contract-tested on SQLite-in-memory ŌĆö MLC-008).
- **Reason:** scale discipline ŌĆö 5ŌĆō20 users, a single transport, no rule sharing across transports. The strict-boundary requirement in `02` that actually carries weight (adapters to 1C/IIS) is honored; layering the app's own CRUD behind use-cases would buy testability it already has.
- **Revision condition (not Locked):** if a **second consumer** of the same rules appears ŌĆö a second transport (gRPC/CLI/scheduled mass-import) or a background worker that must run the same validation/uniqueness/audit outside the HTTP pipeline ŌĆö extract those rules into Application use-cases at that point. That extraction is scoped as a backlog item (**`MLC-011(a)`**), not an implied current obligation; promoting it supersedes this ADR without a revocation ceremony.

## 21. Accepted risk ŌĆö 1C cluster password on the `rac.exe` command line
- **Decision:** The 1C cluster admin password (`Settings.OneC.Cluster.AdminPassword`) is passed to `rac.exe` as the `--cluster-pwd=<password>` **command-line flag** (`RacExecutableRasClusterClient.BuildArgsWithAuth`, alongside `--cluster-user`). On Windows a process's command line is readable by other processes of the same (or a more privileged) account and by local administrators, so the password is briefly visible to a local-admin/process inspector for the lifetime of each `rac.exe` invocation. This is recorded as an **accepted residual risk**, not remediated.
- **Why it is unavoidable:** in the supported 1C 8.3ŌĆō8.5 versions `rac.exe` accepts the cluster password **only** via `--cluster-pwd` ŌĆö there is no stdin prompt, no password-file, and no environment-variable channel. The flag is the sole supported authentication path (see **ADR-3.3** / **ADR-16**). Invocations are short-lived (30 s per-call deadline, Ōēż 26 procs/min), so the exposure window is brief but real.
- **Compensating factors:** (1) **Single-node topology** (Locked Operational Constraints / `04_INFRASTRUCTURE.md` ┬¦3) ŌĆö the command line is never exposed over a network; (2) **admin-only host** ŌĆö anyone able to read another process's command line on this box is already a local administrator with full control of the 1C cluster, MSSQL, and the DPAPI key ring, so the leaked password grants nothing they don't already hold; (3) **encrypted at rest** ŌĆö the password lives DPAPI-encrypted in `dbo.Settings` (**ADR-8**) and is decrypted into the argument list only at spawn time; (4) the host sits behind the network-edge perimeter that also provides the second auth factor (**ADR-15**, `OPERATIONS.md`).
- **Rejected:** stdin / temp-file / environment-variable delivery of the password ŌĆö **not supported** by `rac.exe` (it would simply fail to authenticate); running the cluster with no registered administrators (anonymous mode) to avoid a password entirely ŌĆö only viable where the operator left the cluster unsecured, cannot be mandated; building the long-lived RAS-socket adapter (Strategy B, deferred in `ROADMAP.md`) to cut spawns ŌĆö it reduces invocation frequency but authenticates the same way and does not remove the flag.
- **Reason:** reality discipline ŌĆö the limitation is in `rac.exe`, and on a single-node admin-only host the residual risk is dominated by the operator's existing privileges. Documenting it as an accepted risk (here and in `04_INFRASTRUCTURE.md` ┬¦1) is the honest, lowest-cost answer.
- **Revision signal:** adopt it immediately if a supported `rac.exe` gains a stdin / file / env password channel (the adapter already isolates spawning behind `IRacProcessRunner`, so the switch is local). Re-review if the single-node admin-only topology assumption ever changes ŌĆö the topology-change trigger in the Locked Operational Constraints already mandates a full adapter re-review.

## 22. Transport hardening is config-gated
- **Decision:** Production transport hardening is **opt-in via configuration**, not wired unconditionally into the pipeline. (1) **HTTPS-redirect + HSTS** (`app.UseHsts()` + `app.UseHttpsRedirection()`) run **only** when `Security:EnforceHttps=true` **and** the environment is not Development ŌĆö the decision is the pure helper `TransportSecurity.ShouldEnforceHttps`. (2) **Swagger UI** is served when `Development || Security:EnableSwagger` (`TransportSecurity.ShouldEnableSwagger`) ŌĆö always in Development, closed in Production unless the operator re-opens it. (3) **SQL channel encryption** is hardened to `Encrypt=True` only in `appsettings.Production.json`; the base `appsettings.json` keeps `Encrypt=False` for local dev. (4) **`AllowedHosts`** is narrowed from `*` to the panel hostname in the Production profile. The auth cookie `Secure=Always` outside Development (ADR-7) is unchanged and independent of these flags. Operator-facing procedure is in `OPERATIONS.md` ("Transport hardening").
- **Rejected:** enabling `UseHttpsRedirection`/`UseHsts` **unconditionally** ŌĆö it breaks Development (plain HTTP to a local SQL without TLS) and, on the single-node topology that may sit **behind a terminating TLS reverse proxy** (IIS ARR / Nginx), causes double-redirects or redirect loops (the app on `localhost` only sees the already-terminated HTTP and cannot tell TLS happened upstream); a terminating proxy already owns redirect + HSTS, so the app must not duplicate it. Editing the **base** `appsettings.json` connection string to `Encrypt=True` ŌĆö it is shared with local dev, where the local SQL instance has no TLS certificate, so the dev default must stay `Encrypt=False` and the production override lives in `appsettings.Production.json`/ENV. Removing Swagger outright ŌĆö it is the browsable reference contract the hand-written TS types are synced against (ADR-10.1), so it stays available in Development and re-openable in Production rather than deleted.
- **Reason:** the single-node deployment (Locked Operational Constraints) may or may not have a terminating reverse proxy in front, and the operator (a sysadmin, not a developer) decides per-deployment which side terminates TLS. A config flag keeps the dev default working untouched and lets Production harden ŌĆö HTTPS enforcement, encrypted SQL channel, closed API map, host-header filter ŌĆö without a code change or a rebuild. The panel is still a LAN/VPN-perimeter system (ADR-15); this hardens the transport within that perimeter, it does not authorize public exposure. **Revision signal:** if the deployment standardizes on the service terminating TLS directly (no proxy ever), promote `Security:EnforceHttps` to default `true` in the Production profile; this ADR supersedes without a revocation ceremony.

## 23. Time on the wire is UTC with a `Z` suffix
- **Decision:** every `DateTime` the API serializes carries an explicit UTC designator (`...T09:00:00Z`). Timestamps are **stored** UTC (`TimeProvider.GetUtcNow().UtcDateTime` everywhere ŌĆö `CreatedAt`/`UpdatedAt`/`LastCheckAt`/audit `Timestamp`/settings `UpdatedAt`), **transported** UTC, and **displayed** in the browser's local zone (the SPA's `RelativeTime` and `date-fns format` convert from the UTC instant). There is **no** timezone setting ŌĆö the browser already knows the user's OS zone. Server-side rendering of localized time (emails, PDF, exports) does not exist in v1, so there is nothing that needs a configured zone.
- **Mechanism:** SQL Server `datetime2` columns carry no zone, so EF materializes them as `DateTime` with `Kind=Unspecified`, and `System.Text.Json` then emits them **without** a `Z`. A global read-only value converter in `AppDbContext.ConfigureConventions` (`Properties<DateTime>().HaveConversion<UtcDateTimeConverter>()`, `SpecifyKind(..., Utc)` on materialization) re-stamps every `DateTime`/`DateTime?` as UTC so the `Z` is present on the wire. The converter is read-only (write is identity ŌĆö values are already UTC) and does not change the column type, so it generates **no migration**. In-memory timestamps that never round-trip the DB (RAS session `StartedAtUtc`, dashboard `LastCheckedAtUtc`) already keep `Kind=Utc` and are unaffected. Guarded by `UtcDateTimeConventionTests` (every model `DateTime` property reads back as `Utc`) and `UtcDateTimeJsonTests` (the wire contract ŌĆö `Kind=Utc` ŌåÆ `Z`).
- **Rejected:** a per-property `SpecifyKind` at each contract/endpoint ŌĆö it fixes one symptom and silently rots when a new `datetime2` column is added; a global JSON `DateTime` converter that always appends `Z` regardless of `Kind` ŌĆö it would mask, rather than fix, a value that genuinely is not UTC; an operator-facing **timezone setting** ŌĆö redundant with the browser zone, adds a setting that can disagree with it, and buys nothing for a screen-only admin panel.
- **Reason:** a single source of truth ŌĆö one UTC instant on the wire ŌĆö makes every relative and absolute label across the UI consistent by construction, and the browser localizes for free. The bug this closes (publication ┬½ą¤čĆąŠą▓ąĄčĆąĖčéčī čüąĄą╣čćą░čü┬╗ showing "179 minutes ago" on a UTC+3 host) was precisely a lost `Kind` flag, not a clock disagreement. **Revision signal:** if v1 ever renders localized time **on the server** (scheduled email/report), introduce an explicit display-zone there at that point ŌĆö storage/transport stay UTC regardless.

## 24. IIS lifecycle management from the panel
- **Decision:** The panel manages the IIS **hosting lifecycle** ŌĆö application-pool `recycle` / `start` / `stop`, site `start` / `stop` / `restart`, and server-wide `iisreset` (restart / `/stop` / `/start`) ŌĆö from a dedicated ┬½ąŻą┐čĆą░ą▓ą╗ąĄąĮąĖąĄ IIS┬╗ block **above** the publications list on the Publications page (MLC-047). Targets (pools, sites) are discovered from the **live IIS** (`ServerManager.ApplicationPools` / `Sites` with current state), not from publication rows; the overall IIS status comes from the **`W3SVC` service** (`ServiceController`), not `ServerManager` (it stays readable when IIS is stopped). `iisreset` is server-wide (restart / `/stop` / `/start`). All operations go through a new Application port `IIisLifecycleService` ŌåÆ `OneCIisLifecycleService` (Infrastructure): pool/site commands via `Microsoft.Web.Administration.ServerManager` (runtime commands ŌĆö no `CommitChanges`), `iisreset` by spawning `iisreset.exe` (OEM/CP866 output like `rac.exe`, 90s timeout). The anti-corruption boundary (ADR-20) is unchanged ŌĆö Web never touches `ServerManager`/`Process` directly. New endpoint group `/api/v1/iis/*` (discovery `Viewer`; mutations `Admin`). Every mutation is audited (`IisApplicationPoolRecycled=220` ŌĆ” `IisReset=226`, `IisStopped=227`, `IisStarted=228`, `TenantId=null` ŌĆö server-scope). Destructive operations (recycle, stop, restart, `iisreset` restart/stop) show a plain confirmation dialog in the UI (no typed token ŌĆö these are frequent operational actions; 06_UI_DESIGN ┬¦7) backed by a server-side `Confirm=true` gate on `recycle`/`reset`/`stop` (409 `IIS_CONFIRM_REQUIRED` without it); `start` (pool/site/server) is recovery ŌĆö no dialog, no confirm gate. Destructive operations are serialized by `IIisResetConcurrencyGate` (singleton `SemaphoreSlim(1, 1)`) ŌĆö two `recycle`/`iisreset` never overlap.
- **Accepted trade-off:** these are the first **server-scope** IIS mutations (beyond per-publication `web.config`/`webinst`), so the panel's service identity needs rights to manage IIS pools/sites and to run `iisreset` (start/stop of the `W3SVC`/`WAS` Windows services) ŌĆö see OPERATIONS. `iisreset`/stop interrupts **all** sites on the server, including the panel itself if it is hosted on that IIS; this is surfaced to the operator in the confirmation dialog. Operations are synchronous and bound to the open tab (like the `webinst` batch).
- **Rejected:** extending `IIisPublishingService` (it owns publication read-only status + `webinst`/`web.config`; mixing in server-scope lifecycle would blur ADR-4's read-only invariant and the interface's cohesion); per-row pool/site buttons (the operator manages IIS at the server level ŌĆö all bases share one pool/site ŌĆö so a single ┬½mass┬╗ block fits); PowerShell Remoting (single-node, local IIS ŌĆö `ServerManager` + `iisreset.exe` suffice).
- **Relation to ADR-4:** ADR-4 governs the *publication's desired state* (read-only); ADR-24 governs the *hosting lifecycle*. They are orthogonal ŌĆö ADR-4's read-only invariant stands.

## 25. License usage time-series collection
- **Decision:** The panel collects a **time-series of license consumption** (foundation of the ┬½ą×čéčćčæčéčŗ┬╗ track, MLC-048) ŌĆö the system otherwise keeps no usage history (active sessions live only in the in-memory `ActiveSessionSnapshotStore`; only `AuditLogs` persist, and those are events, not measurements). The **metric** is the instantaneous `Consumed` (license-consuming concurrent sessions) plus the tenant's `Limit` (`MaxConcurrentLicenses`) at sample time. Samples are taken **inside the cold `ReconciliationJob`** reusing the cycle's already-computed `LicenseConsumption.CountByTenant` and the live limit ŌĆö **no new `rac.exe` spawn** (ADR-3.3/16); idle tenants sample `Consumed=0` so min/avg stay honest. A process-wide **singleton accumulator** (`ILicenseUsageAccumulator`, registered like `ColdThrottleState` so state survives the scoped job's per-invocation resolution) folds each sample into the current **15-minute bucket** as running min/max/sum/count per tenant, and on a bucket boundary emits one aggregate row per tenant (`min`/`max`/`avg`, with **`max` mandatory** ŌĆö for concurrent licenses the peak is what hits the limit; `avg` is `double`; `Limit` is the last observed value in the bucket). Rows persist to a telemetry table `dbo.LicenseUsageSnapshots` **outside the enforcement lock** (telemetry never blocks the kill path). **One tier, no rollup** ŌĆö a single bucket granularity is kept and pruned by a **retention setting + nightly job** (`LicenseUsage.RetentionDays`, default 365, range 30ŌĆō3650; `license-usage-retention` at 03:30 UTC, batched delete, no audit row ŌĆö housekeeping). The entity is **telemetry in Infrastructure** (`Infrastructure/Reporting/LicenseUsageSnapshot`, FK to `Tenant` `OnDelete(SetNull)` with nullable `TenantId`), by the `AuditLog` precedent ŌĆö **not** a domain aggregate; the Application layer sees only neutral records (`LicenseUsageSample`/`LicenseUsageBucket`), never the entity (layer boundary).
- **Accepted trade-off:** the current partial (still-open) bucket lives only in the singleton's memory, so a process restart loses it ŌĆö acceptable for best-effort telemetry (no graceful-shutdown flush in v1). Sampling covers active tenants that have Ōēź1 infobase (the cold cycle's data); a tenant with no infobases has no sessions and is not sampled.
- **Rejected:** a domain aggregate / EF-mapped value object (these are measurements, not business state ŌĆö `AuditLog` is the precedent); multi-tier rollups (one 15-min tier is enough for the reports; retention keeps the table bounded); a separate measurement spawn of `rac.exe` (ADR-3.3/16 ŌĆö the cold cycle already has the numbers); persisting under the enforcement lock (telemetry must not contend with the kill path).

## 26. Performance subsystem ŌĆö Live (pull, no persist) + Recording (on-demand table)
- **Decision:** The ┬½ąæčŗčüčéčĆąŠą┤ąĄą╣čüčéą▓ąĖąĄ┬╗ section answers "why is 1C slow?" through a two-level methodology ŌĆö a **saturation traffic-light** (CPU `Processor Queue Length` + %, disk `Avg. Disk sec/Read|Write` latency + queue, RAM `Available MBytes` + `Pages/sec` ŌĆö saturation signals, **not** bare utilization %) plus **attribution by process families** (1ąĪ `rphost`/`ragent`/`rmngr` ┬Ę MSSQL `sqlservr` ┬Ę OS-update ┬Ę antivirus ┬Ę other), drilling into the guilty family. It has **two collection modes**, and this is the load-bearing decision:
  - **Live** ŌĆö always available, **pull-on-demand**: while a browser tab is open the frontend polls (~5s) and the backend reads the sources on each poll. **Nothing is persisted** ŌĆö closing the tab stops collection (zero background cost). The first slice is the host snapshot (`MLC-064`): `GET /api/v1/performance/host`, role `Viewer`. The 1C-load live view (`MLC-066`) adds `GET /api/v1/performance/onec-sessions` (role `Viewer`) ŌĆö perf-rich sessions (`OneCSessionLoad`) + worker processes (`OneCProcessLoad`) read through `IClusterClient` (`ListSessionLoadsAsync` + `ListProcessesAsync`, 2 `rac.exe` spawns/poll, ADR-3.3); it likewise persists nothing. The SQL-DMV live view (`MLC-068`) adds `GET /api/v1/performance/sql` (role `Viewer`) ŌĆö active requests + blocking chains + per-database IO-stall + a `wait_stats` delta read through `ISqlPerformanceProbe`, with databaseŌåÆtenant attribution stitched in the endpoint (vertical slice over `AppDbContext`); it persists nothing.
  - **Recording** ŌĆö turned on/off **manually** for an investigation (`MLC-070`, Phase 4): a process-wide singleton `IPerfRecordingService` ŌåÆ `PerfRecordingService` holds the one active recording (id + start + sample count) and a background `PerfRecordingSamplingService` samples on a timer (`Performance.RecordingSampleIntervalSeconds`, default 15 s), writing each tick ŌĆö the host snapshot + the top-N 1ąĪ/SQL culprits ŌĆö to a **DB table** until the operator stops it (`StopReason=Manual`) or an **auto-stop** fires (`Performance.RecordingMaxDurationMinutes` / `Performance.RecordingMaxSamples`, whichever first). One recording at a time. The telemetry entities `PerfRecording` / `PerfRecordingSample` live in `Infrastructure.Reporting` (next to `LicenseUsageSnapshot` ŌĆö **not** the adapter namespace `Infrastructure.Performance`, which the layer guard forbids Web from touching), so the endpoints read them through a vertical slice (`AppDbContext`); a sample keeps the level-1 host metrics in flat columns and the level-2 attribution (process families) + serialized culprits in JSON columns. On process restart the in-memory active state is lost, so any row left `Active` is reconciled to `Interrupted` at startup (best-effort, like the lost partial bucket of ADR-25). Endpoints under `/api/v1/performance/recordings`: `POST` (start) + `POST ŌĆ”/{id}/stop` + `DELETE ŌĆ”/{id}` are `Admin`; `GET` (list) + `GET ŌĆ”/{id}` (the sample series) are `Viewer`. There is **no always-on collection and no nightly retention job** (a recording is bounded by its own duration). Time flows through `TimeProvider` (deterministic auto-stop tests). Sampling spawns `rac.exe` + DMV reads **periodically while a recording runs** ŌĆö a deliberate operator-initiated cost, bounded by auto-stop, unlike the live pull that runs only while a tab is open (ADR-3.3 ┬½Spawn cadence┬╗).
- **Adapters (ADR-20).** Host metrics go through the Application port `IHostMetricsProbe` ŌåÆ `OneCHostMetricsProbe` (Infrastructure, Windows-only); 1C through the existing `IClusterClient`; SQL through `ISqlPerformanceProbe` ŌåÆ `SqlPerformanceProbe` (Infrastructure, plain ADO.NET ŌĆö **not** Windows-only, like `SqlDatabaseDiscovery`; no `[SupportedOSPlatform]`/`CA1416`). Web never touches WMI / `System.Diagnostics.Process` / SQL DMVs directly ŌĆö but the SQL endpoint may read its **own** `AppDbContext` for the databaseŌåÆtenant join (that is the panel's DB, not a 1C/IIS/SQL-discovery boundary). The SQL probe connects with the parameters inherited from `ConnectionStrings:Default` (same account as the panel, `InitialCatalog=master` ŌĆö DMVs are server-scoped), valid because the single-node topology co-locates the panel DB and the 1C infobase databases on one MSSQL instance; it needs **`VIEW SERVER STATE`** (`HAS_PERMS_BY_NAME` is checked first ŌĆö missing right ŌåÆ `Status=PermissionDenied`, an honest degraded signal, not an empty ┬½all clear┬╗; unreachable SQL ŌåÆ `Status=Unavailable`). 1C-originated SQL is identified by `program_name='1CV83 Server'`; attribution is **database-granular** (`DB_NAME ŌåÆ Infobase.DatabaseName ŌåÆ tenant`) ŌĆö a SQL request is **not** traced to a 1C session/user (all 1CŌåÆSQL traffic shares one `program_name`, MLC-063). The probe is a **singleton** that holds the previous snapshot (per-process CPU times, raw disk perf-counters) so it can compute the **CPU% delta** between polls (pattern `ColdThrottleState`/`IClusterUuidCache`); the **first poll reports `Measuring=true`** (delta-based metrics ŌĆö process CPU%, disk latency ŌĆö are not yet available). The SQL probe follows the same pattern ŌĆö `wait_stats` and IO-stall are cumulative since SQL startup, so it holds the previous read and reports `Measuring=true` on the first poll (the instantaneous active-requests list is always populated). Processes the backend cannot read for lack of OS rights are **counted** (`ProcessesInaccessible` ŌåÆ derived `AttributionIncomplete`), not swallowed, so the section flags incomplete attribution instead of silently mislabelling an under-privileged host's load as ┬½ą¤čĆąŠčćąĄąĄ┬╗ (honest-signal pattern, MLC-064a; see 04_INFRASTRUCTURE ┬¦6 and OPERATIONS).
- **Host metric source = WMI, not `PerformanceCounter`.** `OneCHostMetricsProbe` reads `Win32_PerfFormattedData_*` / `Win32_PerfRawData_*` / `Win32_ComputerSystem` via WQL. Performance-counter **category/counter names are localized on RU Windows** (`ą¤čĆąŠčåąĄčüčüąŠčĆ` / `% ąĘą░ą│čĆčāąČąĄąĮąĮąŠčüčéąĖ ą┐čĆąŠčåąĄčüčüąŠčĆą░`), so the English-name `PerformanceCounter` lookup throws ┬½category does not exist┬╗; WQL perf-class **property names are invariant** (verified on the RU stand). Disk latency (a `PERF_AVERAGE_TIMER`) is cooked by hand from raw counters because the formatted class truncates the sub-millisecond fraction to integer seconds. The processŌåÆfamily map is operator-tunable without redeploy via `Settings.Performance.ProcessFamilyMap` (pattern `OneC.LicenseConsumingAppIds`).
- **Perf events are measurements, not actions ŌåÆ own table, not `AuditLog`.** Recordings/samples persist to a dedicated telemetry table (precedent `LicenseUsageSnapshot`, ADR-25), and recording start/stop is **not** audited in v1 (operational tooling). A future audit need takes a fresh frozen `AuditActionType` in the `600+` series.
- **Rejected (gated on a trigger, per `MLC-063` reconnaissance):** an always-on "black box" of top-level metrics with retention (the Live+Recording split already covers the live question and the investigation question without a permanent write path); stitching a SQL query ŌåÆ 1C session ŌåÆ user (1C does not propagate a session-id to its SQL connections ŌĆö all 1CŌåÆSQL traffic shares `program_name='1CV83 Server'`; attribution works at **database** granularity `database_id ŌåÆ DB_NAME ŌåÆ Infobase.DatabaseName ŌåÆ tenant`, not down to the session ŌĆö empirically confirmed); remote collection from other cluster nodes (single-node topology, Locked Operational Constraints).
- **Reason:** the live question ("what is saturated right now, and is it 1C or not?") needs no history, so paying for an always-on store is waste; the investigation question ("capture what happens over the next N minutes") needs a bounded, restart-surviving record. The two modes answer both with the minimal persistence each requires. **Mirrors ADR-25** (which introduced the first telemetry table) ŌĆö this ADR is authored in the first backend task of the track (`MLC-064`), as ADR-25 was for ┬½ą×čéčćčæčéčŗ┬╗.

## 27. On-demand SQL database backup (operator-initiated, `COPY_ONLY`)
- **Decision:** The panel offers a **per-database, on-demand full backup** triggered from an infobase card ŌĆö a thin convenience over `BACKUP DATABASE ŌĆ” WITH COPY_ONLY, COMPRESSION, CHECKSUM, FORMAT, INIT`. This **amends ADR-15** (which kept all backup out of scope): on-demand is in; scheduled orchestration and restore stay out. All infobase databases run SIMPLE recovery, so only a FULL backup is meaningful.
  - **`COPY_ONLY` is load-bearing.** A normal FULL resets the differential base (the DCM) and would silently break an operator's external differential chain; `COPY_ONLY` is fully restorable yet leaves the DCM untouched, so it is safe whether or not a server runs differentials. Consequently there is **no per-server "diff vs full" flag** ŌĆö one uniform path. (Deployments differ in their *external* backup strategy across installs, but each install is one MSSQL instance ŌĆö see Deployment topology.)
  - **Adapter (ADR-20).** SQL work goes through the Application port `ISqlBackupService` ŌåÆ `SqlBackupAdapter` (Infrastructure, plain ADO.NET ŌĆö **not** Windows-only, like `SqlDatabaseDiscovery`/`SqlPerformanceProbe`). Web never issues `BACKUP` / `xp_*` directly. The adapter connects with the parameters inherited from `ConnectionStrings:Default` (`InitialCatalog=master`), valid under the single-node topology (one panel = one MSSQL instance hosting the 1C infobase databases). It **never throws** for infrastructure failure ŌĆö it returns a typed degraded result (`InsufficientSpace` / `EstimateUnavailable` / `PermissionDenied` / `BackupFailed`), the honest-signal pattern (MLC-064a/068).
  - **Orchestration.** A process-wide singleton `IBackupOrchestrator` + a background `BackupPumpService` (`BackgroundService`) drive the queue, which **is** the `DatabaseBackups` table ŌĆö a telemetry-style entity in `Infrastructure.Reporting` (next to `PerfRecording`/`LicenseUsageSnapshot`, **not** the adapter namespace `Infrastructure.Backups` the layer guard forbids Web to touch). Each pump tick re-reads `Backup.MaxParallel` and, while `running < max`, starts the oldest `Queued` row whose `(server, database)` is not already running ŌĆö enforcing **per-database exclusion**, the **per-server parallel cap**, and **FIFO** in one pass. The long `BACKUP` runs *outside* the gate (backups are concurrent by design; the gate guards only start/finish bookkeeping). `RequestAsync` returns `AlreadyActive` (ŌåÆ `409 BACKUP_ACTIVE`) when that database already has a `Queued`/`Running` row. On process restart, orphaned `Running` rows reconcile to `Failed`/`Interrupted` (the `.bak` may be partial; timestamped filenames + `RESTORE VERIFYONLY`-before-delete keep a partial from being mistaken for good ŌĆö like the lost partial bucket of ADR-25). **Not Hangfire** (no native per-key mutex / dynamic worker cap; Hangfire stays for recurring CRON only) and **not a fixed `SemaphoreSlim`** (cannot honor a dynamic cap, no FIFO, no per-db exclusion). Time flows through `TimeProvider` (deterministic tests).
  - **Storage & retention.** A single configured root (`Backup.FolderPath`, a **local** drive on the SQL host) with a per-database subfolder. **Keep-latest-one per database**, replaced safely as *create new ŌåÆ `RESTORE VERIFYONLY` ŌåÆ then delete the prior file* ŌĆö **never** delete-then-backup (a failed new backup must not leave zero backups). A recurring `BackupRetentionJob` (CRON, pattern of `AuditRetentionJob` / ADR-25) deletes any `.bak` older than `Backup.TtlHours` (default 24) and reaps stale rows. File/dir/free-space operations run **server-side** via `xp_create_subdir` / `xp_fixeddrives` / `xp_delete_file` ŌĆö the SQL service account both writes and deletes, so there is no cross-identity ACL split (and it stays correct even if the folder were remote from the app host).
  - **Disk guard.** Before `BACKUP`, the adapter estimates size = used ROWS pages (`SUM(FILEPROPERTY(name,'SpaceUsed'))`) and requires free space Ōēź estimate + `Backup.DiskSafetyMarginMb`. The used-data figure is a deliberate **over-estimate** (compression shrinks the real `.bak`). **If the estimate cannot be obtained, the backup does not start** (`EstimateUnavailable`) ŌĆö there is no fixed GB floor.
  - **Roles & audit.** Triggering a backup = `Viewer` (operators); deleting a backup and changing any `Backup.*` setting = `Admin`. New frozen `AuditActionType` slots in the 500 maintenance series: `BackupRequested=510`, `BackupSucceeded=511`, `BackupFailed=512`, `BackupDeleted=513`, `BackupsPurged=514`.
- **Permissions.** The panel's SQL login is granted **sysadmin**: `BACKUP DATABASE` needs `db_backupoperator`, but `xp_create_subdir` / `xp_fixeddrives` / `xp_delete_file` need sysadmin. Missing rights ŌåÆ `PermissionDenied` degraded result, not a 500. See `OPERATIONS.md`.
- **Rejected:** scheduling / always-on backups (stays ADR-15 ŌĆö the operator's platform tooling owns the *strategy*); a normal (non-`COPY_ONLY`) FULL (would break external differential chains); restore from the panel (out of scope ŌĆö the operator restores via SSMS); a per-server diff/full flag (`COPY_ONLY` makes it unnecessary); a fixed free-space floor (the per-database estimate is sharper; no estimate ŌåÆ refuse rather than guess).
- **Reason:** operators wanted a one-click full backup of a client base without disturbing their existing (often differential) backup strategy. `COPY_ONLY` + on-demand-only + no-restore keeps the new surface narrow and ADR-15's discipline intact, while the queue / cap / keep-latest / disk-guard make it safe to run against production. Authored as the track's foundation (`MLC-075`), mirroring how ADR-25/26 front-loaded their tracks' decisions.

## 28. Single-host topology ŌĆö one host, one 1C cluster, one SQL instance
- **Decision:** The deployment is **single-host by design and confirmed final** (operator decision, 2026-06-10): one Windows Server host runs the .NET backend, IIS, the MSSQL instance, and the 1C cluster together; there is exactly **one 1C cluster** and **one SQL instance** hosting all client databases. This formalizes the long-standing ┬½Deployment topology ŌĆö single-node┬╗ Locked Operational Constraint (below) and removes the generic multi-server scaffolding the panel carried ┬½just in case┬╗. The cleaned-up rudiments (UX-redesign stage 2, `MLC-087ŌĆ”091`):
  - **One SQL instance, one setting.** The per-base `Infobase.DatabaseServer` column is **dropped** (MLC-088); the SQL instance lives only in `Settings.Sql.Server` (renamed from `Defaults.DatabaseServer`, MLC-087). It is no longer a UI-only form-prefill default but the **single runtime-resolved source**: `SqlDatabaseDiscovery` (database-name discovery) and `BackupsEndpoints` (backup enqueue) read it; an empty value surfaces as `Available:false` on the database picker and `409 SQL_SERVER_NOT_CONFIGURED` on backup.
  - **No server picker step.** `GET /discovery/databases` takes **no `server=` parameter** (MLC-087) ŌĆö it resolves the instance from `Sql.Server`. `GET /discovery/sql-instances` stays (it feeds the ┬½SQL Server┬╗ picker on ┬½ą¤ą░čĆą░ą╝ąĄčéčĆčŗ┬╗, not a per-base choice).
  - **One cluster, one address.** The `OneC.Cluster.Server` setting is **removed** (MLC-089); the `webinst -connstr "Srvr=ŌĆ”"` cluster address is derived from the host of `OneC.RAS.Endpoint` (the cluster and RAS share the host). `ResolveClusterServer` no longer has a ┬½cluster lives elsewhere than RAS┬╗ fallback.
- **Relation to ADR-17.** ADR-17 made the SQL-server / IIS-site / platform-version fields **UI-only form-prefill** and **Locked** that "promoting one to a runtime-resolved global requires revoking ADR-17 first." The confirmed single-host decision is exactly that revision trigger **for the SQL-server field only**: it is now a runtime-resolved global (`Sql.Server`), so ADR-28 **supersedes ADR-17 for that field**. The other two prefill keys (`IIS.DefaultSiteName`, `OneC.DefaultPlatformVersion`) stay UI-only per ADR-17 ŌĆö platform version legitimately varies per base on one host, and the IIS site is a per-publication value. **Relation to ADR-4.** ADR-4's webinst publication is unchanged; only the `Srvr=` source in the connection string moved from a dedicated key to the RAS-endpoint host (above).
- **Rejected:** keeping the generic multi-server scaffolding (a per-base `DatabaseServer` column, a `server=` discovery parameter, a separate `OneC.Cluster.Server` key) "in case the topology ever splits" ŌĆö it modeled a multi-host deployment that has never existed in the field, every value was identical across rows, and it cost real UX friction (a "pick a server first" step, a server column, a redundant settings key) and doc-canon churn for zero realized flexibility. Re-deriving the SQL server from `ConnectionStrings:Default` instead of a setting was also rejected (it would erase the ┬½SQL Server┬╗ section on ┬½ą¤ą░čĆą░ą╝ąĄčéčĆčŗ┬╗ and the discovery picker ŌĆö MLC-083/087); the single setting stays the source of truth.
- **Reason:** reality discipline ŌĆö the panel administers one host's 1C licensing; modeling N hosts it never has is speculative generality. One setting per concern (`Sql.Server`, `OneC.RAS.Endpoint`) is simpler to operate and matches the deployment. **Locked:** re-introducing any multi-host capability ŌĆö a SQL instance separate from the panel host, a second SQL instance for client databases, or a multi-node 1C cluster ŌĆö requires **revoking ADR-28 first**, and triggers the full infrastructure-adapter re-review the ┬½Deployment topology┬╗ Locked Constraint already mandates (per-base server resolution, distributed locks, remote IIS/SQL adapters).

## 29. Discovery-first infobase adding + cluster-base ignore-list
- **Decision:** Cluster bases present in the 1C cluster but **not** entered in the panel are made **visible and actionable** instead of silently invisible. ## 31. GUI installer (Inno Setup) for the single host
- **Decision:** The panel ships a **GUI installer** (`.exe` wizard, **Inno Setup 6**, Unicode) that installs and upgrades the single-host deployment without hand-wiring IIS. It installs the self-contained publish artefact (ADR-30 / MLC-099 ŌĆö `MitLicenseCenter.Web.exe` with the .NET 10 runtime embedded + `wwwroot`), registers and starts **one Windows service** (`MitLicenseCenter`) running the backend that serves both the API and the SPA same-origin (ADR-30), opens the chosen LAN port in the firewall, and writes a working `appsettings.Production.json` **from interactive wizard input** (MLC-101). The installer source is `installer/MitLicenseCenter.iss`; it is built by `scripts/build-installer.ps1` (publish ŌåÆ ISCC). **Inno Setup is a build-machine precondition** (`winget install JRSoftware.InnoSetup`), not a runtime dependency.
  - **Interactive wizard collects credentials + network (MLC-101).** Custom `[Code]` pages gather the SQL instance (default `.`) and database (default `MitLicenseCenter`), the **authentication mode**, the matching credentials, and the network port (default `8080`) + `AllowedHosts` (default `*`). ## Locked Operational Constraints (binding)
- **Kill priority:** when `Consumed > Limit`, kill sessions ordered by `StartedAt DESC` (newest first) until `Consumed == Limit`. Simplest to explain to users and avoids interrupting established sessions.
- **Kill grace period:** a session younger than the kill grace period (`Enforcement.KillGraceSeconds`, default 15 s, range 5ŌĆō120, measured against the snapshot `StartedAt`) is **not** eligible for a limit-kill. 1C only fills `user-name` after authentication, and newest-first aims squarely at the just-connected session ŌĆö without the grace we would systematically terminate sessions in the "connected, holding a license, not yet logged in" window and record an empty user in the audit. Because candidates are newest-first, the youngest is first: hitting one inside the grace **stops** this tenant's cycle (waiting) rather than falling through to kill an older established session. The over-limit state is tolerated for Ōēż grace; the next hot tick (~4ŌĆō5 s) re-evaluates and kills the now-aged session with its `user-name` resolved. An empty `user-name` that survives the grace (genuinely user-less base infobase) renders as ┬½ą▒ąĄąĘ ą┐ąŠą╗čīąĘąŠą▓ą░čéąĄą╗čÅ┬╗ in the audit text ŌĆö see `SessionDisplay`.
- **Idempotent kill protocol:** before a kill, re-fetch the session by ID and verify `(InfobaseId, AppID, StartedAt)` match the snapshot; mismatch ŌåÆ skip and wait for the next cycle; "session not found" ŌåÆ treated as a successful kill. Every kill writes an `AuditLog` entry with reason and snapshot context. **Audit on real outcome only:** the `AuditLog` row is written **iff** `KillSessionResult` is `Killed` **or** `AlreadyGone` (the idempotent "already terminated" success). When RAS is unreachable / `rac.exe` errors (both flags `false`) **nothing is killed and nothing is audited** ŌĆö the immutable log must never record a kill that did not happen. Both paths obey this: the automatic `KillEnforcer` (reason `LimitExceeded`, initiator `System`) logs-and-skips the failed session for the next cycle, and the manual `POST /api/v1/sessions/{id}/kill` (reason `ManualByAdmin`, initiator = admin) returns an error to the operator instead of a false `204` ŌĆö `409 SESSION_STALE` on descriptor mismatch (refresh and retry), `502 CLUSTER_UNAVAILABLE` when RAS is down. `404` still means the session was not in the current snapshot at all.
- **Publication status:** a Hangfire job refreshes publication status every 5 minutes (throttled to `Drift.IntervalMinutes`, **read-only**); `POST /api/v1/publications/{id}/check` refreshes one on demand. There is no auto-correction and no desired-state enforcement ŌĆö (re)publication via `webinst` (`POST /publications/{id}/publish`) and platform change via `web.config` (`POST /publications/{id}/change-platform`) are explicit, audited operator actions.
- **Deployment topology:** single-node ŌĆö backend, IIS, MSSQL and the 1C cluster all on the same Windows Server host. No remoting, no WinRM, no cross-host adapters. Changing this assumption requires re-reviewing every infrastructure adapter.

Supported install path on Windows is the standalone winget package (`winget install pnpm.pnpm`) ŌĆö Corepack works on Linux/macOS or an elevated Windows shell but can't write to `C:\Program Files\nodejs\` for a non-admin user.
- **Node.js minimum:** Node 22.13+ (pnpm 11 fails on older with `No such built-in module: node:sqlite`).
- **Pre-commit hooks:** a custom `.husky/pre-commit` registered via `git config core.hooksPath .husky` (set idempotently by `frontend/scripts/install-git-hooks.mjs`). The `husky` npm package is intentionally not installed. The hook runs `lint-staged` on staged JS/TS/JSON/CSS and `dotnet format --verify-no-changes` on staged `.cs`; requires Git Bash on Windows.
- **PowerShell script encoding:** all `scripts/*.ps1` are saved as **UTF-8 with BOM** ŌĆö Windows PowerShell 5.1 reads BOM-less UTF-8 as cp1251 and corrupts Cyrillic.
- **shadcn on Windows:** add components via `scripts/shadcn-add.ps1` (workaround for `pnpm dlx shadcn` ŌĆö uses `pnpm add --ignore-scripts --config.node-linker=hoisted` in a temp folder).
- **Dev scripts:** `build.ps1` (the "is the project healthy?" gate ŌĆö backend build/test/format + frontend lint/type-check/**test**/build, so frontend tests are part of the full check; each native step passes or fails strictly by `$LASTEXITCODE`, so a tool writing a banner to stderr under Windows PowerShell 5.1 ŌĆö e.g. pnpm's `$ eslint .` ŌĆö never becomes a false failure, even when output is captured to a log via `*>&1`/`Tee-Object`; `db-reset.ps1` guards its `dotnet ef` step the same way), `dev.ps1` (backend `dotnet watch` + frontend `pnpm dev` in parallel), `db-reset.ps1` (drop + migrate + seed admin).
