# UI & Frontend Requirements

This document outlines the architecture and key views for the web-based administrative Control Panel. 
The UI is strictly separated from the infrastructure logic and communicates ONLY with the Backend via REST API.

## Language & Localization

The Control Panel is delivered in **Russian** as the only shipping locale of v1 (target audience — Russian-speaking 1C hosting administrators; see `01_PROJECT_CONTEXT.md`).

- All visible strings — labels, buttons, table headers, tooltips, validation, toasts, dialogs, empty states, error pages — are in Russian.
- Strings are stored in a single `src/i18n/ru.json` and accessed via `react-i18next`. User-facing copy is routed through i18n rather than hardcoded in JSX, so a future locale can slot in without component edits. (One deliberate exception: the product name "MitLicense Center" is a literal in the topbar, not a translation key.)
- Locale: `ru-RU`. Absolute dates/times are formatted with **`date-fns`** + `date-fns/locale/ru` (see the stack table in `06_UI_DESIGN.md`); relative "N сек назад" freshness uses `Intl.RelativeTimeFormat('ru')`. Use these existing helpers — do not roll a custom formatter.
- Date input controls accept and display `ДД.ММ.ГГГГ`; under the hood always serialize as ISO-8601 UTC.
- The backend's user-facing error payloads are already in Russian (per `01_PROJECT_CONTEXT.md`) — the SPA does not translate them, just displays them.

## 1. Frontend Technology Stack
- **Framework:** React (Single Page Application).
- **Language:** TypeScript.
- **State & Data Fetching:** React Query (or similar tool) for caching, background refetching, and polling.
- **Component library:** **shadcn/ui** on top of **Radix UI**, styled with **Tailwind CSS** (see ADR-11). Mixing in other component libraries (MUI, Ant Design, etc.) is not allowed — if a control is missing, build it on top of Radix + Tailwind in the same style. Detailed visual language, status semantics, table patterns, and Russian microcopy are defined in `06_UI_DESIGN.md`.
- **Constraint:** Desktop applications, ASP.NET MVC, and Blazor MUST NOT be used.

## 2. UI Architecture & Communication
- The React application is completely decoupled from the .NET backend.
- It consumes a unified REST API provided by the Backend, versioned at `/api/v1/...` (see ADR-10). OpenAPI/Swagger UI is exposed at `/api/docs`; the TypeScript API types are hand-written in `frontend/src/features/<feature>/types.ts` and kept in sync with that spec by hand (see ADR-10.1) — there is no codegen step. On a few **critical response boundaries** (`/auth/me` + `/auth/login` — role gating; `/sessions/snapshot`; the paginated list envelopes for tenants and infobases) the response is additionally validated at runtime with a Zod schema, and the TS type is derived from that schema (`z.infer`) so there is a single source of truth; a contract mismatch there raises a controlled `ApiSchemaError` rather than a silent wrong type (MLC-016). All other endpoints keep the plain `api<T>()` cast.
- **Polling:** For live monitoring (like active sessions), the UI will poll specific "Snapshot" endpoints periodically (e.g., every 15 seconds) rather than maintaining complex WebSocket connections, keeping the architecture simple and scalable.
- **Authentication:** Cookie-based session auth via ASP.NET Core Identity (see ADR-7). The SPA hits `POST /api/v1/auth/login` with username/password, the server sets an HttpOnly cookie, and subsequent requests are authenticated automatically. Logout clears the cookie. Unauthenticated requests to protected endpoints return `401` and the SPA redirects to the login screen. The `Viewer` role hides destructive UI controls (kill, publish, change platform, edit) and the backend additionally enforces role checks server-side.
- **User menu (topbar):** the signed-in account lives in a topbar dropdown — login + role (the only place the user sees their own role), **«Смена пароля»** (a dialog reusing the same `ChangePasswordForm` as the forced-change screen, §3.7) and **«Выйти»**. There is no separate profile page or `/profile` route.

## 3. Key Views / Pages

### 3.1. Dashboard («Обзор»)
The landing page (`/`) is a true overview — licenses **and** host health — and every block links into its section.
- **KPI cards** (five, clickable): «Клиенты» (active of total) → `/tenants`, «Инфобазы» → `/infobases`, «Активные сеансы» → `/sessions`, «Использовано лицензий» and «Свободно лицензий» (same reporting axis) → `/reports`. The summary data (`GET /api/v1/dashboard/summary`) polls every 5 s.
- **System row:** the **RAS adapter health card** (rac.exe reachable + last error + consecutive-failures count, backed by `RasHealthProbingService`) next to a compact **host-health card** — three simplified gauges (CPU / RAM / disk latency) reusing the `MetricGauge` primitive and the saturation thresholds of `/performance` (§3.8), without the detail sub-lines. The whole card links to `/performance`. The split of responsibility is deliberate: the dashboard answers «есть ли проблема», the Performance section answers «какая и из-за кого» — per-family attribution is **not** shown here.
- The host card is fed by the same `GET /api/v1/performance/host` (same Zod-validated boundary), but polled every **45 s** — an overview does not need the 5-second live cadence of `/performance`. On the first probe (`Measuring=true`) the CPU and disk gauges show «измеряю…» instead of drawing zeros; RAM (instantaneous) renders at once. The dashboard and `/performance` share the React Query cache key, so navigating between them shows the last snapshot instantly.
- **Top tenants by consumption** — consumption bars per client; each tenant name links to its detail page (`/tenants/:id`).

### 3.2. Tenants Management
- List of all Clients. Each row shows a **«Базы»** count (`InfobaseCount` from the list API) which is itself a **link to the filtered infobase list** (`/infobases?tenantId=…`); the client **name is a link** to the tenant detail page (`/tenants/:id`) — navigation is an explicit affordance, not a whole-row click (per the table convention in `06_UI_DESIGN.md`).
- Form to create/edit a Tenant (Name, `MaxConcurrentLicenses`, Status).
- **Tenant detail page (`/tenants/:id`)** — the per-client lens: header with client name, status and license limit, a **«Добавить базу»** button (opens the infobase form with the client pre-selected), and a table of that client's infobases (no «Клиент» column). Edit / delete / **«Перенести в другого клиента»** act per row.

### 3.3. Infobases & Publications («Базы»)
A single **«Базы»** page (`/infobases`) carries both the infobases and their IIS publications — a publication is a mandatory 1:1 part of an infobase (created in the same POST, cascade-deleted; `03_DOMAIN_MODEL.md`), so for the operator they are one object in one place. There is no separate «Публикации» page. The page has two tabs: **«Базы»** (the table below) and **«IIS»** (the IIS lifecycle card).
- **Table:** a flat paginated list with a «Клиент» column and a tenant filter. The filter is serialized to the URL (`?tenantId=`), so a filtered list is linkable — the «Базы» count in the tenants table lands here pre-filtered. There is **no grouped «По клиенту» mode**: the per-client lens is the tenant detail page (`/tenants/:id`).
- **Columns:** Название · Клиент · Статус базы · Публикация (publish-status badge + a provenance **icon** with tooltip — `webinst` / «Конфигуратор» / unknown) · Версия платформы · Проверено · actions. Publication operations live in the per-row action menu and the bulk bar (below); raw IDs (cluster GUID, database name) live in the edit form, not the list.
- Assignment interface: Attach an Infobase to a specific Tenant.
- **Reassignment:** the per-row **«Перенести в другого клиента»** action moves a base to another tenant (`POST /infobases/{id}/reassign`). The client field in the edit form stays locked — moving ownership is this dedicated action only. A name collision in the target client surfaces as a clear error (`409 INFOBASE_NAME_TAKEN_IN_TARGET`).
- **Add/Edit form is intentionally minimal.** The always-visible part has three fields only: **Клиент** (tenant), **База в кластере 1С** (the cluster infobase picked by name — its GUID is hidden, and picking it auto-fills the display name; the picker lists every cluster base and the form checks the picked one's availability up-front via `GET /api/v1/infobases/cluster-id-availability`, flagging «уже привязана к клиенту …» without loading the whole infobase list), and **Имя базы данных (SQL Server)** — a picker over the databases of the configured SQL instance (`GET /discovery/databases`), working immediately (no «pick a server first» step). Everything that is normally identical across all bases lives in a collapsed **«Дополнительно»** disclosure: display-name override, IIS site, virtual path, platform version, physical-path override, status.
- **There is no SQL-server field anywhere in the form.** The single SQL instance is set once on `/settings` («SQL Server» section, below); the form supplies the value invisibly — on **create** from the `Defaults.DatabaseServer` setting, on **edit** the base's current stored value (editing a base never silently migrates its server). If the setting is empty, submit is blocked with a toast pointing the operator at «Параметры».
- The «Дополнительно» fields are pre-filled from settings (`IIS.DefaultSiteName`, `OneC.DefaultPlatformVersion` — see `04_INFRASTRUCTURE.md`). The virtual path is derived from the database name as `"/" + slug(databaseName)`, and the physical path as `{IIS.DefaultVrdRoot}\{databaseName}` (default root `C:\inetpub\wwwroot`, e.g. `C:\inetpub\wwwroot\acme_bp`) — no IIS-site segment. Both are pre-filled as real values (not just placeholders) and stay editable; editing one stops it auto-updating. The disclosure auto-expands when a validation error lands on one of its fields.
- Inside «Дополнительно» the fields are grouped under subsystem sub-headings: **«Инфобаза»** (display name, status) and **«Публикация в IIS»** (site, virtual path, platform version, physical path). The Settings page (`/settings`) mirrors this subsystem grouping (MLC-055/083): **«Подключение к 1С / RAS»** (cluster credentials, RAS **port**, and a single **«Платформа 1С»** picker that sets both the rac.exe path and the default platform version), **«SQL Server»** (`Defaults.DatabaseServer` — the **single place** where the SQL instance hosting client databases is set, a picker over the locally-discovered MSSQL instances `GET /discovery/sql-instances` with a manual fallback; the infobase form and its database-name picker consume the value automatically; the setting key keeps its historical name until the stage-2 backend cleanup), **«Учёт лицензий»** (the license-consuming app-id whitelist), **«Публикации IIS»** (`IIS.DefaultVrdRoot` + `IIS.DefaultSiteName`), **«Опрос 1С и фоновые проверки»**, **«Хранение данных»** (audit + license-usage retention windows), and **«Резервные копии баз»** (the four `Backup.*` keys — folder path, file TTL, parallelism cap, disk safety margin; see 3.9). There is no «Значения по умолчанию для новых баз» section.
- **Publication Settings:** View/Edit publication parameters (Site Name, Virtual Path, 1C Platform Version, physical-path override). OData / HTTP-services / custom VRD are **not** panel-managed (ADR-4) — configured in Designer / a `-descriptor` template.
- **Status:** the «Публикация» column shows `LastCheckStatus` (`Published` / `NotPublished` / `Error` / `Unknown`) as a badge, `LastCheckDetails` on hover, and provenance (`webinst` / `Конфигуратор` / unknown) as an adjacent icon with a tooltip (provenance gates the overwrite warning on publish); «Проверено» shows `LastCheckAt`. Updated by the background status-refresh job (every 5 min) plus an on-demand **«Проверить публикацию»** row action (`POST /api/v1/publications/{id}/check`, read-only).
- **«Опубликовать»** — runs `webinst` (`POST /api/v1/publications/{id}/publish`). Confirmation dialog (type the site+path token); if the publication's source is not `Webinst` it warns that manual config will be overwritten. Audited as `PublicationPublished`.
- **«Сменить платформу»** — picks an installed version and rewrites the `wsisapi.dll` path in `web.config` (`POST /api/v1/publications/{id}/change-platform`, `default.vrd` untouched). Audited as `PublicationPlatformChanged`. Both actions are explicit, admin-only — nothing auto-corrects.
- **Bulk operations (admin, MLC-046):** a leading checkbox column (header = select-all of the current page; the selection itself survives paging and filter changes) drives a multi-selection; a bulk action bar then offers **«Опубликовать выбранные»** and **«Сменить платформу выбранным»**. A batch runs as N idempotent single calls to the same per-publication endpoints through a small client pool (no bulk endpoint), with a live progress dialog (per-publication ожидание/выполняется/готово/ошибка + a final «успешно / с ошибкой / пропущено» summary, cancellable — in-flight items finish, the rest are skipped). Publish shows the overwrite warning **once** for the whole selection, listing the affected non-`Webinst`, currently-published publications. Change-platform picks **one** target version (installed) applied to all selected. Each publication writes its own audit (`PublicationPublished` / `PublicationPlatformChanged`); successful items leave the selection so a failed subset can be retried. The run is bound to the open tab.

- **IIS lifecycle tab (admin, MLC-047):** the **«IIS»** tab of the «Базы» page hosts the **«Управление IIS»** card managing the IIS hosting lifecycle (server-scope, not per-row). It lists application pools and sites discovered from the live IIS, each with a state badge (`Запущен` / `Остановлен` / …) and actions — pool **«Перезапустить пул» (recycle)** / **«Запустить»** / **«Остановить»**, site **«Перезапустить»** / **«Запустить»** / **«Остановить»** — plus, in the card header, the **overall IIS status badge** (read from the `W3SVC` service) and server-wide buttons: **«Перезапустить IIS»** (`iisreset`, neutral) and a single **stop/start toggle** — **«Остановить IIS»** (`iisreset /stop`, red) when IIS is running, **«Запустить IIS»** (`iisreset /start`, green) when it is stopped (same one-button pattern as pools/sites; stop is red, start is green throughout). Viewer sees state read-only; only Admin gets the action buttons. Destructive actions (recycle, stop, restart, `iisreset` restart/stop) ask for confirmation via a plain `AlertDialog` (no typed token — see 06_UI_DESIGN §7); the server-side `Confirm` gate is the backstop. `start` is one-click recovery. Each operation is audited (`IisApplicationPoolRecycled`…`IisStarted`). When IIS is unreachable the card shows the discovery error instead of the lists.

### 3.4. Active Sessions Monitor (The Kill Switch)
- A combined table displaying current `ActiveSessionSnapshot` data.
- **Columns:** Tenant Name, Infobase Name, Session ID, AppID, Duration, `ConsumesLicense` flag.
- **Actions:** Administrators can manually select a session and click "Terminate", which triggers an API call to the backend to kill the session via `rac.exe session terminate`.

### 3.5. Audit Logs
- A read-only, filterable table displaying the `AuditLog` entity data.
- Shows who (or what background job) killed a session, updated a publication, or changed a limit.
- Filters: by `ActionType`, `Initiator`, `TenantId`, date range. For `SessionKilled` rows, the `Reason` column distinguishes `LimitExceeded` from `ManualByAdmin`.

### 3.6. License-Usage Reports
- Route `/reports` (sidebar group «Мониторинг», `Viewer`-readable — not admin-only), backed by the read-only Reports API (`GET /api/v1/reports/license-usage` and `/{tenantId}`, see `03_DOMAIN_MODEL.md`).
- **Period filter** (`from`/`to` date inputs, shared by both sections), plus a **month quick-pick** («Месяц» — an `<input type="month">` flanked by ‹/› step buttons) that fills the same `from`/`to` with the chosen month's bounds (a whole month is always < 31 days, so it never trips the clamp). Omitted bounds default to the last 7 days; a span wider than 31 days is clamped server-side, and the **effective** range echoed in the response (`fromUtc`/`toUtc`) is shown above the summary chart — not the requested one. When the response is `clamped`, a **truncation notice** under the filter states that the requested period exceeded the maximum and only the last N days are shown.
- **Summary section** (all clients): a time-series chart of peak consumption (`consumedMax`, filled area), average consumption (`consumedAvg`, line) and the licence limit (`limit`, dashed ceiling) over 15-minute buckets, plus textual stats (period peak with share-of-limit and moment, period average). A **visible caveat** notes that the per-bucket sum is an overview figure, not a true simultaneous platform peak (tenants peak at different moments — the `MLC-049` decision).
- **Drill-down block** (separate, below the summary): a client selector renders the same chart for one tenant's stored series with its own peak/average stats.
- **Empty-state «данные накапливаются»**: telemetry accrues only since `MLC-048`, so an empty series (a `200` with `buckets: []`, never an error) renders an explanatory empty-state instead of a chart — both in the summary and the drill-down.
- **Export** («Скачать» menu in each section header — the summary and the drill-down export **separately**, hidden while the series is empty). Formats: **CSV** (UTF-8 **with BOM**, `;` separator, decimal comma — RU-Excel-friendly; plain bucket table), **XLSX** (two sheets — «Сводка» metadata + «Данные» with numeric cells so Excel pivots/charts work), **HTML** (a self-contained offline file reproducing the panel's interactive chart — the Chart.js engine is inlined into the document, no CDN; opens in any browser with working tooltips, plus the summary stats) and **PDF** (title + summary + a rendered chart image; a Cyrillic-capable font — Roboto Regular, Apache-2.0 — is embedded, since jsPDF's built-in fonts lack Cyrillic). The HTML and PDF are presentation exports — they carry the summary (period/peak/average/caveat) and the chart, **not** the raw per-bucket table (which would be thousands of rows on long periods); the raw bucket table lives only in the CSV/XLSX exports. Export is **fully client-side** — the already-loaded `LicenseUsageSeriesResponse` is serialized in the browser, no export endpoint; the heavy libraries (`xlsx`, `chart.js`, `jspdf`) are `dynamic import`-ed on click and split into per-format lazy chunks. Lives in `features/reports/export/`.
- First use of `recharts` in the SPA (chart components live in `features/reports/`).

### 3.7. Users
Admin-only route `/users` (sidebar group «Система», `UsersRound` icon, gated on the `Admin` role) manages the panel's accounts from the UI instead of the console `reset-admin` utility. Backed by `/api/v1/users` over the existing ASP.NET Identity (no schema change — `AppUser` is a plain `IdentityUser<Guid>`):
- **List** (`GET /users`, `Viewer`-readable) — every panel account with login, role(s) (`Admin` / `Viewer`), a status badge (**Активен** / **Отключён**) and a **last-login** timestamp (`—` if the account has never signed in).
- **Create** (`POST /users`, `Admin`) — login + a role chosen at creation (radio **Администратор** / **Наблюдатель**). The server generates a temporary password and returns it; the UI shows it **once** in a dialog with a copy button (it is never stored or shown again). The new account is flagged **must-change-password**, so the recipient is forced to set a permanent password on first login (see below). Duplicate login → `409 USER_USERNAME_DUPLICATE` on the login field.
- **Reset password** (`POST /users/{id}/reset-password`, `Admin`) — generates a new temporary password (shown once, same dialog); the old one stops working immediately and the account is re-flagged **must-change-password**.
- **Disable / enable** (`POST /users/{id}/disable` · `/enable`, `Admin`) — "disable" is an Identity lockout (`LockoutEnd = MaxValue`) that blocks login; "enable" lifts it and resets the failed-attempt counter. Two guards return `409` with a clear toast: you cannot disable **your own** account (`USER_CANNOT_DISABLE_SELF`), nor the **last active `Admin`** (`USER_LAST_ACTIVE`) — the count is of `Admin`-role accounts only, so a lone `Viewer` can never leave the panel unmanageable.
- **Change role** (`POST /users/{id}/role`, `Admin`) — switches an existing account between `Admin` and `Viewer` (radio dialog; body `{ role }`, validated `role ∈ Roles.All`; setting the current role is idempotent, no audit). Two guards return `409` with a clear toast: you cannot change **your own** role (`USER_CANNOT_CHANGE_OWN_ROLE` — self-demotion would lock you out), nor demote the **last active `Admin`** (reuses the same `USER_LAST_ACTIVE` last-active-Admin check as disable). The new role takes effect on the target user's **next login** — the role is baked into their cookie until they re-authenticate; the dialog says so.
- **Force password change on first login** — an account flagged `MustChangePassword` (after create or reset) lands on a blocking screen on next login: until it sets a permanent password it cannot reach any other page (the sidebar/app is replaced entirely); only sign-out is allowed. The change goes through the same `POST /auth/change-password` (the user knows the temporary password just used to sign in), which clears the flag. `GET /auth/me` and `POST /auth/login` return the flag so the SPA can gate.
- All mutations are written to `AuditLog` (`UserCreated` / `UserDisabled` / `UserPasswordReset` / `UserEnabled` / `UserRoleChanged`, `tenantId: null`). The generated password is **never** written to the audit log — only the action and who/whom.

The section name is «Пользователи» (renamed from «Администраторы» in MLC-060) because it manages both `Admin` and `Viewer` accounts.

### 3.8. Performance («Быстродействие»)
Route `/performance` (sidebar group «Мониторинг», `Activity` icon, `Viewer`-readable — not admin-only), the live operational answer to «почему 1С тормозит». Backed by the read-only host probe (`GET /api/v1/performance/host`, MLC-064 / ADR-26). The collection model is **live pull-on-demand**: while the tab is open the SPA polls every 5 s (`refetchInterval`, `placeholderData: prev` — the previous snapshot stays on screen between polls, never blanking); nothing is persisted, closing the tab stops collection (ADR-26). The response is runtime-validated by a Zod schema — a critical boundary (ADR-10.1 / MLC-016), since the delta metrics feed the verdict. The start screen is a «сейчас» snapshot with three stacked blocks:
- **Verdict banner** — the section's headline answer. Green «Ресурсы в норме» when no resource is saturated; amber/red «Высокая нагрузка: {ресурс}. Основной потребитель — {семья}» naming the bottleneck resource and its dominant process family (for CPU/RAM; disk names no family). On the first poll it reads «Идёт первичное измерение…» (see `Measuring` below).
- **«Светофор ресурсов»** — three gauges (CPU, RAM, Disk) built on the linear `Progress` primitive (no radial gauge in the kit). State (OK/Warn/Crit) is driven by **saturation, not raw utilization** (ADR-26 methodology): CPU by **processor queue length** (+ %), RAM by **paging** (Pages/sec) alongside used %, Disk by **latency** (avg sec/op, worst of read/write) shown in ms. The gauge bar and value are tinted by the worse of the saturation signal and the utilization band (emerald/amber/red — 06_UI_DESIGN §3); a sub-line shows the saturation signal (queue / free GB + paging / queue).
- **«Кто потребляет»** — attribution of CPU and RAM by **process family**. A horizontal stacked bar (recharts, expand layout — shares 0–100%) over a dense table of absolute values (CPU %, RAM GB, process count) per family. Families arrive as stable **keys** (`OneC` / `Mssql` / `OsUpdate` / `Antivirus` / `Other`) and are localized via i18n + a fixed colour per family (one family, one colour — like statuses); an unknown key from the operator-configurable `Performance.ProcessFamilyMap` degrades to «Прочее»'s label/colour but never drops the row.
- **Attribution-incomplete banner** — an amber warning shown when the probe could not read some processes for lack of OS rights (`processesInaccessible > 0` / `attributionIncomplete`). It names the count and points the operator at running the backend with administrator rights, because an under-privileged backend cannot read another account's `rphost`/`sqlservr` and would otherwise mislabel everything «Прочее». Same honest-signal pattern as IIS-permissions / readiness / `Measuring`; under the normal admin-elevated backend the banner stays hidden (see `OPERATIONS.md` «Быстродействие — required permissions»).
- **`Measuring` flag** — `true` on the very first poll, when metrics that need a delta between two samples (host CPU %, per-process CPU %, disk latency) are not yet ready. The UI shows «измеряю…» for those (CPU and Disk gauges, the CPU attribution column, the verdict) instead of drawing the zeros as real values; RAM (instantaneous available/total) and the RAM attribution render normally. The flag clears on the second poll (~5 s).

**1C-load drill-down («кто грузит внутри 1С»)** — the live answer when the bottleneck is 1С itself (MLC-067), rendered as a section below the host snapshot on the same `/performance` route (a separate live source from the host probe — its own query, loading and error handling). Backed by `GET /api/v1/performance/onec-sessions` (`Viewer`, MLC-066), same pull-on-demand model (~5 s poll, `placeholderData: prev`, nothing persisted, Zod-validated boundary): two dense tables — **sessions** (top by `cpu-time-current` then `duration-current`; columns user / host / appID / CPU / duration / **DBMS share** (`duration-current-dbms`, separating «1С грузит app-сервер» from «грузит SQL») / memory / last-active / state) and **worker processes** (`available-perfomance` with a degradation tint, `avg-call-time`, memory size, pid). A per-row **state badge** flags the load shape — `Заблокирован` (red, `blocked-by-dbms`/`-ls` ≠ 0) ▸ `Долгий вызов` (amber, `duration-current` over the long-call threshold) ▸ `Активен` ▸ `Молчит` (idle past the silent threshold by `last-active-at`) ▸ `Простаивает`. This is **separate** from `/sessions`, whose job is *kill* — here there is no action, only attribution. All perf fields are nullable (the backend «never throws» — absence → `null`) and render as «—», never `0`; `memory-current` keeps its sign (negative during GC). **Empty-state:** both lists empty ⇒ «нет активной нагрузки» (the backend carries no Available flag — a missing signal is not an error; usually rac is not configured or no sessions are active, in which case the RAS-health card on the dashboard is the place to look).

**SQL-load drill-down («1С грузит SQL?»)** — the live answer when the bottleneck is MSSQL (MLC-069), rendered as a section below the 1С-load drill-down on the same `/performance` route (again a separate live source — its own query, loading and error handling). Backed by `GET /api/v1/performance/sql` (`Viewer`, MLC-068), same pull-on-demand model (~5 s poll, `placeholderData: prev`, nothing persisted, Zod-validated boundary). The response is `SqlPerformanceView` = `snapshot` (from the DMV probe) + `databases` (the endpoint's own database→client attribution via its `AppDbContext`, a vertical slice). The status enum (`Ok` / `PermissionDenied` / `Unavailable`) arrives as a string: a non-`Ok` status carries empty lists and renders an **honest degraded banner** — `PermissionDenied` (amber) names the missing `VIEW SERVER STATE` grant and points at `OPERATIONS.md`; `Unavailable` (neutral) reports SQL is unreachable / the connection string is unset — instead of a silent «всё спокойно» (same pattern as the attribution-incomplete / IIS-permissions banners). On `Ok` the section shows: a dense **active-requests** table (top by CPU; columns session / database+client / CPU / elapsed / logical reads / wait / query text — truncated with the full text in a tooltip), with a **1С badge** (`isOneC`, `program_name='1CV83 Server'`) marking 1С-originated SQL and **blocking badges** (`ждёт сеанс N` on a blocked request by `blockingSessionId`, `блокирует` on a session that blocks others) making the **blocking chain** visible; a **«нагрузка по базам»** table joining requests to the attribution (per-database request count, of-which-1С count, summed CPU, client); and a **«конкуренция за ресурсы»** block with the wait-stats delta and per-database IO-stall delta. The contention deltas are cumulative-since-SQL-start, so on the first poll (`Measuring`) the section shows «измеряю…» for them while active requests (instantaneous) render at once. All perf fields are nullable and render as «—», never `0`. Attribution is at **database** granularity (SQL→session→user is impossible — common `program_name`/host, ADR-26); a database with no registered infobase shows client «—». **Empty-state:** `Ok`, not measuring, and no requests / waits / IO ⇒ «нет активности SQL».

**On-demand recording («Запись по требованию»)** — the only **persisted** source of the section (MLC-071), rendered as a section below the three live drill-downs on the same `/performance` route. Backed by `/api/v1/performance/recordings` (MLC-070, ADR-26): unlike the live sources (pull-on-demand, nothing persisted), a recording is started manually for an investigation and the backend samples host metrics + culprits on a timer into the database until a manual stop or auto-stop (time/sample limit). **Reading** the list and viewing a recording is `Viewer`; **управление** (start / stop / delete) is `Admin` (gated by `useMe` — the controls are hidden for a Viewer, matching the route's own `Viewer`-readability). The list query polls every 5 s so the «идёт запись» indicator and sample count stay fresh. The section shows:
- **Controls + indicator** — a `Admin`-only **start** button (or **stop** when a recording is active), plus a pulsing «Идёт запись» indicator visible to everyone while a recording runs. Start surfaces the backend's `409 RECORDING_ACTIVE` (one recording at a time) as a toast; **stop** and **delete** are confirmed via an `AlertDialog` (06_UI_DESIGN — destructive/irreversible actions are confirmed).
- **Investigations list** — a dense table (start / stop time / status `Идёт`/`Остановлена`/`Прервана` with stop-reason / who started / sample count); a row opens the viewer, and `Admin` gets a per-row delete (disabled for an `Active` row — the backend rejects it with `409`).
- **Recording viewer** (dialog) — metadata header + a **host-metrics-over-time chart** (recharts `ComposedChart`: CPU % and memory-used % on a 0–100 left axis, disk latency in ms on a right axis) + **top culprits over the period** (the 1С-load and SQL-load tables from MLC-067/069, fed the peak-per-key slice aggregated across samples — the moment each session/process/request loaded hardest) + **export**. Recording samples carry only the DMV snapshot (no database→client attribution — that lives only in the live endpoint), so the SQL tables show client «—». Export mirrors `features/reports/export/`: a «Скачать» menu with CSV (eager) and Excel (lazy SheetJS chunk) of the host sample series.

The response boundary is Zod-validated (ADR-10.1 / MLC-016); `stoppedAtUtc`/`stopReason` (absent on an active recording) and per-sample `oneC`/`sql` (absent when a source was unconfigured) are declared via `omittable()`, since the backend omits null fields (`JsonIgnoreCondition.WhenWritingNull`).

The gauge primitive (`MetricGauge`) lives in `features/performance/` «по месту»; generalizing the gauge/dense-table primitives is deferred to the holistic UI phase. The «Анализ быстродействия» track ships its three live drill-downs — host snapshot, 1С-load and SQL-load — and on-demand recording above (ADR-26), completing the track.

### 3.9. SQL database backups («Бэкапы базы»)

On-demand `COPY_ONLY` backups of a client's SQL database (MLC-078; backend — `04_INFRASTRUCTURE.md` §8, ADR-27). There is no dedicated route — the feature is a **dialog opened from the infobase row**: a dedicated icon button («Бэкапы базы», `DatabaseBackup` icon with a tooltip) sits in the actions cell of every infobase row, on both the infobases list and the tenant detail page. The button is **visible to both roles** — starting a backup is deliberately `Viewer` (the operator button, ADR-27) — which is why it is *not* an item of the admin-only row dropdown (that dropdown is not rendered for a Viewer at all).

- **The dialog** (`features/backups/BackupsDialog`) shows that infobase's backups, newest first (`GET /backups?infobaseId=…`), polling every 5 s while open (`refetchInterval`, `placeholderData: prev` — the same live-polling idiom as recordings) so the queue progress is visible without manual refresh: the orchestrator moves rows `Queued → Running → Succeeded/Failed` server-side. Columns: status badge (**«В очереди»** neutral / **«Выполняется»** info with a pulsing dot / **«Готов»** success / **«Ошибка»** danger with the `failureReason` localized underneath and the backend's `errorMessage` truncated with a hover title), request / completion times, file size (КБ/МБ/ГБ; the full `filePath` on hover), and who requested. Absent values render as «—», never `0`. Empty state: «Бэкапов ещё нет».
- **«Сделать бэкап»** (header button, Viewer + Admin) POSTs `{infobaseId}` and toasts the outcome. The button is **not** disabled while a backup is active — the server's per-database lock is the single source of truth, and a duplicate request surfaces its `409 BACKUP_ACTIVE` as a clear toast (no UI-side race); an unconfigured folder surfaces `409 BACKUP_FOLDER_NOT_CONFIGURED` as a toast pointing the operator at «Параметры».
- **Delete** is per-row, **Admin-only** (gated by `useMe`, same as recordings), confirmed via an `AlertDialog` (irreversible — the `.bak` is removed from the SQL host's disk). A `Running` row's delete is disabled (the backend rejects it with `409`); a file-deletion failure surfaces `409 BACKUP_DELETE_FAILED` («запись сохранена») as a toast.
- **Settings**: the `/settings` section **«Резервные копии баз»** edits the four `Backup.*` keys (folder path with a `D:\Backups` placeholder, TTL hours, parallelism cap, disk safety margin) through the standard `SettingField` (Admin-only via `PUT /settings/{key}`, ranges mirrored from `SettingDefinitions`).
- The audit journal renders the five backup actions (`BackupRequested` / `BackupSucceeded` / `BackupFailed` / `BackupDeleted` / `BackupsPurged`) in Russian and offers them in the action-type filter.
- The response boundary is Zod-validated (ADR-10.1 / MLC-016): the five nullable `BackupSummary` fields (`startedAtUtc`/`completedAtUtc`/`filePath`/`fileSizeBytes`/`errorMessage`) are omitted on the wire (`JsonIgnoreCondition.WhenWritingNull`) and declared via `omittable()`; an unknown `status`/`failureReason` string from a future backend degrades to a neutral badge with the raw name instead of rejecting the whole list.
