# MitLicense Center — дизайн-система и UX-паттерны

Документ описывает устройство интерфейса: технологическую базу, layout, повторяемые
паттерны страниц и компонентов, правила текстов и доступности.
Источник фактов — код `frontend/src`.

---

## 1. Технологическая база

### Компонентная библиотека

Интерфейс построен на **shadcn/ui** — наборе компонентов над **Radix UI** примитивами,
стилизованных через **Tailwind CSS v4**. Компоненты скопированы в проект
(`frontend/src/components/ui`) и дорабатываются напрямую; внешней зависимости нет.

Используемые примитивы Radix (через shadcn): `Dialog`, `AlertDialog`, `DropdownMenu`,
`Select`, `Popover`, `Tooltip`, `Sheet`, `Tabs`, `Checkbox`, `Separator`.

Иконки — **lucide-react**.

### Тема и токены цвета

Управление темой — **next-themes** (`attribute="class"`, `storageKey="mlc-theme"`).
Доступны три режима: **Светлая** / **Тёмная** / **Системная** (следует за ОС).
Тема по умолчанию — **Тёмная** (`defaultTheme="dark"`, MLC-195): тёмная тема —
основной режим редизайна. Пользователь по-прежнему выбирает любой из трёх режимов через
`ThemeToggle`; выбор сохраняется в `localStorage`.
Класс `.dark` ставится на `<html>`, вариант задан через `@custom-variant dark`.

Токены объявлены в `index.css` в пространстве `oklch`:

| Токен | Назначение |
|---|---|
| `--background` / `--foreground` | Фон страницы и основной текст |
| `--card` / `--card-foreground` | Фон карточек |
| `--primary` / `--primary-foreground` | Акцентные кнопки |
| `--muted` / `--muted-foreground` | Приглушённый фон и вспомогательный текст |
| `--destructive` / `--destructive-foreground` | Деструктивные действия |
| `--border`, `--input`, `--ring` | Рамки, поля, фокус-кольцо |

Дополнительно — **семантическая палитра** для статусов домена:

| Токен | Цвет (light) | Использование |
|---|---|---|
| `--status-success` | emerald | Норма, активно, «Опубликована» |
| `--status-warning` | amber | Предупреждение, пороговое значение |
| `--status-danger` | rose | Превышение, ошибка |
| `--status-info` | sky | Информационное |
| `--status-neutral` | grey | Нейтральный |

**Sidebar-токены строго монохромны** (MLC-195). Отдельный набор `--sidebar-*`
(`index.css`, блоки `:root` и `.dark`) не содержит бренд-акцентного синего: токены
`--sidebar-primary` и `--sidebar-ring` приведены к нейтральному grey (hue 0, sat 0%),
согласованному с `--primary` / `--ring` соответствующей темы. Активный пункт навигации
и фокус-кольцо сайдбара — нейтральные, как и вся палитра.

Компонент `StatusBadge` оборачивает `Badge` с пятью вариантами:
`success`, `warning`, `danger`, `info`, `neutral`. Это **единственный** способ нарисовать цветную
плашку статуса: он гарантирует единый вид статуса на всех экранах и контраст WCAG AA (MLC-138).
Запрет «сырых» цветовых классов в обход компонента машинно-проверяем — кастомное ESLint-правило
`mlc/no-raw-status-badge` (`frontend/eslint-rules/`) ловит сигнатуру чипа `bg-{fam}-500/15` и требует
`StatusBadge`; баннеры, сплошные шкалы и текст-подписи не флагаются. Сам `StatusBadge.tsx` под правило
не попадает (`eslint.config.js` игнорит `src/components/ui/**`).

**Кнопки-действия «запуск/старт» — вариант `success` (MLC-204).** Старт IIS-сайта/пула/
сервера — пара к `destructive` («стоп»): зелёная кнопка через вариант `success` компонента
`Button` (сплошной `emerald-600` + белый текст, контраст ~4.2:1 WCAG AA). Зелёный централизован
в слое UI-примитива (`src/components/ui/button.tsx`, вне правила `no-raw-status-badge`) — в экранах
сырых `bg-emerald-*` нет. Семантический токен `--status-success` остаётся для тинта бейджей/текста
(`StatusBadge`), а не для сплошной заливки под белым текстом, где он контраст не держит.

### Типографика

Основной шрифт — **Inter** (sans-serif fallback-цепочка включает Segoe UI, Roboto).
Моноширинный — **JetBrains Mono** (код, идентификаторы, tabular-nums).
Числа в таблицах применяют класс `tabular-nums` для выравнивания по разрядам.

### Эталонный экран дизайн-системы (`/design`)

`DesignSystemPage` (`features/design/DesignSystemPage.tsx`, маршрут `/design`) — единый
«kitchen-sink» эталон визуального языка Фазы 0 редизайна (MLC-195). Экран **вне навигации**
(в сайдбаре пункта нет), открывается по URL и виден обеим ролям. Данные — статические
фикстуры в файле, без обращений к API. На одном экране секциями `Card`:

- **Типошкала** — зафиксирована визуально: страница 22/500 · раздел 16/500 · текст 14/400 ·
  подпись 12/400 (`text-muted-foreground`); числа — `font-mono tabular-nums`. Каждая строка
  подписана своей ролью.
- **Статусы** — все пять вариантов `StatusBadge` (success/warning/danger/info/neutral)
  с доменными подписями. Цвет статуса — только через `StatusBadge` (правило
  `mlc/no-raw-status-badge`).
- **Две плотности карточек** — тихая KPI-карточка (крупное число + подпись) и плотный
  «ридаут хоста» (строки метрик подряд, компактные паддинги).
- **Чарт-кит** — пример `ComposedChart` (recharts) с темизацией §6: сетка `stroke-border`,
  оси/подписи `text-muted-foreground`, `Tooltip contentStyle={{ fontSize: 12 }}`; цвета серий —
  из семантической палитры статусов (`var(--status-*)`), не hex-зоопарк.
- **Состояния** — загрузка (`Skeleton`), пусто (иконка lucide + заголовок + подсказка),
  ошибка (баннер `bg-destructive/5 border-destructive/40` + ссылка «Обновить»).
- **Полоса квоты лицензий** — `Progress` с цветом индикатора по `severityToProgressClass`
  из `lib/quota.ts`.

## 2. Layout — оболочка приложения

### Структура

```
AppShell
├── SidebarProvider
│   ├── Sidebar          (nav, collapsible="icon")
│   └── SidebarInset
│       ├── Topbar       (sticky, h-14, backdrop-blur)
│       └── main         (px-6 py-6, overflow-y-auto)
│           └── <Outlet> (Suspense + PageFallback)
```

Компонент-оболочка — `AppShell`. Страницы рендерятся через `<Outlet>` внутри `<main>`.
Отложенная загрузка чанков прикрыта `<Suspense fallback={<PageFallback />}>`; пока
чанк грузится, показываются три `<Skeleton>`-блока (без текста, без i18n).

### Сайдбар

Сайдбар (`collapsible="icon"`) при сворачивании показывает только иконки.
**Брейкпоинт мобильного режима — 768 px** (константа `MOBILE_BREAKPOINT` в
`hooks/use-mobile.ts`): при ширине < 768 px сайдбар открывается как `Sheet` (drawer)
поверх контента, а не вытесняет его.

Состояние открытия сохраняется в cookie `sidebar_state` (TTL 7 суток).
Сайдбар переключается горячей клавишей `b`.

Шапка сайдбара содержит логотип «MitLicense Center» (скрывается при сворачивании).

### Навигационные группы

Сайдбар делится на четыре группы и содержит **7 разделов** (MLC-196c, ADR-53: «Отчёты»
растворены в «Сеансы»/«Базы», отдельного пункта меню нет). Видимость пунктов зависит от роли:

| Пункт | Путь | Роль |
|---|---|---|
| **Обзор** (`GaugeIcon`) | `/` | Admin + Viewer |
| — группа **Мониторинг** — | | |
| Сеансы (`MonitorPlayIcon`) | `/sessions` | Admin + Viewer |
| Быстродействие (`ActivityIcon`) | `/performance` | Admin + Viewer |
| — группа **Управление** — | | |
| Клиенты (`Building2Icon`) | `/tenants` | Admin + Viewer |
| Базы (`DatabaseIcon`) | `/infobases` | Admin + Viewer |
| — группа **Система** — | | |
| Аудит (`ScrollTextIcon`) | `/audit` | Admin + Viewer |
| Пользователи (`UsersRoundIcon`) | `/users` | **только Admin** |
| Параметры (`SettingsIcon`) | `/settings` | **только Admin** |

Пункты «Пользователи» и «Параметры» не рендерятся для роли Viewer.

### Подвал сайдбара

`SidebarFooter` показывает приглушённой строкой версию установленной панели
(«Версия {version}»). Источник — рантайм-версия бэкенда из анонимного
`/api/v1/health` (фича `health`, хук `useHealth`). При недоступном `/health`
строка не рендерится (версия неизвестна — подвал отсутствует, шелл не ломается).
Подвал скрывается при сворачивании сайдбара в иконки.

### Топбар

Топбар — sticky, `z-20`, высота 14 (3.5 rem), backdrop-blur эффект.
Содержит:
- `SidebarTrigger` — кнопка сворачивания/раскрытия сайдбара;
- название «MitLicense Center»; в не-production сборках рядом — бейдж `dev` / `test`;
- переключатель темы (`ThemeToggle`) — выпадающее меню с тремя опциями;
- аватар пользователя — кружок с инициалами (первые 2 символа логина), при ширине ≥ `sm`
  — полное имя.

Аватар открывает дропдаун: логин, роль (Admin → «Администратор», Viewer → «Наблюдатель»),
«Смена пароля» (открывает `Dialog` с формой прямо в топбаре), «Выйти».

## 3. Паттерн страниц-списков

### Структура страницы

Все страницы-списки следуют одному шаблону:

```
<div class="space-y-6">
  <header>              // h2 + subtitle + кнопка «Добавить» (только Admin)
  [error banner]        // при isError
  <div class="rounded-md border">
    <Table>             // shadcn Table
  </div>
  <PaginationBar>       // при total > pageSize
  [dialogs]             // FormDialog, DeleteDialog — монтированы, closed по умолчанию
```

### Таблицы

Списки управления строятся на компоненте `DataTable` (`@tanstack/react-table`, ADR-46),
который рендерит через shadcn-примитивы `Table` над нативным `<table>`. Живые виджеты
`/performance` остаются на прямом shadcn `Table` (авто-обновляемые ридауты без фильтров
и пагинации).

**Тулбар таблицы** (`DataTableToolbar`) — строка над таблицей: слева слот фильтров/поиска
страницы, справа два контрола (кнопки `outline`, `size="sm"`):

- **Плотность** (density-toggle) — кнопка с иконкой строк (`Rows2`/`Rows3`) и подписью
  «Плотность». Клик переключает `comfortable` ⇄ `compact`; значение персистится в
  localStorage (`mlc-table-density`), влияет на вертикальные паддинги строк/заголовков.
  Дефолт — `comfortable` (текущий вид). aria-label/title отражают целевой режим.
- **Колонки** — кнопка с иконкой колонок (`Columns3`) и подписью «Колонки» открывает
  dropdown «Видимость колонок» с чекбоксами по скрываемым колонкам; чекбокс не закрывает
  меню (можно скрыть несколько). Колонки `select` (чекбоксы bulk-выбора) и
  `actions`/`action` не скрываются.

Заголовки сортируемых колонок (`DataTableColumnHeader`) — кликабельны, с иконкой
направления: ↑ (asc), ↓ (desc), ↕ (без сортировки, приглушённая). Числа в ячейках —
`tabular-nums`.

### Пагинация

Компонент `PaginationBar` реализует **серверную пагинацию**:
- отображает сводку «N–M из total» (`tabular-nums`);
- при `isFetching && !isLoading` показывает «Обновление…» рядом со сводкой;
- ссылки «Предыдущая» / «Следующая» и номера страниц (функция `pageLinkRange`);
- не рендерится, если `total <= pageSize`.

### Состояние загрузки

При `isLoading === true` тело таблицы заполняется **4 строками-скелетонами**
(компонент `Skeleton`). Форма ячеек повторяет реальные колонки.

### Пустое состояние

При `items.length === 0 && !isLoading && !isError` внутри таблицы рендерится ячейка
`colSpan={все колонки}` с иконкой домена, заголовком и подсказкой из i18n:
`*.empty.title` / `*.empty.hint`. Для Admin — кнопка быстрого создания.

### Состояние ошибки

При `isError` над таблицей появляется баннер с красным фоном (`bg-destructive/5`,
`border-destructive/40`), текстом из `*.errors.loadFailed` и ссылкой «Обновить»
(`Button variant="link"`). При успешном повторе — `toast.success("Обновить")`.

## 4. Формы и диалоги

### Модальные диалоги (формы создания/редактирования)

Форма живёт в `Dialog` (Radix `DialogContent`).
Стек внутри: `react-hook-form` + `zodResolver` + shadcn `Form`/`FormField`/`FormMessage`.

Ширина диалога — `sm:max-w-lg` (формы клиентов, баз, пользователей) или `sm:max-w-md`
(смена пароля).

Структура диалога:
```
DialogHeader (Title + Description)
<form>
  FormField → FormLabel + FormControl + [FormDescription] + FormMessage
  ...
DialogFooter
  Button variant="ghost" → Отмена
  Button type="submit"   → Создать / Сохранить
```

При `isPending` кнопка `submit` и кнопка «Отмена» блокируются; текст кнопки меняется
на `common.loading` («Загрузка…»).

### Валидация и показ ошибок

Клиентская валидация — **Zod-схема**, ошибки выводятся под полем через `<FormMessage>`.
Серверные ошибки после submit — единый порядок **409-code → 400-field → generic-тост**:
- **409 с кодом конфликта** → маппится на конкретное поле через `matchConflictCode`
  → `form.setError(field, { type: "server", message })` → inline под полем;
- **400 ValidationProblem** → `applyFieldErrors(error, form.setError, fieldMap?)`
  разбирает dict `errors` (ключи бэка PascalCase, для публикации с префиксом
  `Publication.`) и ставит **первое** сообщение поля inline (`type: "server"`);
- **прочее** → `toastFormSubmitError(error, t)` (400 → серверное сообщение, иначе generic).

Таким образом, и ошибки **уникальности** (409), и нарушения **длины/символов**
(400 ValidationProblem из барьера `InfobaseValidationRules`, MLC-118) показываются
**inline под полем**; только сетевые/непредвиденные — тостом. Поля свёрнутого блока
«Дополнительно» в форме инфобазы при inline-ошибке раскрывают блок. На экране входа
канал ошибки — form-level inline (`role="alert"`): неверные данные / нет связи.

### Диалоги подтверждения деструктивных действий

Используется `AlertDialog` (Radix). Применяется для удаления объектов (клиент,
инфобаза, бэкап, запись быстродействия) и операций IIS (остановка сайта, iisreset).

Барьер подтверждения дифференцируется по обратимости действия (ADR-45):

- **Обратимые действия** (публикация, снятие публикации — результат восстановим
  переопубликацией) — простое «да/нет»; ручной ввод токена отсутствует, кнопка
  подтверждения активна сразу. Лечащие действия со службой RAS (регистрация /
  перерегистрация / запуск — `RasServiceActionDialog`, ADR-47) идут по этому же
  паттерну и дополнительно показывают точную команду `sc …` (`commandPreview`)
  моноширинным блоком — прозрачность и воспроизводимость оператором вручную.
- **Необратимые действия** (удаление клиента, инфобазы — каскадное, восстановления нет)
  — «да/нет» с **явным предупреждением о необратимости** (`text-destructive`); ручной
  ввод имени убран, но диалог-барьер сохранён (нельзя удалить одним кликом без открытия
  диалога).

Кнопка подтверждения деструктивных действий — **вариант `destructive`** компонента `Button`
(через `AlertDialogAction variant="destructive"`, который проксирует проп в `Button`): красный фон
`bg-destructive` + фокус-кольцо `ring-destructive/20`. Цвет задаётся вариантом, а не сырым
className в экранах (MLC-205) — единый источник для всех деструктивных подтверждений; пара к
варианту `success` для зелёных действий «запуск».

#### Инвариант порядка кнопок и начального фокуса

**Обязательное правило для всех новых деструктивных диалогов:** `AlertDialogCancel`
всегда стоит в разметке **перед** `AlertDialogAction`. Radix UI выставляет начальный
фокус на первый интерактивный элемент `AlertDialogFooter` — то есть автоматически
на «Отмену». Это исключает случайное подтверждение деструктивного действия нажатием
Enter при открытии диалога.

Все существующие диалоги (`DeleteTenantDialog`, `DeleteInfobaseDialog`,
`DeleteBackupDialog`, `UnpublishPublicationDialog`, `KillSessionDialog`,
`StopRecordingDialog`, `DeleteRecordingDialog`, `DisableUserDialog`,
`ResetPasswordDialog`, `IisConfirmDialog`, `RasServiceActionDialog`) соблюдают этот порядок.
Нарушение порядка — дефект.

### Discovery-поля (автоподстановка)

Компонент `DiscoveryField` — контролируемое поле «выбор из обнаруженного ИЛИ ручной
ввод». Состояния:

1. **Список доступен** (`available = true`): `Select` с обнаруженными значениями +
   кнопка «Обновить» + ссылка «Ввести вручную».
2. **Ручной режим** (`manual = true` или `available = false`): `Input` + ссылка
   «Выбрать из списка» (если список стал доступен) или «Повторить» (если недоступен).
3. **Предусловие не выполнено** (`disabledHint` задан): только `Input` с подсказкой,
   список не предлагается.

При загрузке — плейсхолдер «Загрузка…»; пустой список — «Ничего не найдено».
Компонент используется в формах инфобаз и параметров для полей: база кластера 1С,
база SQL, сайт IIS, путь к `rac.exe`, версия платформы, SQL-инстанс.

## 5. Обратная связь: тосты и индикаторы

### Тосты (sonner)

Toaster монтируется в `App.tsx`: `position="top-right"`, `richColors`, `closeButton`.

Sonner темизирован: `--normal-bg → var(--popover)`, `--normal-text → var(--popover-foreground)`,
`--border-radius → var(--radius)`. Тема тостов следует активной теме приложения.

Иконки: `CircleCheckIcon` (success), `OctagonXIcon` (error), `TriangleAlertIcon` (warning),
`InfoIcon` (info), `Loader2Icon animate-spin` (loading).

Паттерны вызовов:
- Успешное действие → `toast.success(t("*.toasts.<action>", { name }))`.
- Серверная или generic ошибка → `toast.error(t("errors.generic"))` или
  `toast.error(error.message)` (400 с понятным сообщением).
- Ошибки уникальности (409) и валидации поля (400) → **inline под полем**, тост не используется.

### Баннер состояния соединения

`ConnectionBanner` (в `AppShell`, над топбаром) показывает `errors.network`
(«Нет связи с сервером.»), пока последний запрос упал с `ApiNetworkError`; снимается
при следующем успехе. Сигнал — module-store `connectionStatus` (взводится из
`classifyError` на кэшах React Query). Это **первичный** индикатор недоступности
бэкенда, общий для всего приложения (отличается от per-таблица-баннеров загрузки).

### Индикатор свежести данных

Компонент `RelativeTime` показывает, сколько времени прошло с момента снятия снимка
(используется в «Сеансах» и «Быстродействии»):
- `< thresholdAmberSec` секунд — `text-muted-foreground` (норма);
- `> thresholdAmberSec` секунд — `text-amber-600 dark:text-amber-400` (данные устарели);
- при `isError = true` — `text-destructive` (ошибка опроса).

Тайтл элемента содержит точную дату-время в формате `dd.MM.yyyy HH:mm:ss` — виден
при наведении (нативный `title`).

**Отдельного баннера «Данные устарели» с кнопкой повтора не существует.**
Единственный индикатор устаревания/ошибки опроса — изменение цвета метки времени
(amber → красный). Не ожидайте повторяющегося вызова действия или заметного UI-блока
при устаревании; их нет в коде. Если задача требует явного баннера — это новый
компонент.

### Контролы «живых» страниц — `LiveControls`

Компонент `LiveControls` (`components/LiveControls.tsx`) — индикатор состояния + пара кнопок
в шапке «живых» страниц («Сеансы», «Быстродействие»), рядом с индикатором свежести:
- **Индикатор состояния** (слева) — цвет кодирует **текущее** состояние, не действие
  (MLC-158): в живом режиме зелёная точка + «Авто-обновление» (`text-muted-foreground`,
  `bg-emerald-500`); на паузе — янтарная плашка `Pause` + «На паузе» (`bg-amber-100`/
  `text-amber-700`, dark `bg-amber-950`/`text-amber-400` — тот же янтарный, что у устаревшей
  свежести). Так «уже на паузе» и «поставить на паузу» не путаются.
- **Пауза/Возобновить** (`Pause`/`Play`) — замораживает авто-обновление таблицы
  (`refetchInterval` → `false`), чтобы строка не «уезжала» под курсором; подпись/иконка по
  `isPaused`. На паузе кнопка «Возобновить» — **зелёная** (`border-emerald-500`/
  `text-emerald-700`), как призыв «вернуть в живой режим»; в живом режиме «Пауза»
  нейтральная (янтарный не используем — ложная тревога, когда всё работает).
- **«Обновить сейчас»** (`RefreshCw`, крутится `animate-spin` при `isRefreshing`) — ручное
  обновление. На «Сеансах» это **живой форс-обход 1С** (`POST /api/v1/sessions/refresh` →
  cold-прогон сейчас → перечитка снимка); на «Быстродействии» — refetch live-источников
  (host/1С/SQL), сам раздел уже live на каждый poll.

Кнопки — shadcn `Button` `variant="outline" size="sm"`, с `aria-label`/`title` (i18n
`common.pause`/`resume`/`refreshNow`/`live`/`pausedStatus`).

В «Быстродействии» первый poll требует двух измерений (дельта-метрика). До получения
дельты `MetricGauge` показывает `animate-pulse` и текст «измеряю…» вместо значения.

### Состояние «фонового обновления»

`PaginationBar` получает `isFetching` и показывает «Обновление…» рядом со сводкой
— без блокировки таблицы.

## 6. Визуализации (recharts)

Библиотека **recharts** используется в двух разделах:

### Отчёты — `LicenseUsageChart`

`ComposedChart` (`ResponsiveContainer width="100%" height={320}`), три серии:
- `consumedMax` — заливка (`Area`, gradient) sky-500 — пик потребления;
- `consumedAvg` — линия поверх emerald-600 — среднее в бакете (округлено до 0.1);
- `limit` — пунктирная линия (`strokeDasharray="6 4"`) rose-500 — лимит клиента.

Данные — 15-минутные бакеты; метка X-оси `dd.MM HH:mm`.

При `isLoading` вместо графика — `<Skeleton className="h-[320px] w-full" />`.

### Быстродействие — `ProcessFamilyAttribution` и `RecordingHostChart`

Экран — воронка: вердикт «почему тормозит» → светофор ресурсов → сегментированный
переключатель слоёв (`Tabs`) Хост/1С/SQL с авто-фокусом по вердикту → единый блок
«Блокировки» → запись по требованию. Слой «Хост» несёт атрибуцию семей процессов,
«1С»/«SQL» — собственные live-разрезы (MLC-207). Светофор кликабелен (MLC-208):
клик по гейджу ресурса наводит drill-down на слой — CPU/RAM на слой доминирующей
семьи, Диск всегда на слой SQL (IO-stall по базам); клик пинит слой, как ручной
выбор вкладки. Блок «Блокировки» (MLC-210) сводит цепочки блокировок SQL и
заблокированные сеансы 1С в одно место; пусто → нейтральное «блокировок нет»
(хороший случай, без красных акцентов).

`ProcessFamilyAttribution`: горизонтальный stacked `BarChart` (семьи процессов в
долях CPU и RAM). Цвет сегментов задаётся через `familyColor(familyKey)`.

`RecordingHostChart` (`ComposedChart`, height={300}): три серии host-метрик записи
во времени — CPU%, RAM занято%, латентность диска (мс). CPU и RAM на левой оси
0–100 (единицы совпадают); латентность — на правой оси с `axisId`.
Цвета: CPU — sky-500, RAM — emerald-600, disk — amber-600.

Темизация: сетка `CartesianGrid` использует класс `stroke-border`; оси и подписи —
`text-muted-foreground`. `Tooltip` получает `contentStyle={{ fontSize: 12 }}`.
Recharts не наследует CSS-переменные автоматически — цвета заданы как hex-литералы,
семантически соответствующие токенам проекта.

### `MetricGauge` (линейный прогресс-бар)

Радиального gauge в наборе нет. В разделе «Быстродействие» три метрики (CPU, RAM,
диск) отображаются через shadcn `Progress` с цветом индикатора по уровню насыщения:
- `ok` → emerald-600 / emerald-500 dark;
- `warn` → amber-600 / amber-500 dark;
- `crit` → red-600 / red-500 dark.

Значение и подпись тоже перекрашиваются по тому же уровню.

## 7. Единый визуальный язык квоты лицензий (MLC-122 / R6)

### Пороги и severity

Все экраны используют единственный источник правил — `frontend/src/lib/quota.ts`:

| Порог | Severity | StatusBadge variant | Цвет прогресс-бара |
|---|---|---|---|
| `percent < 75` | `ok` | `neutral` (бейдж не показывается) | по умолчанию |
| `75 ≤ percent < 90` | `warning` | `warning` (amber) | `bg-amber-500` |
| `percent ≥ 90` | `danger` | `danger` (rose) | `bg-rose-500` |
| `limit ≤ 0` (безлимит) | `ok` | — (бейдж не показывается) | — |

Константы порогов: `QUOTA_WARNING_THRESHOLD = 75`, `QUOTA_DANGER_THRESHOLD = 90`.

### Хелперы

- `quotaPercent(consumed, limit)` — процент (0 при безлимите);
- `quotaSeverity(consumed, limit)` — severity (цвет акцента);
- `quotaLabel(consumed, limit)` — **текст** ярлыка по ФАКТУ `consumed vs limit` (MLC-188):
  `"exceeded"` только при `consumed > limit`; `"atLimit"` при равенстве (`N` из `N` — «лимит
  достигнут», не превышение); `"nearLimit"` ниже лимита, но severity warning/danger; `null` иначе.
  Текст ярлыка НЕ путать с цветом-severity (`danger` уже с ≥90%): «превышение» совпадает с
  определением домена (`03_DOMAIN_MODEL` §enforcement: только `limit > 0` и `consumed > limit`).
- `severityToStatusBadgeVariant(severity)` — вариант `StatusBadge` (цвет);
- `severityToProgressClass(severity)` — Tailwind-класс для `Progress`;
- `quotaDisplay(consumed, limit)` — все значения разом (`percent`, `severity`, `label`, `badgeVariant`, `progressClass`).

### Применение по экранам

| Экран | Что показывает |
|---|---|
| `/` (Дашборд) | Виджет «Требует внимания» — иконки/цвета строк по палитре severity (квоты danger/warning, дрейф, диск, RAS) |
| `/tenants` (список) | Единая колонка «Лицензии»: `consumed / limit (percent%)` + полоса заполнения (`Progress`, цвет по severity) + `StatusBadge` при warning/danger |
| `/tenants/:id` (карточка) | Лицензионная панель: полоса потребления (`Progress`) + `consumed / limit (percent%)` + `StatusBadge`; при `consumed > limit` — текст «превышение на N» |
| «Сеансы» → «Использование за период» / «Базы» → «Размер баз» | Пиковое значение `ReportsStats` + `StatusBadge` при warning/danger |

### Источник потребления на клиента (FE live-оверлей)

На `/tenants` и `/tenants/:id` текущее потребление — live-оверлей, агрегированный на
фронтенде из снапшота сеансов (`GET /api/v1/sessions/snapshot`, обновляется каждые 5 с).

Хук `useTenantConsumption()` строит `Map<tenantId, consumed>` через функцию
`buildConsumedByTenant(items)`: считает записи со `licenseStatus === "Consuming"`, группирует
по `tenantId`. Это намеренное небольшое дублирование канонического backend-метода
`LicenseConsumption.CountByTenant` — на FE это визуальный оверлей, не контракт/не parity-правило.

Клиент без сеансов в снапшоте → consumed = 0. Пока снапшот загружается впервые → скелетон
(нет мигания при poll). Безлимитные клиенты (`maxConcurrentLicenses ≤ 0`) — нейтрально («—»).

### Привязка тестов

Тесты привязываются к `data-variant` атрибуту `StatusBadge` (FE-19, MLC-120) — семантика
`severity → variant`, а не Tailwind-класс. Это защищает тесты от дизайн-рефакторов.

## 8. Правила UI-текстов

### Язык и тон

Все тексты интерфейса — **русский**. Единственный источник строк — словарь
`frontend/src/i18n/ru/*.json`: по одному файлу на top-level namespace (`common`, `nav`,
`sessions`, `reports`, `errors`, …), каждый оборачивает свой ключ. Файлы собираются обратно
в один объект и единый namespace `translation` в `frontend/src/i18n/index.ts` (MLC-027) —
итоговая структура словаря идентична прежнему плоскому `ru.json`. Ниже ссылки на ключи
даются по их namespace-файлу (`errors.*` → `ru/errors.json` и т. п.).
Тон: нейтральный, деловой, без личных обращений в действиях (не «Ваши клиенты», а «Клиенты»).

### Соответствие глоссарию

UI строго следует терминологии из глоссария (01_OVERVIEW.md §5):
- «Клиент» (не «арендатор», не «компания»);
- «Инфобаза» (не «база данных»);
- «Публикация» (не «деплой»);
- «Лимит лицензий» / «Контроль квот»;
- «Бэкап» (не «резервная копия»).

### Тон сообщений об ошибках

Три обязательных правила, выведенных из фактических строк `errors.*` (`ru/errors.json`)
и сообщений форм в словаре `ru/*.json`:

1. **Формальное «Вы», не «ты».** Адресат — пользователь-профессионал; обращение
   всегда в формальной форме, в том числе в подсказках и описаниях. Примеры из кода:
   `"У вас нет прав на это действие."` (`errors.forbidden`),
   `"Вы вошли по временному паролю."` (`auth.forceChange.subtitle`).

2. **Описывать что произошло / что сломалось, а не что пользователь пытался сделать.**
   Сообщение называет факт или техническую причину. Примеры:
   `"Не удалось загрузить список клиентов."` (`tenants.errors.loadFailed`),
   `"Файл бэкапа не удалось удалить — запись сохранена."` (`backups.errors.deleteFailed`).
   Нельзя: «Вам не удалось выполнить удаление» (фокус на действии пользователя).

3. **Не извиняться.** В словаре `ru/*.json` нет ни одного «Извините», «К сожалению» или «Простите».
   Сообщение — нейтральный факт. Единственный вариант «смягчения» — предложение
   действия: `"Попробуйте ещё раз."` (`errors.generic`).

## 8. Доступность

Доступность обеспечивается преимущественно инфраструктурой Radix UI:

- **Фокус-менеджмент**: `Dialog`, `AlertDialog`, `DropdownMenu`, `Select`, `Sheet`
  управляют фокусом автоматически (Radix focus-lock, возврат фокуса при закрытии).
- **ARIA**: Radix-примитивы предоставляют корректные `role`, `aria-expanded`,
  `aria-haspopup`, `aria-labelledby`, `aria-describedby`.
- **`aria-hidden="true"`** проставлен на декоративных иконках lucide-react в ряде мест
  (Topbar, ThemeToggle, пункты `DropdownMenuItem`); сплошного покрытия всех иконок нет.
- **`aria-label` — русский** (MLC-138/UX-40): все `aria-label` и `sr-only`-тексты
  на русском. Компоненты с `t()` используют i18n-ключи (`common.pagination.*`); shadcn-
  обёртки без контекста i18n (pagination.tsx, sidebar.tsx, dialog.tsx) содержат
  русские литералы напрямую. Покрытие: `nav[aria-label]` пагинации → «Пагинация»;
  кнопки навигации «Назад» / «Вперёд»; sr-only эллипсис «Ещё страницы»; кнопка
  закрытия диалога «Закрыть»; `SidebarRail` → «Свернуть/развернуть боковую панель».
  `PaginationBar` передаёт `aria-label` через ключи `common.pagination.previous` /
  `common.pagination.next`.
- **`sr-only`** используется для скрытого текста кнопок с иконкой (например,
  `<span className="sr-only">{t("common.details")}</span>` в DropdownMenuTrigger).
- **Состояние загрузки** страницы: `<PageFallback>` оборачивается в
  `role="status" aria-busy="true"`.
- **Disabled-состояние пагинации**: кнопки получают `aria-disabled` + `pointer-events-none
  opacity-50` (не `disabled` на `<a>`, чтобы не ломать семантику ссылок).
- **Keyboard shortcut** сайдбара (`b`) зарегистрирован через shadcn SidebarProvider.

### Контраст WCAG AA (MLC-138/UX-28)

Все варианты `StatusBadge` проверены инструментально (oklch → sRGB → WCAG 2.1 relative
luminance). Целевой минимум для мелкого текста бейджа — **4.5:1**.

| Вариант | Светлая тема (ДО → ПОСЛЕ) | Тёмная тема |
|---|---|---|
| `success` | 4.76:1 (без изменений) | 10.71:1 |
| `warning` | 4.48:1 → **6.32:1** (amber-700→amber-800) | 10.98:1 |
| `danger` | 5.16:1 (без изменений) | 9.17:1 |
| `info` | 5.10:1 (без изменений) | 9.92:1 |
| `neutral` | 4.34:1 → **7.09:1** (muted-fg→zinc-700/zinc-400) | 5.83:1 → 5.90:1 |

Токены `index.css` не изменялись. Правки изолированы в `StatusBadge.tsx` (классы
`text-amber-800` вместо `text-amber-700` и `text-zinc-700 dark:text-zinc-400` вместо
`text-muted-foreground`).

### Деструктивный стиль (MLC-138/UX-26)

Кнопка подтверждения всех деструктивных диалогов использует `variant="destructive"`
(через `AlertDialogAction`) или явные классы `bg-destructive text-destructive-foreground`.
Покрытие: `DeleteTenantDialog`, `DeleteInfobaseDialog`, `DeleteBackupDialog`,
`UnpublishPublicationDialog`, `KillSessionDialog`, `DeleteRecordingDialog`,
`DisableUserDialog` — явные `bg-destructive`-классы. `StopRecordingDialog` и
`IisConfirmDialog` — `variant="destructive"` на `AlertDialogAction` (добавлено MLC-138).
