# MitLicense Center — Roadmap

Документ описывает актуальные планы развития и отложенный бэклог.
Горизонт — пострелизный цикл (v1.x и далее). Продуктовый контекст —
`01_OVERVIEW.md`; архитектурные ограничения — `DECISIONS.md`.

## Релиз `0.6.0-beta` — в работе (поставлен 2026-06-15)

Минор по запросу владельца — трек «Релиз `0.6.0-beta`» (`MLC-164..166`). Тема — достоверность
лицензий и читаемость того, что видит оператор.

- `MLC-166` (ядро, **ADR-48**) — **`ConsumesLicense` = факт `rac --licenses`**, а не эвристика по
  white-list `app-id`. Контроль лицензий — главная функция панели, а договорённость с клиентами
  («сеанс держит клиентскую лицензию 1С → платит за место») — это факт от 1С, который раньше мы лишь
  угадывали по типу клиента (ошибка на неоднозначных типах: веб-сервисы и пр.). Холодный тир делает
  второй вызов `--licenses` и сшивает по `session`-GUID; горячий — берёт классификацию из холодного
  снимка. Инвариант «grace ≥ холодного интервала» гарантирует: каждое завершение — на факте. White-list
  `app-id` удалён из подсчёта (остаётся как атрибут сеанса).
- `MLC-164` — аудит: убрать ложную запись «публикация создана» при добавлении базы (webinst не
  запускается → публикации на IIS нет; реальное событие логируется отдельно `PublicationPublished`).
- `MLC-165` — «Сеансы»: фильтр по типам сеансов (мультивыбор `app-id` с человеческими именами).

**Спайк rac (стенд 8.5, 2026-06-15).** `rac session list --licenses` — отдельная проекция: даёт
per-session факт лицензии (`license-type`, `series`, `short-presentation`), но **без поля `infobase`**
(нужно для привязки к тенанту) → отсюда два вызова на холодном тире. Нелицензионный сеанс в вывод
`--licenses` **не попадает вовсе** (на стенде: `BackgroundJob` отсутствует, `3×1CV8C` с `HASP` —
присутствуют) → правило сшивки = членство `session`-GUID в выводе. Вариант «`--licenses` на горячем
тире каждые 4 с» отклонён — двойные спавны на горячем пути ради косметики (см. ADR-48).

> Живая проверка на стенде (за владельцем): после апгрейда на `0.6.0` — потребление лицензий по
> «Сеансам»/дашборду совпадает с фактом 1С; завершение лишних сеансов идёт по реально лицензионным.

## Релиз `0.5.6-beta` — выпущен (2026-06-15)

Патч-фикс: `MLC-163` — «Проверить публикацию» падало 500 (`DbUpdateConcurrencyException`), а фоновое
обновление статусов публикаций молча не работало с релиза `0.5.0`. Причина — `PublicationStatusRefreshJob`
аттачил probe-публикацию без concurrency-токена `RowVersion` (добавлен в MLC-151) → targeted-UPDATE по
`Id AND RowVersion` не находил строку. Фикс — грузить `RowVersion` в снимок + устойчивость ручной
проверки к реальному конфликту. Не связано с RAS-работой (другая подсистема). Подтверждено
регрессионным тестом на SQLite (без фикса падает).

> **Живая проверка в UI (за владельцем):** после апгрейда панели на `0.5.6` — «Проверить публикацию»
> отдаёт результат (не «произошла ошибка»).

## Релиз `0.5.5-beta` — выпущен (2026-06-15)

Патч-фикс: `MLC-162` — обнаружение службы RAS не находило существующую службу (детектор всегда
«не зарегистрирована»). Причина — вызов `sc.exe` из-под `Process.ArgumentList` для энумерации
`state=` на реальной ОС не отрабатывал (поймано на dev-сервере; юнит-тесты мокали `sc`).
Обнаружение переведено на чтение реестра служб (`…\Services\*\ImagePath`, фильтр `ras.exe`) +
`ServiceController` — один проход, без спавнов (снимает и перф-риск из ревью MLC-159); команды
действий собираются одной raw-строкой под нестандартный парсер `sc.exe` (как установщик). ADR-47
Update-нота. Фикс наземно проверен на dev-сервере: реестровый детектор находит существующую службу
`1C:Enterprise 8.5 Remote Server` (Running) — там, где sc-перебор давал ложное «не зарегистрирована».

> **Живая проверка в UI обновлённой панели (за владельцем):** блок RAS в Настройках → «ок»/«устарела»;
> при желании — действия регистрации/обновления/запуска на стенде.

## Релиз `0.5.4-beta` — выпущен (2026-06-15)

Минор по запросу владельца: трек **«Авто-регистрация службы RAS»** (`MLC-159..161`). Панель сама
обнаруживает, что на хосте не зарегистрирована (или указывает на устаревшую платформу) служба
`ras.exe`, и чинит это в один клик с подтверждением — закрывает «забытую» ручную операцию при
разворачивании на новом сервере и при обновлении платформы 1С. Дом действия — секция «Подключение
к 1С/RAS» в `/settings` (Admin-only); дашборд только сигналит недоступность со ссылкой в настройки.
Архитектура — **ADR-47** (новый порт `IRasServiceManager` поверх `sc.exe`, диагностика из 4
состояний, аудит-серия 600). `rac.exe` остаётся единственным адаптером кластера — управление службой
ортогонально (Strategy B остаётся отклонённой).

- `MLC-159` — BE: обнаружение службы RAS (по `binPath` с `ras.exe`) + адаптер `IRasServiceManager`
  (register/update/start через `sc.exe`), эндпоинты `/ras-service/*`, аудит 600/601/602.
- `MLC-160` — FE: блок диагностики RAS (4 состояния) + лечение в Настройках, диалоги с точной
  командой `sc …`, parity-схемы.
- `MLC-161` — FE: сигнал недоступности RAS на дашборде из дешёвого health-снимка `summary.ras`
  (ссылка-переход в «Параметры»; дорогой `/ras-service/status` с дашборда не дёргается) + бамп
  версии `0.5.4-beta`.

> Живая проверка на стенде (регистрация/перерегистрация/запуск службы RAS, обе роли) — за владельцем.

## Актуальные планы (вне релиза 0.4.0-beta)

### 1. ESLint-правило для StatusBadge

`StatusBadge` — единственный способ отображать статусы в UI; правило его применения
поддерживается ревью, не линтером. Запланировано: кастомное ESLint-правило,
форсящее `StatusBadge` вместо сырых цветовых классов.

### 2. RAS Strategy B — **отклонено** (2026-06-14)

Текущее состояние: `rac.exe` запускается как отдельный процесс на каждый цикл.
Кросс-вызовой кэш UUID кластера уже сократил steady-state spawn rate примерно
вдвое; бюджет ≤ 26 proc/min выдерживается с запасом.

«Долгоживущий TCP-сокет на порту 1545» (Strategy B) означал бы **переписать закрытый
бинарный протокол RAS на .NET** (реверс-инжиниринг, ломкость по версиям 8.5.x, потеря
страховки официального `rac.exe`). Измеренной проблемы латентности нет (бюджет держится
с запасом), поэтому решением владельца (2026-06-14) **отклонено**. Регистрация `ras.exe`
службой Windows — ортогональна (серверная сторона), не замена Strategy B; взята отдельным
треком в релиз `0.5.4-beta` (ADR-47, `MLC-159..161`). Официальный `rac.exe` остаётся единственным
адаптером кластера (ADR-16). Вернуть Strategy B можно только при измеренной проблеме латентности
на боевой нагрузке.

### 3. Multi-cluster / multi-node топология

Все адаптеры предполагают single-node (зафиксировано ADR-28). Расширение требует
полного пересмотра каждого адаптера и операционной модели. Брать только при явном
запросе со стороны оператора и после отзыва ADR-28.

> Бывшие пункты «Серверный фильтр «Базы»», «Optimistic concurrency на Infobase/Publication» и
> «Low-priv аккаунт службы» взяты в трек `0.5.0-beta` (`MLC-150`/`MLC-151`/`MLC-152`).

## Permanently out of scope (ADR-15)

Планировщик резервного копирования, restore-оркестрация внутри панели и встроенная
2FA. Исключение: on-demand `COPY_ONLY`-бэкап (ADR-27). Вернуть любой пункт можно
только явным отзывом ADR-15.
