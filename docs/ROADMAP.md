### 1. Разобраться, почему время на бэкенде в UTC

Бэкенд хранит и планирует время в **UTC**, а пользователь работает в местном поясе (Москва,
UTC+3). Это всплывает в ночных рекуррентных джобах: например, сбор размера баз идёт по cron
`0 2 * * *` = 02:00 UTC = **05:00 по Москве** — неочевидно, если ждёшь «в 2 ночи». То же у
ретеншенов (03:00–04:00 UTC) и у меток времени в БД.

**Откуда это.** В проекте принята конвенция «всё время в БД — UTC» (тест-инвариант
`UtcDateTimeConventionTests`), а перевод в местное время делается на границе: фронтенд показывает
локальные сутки браузера (для отчётов это уже разобрано в **ADR-23**, задача `MLC-177`). Cron
рекуррентных Hangfire-джоб по умолчанию тоже считается в UTC.

**Что разобрать.** (1) Подтвердить и явно задокументировать, почему backend в UTC (единое
хранение без неоднозначностей перехода на зимнее/летнее время, конвертация на фронтенде) —
вероятно это осознанный выбор, но он нигде прямо не оформлен как решение/ADR. (2) Решить, нужно
ли ночные cron-джобы привязать к местному поясу хоста (`TimeZoneInfo`), чтобы «ночные» работы
реально шли ночью по местному времени, либо оставить UTC и просто показывать оператору локальное
время (Hangfire-дашборд / описание джоб в `OPERATIONS.md`). Итог — нота/ADR в `DECISIONS.md` +
при необходимости правка расписаний. Преимущественно backend; объём определит постановочный PR.

### 2. Алерт квоты в виджете «Требует внимания» путает «превышение» и «достигнут лимит»

Виджет «Требует внимания» на «Обзоре» при severity `danger` пишет «N клиентов **превысили квоту
лицензий**». Но это `danger`-бакет (`percent ≥ 90`), куда попадают и 90–99 % (ещё ниже лимита),
и ровно `N/N` (лимит **достигнут**), и только потом — реальное превышение (`consumed > limit`).
Это тот же конфляут, что **`MLC-188`** уже исправил для ярлыков клиентов: превышение — только
`consumed > limit`; `== limit` — «лимит достигнут»; `≥ warning` и `< limit` — «близко к лимиту».
То есть виджет рассинхронен с доменным определением (`03_DOMAIN_MODEL` §enforcement) и с
`quotaLabel()` (`lib/quota.ts`).

**Где.** Текст — `i18n dashboard.json` (`alerts.quotaDanger_*` «превысили квоту»); счётчик —
бэкенд `DashboardEndpoints` (`quotaDanger` = `percent ≥ QuotaDangerThreshold`), контракт
`DashboardAlertsResponse` отдаёт лишь два бакета (`quotaWarning`/`quotaDanger`), оба по severity-%.

**Что сделать.** Привести алерт в соответствие с `MLC-188`: либо (минимум) переформулировать
`quotaDanger` так, чтобы он не утверждал «превысили» для `≤ limit`; либо (консистентно) развести
бакеты по факту — превышение (`consumed > limit`) / лимит достигнут (`== limit`) / близко
(`≥ warning`, `< limit`) — зеркаля `quotaLabel`. Второй путь трогает контракт `/dashboard/alerts`
(новые счётчики), Zod-parity и i18n-плюрализацию. BE+FE; объём определит постановочный PR.

### 3. Доносить пользователю причину завершения сеанса + контакты для обращения

Когда панель принудительно завершает сеанс 1С по лимиту лицензий (enforcement, `KillEnforcer` →
`rac session terminate`), пользователь просто «вылетает» без объяснения — не понимает, почему его
отключили и куда обращаться. Для хостинга это плохой опыт: звонки «у меня всё упало» вместо
понятного «лимит лицензий, обратитесь к провайдеру».

**Идея.** У 1С `rac session terminate` есть параметр `--error-message="<текст>"` — клиент
показывает этот текст пользователю как причину завершения. Сейчас вызов передаёт только
`--cluster`/`--session` (`RacExecutableRasClusterClient`), сообщение не задаётся. Добавить
настраиваемый текст причины с **телефоном и названием организации провайдера**, куда обращаться.

**Что сделать.** (1) Новая настройка (`Setting`, раздел `/settings`) — шаблон сообщения, который
заполняет провайдер (причина + телефон + организация; разумный дефолт). (2) Прокинуть его как
`--error-message` в `rac session terminate` на enforcement-пути (`KillEnforcer`); решить, нести ли
сообщение и при ручном завершении сеанса оператором из панели. **Гочи:** кодировка аргументов
`rac` (OEM/CP866, кириллица — есть `SystemProcessRacRunnerEncodingTests`) и экранирование текста с
пробелами/кавычками; доступность `--error-message` зависит от версии платформы 1С — деградировать
мягко, если не поддерживается. Parity новой настройки (`SettingDefinitions` + enum + FE-поле +
i18n). Преимущественно backend; объём и формат шаблона определит постановочный PR.

### 4. «Аудит» — два поля поиска, похоже, дублируют друг друга

На странице «Аудит» (`/audit`, `AuditFiltersBar`) фильтр-бар несёт несколько перекрывающихся
механизмов поиска, и часть выглядит избыточной (наблюдение владельца). Состав сейчас:

- **«Тип действия»** — `SearchableSelect` (выпадающий список + поле поиска внутри него для выбора
  типа действия);
- **«Клиент»** — `SearchableSelect` (то же, для выбора клиента);
- **«Поиск»** — свободное текстовое поле; бэкенд (`AuditEndpoints`) матчит его по описанию И
  инициатору (`Description.Contains || Initiator.Contains` → `LIKE`).

То есть на одном баре до трёх «полей поиска» (две внутри выпадашек + одно свободное) — для
пользователя это читается как дубль. Дополнительно: бэкенд принимает ещё параметр `initiator`
(точное совпадение по инициатору), который фронтенд **не использует** — мёртвая половина контракта.

**Что разобрать.** Уточнить с владельцем, какие именно два поля воспринимаются дублем, и упростить
фильтр-бар: убрать/объединить избыточное (например, оставить выпадашки-фильтры + одно свободное
поле с понятной подсказкой «по чему ищем»), решить судьбу неиспользуемого `initiator`. Менять
поведение поиска — обновлять обе стороны (Zod-parity + бэкенд) и тесты. Преимущественно FE +
небольшой backend; объём определит постановочный PR.

### 5. Конфигурируемость подключения к 1С: агентский порт (1540) + сбор в установщике

**Что уже есть в `/settings`:** endpoint RAS-сервера `host:port` (`OneC.RAS.Endpoint`, дефолт
`localhost:1545`) и учётка администратора кластера (`OneC.Cluster.AdminLogin/Password` →
`--cluster-user`/`--cluster-pwd`). То есть нестандартный RAS-порт и защищённый кластер панель уже
поддерживает — но настраивается это пост-установочно.

**Пробел 1 — агентский порт `ragent` (1540) захардкожен.** При авто-регистрации службы RAS
(ADR-47) панель строит `ras cluster --service --port=<RAS-порт> localhost:1540`, где `localhost:1540`
— **константа** `ScRasServiceManager.LocalAgentAddress` (1540 — стандартный порт агента кластера).
Если центральный сервер/агент 1С слушает **нестандартный** порт, авто-регистрация RAS соберёт
неверный адрес агента → служба RAS не подключится к кластеру → панель не видит сеансы. Хост всегда
`localhost` (single-host, ADR-28), меняться может только порт. Сделать **порт агента настройкой**
(дефолт 1540), читать так же, как RAS-порт.

**Пробел 2 — установщик не собирает параметры 1С.** Мастер берёт SQL, порт *панели* и AllowedHosts,
но не RAS-порт, не агентский порт, не учётку кластера — всё задаётся вручную в `/settings` после
установки (дефолты `localhost:1545`/`1540`, аноним). Для нестандартного окружения это лишний шаг.
Рассмотреть страницу «Подключение к 1С» в мастере, засевающую эти настройки при установке.

**Гочи.** Настройки живут в БД (`dbo.Settings`, сидятся на старте приложения); **пароль кластера
шифруется** DataProtection ключом приложения — установщик не может легко произвести шифртекст
(не-секретные RAS-/агент-порт засеять проще; пароль — либо оставить `/settings`, либо bootstrap с
шифрованием на первом старте). Parity новых настроек (`SettingDefinitions` + enum + FE-поле + i18n).
Backend (агент-порт → настройка, замена константы) + опционально установщик; объём определит
постановочный PR.
