Этап А6 пред-релизного аудита. Дата: **2026-06-12**, build-машина (Windows 11 Pro), репо
`F:\dev\MitLicense Center`, HEAD `e6b1317` (тот же, что в А0). Метод: **сборка фактом**
(publish + инсталлятор реально собраны на этой машине), анализ инсталлятора/эксплуатации/
бэкапа/зависимостей/CI — независимыми субагентами (sonnet/opus), все High-кандидаты
прогнаны через субагентов-скептиков (включая веб-проверку CVE/лицензий по первоисточникам).
Версии зависимостей взяты из A0-BASELINE §2.5/2.6 (не пересобирались). Код не правился.

## 2. Кто и когда катит миграции БД (факт по коду)

Не инсталлятор и не оператор — **приложение на старте службы** (fail-fast, ADR-18):
`Program.cs:167–179` — `DatabaseBootstrapper.EnsureDatabaseCreatedAsync` (сырой
`CREATE DATABASE` к `master`, нужны права dbcreator+), затем `Program.cs:335–344` →
`IdentitySeeder` → `db.Database.MigrateAsync()` → сидеры. Любая ошибка → `LogCritical` +
`throw` → процесс падает, служба не входит в Running. Инсталлятор миграции не катит и
совместимость схемы не проверяет.

## 3. FINDINGS

### High

**[REL-01] · High · Evidence:** `scripts/publish-release.ps1:83–139` (нет очистки OutputDir),
`installer/MitLicenseCenter.iss:69` (`Source: "{#PublishDir}\*"`), лог ISCC 2026-06-12
(сжаты Hangfire.Core.dll, Microsoft.EntityFrameworkCore.dll и т.д.), дифф чистого публиша
(57 файлов) против `artifacts\0.1.0-beta\backend` (82 лишних).
**Суть:** релизный артефакт недетерминирован — `dotnet publish` идёт в накопительный каталог,
и всё, что там лежало (включая остатки `-FrameworkDependent`-публиша), упаковывается в
Setup.exe. Доказано фактом: собранный сегодня инсталлятор содержит 82 чужих файла, среди них
`MitLicenseCenter.Web.runtimeconfig.json`/`deps.json`/`Web.dll` старого FD-формата **рядом с
single-file exe** — эти файлы хост .NET читает при старте, поведение на машине заказчика без
установленного .NET непредсказуемо (вплоть до отказа запуска).
**Риск:** заказчику уезжает раздутый (+~26 МБ) артефакт с устаревшими DLL и конфигами чужого
режима публиша; состав поставки зависит от истории конкретной build-машины.
**Рекомендация:** в `publish-release.ps1` чистить/пересоздавать OutputDir перед `dotnet publish`
(или публишить во временный каталог и атомарно подменять); в `build-installer.ps1` — sanity-чек
состава (отсутствие `*.deps.json`/`runtimeconfig.json` при self-contained).
**Confidence:** High (факт упаковки доказан логом; влияние соседнего runtimeconfig.json на
запуск single-file exe — Medium, экспериментально не проверялось). *Скептик: подтверждён High.*

**[REL-02] · High · Evidence:** `installer/MitLicenseCenter.iss:575–585` (провал `sc start` →
MsgBox **без** Abort), `iss:65–69` (файлы заменяются в `[Files]` до старта), `Program.cs:335–344`
(миграции на старте приложения).
**Суть:** обновление поверх не имеет согласованного отката файлы↔БД. Файлы заменяются на новую
версию ДО старта службы; миграции катит новая версия на старте; при N>1 миграций EF применяет
их последовательно (каждую в своей транзакции) — падение на середине оставляет **частичную
схему**, файлы уже новые, отката нет. При этом провал старта службы **не прерывает установку**:
инсталлятор завершается «успехом» с неработающей панелью. Принудительного бэкапа/предупреждения
перед апгрейдом нет.
**Риск:** типовой сценарий «заказчик обновился через версию-другую» при сбое миграции оставляет
систему в нерабочем несогласованном состоянии; путь назад — только ручной restore БД +
переустановка старой версии, на которую инструкции нет (REL-05).
**Рекомендация:** минимум — экран мастера на апгрейде «сделайте бэкап БД + key ring» и Abort
(или явный экран ошибки) при провале старта службы; идеал — pre-flight проверка совместимости
схемы. **Confidence:** High. *Скептик: подтверждён High (одиночная миграция атомарна, но
сигнализация «успеха» при нерабочей службе подтверждена кодом без оговорок).*

### Medium

**[REL-03] · Medium · Evidence:** grep `sc failure|FailureActions|ChangeServiceConfig2|depend=`
по `installer/` и `scripts/` — 0 совпадений; `iss:472–477` — `sc create … start= auto` без
`depend=`.
**Суть:** recovery-политика службы не настраивается и зависимости от SQL Server нет. После
ребута хоста, если MSSQL поднимается медленнее, служба падает (fail-fast bootstrap) в Stopped
и **не перезапустится** до ручного вмешательства; то же при любом рантайм-краше.
**Риск:** панель лежит до утра, оператор узнаёт от пользователей.
**Рекомендация:** в `ConfigureService` добавить `sc failure … reset= 86400 actions=
restart/30000/restart/60000/restart/120000` и/или `depend=` на службу SQL (для локального SQL);
как минимум — инструкция в OPERATIONS.md (services.msc → Recovery).
**Confidence:** High (факт), Medium (величина риска). *Скептик: понижен High→Medium.*

**[REL-04] · Medium · Evidence:** `Infrastructure/DependencyInjection.cs:258–274` —
`PersistKeysToFileSystem` **без** `ProtectKeysWith*`; официальный док Microsoft («If you specify
an explicit key persistence location … keys are no longer encrypted at rest»); фактическая
структура dev key ring XML: `<masterKey><value>` без `<encryptedSecret>`.
**Суть:** канон (CLAUDE.md, OPERATIONS.md, описание настройки «Хранится зашифрованным DPAPI»)
утверждает «DPAPI key ring» — фактически ключи Data Protection лежат **plaintext XML** в
`%ProgramData%\MitLicenseCenter\keys` (защита — только ACL). Скептик-проверка разрешила
противоречие двух аудиторов: автоматического DPAPI-fallback при явном PersistKeysToFileSystem
**не существует**. Следствия: (а) заявленная модель безопасности не соответствует коду;
(б) **позитив:** бэкап key ring + БД полностью переносим на новое железо — «единый бэкап-юнит»
работает (единственный шифруемый секрет — `OneC.Cluster.AdminPassword`, при полной утрате ключей
восстанавливается ручным повторным вводом в UI). Важный trade-off: «просто добавить
`ProtectKeysWithDpapi`» закроет plaintext, но **сломает** перенос бэкапа на новое железо;
совместить шифрование at rest и переносимость может только `ProtectKeysWithCertificate` (+бэкап .pfx).
**Риск:** доступ к файлам keys (бэкап-архив, теневой админ) = пароль кластера 1С; ложные
обещания канона при аудите заказчиком.
**Рекомендация:** осознанное решение через ADR: либо оставить как есть и исправить канон/описание
настройки («шифрование Data Protection, ключи защищены ACL, переносимы»), либо сертификат.
Плюс: ловить `CryptographicException` в `SettingsStore` с понятным сообщением про key ring
(сейчас оператор увидит голый стектрейс в Event Log).
**Confidence:** High (~0.98 по скептику с тройным подтверждением: код + док + XML).

**[REL-05] · Medium · Evidence:** `docs/OPERATIONS.md:83–103` («что бэкапить» есть; «как
восстановить» — нет), частичный откат релиза — только OPERATIONS.md:44–53.
**Суть:** пошаговой инструкции восстановления панели нет — ни «та же машина», ни «новое железо»,
ни совместное восстановление key ring + БД, ни действия при утрате ключей (есть лишь
предупреждение «re-enter every secret setting»). Аудитория — оператор 1С-хостера без DBA.
**Риск:** при реальной аварии оператор не знает порядок действий; в сочетании с REL-02 это
единственный путь спасения после неудачного апгрейда.
**Рекомендация:** раздел «Восстановление» в OPERATIONS.md: (а) та же машина; (б) новое железо
(restore БД → Setup.exe → keys скопировать (переносимы, см. REL-04) или заново ввести пароль
кластера); (в) restore-drill и для key ring. **Confidence:** High. *Скептик: понижен High→Medium.*

**[REL-06] · Medium · Evidence:** `iss:457–465` (`GrantServiceAccountRights` — только Modify на
`%ProgramData%\MitLicenseCenter`), `iss:592–671` + метка `iss:770` (тест SQL в режиме A идёт под
Integrated Security **установщика-админа**, не сервис-аккаунта); гоча CLAUDE.md/OPERATIONS.md про
`inetsrv\config` (читаем только Administrators/SYSTEM).
**Суть:** режим A (Windows-аккаунт под службой) ставится с двумя зазорами: (1) аккаунту не
выдаются права на IIS-конфиг → **все публикации после установки в статусе «Ошибка проверки»**;
(2) «Проверить подключение» зелёное не гарантирует, что у сервис-аккаунта есть доступ к SQL —
служба создастся и упадёт на старте.
**Риск:** «установка прошла, а половина продукта не работает» — худший первый опыт заказчика;
диагностика требует знания внутренностей.
**Рекомендация:** в мастере (режим A) — явный экран с требованиями к аккаунту (права на SQL,
членство в Administrators или read на inetsrv для IIS-функций); пост-установочный чек-лист в
OPERATIONS.md. **Confidence:** High.

**[REL-07] · Medium · Evidence:** `iss:368–397` (`WriteProductionConfig` — режим B пишет
`Password=…` открытым текстом в `{app}\appsettings.Production.json`), ACL файла наследуется от
Program Files (Users: Read), инсталлятор ACL не ужесточает.
**Суть:** в режиме B SQL-пароль читается любым локальным пользователем сервера.
**Риск:** локальный пользователь без прав получает доступ к БД панели (логины/настройки/аудит).
**Рекомендация:** `icacls` на файл в режиме B (SYSTEM/Administrators/сервис-аккаунт) либо
задокументированная рекомендация режима A. **Confidence:** High.

**[REL-08] · Medium · Evidence:** состав чистого публиша (4×`*.pdb`,
`appsettings.Development.json`, `web.config`); `iss:69` исключает только
`appsettings.Production.json`.
**Суть:** заказчику ставятся отладочные символы (pdb = имена/пути исходников — облегчает реверс
закрытого продукта). `Development.json` безвреден (секретов нет, в Production игнорируется),
`web.config` мёртвый (хостинг — служба, не IIS).
**Риск:** information disclosure для B2B-поставки; непрофессиональный вид артефакта.
**Рекомендация:** `-p:DebugType=none` (или не копировать pdb) + Excludes в .iss для
Development.json/web.config. **Confidence:** High. *Скептик: понижен High→Medium.*

**[REL-09] · Medium · Evidence:** в `installer/MitLicenseCenter.iss` нет `SignTool`-директив,
в `scripts/` нет signtool-вызовов; лог ISCC — без шага подписи.
**Суть:** ни Setup.exe, ни `MitLicenseCenter.Web.exe` не подписаны Authenticode.
**Риск:** SmartScreen «Неопознанный издатель» при каждом запуске установщика у заказчика;
антивирусы агрессивнее к неподписанным self-contained single-file exe (типовой FP-профиль);
невозможно верифицировать целостность поставки.
**Рекомендация:** сертификат подписи кода (OV/EV) + `SignTool=` в .iss и подпись exe до
упаковки. **Confidence:** High (факт отсутствия).

**[REL-10] · Medium · Evidence:** Hangfire.Core — **LGPL v3** (hangfire.io/licenses.html);
xlsx (SheetJS CE) — Apache 2.0 (атрибуция обязательна); в репо/инсталляторе нет
`THIRD_PARTY_LICENSES` (grep по репо — отсутствует).
**Суть:** дистрибутив не сопровождается текстами лицензий и атрибуцией. LGPL v3 при
NuGet-линковке **не требует** открытия кода продукта (коммерческая лицензия Hangfire нужна
только при распространении модифицированного Hangfire), но требует: уведомление об
использовании, тексты GPLv3+LGPLv3 в поставке, возможность замены библиотеки. Apache 2.0
(xlsx) требует атрибуцию. Microsoft.Web.Administration — проприетарная MS, redistribution в
составе .NET-приложений разрешён. Остальной стек — MIT/BSD, чисто.
**Риск:** формальное нарушение LGPL/Apache в коммерческой поставке; блокер при лицензионном
аудите enterprise/госзаказчика.
**Рекомендация:** `THIRD_PARTY_LICENSES.txt` в `[Files]` инсталлятора (Hangfire LGPLv3 + тексты,
SheetJS Apache 2.0, перечень MIT). **Confidence:** High.

**[REL-11] · Medium · Evidence:** `Directory.Packages.props` — нет file-sink (Serilog/NLog
отсутствуют); `Program.cs:27` `UseWindowsService` (логи → Event Log «Application», источник
MitLicenseCenter); `appsettings.Production.json` — секции Logging нет (уровни из базового
`appsettings.json`: Default=Information); OPERATIONS.md — нет разделов «как достать логи»,
«порт занят», «SQL упал в рантайме».
**Суть:** единственный канал логов в prod — общий Windows Event Log Application (дефолт ~20 МБ,
перезапись по кругу) без инструкции по чтению/экспорту/увеличению размера; runbook'ов типовых
отказов нет. Health-endpoint'ы при этом в порядке: `/api/v1/health` (liveness) и `/health/ready`
(database → 503; ras/hangfire → degraded при HTTP 200) — пригодны для мониторинга, факт
сверен с кодом (`HealthEndpoints.cs`, `ReadinessEvaluator.cs`).
**Риск:** при инциденте история может быть уже затёрта; оператор без runbook не диагностирует
«порт занят» (голый SocketException в Event Log) или «SQL упал в рантайме» (500/503 без
инструкции следующего шага).
**Рекомендация:** OPERATIONS.md — раздел «Логи» (PowerShell-однострочник выборки по источнику,
увеличение журнала до 50–100 МБ) + runbook «панель не отвечает» (health/ready → Event Log →
netstat → действия); рассмотреть rolling-file sink вторым провайдером; примечание: `degraded`
по hangfire отдаёт 200 — мониторингу заказчика проверять тело, не только код.
**Confidence:** High.

**[REL-12] · Medium · Evidence:** `.github/workflows/ci.yml` — нет jobs для
publish-release/инсталлятора; `build.ps1` их тоже не гоняет; pre-commit — только lint-staged.
**Суть:** релизный путь (publish-release.ps1 → ISCC → Setup.exe) не проверяется **нигде**
автоматически; .iss никем не валидируется; EF-миграции не прогоняются против реального MSSQL
(только юнит-провайдеры); e2e-тестов нет как класса. Сам ci.yml при этом корректно
воспроизводит build.ps1 шаг-в-шаг (единственное расхождение — CI фильтрует
`Category!=Smoke`, build.ps1 нет; Smoke-тесты — живой rac.exe, на dev-стенде проходят);
известный красный Actions по биллингу — вне скоупа, не finding.
**Риск:** несобираемость релизного артефакта обнаружится в день поставки (сегодняшний прогон
собрался, но гарантия разовая).
**Рекомендация:** job/локальный гейт «собрать publish + скомпилировать .iss» перед релизом;
зафиксировать в OPERATIONS.md релизный чек-лист. **Confidence:** High.

**[REL-13] · Medium · Evidence:** `iss:65–69` — `[Files]` с `ignoreversion recursesubdirs`,
очистки `{app}\wwwroot` перед копированием нет.
**Суть:** при обновлениях старые хэшированные ассеты SPA (`assets/*-<hash>.js`) предыдущих
версий не удаляются — копятся orphan-файлы; `index.html` перезаписывается и ссылается на новые,
так что поломки нет, но при сбое копирования возможен рассинхрон (белый экран).
**Риск:** мусор у заказчика, труднодиагностируемый рассинхрон в редком сценарии.
**Рекомендация:** `[InstallDelete] Type: filesandordirs; Name: "{app}\wwwroot"` перед
копированием. **Confidence:** Medium.

### Low

**[REL-14] · Low · Evidence:** dotnet.microsoft.com support policy; `project.assets.json`
(runtime packs `[10.0.9]`); `dotnet --version` → 10.0.301 при `global.json` 10.0.100 +
`rollForward latestFeature`.
**Суть:** .NET 10 — **LTS до 2028-11-14** (не STS — опасения снимаются); сегодняшняя сборка
вшивает актуальный runtime 10.0.9 (июньские CVE-2026-45491/45591 закрыты; CVE-2026-45490 —
SDK-only, заказчику не едет). Но вшиваемый патч определяется SDK build-машины — процессного
шага «обновить SDK перед релизом» нет.
**Риск:** релиз со «вчерашней» build-машины унесёт заказчикам устаревший рантайм; security-патч
рантайма всегда требует перевыпуска продукта (свойство self-contained).
**Рекомендация:** пункт релизного чек-листа «обновить SDK 10.0.x до текущего» + ежемесячный
взгляд на Patch Tuesday. **Confidence:** High. *Скептик: понижен High→Low (CVE фактически закрыты).*

**[REL-15] · Low · Evidence:** NuGet: Swashbuckle.AspNetCore актуальная 10.2.1 (в проекте 7.2.0,
3 мажора позади; CVE по 7.x не зафиксированы; Swagger в prod выключен `EnableSwagger=false`);
Hangfire 1.8.18 → 1.8.23 (минорные фиксы, CVE нет, проект жив).
**Суть/Рекомендация:** плановые обновления после релиза; не блокер. **Confidence:** Medium.

**[REL-16] · Low (понижен с High) · Evidence:** официальный advisory SheetJS CVE-2023-30533:

**Суть:** известные CVE замороженного npm-пакета xlsx 0.18.5 — в **парсере**; проект write-only
→ CVE неприменимы. Останется флагом в сканерах зависимостей заказчика.
**Рекомендация:** lint-запрет `XLSX.read`/`readFile` (guard от будущей регрессии); при случае —
миграция на exceljs. **Confidence:** High. *Скептик: понижен по официальному advisory.*

**[REL-17] · Low · Evidence:** баннер ISCC 6.7.3 «Non-commercial use only»;
jrsoftware.org/files/is/license.txt («including commercial applications»), isorder.php.
**Суть:** лицензия Inno Setup разрешает коммерческое использование бесплатно; с 6.5.0 введена
*добровольная* платная коммерческая лицензия (порог — выручка >$5k/год), баннер означает
отсутствие введённого ключа и в собранный Setup.exe ничего не вшивает. Юридически сборка
легальна; этически/репутационно — разработчики просят коммерческих пользователей купить ключ.
**Рекомендация:** купить single-user ключ для build-машины (баннер исчезнет).
**Confidence:** High (факты), Medium (трактовка «request»).

**[REL-18] · Low · Evidence:** `iss:363,615,626,698,702`.
**Суть:** все строки подключения генерятся с `Encrypt=True;TrustServerCertificate=True` —
канал шифруется, но сертификат SQL не проверяется (MITM не детектируется). Для типового
single-host (SQL локально) приемлемо, но выбора у оператора нет.
**Рекомендация:** зафиксировать как принятый риск в SECURITY.md. **Confidence:** High.

**[REL-19] · Low · Evidence:** `iss:407–421, 547, 1069`.
**Суть:** при откате неудачной установки (`RaiseException` на `sc create`) файлы `[Files]`
откатываются, но созданные через `[Code]` `initial-admin.secret` (пароль admin открытым
текстом), ACL и ярлык остаются — вне трекинга Inno. ACL каталога ограничивает доступ;
перезатираются при повторной установке.
**Рекомендация:** чистить secret-файл перед `RaiseException`. **Confidence:** Medium.

**[REL-20] · Low · Evidence:** `scripts/build-installer.ps1:78` — захардкоженный путь
`C:\Users\<user>\AppData\Local\Programs\Inno Setup 6\ISCC.exe` в трекаемом скрипте.
**Суть:** путь конкретной dev-машины в репо; работает и без него (строкой выше тот же кандидат
через `$env:LOCALAPPDATA`). **Рекомендация:** удалить строку. **Confidence:** High.

**[REL-21] · Low · Evidence:** `.github/workflows/ci.yml`, `.husky/`, отсутствие
`.github/dependabot.yml`.
**Суть (CI-гигиена, пакетом):** `pnpm format:check` не гоняется нигде (только lint-staged по
staged); dependabot/`pnpm audit`/`dotnet list package --vulnerable` — нигде; paths-фильтров нет
(доки триггерят полный прогон); actions пинены major-тегом, не SHA; Node жёстко 22.13 (LTS до
2027-04); расхождение `Category!=Smoke` (CI) vs без фильтра (build.ps1) не задокументировано.
**Рекомендация:** dependabot (npm+nuget+actions) + format:check шаг + заметка про Smoke.
**Confidence:** High.

**[REL-22] · Low · Evidence:** `Program.cs:138`, `Web/Hangfire/JobRetentionStateFilter.cs`,
`IReconciliationJob.cs:21`.
**Суть:** у джобов нет явной retry-политики (дефолт Hangfire — 10 ретраев с backoff), Failed-джобы
не экспирируют — копятся красным в дашборде; оператору не отличить «ретрай нормы» от постоянного
сбоя. **Рекомендация:** осознанный `[AutomaticRetry]` + абзац в OPERATIONS.md. **Confidence:** Medium.

## 4. Ответы на вопросы постановки (сводно)

- **Сборка фактом:** publish и Setup.exe собираются, exit 0 (см. §1). Обломов нет → Blocker'ов
  уровня «не собирается» нет.
- **Обновление поверх:** служба останавливается с ожиданием (до ~30 с), `appsettings.Production.json`
  и key ring сохраняются (Excludes + `uninsneveruninstall`), файлы перезаписываются; миграции
  катит новая версия на старте; отката нет, провал старта не абортит установку (REL-02).
- **Удаление:** `sc stop/delete` + снятие firewall-правила; БД не трогается; `%ProgramData%`
  (key ring) — только по явному согласию (дефолт «сохранить»); генерённый
  appsettings.Production.json удаляется.
- **Бэкап/восстановление:** «key ring + БД = единый юнит» — работает, причём **переносим на новое
  железо** (ключи не DPAPI, plaintext — REL-04); при полной потере keys теряется один секрет
  (пароль админа кластера 1С), восстанавливается вводом в UI. Инструкции восстановления нет (REL-05).
- **Сервер умер, есть бэкап БД (+keys):** панель на новом железе **поднимется**, все данные в БД
  сохранны; с бэкапом keys — без потерь вообще; без keys — заново ввести пароль кластера.
- **Стек:** .NET 10 = LTS до 2028-11-14; runtime в артефакте 10.0.9 (актуален); React 19 /
  Tailwind 4 / Vite 8 / EF Core 10 — текущие мажоры; Hangfire жив (1.8.23); применимых открытых
  CVE в поставке после сегодняшней пересборки **не выявлено** (xlsx-CVE неприменимы — write-only).
  Лицензии: всё MIT/Apache/BSD, кроме Hangfire **LGPL v3** (совместимо при NuGet-линковке, но
  нужен THIRD_PARTY_LICENSES — REL-10) и Microsoft.Web.Administration (redistribution разрешён).
- **CI:** ci.yml корректно зеркалит build.ps1; дыры покрытия — релизный путь, миграции на живом
  MSSQL, e2e, security-сканы (REL-12, REL-21).

## 5. Итог

### Топ-5 рисков релиза

1. **REL-01 — недетерминированный состав поставки:** инсталлятор пакует всё содержимое
   накопительного каталога; сегодня фактом уехало 82 чужих файла, включая конфиги, способные
   повлиять на запуск exe. Чинится тривиально (очистка каталога), но до фикса каждый собранный
   Setup.exe подозрителен.
2. **REL-02 — апгрейд без страховки:** нет бэкапа перед миграциями, нет отката, установка
   рапортует «успех» при неработающей службе. Первый же неудачный апгрейд у заказчика — инцидент
   без инструкции спасения (усугубляется REL-05).
3. **REL-06 — режим A даёт полуработающий продукт:** IIS-публикации в «Ошибка проверки» сразу
   после установки, ложно-зелёный тест SQL. Худший первый опыт + нагрузка на поддержку.
4. **Операционная слепота (REL-03 + REL-11):** служба не самовосстанавливается, логи — только
   общий Event Log без инструкции, runbook'ов нет. Любой ночной инцидент = многочасовой простой.
5. **REL-10 — LGPL/Apache-формальности не закрыты:** для коммерческой поставки enterprise/гос —
   потенциальный стоп при лицензионном аудите. Дёшево закрывается одним файлом.

### Вердикт: **ГОТОВО С ОГОВОРКАМИ**

Сборка и установка работают фактом: publish OK, Setup.exe собрался, happy-path мастера выстроен
аккуратно (rc-проверки sc.exe, стоп службы с ожиданием, сохранение конфига/ключей на апгрейде,
keep-data-диалог на удалении). Blocker'ов нет. Но **до первой поставки заказчику** надо закрыть
минимум: REL-01 (очистка каталога публиша — копеечный фикс), REL-02 (бэкап-предупреждение +
Abort при провале старта), REL-10 (THIRD_PARTY_LICENSES), REL-09 (подпись кода — иначе
SmartScreen встретит каждого заказчика), и решить ADR-вопрос REL-04 (канон «DPAPI» vs реальность
plaintext — сейчас документация обещает не то, что делает код).

### Что не успел покрыть

- **Реальная установка Setup.exe на чистую ВМ** — мастер, апгрейд и удаление проанализированы
  по .iss (opus) и подтверждены компиляцией, но интерактивно не прогонялись; поведение
  «порт занят», «sc create 1057», апгрейд-цикл двух версий не проверены фактом.
- **Запуск собранного exe из артефакта** (включая эксперимент «single-file exe рядом со stale
  runtimeconfig.json» — влияние на старт оценено аналитически, Medium confidence).
- **EF-миграции против чистого живого SQL Server** из релизного артефакта.
- Лицензии **транзитивных** зависимостей (проверен только верхний уровень A0).
- История git на секреты (вне скоупа и здесь, и в А0).
