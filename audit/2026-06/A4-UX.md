# A4 — UX/UI и тексты

Этап А4 пред-релизного аудита. Дата: **2026-06-12**, стенд: локальный dev (SPA :5173 + backend :5080,
БД пересоздана `db-reset` + `perf-seed -Realistic`). Данные стенда: 21 клиент (20 синтетических +
реальный `mitpro`), 51 инфобаза, **реальный кластер 1С 8.5** (rac.exe 8.5.1.1302) с 1 живым сеансом,
100 000 записей аудита и год истории лицензий. Метод: обход живых экранов субагентами (sonnet) по
зонам — мониторинг, управление, система, визуал/доступность, тексты (словарь i18n +
backend-сообщения), ошибки форм и поведение при недоступном backend, плюс сквозной сценарий
«новый клиент» на реальной базе кластера `test` (выполнен дополнительным прогоном по решению
владельца, findings UX-42…47); итоговая калибровка и приоритизация — opus. Оценка выполнена «глазами
стороннего UX-эксперта»; доки `docs/05/06` не использовались как эталон.

**Шкала severity:** Blocker — оператор не может выполнить ключевую задачу или теряет данные;
High — регулярное серьёзное трение; Medium/Low — полировка.

---

## 1. Сводная оценка

MitLicense Center производит впечатление **архитектурно зрелого, но операционно недоговорённого**
продукта. Базовая ткань UI сделана добросовестно: темы переключаются, диалоги имеют focus-trap и
закрытие по Esc, удаление защищено вводом имени, аудит остаётся быстрым на 100 тысячах записей,
экспорт отчётов работает в четырёх форматах, пустые состояния в большинстве мест информативны.
Backend честно защищает критичные инварианты (нельзя отключить или разжаловать самого себя,
серверные ошибки санитизированы — в UI нет ни `[object Object]`, ни голых кодов, ни английских
заглушек). Продукт не выглядит как сырой прототип.

Однако **главная слабость системная и бьёт точно в сердце ценностного предложения**. MitLicense
Center существует ради контроля квот лицензий — а именно нарушителя квоты в нём труднее всего
увидеть. В списке клиентов нет колонки потребления, в карточке клиента нет текущего расхода и ссылок
на отчёты, а в самих отчётах превышение «6 из 5 (120 %)» выводится обычным текстом без какого-либо
визуального акцента. Оператор не может с одного взгляда ответить на свой главный вопрос — «кто
сейчас за пределами лимита?». Это не косметика, это разрыв между обещанием и интерфейсом.

Вторая системная слабость — **хрупкость при недоступном backend и «молчание» об ошибках**. Холодная
загрузка при мёртвом сервере показывает обычную форму входа без намёка на проблему (наблюдено
вживую); попытка входа даёт безликое «Произошла ошибка. Попробуйте ещё раз.» (наблюдено); по коду —
залогиненного пользователя reload молча выкидывает на /login, а данные в открытой сессии «замирают»
без предупреждения. В одном подтверждённом случае (создание пользователя с пробелом в логине → 500)
интерфейс не сказал вообще ничего. В словаре даже есть готовая строка «Нет связи с сервером.»
(`errors.network`) — она нигде не используется.

Третья тема легче — **масштабируемость и полировка**: списки без поиска/сортировки/пагинации
поплывут при росте; терминология плавает («клиент/арендатор», «сеанс/сессия», английские роли в
русских сообщениях); остались протухшие тексты с жаргоном разработки («появится в следующих PR
Stage 2», «появится в Stage 3 (admin-driven Reconcile)»); есть реальные проблемы доступности
(контраст бейджей светлой темы 1.6–2.2:1, англоязычные accessible-имена). По отдельности не
блокирует, вместе — ощущение незавершённости.

## 2. Топ-5 проблем по влиянию на оператора

1. **[UX-02] Нарушитель лимита визуально невидим.** Главная ежедневная задача продукта: оператор
   открывает панель, чтобы найти, кто вышел за квоту — и не может сделать это с одного взгляда.
   Прямой удар по ценности продукта.
2. **[UX-01] Молчаливый сбой формы (500 без обратной связи).** Оператор не знает, выполнилось ли
   действие; рискует повторить операцию или принять неверное решение. Единственный Blocker аудита.
3. **[UX-03] Молчание при недоступном backend.** Администратор инфраструктуры особенно чувствителен
   к «застывшим» данным и необъяснённым разлогинам — подрывает доверие к панели как к источнику
   правды.
4. **[UX-06] Параметры без разметки «обязательно для старта».** Бьёт в самый хрупкий момент —
   первый запуск: без документации оператор не поймёт, что без RAS/SQL/папки бэкапов система
   неработоспособна.
5. **[UX-42] Форма привязки базы предвыбирает чужого клиента.** В главном сценарии онбординга
   клиента легко привязать базу не тому тенанту — её сеансы будут считаться в чужой лимит, ошибка
   обнаружится не сразу (выявлено сквозным прогоном).

Следом за пятёркой: **[UX-05]** нет поиска по клиентам (постоянное трение, растёт с каждым
тенантом) и **[UX-43]** опубликовать через панель можно, снять публикацию — нельзя.

## 3. Вердикт

**Готово с оговорками** к v1 (0.1.0-beta) — при условии короткого обязательного блока до релиза.

**Must-fix до релиза:**
- **UX-01** — гарантированный fallback-фидбек на любой 5xx во всех формах + клиентская валидация логина;
- **UX-02** — минимум: цветовой акцент на превышении лимита везде, где оно выводится, и колонка
  потребления в списке клиентов;
- **UX-03** — минимум: глобальный индикатор потери связи, задействовать `errors.network`, не
  разлогинивать при сетевой (не 401) ошибке;
- **UX-06** — пометить обязательные для старта настройки;
- **UX-10** — убрать жаргон «Stage 2 / Stage 3» из видимых текстов (дёшево, но бьёт по доверию);
- **UX-11** — клиентская валидация имени SQL-базы (та же природа, что UX-01);
- **UX-42** — не предвыбирать клиента в форме привязки базы (риск привязать базу чужому клиенту в
  главном сценарии; выявлено сквозным прогоном).

Сильный кандидат в первый патч после релиза: **UX-43** (нет «снять публикацию» — операция
необратима через UI; вручную снимается в IIS, поэтому не блокер).

**Можно после релиза:** весь Medium/Low (терминология, сортировка/пагинация, расшифровки кодов,
destructive-стили, контраст, aria). После must-fix имеет смысл короткий повторный обход двух путей:
«найти нарушителя лимита» и «форма под сбоем backend».

## 4. Findings

### Blocker

**[UX-01] Молчаливый сбой формы при серверной 500-ошибке** · Blocker · /users, форма «Добавить
пользователя» · *(S7-01 + S4-08)*
Логин с пробелом «a b» проходит клиентскую валидацию, сервер отвечает 500
(`"Произошла непредвиденная ошибка…"` + traceId), а UI не показывает **ничего** — ни тоста, ни
инлайна; форма открыта, кнопка снова активна. Подтверждено вживую.
**Рекомендация:** клиентский whitelist символов логина + гарантированный fallback-тост на любой
необработанный 5xx во всех мутациях.

### High

**[UX-02] Нарушитель лимита визуально невидим** · High (системная тема) · /reports, /tenants,
карточка клиента · *(S2-01 + S3-02 + S3-03; подтверждено сквозным
сценарием S9-05)*
«Пик за период: 6 из 5 (120%)» — обычный текст без цвета/иконки; в списке клиентов нет колонки
потребления (только статический лимит); карточка клиента не показывает текущий расход и не ссылается
на отчёты/аудит клиента. В сквозном сценарии ответ «сколько сейчас потребляет клиент» нашёлся только
на дашборде (и то лишь пока клиент в топ-N) либо через фильтр в «Сеансах».
**Рекомендация:** красный акцент при ≥100% везде; колонка «использовано/лимит» с прогресс-баром в
списке; блок текущего потребления со ссылками в карточке.

**[UX-03] Хрупкость и «молчание» при недоступном backend** · High (системная тема) · все экраны · *(S8-01…06)*
Наблюдено: холодная загрузка при мёртвом backend → обычная форма входа без намёка на сбой; попытка
входа → тост «Произошла ошибка. Попробуйте ещё раз.» (~4 с). По коду (однозначно): reload
залогиненного → редирект на /login (`ProtectedRoute`: isError→Navigate); данные с
`placeholderData` «замирают» молча до 30+ с; глобального индикатора «нет связи» нет; строка
`errors.network` = «Нет связи с сервером.» существует в словаре, но не используется; «Обновлено N
сек назад» исчезает при ошибке. Постраничные банеры «Не удалось загрузить…» появляются после
ретраев (~5–15 с).
**Рекомендация:** глобальный баннер потери связи; различать сетевую ошибку и 401 в `useMe`;
задействовать `errors.network`; явный признак «данные устарели».

**[UX-04] Несогласованная подача ошибок форм** · High · /login vs остальные формы · *(S7-04 + S8-03)*
На /login ошибка — тостом (поля не подсвечены), во всех остальных формах — инлайн под полем; все
сетевые ошибки сворачиваются в один generic-тост.
**Рекомендация:** единый паттерн (инлайн для валидации, тост для глобальных/сетевых с осмысленным
текстом).

**[UX-05] Нет поиска по клиентам** · High · /tenants · *(S3-01)*
Уже на 23 клиентах список листается вручную; фильтров и поиска нет.
**Рекомендация:** строка поиска по названию.

**[UX-06] Параметры: не размечены обязательные для старта** · High · /settings · *(S4-10)*
35 параметров выглядят равнозначно; нет маркировки, что без RAS ExePath / SQL-инстанса / папки
бэкапов ключевые функции не работают.
**Рекомендация:** блок «Необходимо для запуска» или бейджи + баннер на дашборде, пока не заданы.

**[UX-07] Tooltip графиков — белый фон в тёмной теме** · High · /reports · *(S2-02)*
Recharts-тултип с жёстко-белым фоном нечитаем в тёмной теме (тема по умолчанию — тёмная).
**Рекомендация:** стилизовать `contentStyle` под CSS-переменные темы.

**[UX-08] Сайдбар не сворачивается при ширине ≤1024px** · High · все разделы · *(S5-02)*
Фиксированный сайдбар (~252px) перекрывает колонки таблиц на 1024px и 768px. На 1366×768 — всё ок,
горизонтального скролла нет.
**Рекомендация:** брейкпоинт ~1024px с авто-сворачиванием в иконки/оверлей.

**[UX-09] UI предлагает self-действия, которые сервер запретит** · High→Medium по сути, выделено
из заявленных Blocker · /users · *(S4-06 + S4-07, перекалибровано)*
Admin видит «Отключить» и «Сменить роль» применительно к себе без предупреждений. Верифицировано:
backend блокирует оба действия (409, guard'ы `UserCannotDisableSelf`/`UserCannotChangeOwnRole` в
`UsersEndpoints.cs`) — потери доступа не будет, но UI ведёт в тупик; качество показа 409 не
проверялось.
**Рекомендация:** скрывать/гасить self-действия + дружелюбный текст при 409.

**[UX-10] Протухшие тексты с жаргоном разработки** · High · формы/заглушки · *(S6-01 + S6-02)*
`common.comingSoon` = «Этот раздел появится в следующих PR Stage 2.»;
`infobases.form.sectionPublicationHint` = «…Сама публикация в IIS появится в Stage 3 (admin-driven
Reconcile).» — при том что публикация давно реализована.
**Рекомендация:** убрать/переписать без внутренних терминов.

**[UX-11] Имя SQL-базы не валидируется на клиенте** · High · /infobases, форма «Новая инфобаза» ·
*(S7-02)*
Кириллица и пробелы («моя база данных») не дают клиентской ошибки — уходят на сервер.
**Рекомендация:** клиентский паттерн имени БД с инлайн-сообщением «Только латиница, цифры и _»,
синхронно с `InfobaseValidationRules.cs`.

**[UX-42] Форма привязки базы предвыбирает чужого клиента** · High · /infobases, «Разобрать» →
«Назначить» ·
*(S9-01)*
При назначении нераспределённой базы кластера диалог «Новая инфобаза» предзаполняет поле «Клиент»
первым клиентом списка (на стенде — `mitpro`), а не оставляет выбор оператору. Невнимательный
оператор привяжет базу к чужому клиенту — и её сеансы будут считаться в чужой лимит; ошибка выглядит
правдоподобно и обнаружится не сразу.
**Рекомендация:** не предвыбирать клиента (placeholder «Выберите клиента» + required) либо
наследовать активный фильтр по клиенту.

**[UX-43] Нет действия «снять публикацию из IIS»** · High · /infobases, меню базы · *(S9-02)*
В меню есть «Опубликовать», но обратного действия нет. Удаление инфобазы честно говорит «IIS не
затрагивается» — но в результате публикация остаётся в IIS бесхозной (default.vrd + web.config), и
снять её через панель невозможно ни до, ни после удаления записи.
**Рекомендация:** пункт «Снять публикацию» (webinst unpublish) + при удалении инфобазы — явный выбор
«также снять из IIS».

## 5. Сквозной сценарий «новый клиент» (выполнен на реальной базе кластера)

Прогон: создать клиента «UX-Аудит Сквозной» (лимит 2) → привязать реальную базу кластера `test` →
опубликовать → найти потребление → удалить всё в обратном порядке.

**Что хорошо.** Создание клиента — 2 клика от дашборда, форма без ловушек. Путь к привязке базы
подсказывается баннером «2 базы кластера не заведены в панель → Разобрать»; диалог «Назначить»
предзаполняет имя базы кластера и имя SQL-БД из кластера. Жизненный цикл замыкается честно: после
удаления записи база вернулась в «нераспределённые», баннер восстановился; потребление нового
клиента появилось на дашборде («UX-Аудит Сквозной: 1/2 (50%)») без дополнительных настроек.
Диалоги удаления — с вводом имени и честным «IIS не затрагивается».

**Где оператор застрянет без документации** (по итогам прогона): предвыбранный чужой клиент в форме
привязки (UX-42); ошибка публикации с техническим ключом `OneC.RAS.Endpoint` и незакрывающимся
диалогом (UX-44); желание снять базу из IIS — действия нет (UX-43); ответ «сколько сейчас потребляет
клиент» из таблицы клиентов не получить (UX-02/46). После создания клиента следующее действие
(«добавьте базу») ничем не подсказывается.

**Ограничение прогона:** сама публикация webinst не выполнилась (стенд без заданного
`OneC.RAS.Endpoint`) — наблюдалась существующая публикация `/test` (source=Unknown) и UX ошибки;
живое нарушение лимита не воспроизводилось.

## 7. Что работает хорошо

- **Защита опасных действий**: удаление клиента/базы требует ввода точного имени; завершение
  сеанса — ввода идентификатора приложения + предупреждение о потере несохранённых данных; публикация —
  ввода токена «сайт+путь». Диалог удаления базы явно говорит «IIS не затрагивается».
- **Сообщения об ошибках без классики**: ни «[object Object]», ни голых HTTP-кодов, ни английских
  fallback-текстов; backend-ошибки санитизированы, с correlationId (не светится в UI).
- **Инлайн-валидация форм** живая (ошибка исчезает при исправлении), тексты императивные и понятные;
  требования к паролю показаны заранее; серверный 400 смены пароля показан инлайн корректно.
- **Аудит быстрый на 100k записей**, сценарий «кто завершил сеансы Ромашки» решается за 4 клика,
  тексты событий информативны (GUID сеанса, пользователь, причина «превышен лимит»).
- **Отчёты**: экспорт CSV/Excel/HTML/PDF, честное сообщение об ограничении диапазона 31 день,
  корректные «ёлочки» в именах.
- **Навигация**: KPI-карточки дашборда кликабельны; баннеры о расхождениях с кластером на /infobases —
  проактивные; форма создания базы прячет редкие поля в «Дополнительно», есть «Ввести вручную» и
  «Обновить» у списков.
- **Каркас доступности**: focus-trap и Esc в диалогах, единый паттерн заголовков, последовательная
  семантика цветов бейджей, 1366×768 без горизонтального скролла.

## 8. Что не успели посмотреть / ограничения

- **Сквозной сценарий выполнен** (дополнительный прогон на реальной базе кластера `test`, раздел 5),
  но **фактическая публикация webinst не состоялась** — упёрлась в незаданный `OneC.RAS.Endpoint`
  (само сообщение об ошибке — finding UX-44); наблюдалась уже существующая публикация.
- **Живое нарушение лимита не воспроизводилось** (на реальном кластере 1–2 сеанса; enforcement
  наблюдался только по следам в аудите). Оценка видимости нарушений — по экранам с историческими
  данными.
- **Не выполнялись вживую**: запуск/удаление бэкапа, kill сеанса, операции IIS
  (перезапуск пулов/сайтов), logout, смена пароля (только валидация).
- **Роль Viewer не проверялась** (обход только под Admin).
- **S8-01/02 (замирание данных, редирект на /login после reload)** — выведены из кода
  (`queryClient`, `ProtectedRoute`), вживую подтверждены только холодная загрузка и generic-тост
  логина.
- **Не проверено**: накопление тостов, UX 409-ответов self-guard'ов, потенциальный «вечный спиннер»
  на медленном (не мёртвом) backend.
- Параллельно с аудитом владелец вёл на стенде живую работу (создание `mitpro`, переключение RAS на
  реальный rac.exe) — сид-сценарий с 462 живыми сеансами был активен лишь первые минуты, поэтому
  поведение таблицы сеансов под нагрузкой оценено только по признакам UI (отсутствие
  пагинации/сортировки).
