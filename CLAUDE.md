## Рабочий процесс

### Doc-driven

Документация в `docs/` — present-tense, единственный источник правды.
Любое изменение кода правит документацию в **том же** коммите/PR — без changelog-хвостов
и секций «история изменений». Перед тем как писать код — согласовать подход в docs/.

### Бэклог

- `docs/PROJECT_BACKLOG.md` — тонкий ссылочный индекс; одна NEXT TASK за раз.
- Закрытые задачи с отчётами — в `docs/PROJECT_BACKLOG_ARCHIVE.md`; поиск по `MLC-NNN`.
- Статус задачи: `NEXT TASK` → `IN PROGRESS` → закрыта (перемещена в архив).

### Кураторская модель

Claude-куратор: планирует, ставит NEXT TASK постановочным PR, раздаёт задачи
субагентам-исполнителям (worktree), ревьюит и вливает.
Правки канона (`docs/`) идут в том же ходу, что и код.
Субагент-исполнитель не вносит правки в docs/ самостоятельно без задания куратора.

### Git-дисциплина

- Ветка — от `main`; перед работой `git fetch + FF`.
- Постановочный PR куратора вливается **до** исполнителя.
- Один PR на одну задачу; название PR содержит `MLC-NNN`.

### Гейт качества

Тесты обязательны. Локальный прогон `build.ps1` — штатный гейт качества.
CI на GitHub Actions может быть недоступен по биллингу — в таком случае гейт остаётся
локальным; слияние без CI-зелёного допустимо.
Конкретные команды сборки и тестирования — в [`docs/DEVELOPMENT.md`](docs/DEVELOPMENT.md).

### Инварианты — нарушать только явно

Следующие ограничения нельзя снимать молча (нужно решение в `docs/DECISIONS.md`):

- **Enum аудита заморожен:** `AuditActionType` — int-значения неизменяемы; новое действие
  получает новое число, существующие не переназначаются (исторические записи в БД).
- **Parity BE↔FE:** Zod-схемы фронтенда отражают контракты API; при изменении API
  обновляются оба слоя в одном PR.
- **Single-host:** панель управляет только тем узлом, на котором установлена (ADR-28).

## Инструменты

Перед работой с любой внешней библиотекой, фреймворком, SDK, CLI или облачным
сервисом (например React, EF Core, ASP.NET, Zod, React Query, Tailwind) —
обращаться к **MCP Context7** за актуальной документацией (API, конфигурация,
миграция версий), не полагаясь только на знание модели. Использовать по
необходимости, а не для общих вопросов программирования или бизнес-логики.
