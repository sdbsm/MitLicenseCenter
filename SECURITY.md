# Модель безопасности

Документ описывает текущую модель безопасности MitLicense Center: границы доверия,
аутентификацию и авторизацию, защиту секретов, права службы, транспорт и журналирование.
Каждый **принятый риск** перечислен явно — с обоснованием и смягчением.

Термины — по `docs/01_OVERVIEW.md` (глоссарий). Решения упоминаются номерами ADR;
полные формулировки — в `docs/DECISIONS.md`.

---

## 1. Границы доверия и модель угроз

MitLicense Center — административный инструмент **одного хоста** (single-host, ADR-28).
Панель и её служба Windows работают на том же сервере, где живут кластер 1С, IIS и
SQL Server, и управляют именно этим узлом.

Ключевое следствие для модели угроз: **роль Admin фактически равна полному контролю над
хостом.** Через панель Admin публикует и снимает публикации IIS, перезапускает пулы и
сайты, выполняет `iisreset`, завершает сеансы 1С, запускает `BACKUP DATABASE` на SQL и
меняет параметры подключения к RAS/SQL/IIS. Это не побочный эффект, а назначение продукта:
управление инфраструктурой хоста — его основная функция (см. §6).

Граница доверия проходит по **аутентифицированному пользователю панели**. Внутри границы
различаются две роли (Admin / Viewer, §3). Снаружи — анонимный сетевой клиент, которому
доступны только логин, health-проверки и SPA-оболочка.

Из этого вытекает основная операционная рекомендация: **держать панель в доверенном
сетевом периметре** (LAN или VPN), а не публиковать её в открытый интернет. По умолчанию
панель слушает HTTP без TLS (§7) — рекомендуется обратный прокси с TLS или флаг
`Security:EnforceHttps`. Security-заголовки и rate-limiting на логин добавлены (§8).

## 2. Аутентификация

Аутентификация — cookie-сессия на базе ASP.NET Core Identity. Полный пайплайн собран
вручную (`AddIdentityCore`), а не через `AddIdentity`, поэтому ряд механизмов
конфигурируется явно (см. ниже security-stamp).

### Cookie сессии

Настройки куки (`Program.cs`, `AddCookie`):

- **Имя:** `mlc.auth`.
- **`HttpOnly = true`** — кука недоступна из JavaScript (защита от кражи через XSS).
- **`SameSite = Strict`** — кука не отправляется в кросс-сайтовых запросах (основа защиты
  от CSRF, §4).
- **`SecurePolicy`:** в Production — `Always` (кука только по HTTPS); в Development —
  `SameAsRequest`. Secure-политика прода **не зависит** от флага `Security:EnforceHttps` —
  она включена всегда.
- **Срок жизни:** `ExpireTimeSpan = 8 часов`, `SlidingExpiration = true` — скользящее окно
  продлевается при активности.
- Для SPA редиректы заменены статус-кодами: невход → `401`, нет прав → `403`
  (вместо HTML-редиректа на форму логина).

### Парольная политика

`AddIdentityCore` (`Infrastructure/DependencyInjection.cs`):

- Минимальная длина — **12 символов** (`RequiredLength = 12`).
- Обязательны: цифра, строчная буква, заглавная буква, спецсимвол.

Та же длина 12+ повторно проверяется при смене пароля в `/auth/change-password`.

### Блокировка при подборе (lockout) — ADR-36

- **5 неудачных попыток** (`MaxFailedAccessAttempts = 5`) → блокировка на **15 минут**
  (`DefaultLockoutTimeSpan`). Вход выполняется с `lockoutOnFailure: true`, то есть
  неудачи действительно инкрементят счётчик и приводят к блокировке.

### Rate limiting на логин — ADR-42

Поверх Identity-lockout действует per-IP rate-limiting на `POST /api/v1/auth/login`
(MLC-125, SEC-08):

- **PermitLimit = 10 запросов** за **1 минуту**, партиционирование по `RemoteIpAddress`.
- Превышение → **429 Too Many Requests** ДО выполнения тела хендлера (до SignInManager).
  Аудит `LoginFailed` **не пишется** — rate-limit reject не является «неудачным входом».
- Значения зашиты в коде (не tuneable оператором).
- Взаимодействие двух слоёв: lockout защищает учётку (но даёт lockout-DoS), rate-limit
  троттлит источник (не блокирует учётку). Дефенс-ин-депт: атакующий получает 429 ещё
  до бизнес-логики.

### Принудительная смена пароля

Флаг `MustChangePassword` (`AppUser`) включается на учётках, **созданных** администратором
и при **сбросе пароля** — пользователь входит по временному паролю и обязан сменить его при
первом входе. Первый (seed) администратор создаётся **без** этого флага: сидер
(`IdentitySeeder`) не выставляет `MustChangePassword`. Успешная смена пароля снимает флаг.

### Немедленный отзыв доступа (security-stamp)

Скользящая 8-часовая кука сама по себе означала бы, что отзыв роли/учётки/пароля не
прерывает уже активную сессию. Это закрыто через `SecurityStampValidator`:

- Кука валидируется на каждом запросе, но stamp из БД сверяется **не чаще раза в 2 минуты**
  (`SecurityStampValidatorOptions.ValidationInterval = 2 мин`). Это верхняя граница
  задержки отзыва.
- Операции, которые **ротируют** `SecurityStamp` (после чего старая кука отвергается в
  пределах интервала):
  - **отключение** учётки (`disable`) — явный `UpdateSecurityStampAsync`;
  - **смена роли** (`role`) — явный `UpdateSecurityStampAsync` (без него разжалованный Admin
    ходил бы со старыми claims до истечения куки);
  - **сброс пароля** (`reset-password`) — через штатный `ResetPasswordAsync`, который
    ротирует stamp внутри Identity.
- **Само-смена пароля** (`/auth/change-password`) переиздаёт куку свежим stamp'ом
  (`RefreshSignInAsync`) — собственная сессия пользователя не рвётся.

`ValidationInterval = 2 мин` — компромисс «свежесть отзыва ↔ нагрузка на БД».

### LastLoginAt

Время последнего успешного входа фиксируется в `AppUser.LastLoginAt` (UTC) и
показывается в разделе «Пользователи».

## 3. Авторизация

Две политики (`Program.cs`):

- **`Admin`** = `RequireRole("Admin")` — только Admin.
- **`Viewer`** = `RequireRole("Admin","Viewer")` — любая из двух ролей.

Глобального `FallbackPolicy` / `RequireAuthorization` на корне **нет** — требования заданы
на каждом эндпоинте (или на группе). Карта `/api/v1` (72 эндпоинта) по классам доступа:

| Класс доступа | Кол-во | Что это |
|---|---|---|
| **Anonymous** | 3 | `POST /auth/login`, `GET /health`, `GET /health/ready` |
| **Authenticated без роли** | 3 | `POST /auth/logout`, `GET /auth/me`, `POST /auth/change-password` |
| **Viewer** (Admin\|Viewer) | 23 | чтение данных всех разделов + `POST /backups` (запуск бэкапа доступен обеим ролям) |
| **Admin** | 43 | все мутации: CRUD клиентов/инфобаз, публикации, IIS-операции, kill сеанса, параметры, пользователи, discovery, unassigned |

Группы `/discovery` и `/infobases/unassigned` закрыты **Admin на уровне группы**.

Маршруты вне `/api/v1`:

- **`/hangfire`** (дашборд фоновых заданий) — `AdminOnlyDashboardAuthorizationFilter`:
  доступ только при `IsAuthenticated && IsInRole("Admin")`.
- **`/api/docs`** (Swagger UI) и `swagger.json` — отдаются **только** в Development **или**
  при `Security:EnableSwagger=true`; в проде по умолчанию закрыты (см. §7).
- **Статика** (`/assets/*`, SPA-бандлы) — без авторизации (логин-страница грузится анониму).
- **SPA-fallback** (`MapFallback`) — `AllowAnonymous`, отдаёт `index.html`. Зарезервированные
  префиксы `/api/*` и `/hangfire/*` из fallback исключены: неизвестный `/api/*` честно
  возвращает **404**, а не маскируется под HTML.

Маршрутная защита страниц «Параметры»/«Пользователи» на фронтенде дублируется политикой
`Admin` на API — фронтенд-проверка не является границей безопасности.

## 4. CSRF

Antiforgery-токены **не используются** (поиск `antiforgery`/`AddAntiforgery` по backend
даёт 0 совпадений). Защита от CSRF обеспечивается **`SameSite=Strict`** на куке сессии:
браузер не прикладывает её к кросс-сайтовым запросам, а SPA и API работают same-origin
(Kestrel отдаёт собранный фронтенд из того же процесса, ADR-30). Отказ от antiforgery в
пользу `SameSite=Strict` для same-origin SPA — осознанное решение (см. соответствующий ADR
в `docs/DECISIONS.md`).

## 5. Защита секретов

### Шифрование секретных параметров

Значения настроек, помеченных `IsSecret`, шифруются через **ASP.NET Core Data Protection**
(`SettingsStore`, protector с purpose **`mlc.settings.v1`**) и хранятся в БД зашифрованными;
несекретные значения лежат как `ValueText` в открытом виде.

Фактически секретным помечен **единственный** параметр — **пароль администратора кластера
1С** (`OneC.Cluster.AdminPassword`, `IsSecret: true` в `SettingDefinitions`). Остальные
параметры (endpoints, пути, лимиты, app-id, расписания) не секретны.

### Key ring (ключи Data Protection)

Data Protection хранит key ring в **файловой системе в переносимом открытом формате**
(`PersistKeysToFileSystem`, `SetApplicationName("MitLicenseCenter")`) — **без** шифрования
ключей DPAPI/сертификатом. Защита ключей — **только NTFS ACL** на каталоге (ADR-8).
Решение принято ради переносимости резервной копии: открытый key ring можно перенести вместе
с БД на другой хост, и зашифрованные настройки расшифруются.

Пути key ring:

- **Production:** `%ProgramData%\MitLicenseCenter\keys` (`CommonApplicationData`).
- **Development:** `%LOCALAPPDATA%\MitLicenseCenter\keys` (`LocalApplicationData`).

Инсталлятор ужесточает ACL каталога `%ProgramData%\MitLicenseCenter` (`MitLicenseCenter.iss`):
`icacls /inheritance:r` снимает наследование от ProgramData, далее full только для
**SYSTEM** (`*S-1-5-18`) и **Administrators** (`*S-1-5-32-544`); **учётной записи службы**
(виртуальной `NT SERVICE\MitLicenseCenter` или именованной/gMSA, ADR-49) даётся Modify. Грант
выполняется по имени `NT SERVICE\…` (префикс не локализуется) **после** регистрации службы
(`sc create`), когда учётка и её SID уже существуют. Сбой `icacls` выводит предупреждение с
путём, но **не** прерывает установку.

### appsettings.Production.json

Строка подключения использует **доверенное подключение Windows** (`Trusted_Connection=True`)
для обеих строк (`Default` и `Hangfire`) — **SQL-логин и пароль в конфиг не записываются**
(ADR-49). Служба ходит к локальному SQL под своей учётной записью; единственный секрет SQL
на диске устранён. Файл генерируется инсталлятором в `[Code]` (не секцией `[Files]`) и при
деинсталле удаляется. ACL: инсталлятор режет наследование (`Users:RX` от Program Files) — full
только SYSTEM / Administrators, плюс **учётной записи службы** даётся Read.

### initial-admin.secret (ADR-31)

Пароль первого администратора, заданный оператором в мастере установки, передаётся бэкенду
через **одноразовый файл** `%ProgramData%\MitLicenseCenter\initial-admin.secret`. Сидер при
первом старте читает файл, создаёт администратора и **best-effort удаляет файл** сразу после
чтения (контракт одноразовости). Если пользователи уже существуют (повторный старт/апгрейд),
сидер тоже удаляет висящий файл. Пароль **нигде не логируется**. Файл создаётся под тем же
ужесточённым ACL каталога `%ProgramData%\MitLicenseCenter` (SYSTEM/Administrators + учётка службы).
Если файла нет (dev / `db-reset` / ручной деплой), пароль генерируется случайно и пишется
в лог — это fallback-путь вне инсталлятора.

## 6. Права службы

Служба `MitLicenseCenter` (Windows-служба, `UseWindowsService`) ставится инсталлятором
(`sc create … start= auto`, ADR-49) под одной из учётных записей:

- **Виртуальная учётная запись `NT SERVICE\MitLicenseCenter` (по умолчанию):** без пароля
  (им управляет Windows), работает одинаково в рабочей группе и в домене. SCM материализует
  учётку и её SID при `sc create … obj= "NT SERVICE\MitLicenseCenter"` и выдаёт право
  «Вход в качестве службы».
- **Именованная Windows-учётка / gMSA (доменный сценарий, выбор в мастере):** обычная учётка
  регистрируется с `obj=`/`password=`; gMSA — без пароля (явный чекбокс в мастере).

Во всех случаях подключение к **локальному** SQL — по **доверенному подключению Windows**
(`Trusted_Connection=True`); SQL-логин/пароль в конфиге **не используются** (SQL-аутентификация
убрана). Учётка службы должна иметь роль **`sysadmin`** на экземпляре SQL — **инсталлятор
создаёт этот логин автоматически** (`CREATE LOGIN [NT SERVICE\MitLicenseCenter] FROM WINDOWS`
+ членство `sysadmin`, идемпотентно; **всегда Windows-логин**), потому что SID виртуальной
учётки появляется только при регистрации службы.

Назначить роль `sysadmin` может только тот, кто сам `sysadmin` (правило SQL) — поэтому
провижининг идёт под некоторой `sysadmin`-личностью **на выбор** (страница мастера
«Подключение установщика к SQL», MLC-171):

- **(по умолчанию) Integrated Security запускающего администратора** — без ввода пароля;
  работает и на экземплярах **«только Windows-аутентификация»**;
- **(запасной) введённый SQL-логин с ролью `sysadmin`** (`sa` или иной) — для экземпляров в
  **смешанном режиме** (разрешена SQL-аутентификация), когда запускающий администратор не
  является `sysadmin` на SQL.

Отсюда **предусловие установки** (смягчено относительно «запускающий админ обязан быть
`sysadmin`»):

> Установщику должна быть доступна **какая-либо `sysadmin`-личность на локальном экземпляре
> SQL Server** — либо запускающий администратор сам `sysadmin` (Integrated Security, по
> умолчанию), либо вводится SQL-логин с ролью `sysadmin` (требует экземпляра в смешанном
> режиме). Без этого авто-создание SQL-логина учётки службы невозможно — установщик
> прерывается с откатом (а не оставляет службу, падающую на старте с 18456).

**SQL-секрет на диске не появляется.** Введённый SQL-логин (`sa`) — это **транзиентный «ключ»**
провижининга: используется **однократно** для создания Windows-логина службы, **не пишется** в
`appsettings.Production.json` и нигде не сохраняется (передаётся тем же приёмом, что тест
подключения — через временный `.ps1` с `'...'`-литералом, не в командной строке, файл удаляется
сразу). Создаётся **всегда Windows-логин** (`FROM WINDOWS`) — SQL-логинов с паролем не создаётся
ни для службы, ни иначе. Инвариант «секрета SQL на диске нет» сохраняется.

**SQL-доступ:** мастер автоматически тестирует достижимость инстанса при нажатии «Далее» на
страницах «Подключение установщика к SQL» (отдельной кнопки нет, MLC-175) под **выбранной
личностью провижининга**, а **не** под учёткой службы. Сам SQL-логин учётки службы создаётся
установщиком на этапе установки.

Для управления IIS (публикация через `Microsoft.Web.Administration.ServerManager`,
recycle/iisreset) учётка службы должна входить в **локальную группу Administrators** хоста.
Инсталлятор добавляет её туда автоматически по well-known SID `S-1-5-32-544`
(`Add-LocalGroupMember`, локаленезависимо) — best-effort: при сбое панель и SQL работают,
деградируют только IIS-функции (публикации будут в статусе «Ошибка проверки»).

**Эти привилегии — выше минимально необходимых, и это принятый риск (ADR-44).** ADR-49 меняет
только идентичность службы (виртуальная учётка вместо LocalSystem / именованного админ-аккаунта),
но **не объём прав**: `sysadmin` на SQL и admin-эквивалент на хосте остаются штатным профилем.
Обоснование:
функции панели — публикация IIS, recycle/iisreset, бэкап SQL-баз через `BACKUP DATABASE`,
чтение DMV производительности — требуют высоких прав по своей природе. Управление хостом
и есть назначение продукта (§1). Снижение привилегий до «минимума» требует runtime-трассировки
реальных чтений `ServerManager` на стенде; откладывается как follow-up (см. ADR-44).

## 7. Транспорт

По умолчанию служба слушает **HTTP без TLS**.

Прод-хардненинг транспорта вынесен за флаг **`Security:EnforceHttps`** (по умолчанию
`false`). Логика — `TransportSecurity.ShouldEnforceHttps`: HSTS + HTTPS-redirect включаются
**только** вне Development **и** только при `Security:EnforceHttps=true`. Флаг ставят
**только** когда сервис сам терминирует TLS на Kestrel (есть сертификат, перед сервисом нет
терминирующего реверс-прокси). За терминирующим прокси флаг оставляют `false` — иначе
двойной редирект / петли.

Инсталлятор открывает в firewall **входящее правило** на введённый порт с **`profile=domain,private`**
(SEC-06/MLC-126): `netsh advfirewall firewall add rule … profile=domain,private dir=in action=allow
protocol=TCP localport=<порт>`. Профиль Public остаётся закрытым — порт недоступен из
недоверенных сетей. Правило удаляется при деинсталле (по имени правила).

**Опция localhost-bind:** при деплое за реверс-прокси (штатный путь для TLS) рекомендуется
задать `"Urls": "http://localhost:<порт>"` в `appsettings.Production.json` — тогда Kestrel
принимает соединения только с loopback-адреса, firewall-правило на этот порт не нужно вовсе
(подробнее — в `OPERATIONS.md`, §2 «Firewall и bind»).

Рекомендация: разворачивать панель за обратным прокси с TLS **или** включать
`Security:EnforceHttps` с сертификатом на Kestrel; держать панель в LAN/VPN (§1).

## 8. Security headers и rate limiting (MLC-125)

### Security response headers — ADR-41

Middleware `SecurityHeaders` ставит четыре заголовка на **все** HTTP-ответы (SPA, API, статика,
fallback), включая ответы об ошибке. Порядок в пайплайне: сразу после `UseExceptionHandler`,
до `UseStaticFiles` и аутентификации.

| Заголовок | Значение | Все пути? |
|---|---|---|
| `X-Content-Type-Options` | `nosniff` | Да |
| `Referrer-Policy` | `no-referrer` | Да |
| `X-Frame-Options` | `DENY` | Нет `/api/docs` |
| `Content-Security-Policy` | см. ниже | Нет `/api/docs` |

CSP для same-origin SPA:
```
default-src 'self'; base-uri 'self'; object-src 'none'; frame-ancestors 'none';
img-src 'self' data:; font-src 'self' data:; style-src 'self' 'unsafe-inline';
script-src 'self'; connect-src 'self'
```

Исключение `/api/docs/*`: Swagger UI использует inline-скрипты — CSP и X-Frame-Options
не ставятся на этих путях. `nosniff` и `Referrer-Policy` остаются на всех путях.
Значения зашиты в коде (не tuneable оператором). HSTS управляется отдельно (`EnforceHttps`, §7).

### Login rate limiting — ADR-42

Per-IP fixed window на `POST /api/v1/auth/login`: 10 запросов / 1 минута.
При превышении → 429. Подробнее — в §2 «Rate limiting на логин».

## 9. Принятые риски

Каждый пункт — фактическое состояние по коду, с обоснованием и смягчением.

1. **Высокие привилегии службы** — `sysadmin` на SQL и admin-эквивалент на хосте; служба
   работает под виртуальной учётной записью `NT SERVICE\MitLicenseCenter` (или именованной/gMSA),
   входящей в локальную группу Administrators (§6, ADR-44/ADR-49). *Обоснование:* управление
   хостом — функция продукта; снижение до «минимума» признано недостижимым без потери функций
   (ADR-44). *Смягчение:* single-host в LAN/VPN (ADR-28). *Принят.*

4. **Firewall-правило без сужения по источнику (`remoteip=`).** Инсталлятор открывает порт
   с `profile=domain,private` (Public закрыт, SEC-06/MLC-126 — §7), но без ограничения
   `remoteip=`: порт достижим со всех адресов в Domain/Private-сети. Ограничение по источнику
   не задаётся намеренно: multi-subnet LAN — частый сценарий; `remoteip=localsubnet` отрежет
   клиентов из других подсетей. *Смягчение:* рекомендация держать хост в LAN/VPN; оператор
   может сузить правило вручную (`remoteip=localsubnet`). *Принят.*

5. **Plaintext HTTP по умолчанию.** TLS включается только опционально (§7). *Смягчение:*
   флаг `Security:EnforceHttps` / обратный прокси; cookie `Secure=Always` в проде. *Принят:*
   разворачивание во внутренней сети, TLS — на усмотрение оператора.

6. **Key ring в открытом виде на диске** (ADR-8, §5). *Обоснование:* переносимость
   резервной копии секретных настроек. *Смягчение:* NTFS ACL (только SYSTEM/Administrators,
   инсталлятор `icacls /inheritance:r`). *Принят.*

7. **SQL-пароль в конфиге — ЗАКРЫТ/УСТРАНЁН (ADR-49).** Прежний режим B (LocalSystem +
   SQL-логин), писавший SQL-пароль в открытом виде в `appsettings.Production.json`, **убран**.
   Служба работает под Windows-учётной записью и подключается к локальному SQL по
   `Trusted_Connection=True` — SQL-логина и пароля в конфиге **больше нет** (§5/§6). Единственный
   секрет SQL на диске устранён, в том числе из бэкапа конфига. Запасной вариант провижининга
   (SQL-логин `sa`/иной с ролью `sysadmin`, MLC-171) **не меняет** этого: пароль вводится в
   мастере, используется **однократно** для создания Windows-логина службы и **не сохраняется**
   (в конфиг не пишется; передаётся через временный `.ps1`-литерал, не в командной строке, файл
   удаляется сразу) — на диск ничего не оседает. *Статус: разрешён.*

8. **Пароль администратора кластера 1С в командной строке `rac.exe`** (ADR-21) — передаётся
   аргументами процесса (`--cluster-pwd`); альтернативного канала у `rac.exe` нет. *Смягчение:*
   значение шифруется Data Protection в БД (§5) и расшифровывается только в памяти службы на
   время вызова; среда — single-host, доступ к процессам узла только у администраторов хоста.
   *Принят.*

## 10. Журналирование безопасности

Значимые операции пишутся в модуль **Аудит** (`AuditActionType`, раздел «Аудит»):
создание/изменение/удаление объектов, входы и выходы, смены ролей, отключение учёток, сброс
и смена паролей, kill сеансов, операции IIS, бэкапы, изменения параметров.

**Неудачные попытки входа фиксируются** — `AuditActionType.LoginFailed`: при отказе
`PasswordSignInAsync` пишется запись с введённым именем и причиной (неверные учётные данные
либо блокировка учётки). **Пароль в описание никогда не попадает.** Успешный вход пишется как
`AdminLoggedIn`.

При запуске **службой Windows** (`UseWindowsService`) логи приложения направляются в
**Windows Event Log**. Внутренние детали ошибок наружу не отдаются: непойманные исключения
возвращаются клиенту как санитизированный русский `ProblemDetails` (без stack trace и текста
исключения), а полное исключение логируется на сервере.
