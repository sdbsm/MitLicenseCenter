## Центральная сущность — `Investigation` («Дело»)

Объединяет окно времени, сценарий, собранные данные и отчёт.

| Поле | Тип | Смысл |
|---|---|---|
| `Id` | Guid | PK |
| `Scenario` | enum int | `Blocking` / `LongQueries` / `Exceptions` / `General` (новый enum; значения фиксируем) |
| `Status` | enum int | `Collecting` → `Analyzing` → `Completed`; плюс `Interrupted` (рестарт), `Failed` |
| `StartedAtUtc` / `StoppedAtUtc` | datetime | окно сбора |
| `StartedBy` | string | оператор (Admin) |
| `TenantId` / `InfobaseId` | Guid? | опц. фильтр расследования на конкретного арендатора/инфобазу |
| `StopReason` | enum int | `Manual` / `TimeLimit` / `DiskLimit` / `Error` |

## `CollectionConfig` — снимок того, что включали (аудит/воспроизводимость)

| Поле | Смысл |
|---|---|
| `InvestigationId` | FK |
| `LogcfgLocation` | каталог сбора ТЖ |
| `Events` | список включённых событий (`TLOCK,TTIMEOUT,…`) |
| `DurationFilterMs` | порог по длительности |
| `Format` | `json` (см. `40_TECHLOG.md`) |
| `HistoryHours` | лимит ротации |

Зачем: после расследования видно, **что именно** собирали; и можно безопасно снять ровно то,
что ставили.

## `ResourceSample` — метрики хоста/1С/SQL во времени

Переиспользуем существующую модель `PerfRecordingSample` (`20_CURRENT_STATE.md`): плоские
метрики хоста + JSON топ-виновников 1С/SQL, привязка к `InvestigationId` вместо `RecordingId`.
**Целевой дизайн: `Investigation` приходит на смену нынешнему `PerfRecording`** (миграция, не
параллельная сущность).

## Результат анализа ТЖ — `Finding` (агрегаты под сценарий)

Хранить структурированно (таблицы) или как версионированный JSON-результат разбора. Кандидаты:

- **Блокировки:** дерево/рёбра «ждущий сеанс → блокирующий сеанс → ресурс», список дедлоков.
  Поля: ждущий/блокирующий `SessionID`+`Usr`, ресурс, длительность ожидания, инфобаза/арендатор.
- **Долгие запросы:** длительность, текст `Sql`, контекст, инфобаза/арендатор, частота похожих.
- **Исключения:** текст `Descr`, контекст, частота, инфобаза/арендатор.

Каждый `Finding` несёт **привязку к арендатору/инфобазе** (через свойства события + маппинг
база→клиент).

## `Report` — итог «дела»

Сводка: ранжированные `Finding` + рекомендации со ссылками на стандарты (`10_SOURCES.md` C4).
Доступен post-mortem. Кандидат — вычисляемое представление поверх `Finding` + шаблоны рекомендаций,
а не отдельная тяжёлая сущность.

## Жизненный цикл и хранение

- Один **активный** сбор за раз (инвариант безопасности — `60_SAFETY.md`).
- Сырые JSON-файлы ТЖ: собрать → распарсить в `Finding` → **удалить сырьё** (по умолчанию),
  опц. оставлять по флагу (влияет на место). `⚠` решить на этапе C.
- Retention: переиспользовать идею ночного job'а (как `PerfRecordingRetentionJob`).
- Восстановление осиротевших `Investigation` при рестарте → `Interrupted` + гарантированное
  снятие `logcfg.xml` (см. `60_SAFETY.md`).
