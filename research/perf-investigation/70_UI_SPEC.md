# UI-спека: раздел «Производительность» (целевой вид)

> DRAFT, трек **версии 1.2** (ветка `release/1.2`). Согласованный с владельцем (2026-06-20)
> визуальный вид целевого раздела — **основа для реализации фронтенда** (агент-реализатор строит
> ровно по ней, а не выдумывает заново). Дополняет: `30_TARGET_DESIGN` (концепция), `40_TECHLOG`
> (данные ТЖ), `50_DATA_MODEL` (сущности), `60_SAFETY` (инварианты сбора), `20_CURRENT_STATE`
> (что переиспользуем). Сверён по интерактивным макетам, показанным владельцу.

---

## Общая структура

- **Один раздел «Производительность»** (маршрут `/performance`). Единый хедер: заголовок, узел
  (`Andrey-pc · single-host`), бейдж платформы (`8.5`).
- **Переключатель режимов** (сегмент-контрол): **«Наблюдение»** | **«Расследование»**.
- «Наблюдение» = переосмысленная нынешняя «Быстродействие» (переиспользуем probe и источники
  данных, см. `20_CURRENT_STATE`). «Расследование» = новое (сбор/парсинг ТЖ, дело, отчёт).

---

## Экран 1 — «Наблюдение» (live, триаж)

Назначение: что происходит **сейчас**. Источники: WMI/perfmon, `rac`, DMV (как сейчас).

Блоки сверху вниз:
- **Live-статус:** индикатор «в реальном времени», время последнего обновления, пауза/обновить
  (pull ~5 с).
- **Вердикт:** «почему тормозит сейчас» (CPU/RAM/диск + основной потребитель-семья).
- **Светофор ресурсов:** 3 плитки (Процессор / Память / Диск): значение, статус-точка (норма/
  предупреждение), бар, вторичная метрика (очередь / paging / латентность); клик → фокус на слой.
- **Drill-down вкладки:** Хост / 1С / SQL.
  - Хост: атрибуция по **семьям процессов** (1С / SQL / служба / прочее): %CPU, память, число процессов.
  - 1С: сеансы (состояние, CPU, длительность вызова, блокирован) + рабочие процессы.
  - SQL: активные запросы (DMV), нагрузка по базам, ожидания (wait-stats), IO-stall.
- **«Блокировки сейчас»:** компактный live-сигнал (1С / SQL цепочки).
- **Мост к расследованию:** CTA «начать расследование» — заводит дело с записью ТЖ по текущему
  симптому (бесшовный переход в режим «Расследование»).

---

## Экран 2 — «Мастер запуска расследования»

Назначение: вход в воронку; безопасно настроить **целевой** сбор ТЖ.

1. **Сценарий** (один из): Блокировки и дедлоки / Долгие запросы / Исключения / Общая медленная
   работа. Маппинг на события ТЖ — `40_TECHLOG` §6.
2. **На что нацелить (scope):**
   - «Весь узел» (все инфобазы) — больше данных и нагрузки;
   - «Конкретная инфобаза» — **выпадающий список из реестра панели** (Tenants↔Infobases; ничего
     вводить вручную). Выбор → фильтр **`p:processName=<имя ИБ>`** = изоляция: ТЖ только этой базы,
     без данных чужих арендаторов (инвариант `60_SAFETY` §2). На стенде имя процесса = имя ИБ (`mitpro`).
3. **Сбор и лимиты:** окно сбора (авто-стоп, мин), лимит места (ГБ), порог длительности (сек —
   для запросов/общего). Поля с безопасными дефолтами.
4. **Плашка безопасности:** «только события сценария» (не «весь журнал»), авто-снятие `logcfg` по
   окну/лимиту, восстановление конфигурации после перезапуска панели.

Кнопки: Отмена / Запустить.
**После «Запустить»:** панель ставит `logcfg` → сбор (виден прогресс, можно остановить) →
авто-снятие → парсинг → открывается **дело** и **отчёт**.

## Экран 3 — «Дело» (рабочий результат расследования)

Сущность `Investigation` (`50_DATA_MODEL`). Блоки:
- **Шапка дела:** №, название, бейджи (статус, сценарий, период, арендатор/ИБ). Кнопка «Отчёт».
- **Вердикт** — человеческим языком, с серьёзностью.
- **Сводка-метрики:** событий ТЖ, блокировок, долгих запросов, исключений.
- **Цепочка блокировок (1С-уровень):** держащий сеанс (ресурс, длительность транзакции) → ждущие
  сеансы (подсветка таймаута). Из `TLOCK/TTIMEOUT/TDEADLOCK`. ⚠ СУБД-уровень — **отдельный блок**
  (`lkX`/`dbmslocks`, `40_TECHLOG` §5), не смешивать.
- **Топ долгих запросов:** таблица — длительность, контекст (строка кода 1С из `Context`), текст
  SQL. Клик по строке → план (`plansql`). Из `DBMSSQL`.
- **Исключения:** топ `EXCP` (учесть: дедлок СУБД = пара `EXCP`).
- **Рекомендации:** со ссылками на стандарты ИТС.

Состояния дела (UI): `Collecting` (идёт сбор, прогресс) / `Analyzing` / `Completed` / `Interrupted`.

---

## Экран 4 — «Отчёт» (документ)

Документ-вид дела для выгрузки/показа (PDF / копия). Блоки:
- **Шапка:** №, сформирован, период, узел, кто запустил, арендатор. Действия: экспорт PDF / открыть дело.
- **Резюме:** вердикт + серьёзность.
- **«Что собрано» (воспроизводимость):** события, фильтр `p:processName`, формат JSON, объём,
  сырьё удалено после разбора, авто-снятие/лимит соблюдён. = `CollectionConfig` (`50_DATA_MODEL`) +
  «перечень данных» методики ИТС (A5).
- **Находки** (ранжированы по серьёзности): каждая — серьёзность, заголовок, **доказательная
  цепочка событий ТЖ**.
- **Рекомендации:** с ожидаемым эффектом и ссылкой на стандарт.
- **Подвал:** сырьё удалено по retention; дело доступно post-mortem; источники методик — ИТС.

## Экран 5 — «Список дел» (история расследований)

Назначение: видеть прошлые расследования и открывать их post-mortem (аналог нынешней «Записи»).
Точка входа в режим «Расследование» по умолчанию, если активного дела нет.

Блоки:
- **Заголовок + действие:** «Расследования» · кнопка «+ Новое расследование» (→ Мастер, экран 2).
- **Активный сбор (если есть):** строка-баннер сверху с состоянием `Collecting` и прогрессом
  (см. экран 6); один активный сбор за раз (инвариант `60_SAFETY`).
- **Таблица дел** (пагинация, движок `DataTable` как в остальных списках панели): № · дата/период ·
  сценарий · арендатор/ИБ · статус (`Завершено`/`Прервано`/`Идёт`) · краткий вердикт · кто запустил.
  Клик по строке → экран «Дело» (3). Действие в строке — удалить (с подтверждением).
- **Фильтры:** по арендатору, по сценарию, по статусу.
- **Retention-сноска:** старые дела удаляются по политике хранения (ночной job, как у «Записи»).

## Экран 6 — «Прогресс сбора» (состояние `Collecting`)

Назначение: пока идёт сбор ТЖ — показывать, что происходит, и дать остановить. Это **состояние дела**,
а не отдельная страница: показывается в шапке дела (экран 3) и строкой в списке (экран 5).

Блоки:
- **Статус:** «Идёт сбор» + сценарий + арендатор/ИБ.
- **Прогресс окна:** прошло X из N минут (бар), осталось до авто-стопа.
- **Лимиты вживую:** занято места из лимита (бар; при достижении — авто-стоп `DiskLimit`),
  собрано событий/файлов.
- **Что собирается:** список целевых событий сценария (напоминание, что не «весь журнал»).
- **Действие:** «Остановить сейчас» (ручной стоп → `StopReason=Manual`, дальше парсинг и отчёт).
- **Гарантия:** подпись «logcfg снимется автоматически по окончании/лимиту; восстановится после
  перезапуска» (инварианты `60_SAFETY`).

## Воронка / переходы

`Наблюдение` —(CTA «начать расследование»)→ `Мастер` —(Запустить)→ `Сбор/прогресс` →
`Дело` —(«Отчёт»)→ `документ`. Всё — внутри одного раздела «Производительность».

## Привязка блоков к данным (cross-ref)

| Блок UI | Источник |
|---|---|
| Светофор, семьи процессов, сеансы, активные запросы | probe WMI/Process, `rac`, DMV (`20_CURRENT_STATE`) |
| Дерево блокировок 1С | `TLOCK/TTIMEOUT/TDEADLOCK` (`40_TECHLOG` §5) |
| СУБД-блокировки | `lkX` / `<dbmslocks/>` (`40_TECHLOG` §5) |
| Топ долгих запросов | `DBMSSQL` + `<plansql/>` |
| Исключения | `EXCP` |
| Привязка к арендатору | `p:processName` + маппинг база→клиент |

## Открытые вопросы UI (решить при реализации)

- **СУБД-блокировки** (`lkX`) — отдельная вкладка/блок в деле (в макетах показан 1С-уровень).
- **Apdex/замеры** (этап D) — в макеты пока не входят.
