- `history` Min=1 (настройка `TechLog.HistoryHours`) — **0 запрещён** ⇒ нет «бесконечного» хранения
  (спека: `history=0` = не удалять, переполнение диска). ✅ Совпадает с политикой безопасности.
- Один `<log>` (≤20 по спеке). ✅
- **`<plansql>`/`<dump>`/`<dbmslocks>` на уровне `<config>`** (MLC-245, и `<dbmslocks/>` — MLC-236) —
  соответствует спеке (config-level директивы-сборщики, не внутри `<log>`; спека §2/§7/§8). ✅
- `<dump>` с `location` (отдельный подкаталог `\dumps` ≠ `<log>.location`), `create="true"`,

- Отбор событий `<event><eq property="name" value="…"/></event>` — корректная форма «по ИЛИ». ✅

  используются. ✅
- **Сценарий `DbmsLocks` (MLC-236):** событие `DBMSSQL` + config-level `<dbmslocks/>` (формирует поля
  `lkX`, спека §8) + `<property name="all"/>` (выводит `lkX`) — соответствует. Отдельные
  `<property name="lkX"/>` не нужны (их покрывает `all`). ✅ ⚠ Структура полей `lkX` в JSON-ТЖ 8.5 —
  за стенд-приёмкой (источник — infostart 1431026, не снимок 8.5).
- `placement/rotation/compress` по умолчанию (folders/period/none) ⇒ файлы `<процесс>_<pid>/ГГММДДЧЧ.log`. ✅

## Расхождения ❌

### F-1 (P1, **Security**) — изоляция арендатора структурно не работает — ✅ ИСПРАВЛЕНО (MLC-246, 2026-06-21)
**Исправление (MLC-246):** условие `<eq property="p:processName">` теперь кладётся **внутрь каждого
`<event>`**, рядом с `name` (объединение по «И»). Тест `Build_isolation_processName_lives_inside_each_event_next_to_name`
проверяет СЕМАНТИКУ: родитель каждого `p:processName`-`<eq>` — `<event>` (не `<log>`), и в каждом
`<event>` есть И `name`, И `p:processName`. Канон: `40_TECHLOG` §6, `60_SAFETY` №2.

**Исходная находка:** `<eq property="p:processName">` добавлялся **прямым ребёнком `<log>`** (LogcfgBuilder.cs ~стр.42),
а не внутри `<event>`. Комментарий «Один фильтр на весь `<log>`: режет ВСЕ события» — **неверная
модель**: в `logcfg` нет «фильтра уровня `<log>`».
**Спека (§4):** в `<log>` вкладываются только `<event>`/`<property>`; сравнения (`<eq>`) — **внутри
`<event>`**; условия одного `<event>` объединяются по «И». Bare-`<eq>` под `<log>` структурно
недопустим ⇒ **игнорируется** ⇒ изоляции нет ⇒ собирается ТЖ **всех арендаторов** (нагрузка +
**утечка чужих данных** — нарушение инварианта `60_SAFETY` №2).
**Верно:** условие `p:processName` должно лежать **в каждом** `<event>`, рядом с `name`:
```xml
<event>
  <eq property="name" value="DBMSSQL"/>
  <eq property="p:processName" value="ut11_saratov"/>
</event>
```
**Тест-пробел:** `LogcfgBuilderTests` проверяет наличие `<eq p:processName>` где угодно в документе,
а НЕ внутри `<event>` ⇒ ловушка прошла гейт (аналог «ISCC компилируется, T-SQL семантика — нет»).

### F-2 (P1, **Correctness**) — свойства событий не пишутся (нет `<property>`) — ✅ ИСПРАВЛЕНО (MLC-246, 2026-06-21)
**Исправление (MLC-246):** генератор добавляет `<property name="all"/>` внутрь `<log>` — пишутся все
свойства УЖЕ ОТОБРАННЫХ (по типу события + арендатору) событий (property-scope; НЕ «полный ТЖ»).
Тесты `Build_emits_all_property_inside_log_so_event_properties_are_written` (наличие) и
`Build_keeps_event_scope_targeted_never_full_techlog` (event-scope остаётся целевым: нет `<ne>`,
события строго из сценария). Канон: `40_TECHLOG` §6, `60_SAFETY` №1.

**Исходная находка:** генератор **не emit'ил ни одного `<property>`** (явный комментарий «без `<property
name="all"/>"`). 
**Спека (§4.2):** «Свойство события выводится, ТОЛЬКО если для него присутствует элемент
`<property>`». Без `<property>` событие пишется **лишь с базовыми полями** `ts/duration/name/depth/
level/process`. ⇒ В нашем собранном ТЖ **не будет** `Sql`, `Context`, `Locks`/`Regions`/
`WaitConnections`, `Descr`/`Exception`, `p:processName`, `Dbpid`, `Rows` и пр. ⇒ **парсер (этап B)
не извлечёт ничего полезного** (дерево блокировок/топ запросов/исключения — пусто).
**Дымящийся пистолет:** все снимки структуры полей MLC-229 (DBMSSQL с `Sql`/`Context`/…) сняты с
**`<property name="all"/>"`** (`60_SAFETY` стр.18) — наш генератор его не ставит, поэтому реальный
сбор нашим конфигом этих полей бы не дал. На стенде это всплыло бы как «пустые дела».
**Верно:** добавить `<property name="all"/>` (объём режется фильтрами события/арендатора, не
отсутствием свойств) ЛИБО точечный набор `<property>` под сценарий (Sql, Context, planSQLText,
Locks, Regions, WaitConnections, Descr, Exception, p:processName, Dbpid, Rows, …) + для SlowQueries
обязательно `<property name="planSQLText"/>` и `<property name="sql"/>` (иначе `<plansql>` бесполезен).
**Тест-пробел:** `Build_never_emits_all_property` **закрепляет** ошибку (явно запрещает `all`).

### F-3 (P2, **Doc/Performance**) — «фильтр длительности в JSON не работает» — вероятно, неверный диагноз
**Наш канон (40_TECHLOG §6/§8, 60_SAFETY №1):** «`<ge property="Dur">` в JSON ловит 0 ⇒ фильтр
длительности не работает, объём режем только типом события и арендатором».
**Спека (§5):** свойство называется **`Duration`** (сотни µs) / **`Durationus`** (µs); **`Dur` —
не существует** (в перечне нет) ⇒ `<… property="Dur">` ни с чем не сопоставится и вернёт 0 — это
объясняет «ловит 0» как **ошибку имени свойства**, а не «сломанный в JSON фильтр». Офиц. примеры
прямо используют `<gt property="duration" value="10000"/>` (=1 c).
**Следствие:** рабочий фильтр длительности **в `logcfg`** мог бы резать объём **в источнике** (важно:
сейчас объём режется лишь типом+арендатором, фильтр длительности убран). Не баг кода (убрать фильтр
безопасно — порог делает парсер), но **рационал неверен** и упущена оптимизация объёма.
**Действие:** стенд-ретест с `<event><eq property="name" value="DBMSSQL"/><ge property="Duration"
value="10000"/></event>` (8.5, JSON) — подтвердить, работает ли. До ретеста канон НЕ переписывать,
лишь пометить расхождение.

### F-4 (P3, заметка парсеру B) — терпимость к ротации/сжатию
Спека §9: при `rotation="size"`/`compress="zip"` имена файлов — `*.log.N`/`*.log.zip`. Наш генератор
их не включает (умолч. period/none), но оператор/иной инструмент может ⇒ парсер должен искать
`*.log*` и пропускать/распаковывать `.zip`, не падать на `.log.1`.

### F-5 (P3, проверить) — `<dump type="3">` объём
Спека §6: `type=3` = «дополнительный сегмент + вся память процесса» — для большинства случаев
достаточно, но full-dump тяжёлый. Уже помечено в коде/`40_TECHLOG` как стенд-приёмка. Рассмотреть
`externaldump="true"` для серверов без ежесуточного обслуживания (исключает зависание при дампе).
Плюс: каталог `\dumps` требует **полных прав агента `.\mitpro`** (как и каталог сбора) — учесть в ACL.
