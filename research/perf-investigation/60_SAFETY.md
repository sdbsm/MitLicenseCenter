# Политика безопасного сбора технологического журнала

> DRAFT. Кандидат в `SECURITY.md` + ADR при канонизации. Это **самый критичный** раздел
> дизайна: неосторожный сбор ТЖ сам становится инцидентом производительности.

---

## Главный риск (предупреждение из методик ИТС)

«Полный журнал» (`<property name="all"/>` без фильтров) на проде:
- **бьёт по производительности** самой системы, которую расследуем;
- **забивает диск** за минуты/часы (десятки ГБ).

Методика B1/B2 (`10_SOURCES.md`) прямо предупреждает: на рабочих серверах собирать только
нужные события, следить за местом. Наша панель должна делать это **за оператора**, не оставляя
возможности «выстрелить себе в ногу».

**Измерено на стенде 8.5 (2026-06-20):** полный ТЖ (`<property name="all"/>`) при одном активном
сеансе (проведение документа) дал файл `rphost` ≈ **260 МБ за ~100 секунд** (+ `rmngr` ≈ 67 МБ).
В пересчёте — гигабайты в час с одного процесса. Лимит места + авто-стоп — не «на всякий случай»,
а обязательны.

---

## Инварианты безопасности (закладываем в дизайн)

1. **Только целевой сбор.** Панель никогда не ставит «полный ТЖ». `logcfg.xml` всегда
   ограничен событиями сценария + фильтром по длительности (`40_TECHLOG.md` §6).
   ⚠ Сценарий СУБД-блокировок (`<dbmslocks/>`) собирается **без отборов по умолчанию** и особенно
   прожорлив (автор 1431026 получал файлы по часам >6 ГБ) — для него обязательны короткое окно,
   сторож места и авто-стоп.
2. **Изоляция арендатора обязательна.** `logcfg.xml` действует на **весь кластер** (все ИБ).
   Если расследование привязано к арендатору (`InfobaseId`), `logcfg` **обязан** содержать фильтр
   `<eq property="p:processName" value="<имя ИБ>"/>`. Без него собирается ТЖ всех арендаторов —
   это и лишняя нагрузка, и **утечка чужих данных** (тексты запросов/данные других ИБ). Сбор без
   фильтра при заданном `InfobaseId` — запрещён.
   ⚠ Нюанс (infostart 2718175): `p:processName` при динамическом обновлении пишется дважды
   (имя + ID обновления), и у разных событий несёт разный смысл — проверить, что фильтр изоляции
   надёжно ловит нужную ИБ на стенде, и подстраховать парсингом `p:processName` уже на разборе.
2. **Бокс по времени.** У сбора всегда есть окно и **авто-снятие** `logcfg.xml` по таймауту —
   даже если оператор забыл остановить (аналог авто-стопа нынешней «Записи»).
3. **Лимит места + сторож диска.** Перед стартом — проверка свободного места; во время сбора —
   мониторинг каталога; при достижении порога — **авто-стоп** (`StopReason=DiskLimit`).
4. **Один активный сбор за раз.** Нельзя запустить несколько `logcfg` одновременно.
5. **Гарантированное снятие + сторож на старте.** При остановке/ошибке/рестарте панели
   `logcfg.xml` **обязательно** возвращается в исходное состояние (восстановление осиротевших →
   `Interrupted`, как сейчас у «Записи»). **Дополнительно:** orphan-recovery закрывает запись в БД,
   но при крахе **ОС целиком** (не процесса панели) `logcfg.xml` останется в `conf`, и платформа
   продолжит писать ТЖ без панели. Поэтому на старте панель **всегда сверяет фактический
   `logcfg.xml` в `conf` с ожидаемым состоянием** и принудительно восстанавливает «снятый» вид,
   если нет активного `Investigation`. Никаких «забытых» конфигов на проде.
6. **Идемпотентность.** Повторный старт/снятие не ломает состояние; панель хранит снимок
   того, что ставила (`CollectionConfig`, `50_DATA_MODEL.md`).
7. **Только Admin.** Запуск/остановка сбора — роль Admin; Viewer только читает результаты.
8. **Аудит.** Каждая постановка/снятие `logcfg.xml` — запись аудита (новый
   `AuditActionType` = новое int-значение, существующие не переназначать — инвариант проекта).

## Данные ТЖ — чувствительность

- ТЖ может содержать **тексты запросов, имена, фрагменты данных** инфобаз арендаторов.
  Обращаться как с чувствительными: сырьё и `Finding` доступны соответствующим ролям,
  не отдаются наружу за пределы панели.
- **Репозиторий публичный** (память `repo-public`): фикстуры/примеры ТЖ для тестов —
  только обезличенные, без реальных данных арендаторов.

## NTFS ACL / расположение — проверено на стенде 8.5 (2026-06-20)

- **`logcfg.xml` = `C:\Program Files\1cv8\conf\logcfg.xml`.** Право записи в `conf`/`logcfg.xml` —
  только `NT AUTHORITY\SYSTEM`, `BUILTIN\Администраторы`, `TrustedInstaller`. Все прочие — чтение.
- **Сервисный аккаунт панели `NT SERVICE\MitLicenseCenter` права записи НЕ имеет** (как и аккаунт
  агента 1С `.\mitpro`). ⇒ **управление `logcfg.xml` под текущим аккаунтом невозможно.**
- **Решение для трека (выбрать при канонизации, кандидат в ADR):** одно из —
  (а) грант ACL `Modify` на `conf`/`logcfg.xml` сервисному аккаунту панели (делает установщик),
  (б) запуск операции под повышенными правами,
  (в) запуск службы панели под аккаунтом с правами. Вариант (а) — наименее привилегированный,
  предпочтителен; зафиксировать в `SECURITY.md` (модель ACL — ADR-8).
- Каталог сбора ТЖ (`location`) — отдельный, под контролем панели, с ограниченным ACL (туда платформа
  пишет под аккаунтом агента `.\mitpro` — учесть права на запись для агента).

## Чек-лист перед стартом сбора (для реализации этапа A)

- [ ] свободного места ≥ порога;
- [ ] нет другого активного сбора;
- [ ] события ограничены сценарием + фильтр длительности задан;
- [ ] заданы окно (авто-снятие) и лимит места (авто-стоп);
- [ ] исходный `logcfg.xml` сохранён для восстановления;
- [ ] запись аудита о постановке.
