# Технологический журнал 1С: формат, события, сбор, парсинг

> DRAFT. Справочник для реализации слоя ТЖ. Основан на `10_SOURCES.md` (B1 — bash-методика
> ИТС даёт формат txt-событий и приёмы разбора; B2–B8 — настройка и события).
> **Помечено `⚠ verify`** — уточнить на стенде с платформой 8.5, не закладываться вслепую.

---

## Решения трека (зафиксировано владельцем 2026-06-20)

- **Целевая платформа — 8.5** (актуальная). Платформы без поддержки JSON-ТЖ (до 8.3.25) —
  вне фокуса; под них не закладываемся.
- **Целимся в JSON** как основной формат. Возможность записи ТЖ в JSON — фича платформы
  (появилась в **8.3.25**, есть в 8.5), включается атрибутом `format` у элемента `<log>` в
  `logcfg.xml`. `⚠ verify` на стенде 8.5: **точное значение атрибута** (ожидается `format="json"`,
  но открытые источники без подписки этого не цитируют — дока за пейволом) и **структуру JSON**
  (NDJSON vs массив, имена полей).
- **Почему JSON, а не txt:** JSON — структурированный, читается `System.Text.Json` напрямую;
  не нужен хрупкий разбор многострочных событий, удаление BOM и regex-эквилибристика, без
  которых txt-формат не распарсить надёжно (см. ниже раздел про txt).
- **НО fallback на txt держим в уме** до подтверждения JSON-схемы на стенде: фича JSON свежая
  (8.3.25), и если на 8.5 схема окажется неудобной/нестабильной — txt-парсер может стать не
  «легаси на всякий случай», а необходимым. Решение «только JSON» фиксируем **после** проверки
  на живой 8.5, а не вслепую.

## 1. Где лежит и как включается

- Конфигурация — `logcfg.xml` в каталоге `conf` платформы. **Проверено на стенде 8.5.1.1302
  (2026-06-20):** путь — `C:\Program Files\1cv8\conf\` (общий каталог, `conf.cfg` без
  `ConfLocation`-переопределения, только `SystemLanguage`). На стенде уже лежит `logcfg.xml`,
  но в нём **только секция `<dump>`** — сбор событий ТЖ не настроен.
- **⚠ Доступ — НЕ автоматический (важно для дизайна).** Каталог в `Program Files`; право записи в
  `conf`/`logcfg.xml` — только у `SYSTEM`, `Администраторы`, `TrustedInstaller`. Сервисный аккаунт
  панели `NT SERVICE\MitLicenseCenter` и аккаунт агента 1С (`.\mitpro`) имеют **только чтение**.
  ⇒ панель сама `logcfg.xml` под текущими правами **не запишет** — нужно повышение прав или явный
  грант ACL (см. `60_SAFETY.md`). Single-host (ADR-28) даёт локальность, но не доступ.
- Платформа перечитывает `logcfg.xml` периодически (до ~минуты); в каталоге из `location`
  начинают появляться файлы ТЖ.
- Включение JSON: атрибут `format` у `<log>` → значение для JSON. `⚠ verify` точное значение
  (ожидается `format="json"`) и поведение на 8.5 на стенде.

## 2. Структура каталога и файлов (общая)

- `location` → **подкаталоги по процессам**: `<имя процесса>_<PID>/`
  (`rphost_1234/`, `rmngr_…/`, `ragent_…/`). Один сеанс «живёт» в rphost-процессе.
- Файлы **по часам**, имя `YYMMDDHH` (пример из методики: `16091011.log` = 2016-09-10, 11:00).
- Ротация — атрибут `history` (хранить N часов). Превышение = старые часы удаляются платформой.
- **Подтверждено на стенде 8.5 (2026-06-20):** подкаталоги `<процесс>_<PID>/` (`rphost_39992`,
  `rmngr_27996`, `ragent_42296`, `ras_6360`, `1cv8c_2716`, плюс `rac_*` от вызовов), файлы по часам
  `ГГММДДЧЧ.log`; JSON-формат — NDJSON (объект на строку, §4), а не массив.

## 3. Формат txt (справочно — почему мы его НЕ берём)

Строка-событие: `mm:ss.ffffff-<длительность_мкс>,<СОБЫТИЕ>,<уровень>,<ключ>=<значение>,…`
- длительность в **микросекундах** (`20000000` = 20 c);
- **многострочные** события: `DBMSSQL` содержит текст запроса и `Context`; новое событие —
  строка, начинающаяся с метки времени `\d\d:\d\d\.\d+`; всё между метками — продолжение;
- встречается **BOM** (`\xef\xbb\xbf`), ломающий наивный разбор; кодировка UTF-8.

Именно эта хрупкость (склейка многострочных событий регуляркой по метке времени, чистка BOM,
ручной разбор `key=value` с запятыми внутри значений) — причина выбрать JSON. Приёмы txt-разбора
из методики B1 ценны как **логика агрегаций**, но переносим их в C# поверх JSON, а не воспроизводим
bash/regex-конвейер.

## 4. Формат JSON — СНЯТО С ЖИВОЙ 8.5.1.1302 (2026-06-20)

Снято прямым чтением `rphost`-файла под нагрузкой (проведение документа УНФ). ⚠ **Нативный формат
8.5 отличается от README `medigor/tech-log-parser`** — там описан нормализованный выход его
конвертера (массив, `Props`=пары). Платформа 8.5 пишет иначе:

- **NDJSON: один JSON-объект на строку** (НЕ массив `[…]`). Файл начинается с BOM, далее объекты,
  разделённые переводом строки. (Совпадает с infostart 2718175 «каждое событие в отдельной строке»;
  массив из README medigor — это 8.3.25 / выход его конвертера, не нативная 8.5.)
- **Событие — ПЛОСКИЙ объект; свойства — поля верхнего уровня** (а НЕ массив `Props`). Базовые поля:
  `ts`, `duration`, `name`, `depth`, `level`, `process`; далее свойства события как обычные ключи.
- **Все значения — СТРОКИ** (даже числа: `"duration":"1"`, `"Rows":"1"`, `"SessionID":"1"`).
- `Sql`/`Context` — внутри строкового значения с экранированием `\n`/`\r\n`; склейка многострочности
  (как в txt) НЕ нужна — главный выигрыш JSON.

Реальное событие со стенда (DBMSSQL, обрезано):
```json
{"ts":"2026-06-20T03:04:58.858004","duration":"1","name":"DBMSSQL","depth":"5","level":"DEBUG",
 "process":"rphost","p:processName":"mitpro","OSThread":"48908","t:clientID":"17",
 "t:applicationName":"1CV8C","t:computerName":"Andrey-pc","t:connectID":"3","SessionID":"1",
 "Usr":"Андрей","AppID":"1CV8C","DBMS":"DBMSSQL","DataBase":"localhost\\mitpro","Trans":"1",
 "dbpid":"65","Sql":"SELECT ...","Rows":"1","RowsAffected":"-1","Context":"Данные.Провести\r\n..."}
```

- **`p:processName` = имя инфобазы** (на стенде `"mitpro"`, `DataBase`=`localhost\mitpro`) —
  подтверждает фильтр изоляции арендатора по `p:processName` вживую (`60_SAFETY.md`).
- **`duration` — строка.** На образце DBMSSQL `duration`=1..3; единица — вероятно **миллисекунды**
  (как и в README medigor), но события образца быстрые → ⚠ подтвердить на заведомо медленной
  операции. Нормализовать в модели.
- **Дубли ключей** (риск infostart 2718175: `Func` дважды у завершения транзакции) в плоском объекте
  = дубли ключей JSON. `System.Text.Json` по умолчанию берёт последний → потеря. ⇒ для событий с
  возможным повтором свойства — потоковый/толерантный reader, сохраняющий все вхождения (это и есть
  причина, по которой конвертер medigor нормализует свойства в массив пар).
- **Чтение:** NDJSON → построчно, каждую строку парсить как объект (память не растёт; проблема ×10
  из README medigor про «массив целиком» у нас не возникает).
- **Имя файла остаётся `ГГММДДЧЧ.log`** при `format="json"` (расширение НЕ меняется — формат меняет
  содержимое; парсер ищет `*.log`). Атрибут `format="json"` платформой принят без ошибки.

## 5. Ключевые события (Event → смысл → важные свойства)

Под наши сценарии достаточно узкого набора. Полный справочник — B3 (Infostart), сверять с
платформенной докой.

| Событие | Смысл | Важные свойства | Сценарий |
|---|---|---|---|
| `TLOCK` | Установка управляемой блокировки | ресурс/пространство блокировки, процесс, поток | блокировки |
| `TTIMEOUT` | Таймаут ожидания блокировки («Lock request timeout») | кто ждёт, кого ждёт, ресурс | блокировки |
| `TDEADLOCK` | Взаимоблокировка (управляемые) | участники, ресурсы | блокировки |
| `DBMSSQL` | Обращение к СУБД | `Sql` (текст), длительность, rows, dbpid | долгие запросы |
| `SDBL` | Обращение к модели данных 1С | `Func` (Commit/RollbackTransaction…), `Context` | транзакции/долгие |
| `EXCP` | Исключение платформы | `Descr` (текст ошибки), `Context` | исключения/падения |
| `CALL`/`SCALL` | Серверный вызов | длительность, `Memory`, контекст | общая медленная работа |
| `VRSREQUEST`/`VRSRESPONSE` | Взаимодействие веб-расширения/клиента | длительность | веб/долгие операции |

**⚠ Два РАЗНЫХ уровня блокировок — не путать (важно для «дерева блокировок»):**
- `TLOCK/TTIMEOUT/TDEADLOCK` — это **управляемые блокировки уровня 1С** (менеджер блокировок
  платформы). Дерево «кто кого ждёт» по ним — только про 1С-блокировки.
- **Блокировки и дедлоки уровня СУБД** этими событиями НЕ видны. Их источник — отдельный тег
  logcfg **`<dbmslocks/>`** (НЕ `<event>`) с полями `lkX`; пишет сервис вспомогательных функций
  кластера по запросу к `sysprocesses`. Плюс `DBMSSQL`+`<plansql/>` и **внешние средства MS SQL**
  (Extended Events / Profiler / отчёт «Top Transactions by Locks»).

  Семантика полей `lkX` (по infostart 1431026, по мотивам вебинара В. Богачёва):

  | Поле | Смысл |
  |---|---|
  | `lka=1` | поток — **источник** блокировки |
  | `lkp=1` | поток — **жертва** блокировки |
  | `lkpid` | номер запроса к СУБД (у **жертвы**: кто её заблокировал) |
  | `lkaid` | список номеров запросов к СУБД (у **источника**) |
  | `lksrc` | **номер соединения источника** (у жертвы) — связка «жертва → виновник» |
  | `lkpto` / `lkato` | секунд с момента, как поток признан жертвой / источником |

  **Алгоритм дерева СУБД-блокировок:** `жертва.lksrc` → `источник.connectID`. Связка с MS SQL —
  `sys.dm_exec_requests.blocking_session_id` + `dm_exec_sql_text` + `dm_exec_query_plan`.
- Поэтому «дерево блокировок» в дизайне — это **два класса**, а не один. Парсер по `TLOCK`-семейству
  даёт полноту только по 1С-уровню; СУБД-уровень — через `<dbmslocks/>`/`lkX`. Не обещать единое
  дерево, иначе ложная полнота.

**Привязка к арендатору/инфобазе.** Сквозные свойства событий: `SessionID`, `ClientID`,
`Usr` (пользователь), `p:processName` (имя процесса = **имя инфобазы**), PID (из подкаталога),
идентификатор соединения. Ключевой для нас — **`p:processName`**: именно по нему ТЖ и фильтруется,
и связывается с инфобазой/арендатором (маппинг база→клиент у нас уже есть, см. `20_CURRENT_STATE.md`).

## 6. Целевые шаблоны logcfg.xml по сценариям (принцип, не «полный журнал»)

**⚠ `logcfg.xml` действует на ВЕСЬ кластер/агента** (ragent, rmngr, все rphost всех инфобаз),
а не на одну ИБ. Поэтому:
- фильтр по длительности — `<ge property="Dur" value="<микросекунды>"/>` (значение в **мкс**:
  1 c = 1 000 000);
- если расследование привязано к арендатору (`InfobaseId`), шаблон **обязан** содержать фильтр
  по инфобазе `<eq property="p:processName" value="<имя ИБ>"/>` — иначе пишется ТЖ ВСЕХ
  арендаторов (лишняя нагрузка + чужие данные). Это инвариант безопасности — см. `60_SAFETY.md`.

Каждый шаблон: `format="json"`, `location` в наш каталог сбора, ограниченный `history`,
**только нужные события** под сценарий (минимум влияния на прод).

- **Блокировки/дедлоки (1С-уровень):** `TLOCK`, `TTIMEOUT`, `TDEADLOCK` + `SDBL` с
  `Func`=`Begin/Commit/RollbackTransaction` (контекст транзакций).
- **Блокировки (СУБД-уровень):** отдельный тег **`<dbmslocks/>`** с `<property name="lka/lkp/lkpid/
  lkaid/lksrc"/>` (§5). ⚠ Автор 1431026 отмечает: dbmslocks собирается **без отборов** → объём
  большой (у него файлы по часам >6 ГБ) — см. `60_SAFETY.md`.
- **Долгие запросы + планы:** `DBMSSQL`/`SDBL` с фильтром длительности + тег **`<plansql/>`** —
  планы запросов по умолчанию НЕ собираются, нужен явно.
- **Исключения/падения:** `EXCP` + `EXCPCNTX` (контекст) + `<dump .../>` для дампов аварий.
- **Общая медленная работа (серверная часть):** `CALL/DBMSSQL` дольше порога, не «всё подряд».

**⚠ Имя поля длительности в фильтре непостоянно:** в готовых шаблонах infostart встречается
`<ge property="duration" value="100000"/>` (мкс), в других — `Dur`; в JSON поле зовётся `Duration`
(мс, §4). На стенде 8.5 проверить точное имя/регистр/единицы для logcfg-фильтра.

Готовые рабочие шаблоны под все сценарии — infostart 2020498 (`10_SOURCES.md` B4). Точные XML под
наш контур — в реализации (этап A). Здесь — состав и принцип «целевой, не полный».

## 7. Что это значит для C#-парсера (этап B)

- Читаем `*.log`-файлы из подкаталогов процессов **построчно** (NDJSON: объект на строку, §4);
  снять BOM; окно времени — по имени файла (час) + полю `ts` события.
- Все значения — **строки**; `duration` парсить из строки и нормализовать единицы (§4).
- Свойства — поля верхнего уровня объекта. Для событий с **возможными дублями ключей** (§4) —
  толерантный reader, сохраняющий все вхождения (не `Dictionary` вслепую).
- Агрегаты под сценарий:
  - **дерево 1С-блокировок** из `TTIMEOUT/TDEADLOCK/TLOCK`; **дерево СУБД-блокировок** — из `lkX`
    (§5), это два разных механизма;
  - **топ долгих** `DBMSSQL` по длительности (текст + контекст + план из `<plansql/>`);
  - **топ** `EXCP` по частоте (текст + контекст).
- Привязка к инфобазе/арендатору через свойства события + существующий маппинг база→клиент.
- На txt **не закладываемся** (но §Решения — fallback до подтверждения JSON на 8.5).

**Грабли парсера — чек-лист устойчивости (infostart 2718175, июнь 2026):**
- **Дубли ключей реальны:** `t:clientID` дублируется; у `SDBL`-завершения транзакции `Func`
  встречается ДВАЖДЫ с разными значениями. Брать «последнее значение» — терять данные (отсюда
  `Props`=список, §4).
- **`p:processName` ненадёжен как имя ИБ:** при динамическом обновлении пишется дважды (имя + ID
  обновления); у разных событий — разный смысл. **Влияет на наш инвариант изоляции** (`60_SAFETY.md`):
  фильтр `<eq property="p:processName">` сработает по одному из значений — учесть.
- **«Поля-призраки»:** `escalating` у `TLOCK` и семейство `LKP` появляются только при `=true`;
  даже `Sql` у `DBMSSQL` иногда отсутствует. Нельзя рассчитывать на постоянное присутствие поля.
- **Варианты имён:** пользователь — обычно `Usr`, иногда `UserName`; компьютер — `t:computerName`,
  иногда `ClientComputerName`; `dbpid` бывает null/пустым; `SessionID` иногда с «(родителем)».
- **Дубли событий:** дедлок СУБД пишет **ДВА `EXCP`** (фильтр `Exception=DataBaseException`);
  ложные `EXCP` при окне авторизации. `EXCP` «вездесущ» — пролезает в любую выборку; парсер не
  должен падать на чужом наборе полей.
- **Контекст отделён:** иногда `Context` — отдельное событие, иногда РАЗОРВАН (часть в событии,
  часть в отдельном `Context`); связывать по `clientID`.
