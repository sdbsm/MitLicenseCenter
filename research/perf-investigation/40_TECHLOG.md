# Технологический журнал 1С: формат, события, сбор, парсинг

> DRAFT. Справочник для реализации слоя ТЖ. Основан на `10_SOURCES.md` (B1 — bash-методика
> ИТС даёт формат txt-событий и приёмы разбора; B2–B8 — настройка и события).
> **Помечено `⚠ verify`** — уточнить на стенде с платформой 8.5, не закладываться вслепую.

---

## Решения трека (зафиксировано владельцем 2026-06-20)

- **Целевая платформа — 8.5** (актуальная). Платформы без поддержки JSON-ТЖ (до 8.3.25) —
  вне фокуса; под них не закладываемся.
- **Целимся в JSON** как основной формат. Возможность записи ТЖ в JSON — фича платформы
  (появилась в **8.3.25**, есть в 8.5), включается атрибутом `format` у элемента `<log>` в
  `logcfg.xml`. `⚠ verify` на стенде 8.5: **точное значение атрибута** (ожидается `format="json"`,
  но открытые источники без подписки этого не цитируют — дока за пейволом) и **структуру JSON**
  (NDJSON vs массив, имена полей).
- **Почему JSON, а не txt:** JSON — структурированный, читается `System.Text.Json` напрямую;
  не нужен хрупкий разбор многострочных событий, удаление BOM и regex-эквилибристика, без
  которых txt-формат не распарсить надёжно (см. ниже раздел про txt).
- **НО fallback на txt держим в уме** до подтверждения JSON-схемы на стенде: фича JSON свежая
  (8.3.25), и если на 8.5 схема окажется неудобной/нестабильной — txt-парсер может стать не
  «легаси на всякий случай», а необходимым. Решение «только JSON» фиксируем **после** проверки
  на живой 8.5, а не вслепую.

## 1. Где лежит и как включается

- Конфигурация — `logcfg.xml` в каталоге `conf` платформы (`...\1cv8\conf\` либо `conf`
  рядом с каталогом кластера). Single-host (ADR-28): каталог **локальный**, панель имеет
  к нему доступ — это и есть наш «движок сбора».
- Платформа перечитывает `logcfg.xml`; в каталоге из `location` начинают появляться файлы ТЖ.
- Включение JSON: атрибут `format` у `<log>` → значение для JSON. `⚠ verify` точное значение
  (ожидается `format="json"`) и поведение на 8.5 на стенде.

## 2. Структура каталога и файлов (общая)

- `location` → **подкаталоги по процессам**: `<имя процесса>_<PID>/`
  (`rphost_1234/`, `rmngr_…/`, `ragent_…/`). Один сеанс «живёт» в rphost-процессе.
- Файлы **по часам**, имя `YYMMDDHH` (пример из методики: `16091011.log` = 2016-09-10, 11:00).
- Ротация — атрибут `history` (хранить N часов). Превышение = старые часы удаляются платформой.
- `⚠ verify` на 8.5: расширение/структура JSON-файла — один JSON-объект на строку (NDJSON-подобно)
  или массив объектов на файл; точные имена полей.

## 3. Формат txt (справочно — почему мы его НЕ берём)

Строка-событие: `mm:ss.ffffff-<длительность_мкс>,<СОБЫТИЕ>,<уровень>,<ключ>=<значение>,…`
- длительность в **микросекундах** (`20000000` = 20 c);
- **многострочные** события: `DBMSSQL` содержит текст запроса и `Context`; новое событие —
  строка, начинающаяся с метки времени `\d\d:\d\d\.\d+`; всё между метками — продолжение;
- встречается **BOM** (`\xef\xbb\xbf`), ломающий наивный разбор; кодировка UTF-8.

Именно эта хрупкость (склейка многострочных событий регуляркой по метке времени, чистка BOM,
ручной разбор `key=value` с запятыми внутри значений) — причина выбрать JSON. Приёмы txt-разбора
из методики B1 ценны как **логика агрегаций**, но переносим их в C# поверх JSON, а не воспроизводим
bash/regex-конвейер.

## 4. Формат JSON (8.5 — наш выбор)

- Событие — JSON-объект: имя события, длительность, уровень и свойства как поля объекта.
- Парсится `System.Text.Json` в типизированные модели; текст запроса и контекст — обычные
  строковые значения с штатным экранированием (никакой склейки строк).
- `⚠ verify` на стенде: точная схема объекта (имена полей, вложенность, как отдаётся длительность),
  NDJSON vs массив, кодировка.

## 5. Ключевые события (Event → смысл → важные свойства)

Под наши сценарии достаточно узкого набора. Полный справочник — B3 (Infostart), сверять с
платформенной докой.

| Событие | Смысл | Важные свойства | Сценарий |
|---|---|---|---|
| `TLOCK` | Установка управляемой блокировки | ресурс/пространство блокировки, процесс, поток | блокировки |
| `TTIMEOUT` | Таймаут ожидания блокировки («Lock request timeout») | кто ждёт, кого ждёт, ресурс | блокировки |
| `TDEADLOCK` | Взаимоблокировка (управляемые) | участники, ресурсы | блокировки |
| `DBMSSQL` | Обращение к СУБД | `Sql` (текст), длительность, rows, dbpid | долгие запросы |
| `SDBL` | Обращение к модели данных 1С | `Func` (Commit/RollbackTransaction…), `Context` | транзакции/долгие |
| `EXCP` | Исключение платформы | `Descr` (текст ошибки), `Context` | исключения/падения |
| `CALL`/`SCALL` | Серверный вызов | длительность, `Memory`, контекст | общая медленная работа |
| `VRSREQUEST`/`VRSRESPONSE` | Взаимодействие веб-расширения/клиента | длительность | веб/долгие операции |

**⚠ Два РАЗНЫХ уровня блокировок — не путать (важно для «дерева блокировок»):**
- `TLOCK/TTIMEOUT/TDEADLOCK` — это **управляемые блокировки уровня 1С** (менеджер блокировок
  платформы). Дерево «кто кого ждёт» по ним — только про 1С-блокировки.
- **Блокировки и дедлоки уровня СУБД** этими событиями НЕ видны. Их источники иные: поля
  `lka/lkp/lkpid/lkaid/lksrc/lkpto` (пишет сервис вспомогательных функций кластера по запросу к
  `sysprocesses`), `DBMSSQL` + `<plansql/>`, либо **внешние средства MS SQL** (Extended Events /
  Profiler / отчёт «Top Transactions by Locks»).
- Поэтому «дерево блокировок» в дизайне — это **два класса**, а не один. Парсер по `TLOCK`-семейству
  даст полноту только по 1С-уровню; СУБД-уровень требует отдельного механизма. Не обещать единое
  дерево, иначе ложная полнота.

**Привязка к арендатору/инфобазе.** Сквозные свойства событий: `SessionID`, `ClientID`,
`Usr` (пользователь), `p:processName` (имя процесса = **имя инфобазы**), PID (из подкаталога),
идентификатор соединения. Ключевой для нас — **`p:processName`**: именно по нему ТЖ и фильтруется,
и связывается с инфобазой/арендатором (маппинг база→клиент у нас уже есть, см. `20_CURRENT_STATE.md`).

## 6. Целевые шаблоны logcfg.xml по сценариям (принцип, не «полный журнал»)

**⚠ `logcfg.xml` действует на ВЕСЬ кластер/агента** (ragent, rmngr, все rphost всех инфобаз),
а не на одну ИБ. Поэтому:
- фильтр по длительности — `<ge property="Dur" value="<микросекунды>"/>` (значение в **мкс**:
  1 c = 1 000 000);
- если расследование привязано к арендатору (`InfobaseId`), шаблон **обязан** содержать фильтр
  по инфобазе `<eq property="p:processName" value="<имя ИБ>"/>` — иначе пишется ТЖ ВСЕХ
  арендаторов (лишняя нагрузка + чужие данные). Это инвариант безопасности — см. `60_SAFETY.md`.

Каждый шаблон: `format="json"`, `location` в наш каталог сбора, ограниченный `history`,
**только нужные события** под сценарий (минимум влияния на прод).

- **Блокировки/дедлоки (1С-уровень):** `TLOCK`, `TTIMEOUT`, `TDEADLOCK` (+ опц. `DBMSSQL` для
  контекста). СУБД-дедлоки — отдельно (см. §5, требуют `lkX`-событий или внешних средств MS SQL).
- **Долгие запросы:** `DBMSSQL` с `<ge property="Dur" .../>` (напр. 1–3 c в мкс).
- **Исключения/падения:** `EXCP` (+ параметры дампов при необходимости).
- **Общая медленная работа (серверная часть):** `CALL/DBMSSQL` дольше порога, не «всё подряд».

Точные XML-шаблоны — в реализации (этап A). Здесь — состав и принцип «целевой, не полный».

## 7. Что это значит для C#-парсера (этап B)

- Читаем JSON-файлы из подкаталогов процессов; окно времени — по имени файла (час) + метке
  времени внутри события.
- Десериализация в типизированные события → агрегаты под сценарий:
  - **дерево блокировок** из `TTIMEOUT/TDEADLOCK/TLOCK` («кто кого ждёт, на каком ресурсе»);
  - **топ долгих** `DBMSSQL` по длительности (текст + контекст);
  - **топ** `EXCP` по частоте (текст + контекст).
- Привязка к инфобазе/арендатору через свойства события + существующий маппинг база→клиент.
- На txt **не закладываемся**.

## 8. Открытые вопросы для стенда (8.5)

1. Точное значение атрибута `format` (ожидается `json`) и структура JSON-файла (NDJSON vs массив).
2. Точные имена полей события в JSON (могут отличаться от txt `key=value`).
3. Путь каталога `conf` для нашей инсталляции платформы; права записи `logcfg.xml`.
4. Поведение `history` и объёмы под целевые фильтры (для политики места — `60_SAFETY.md`).
