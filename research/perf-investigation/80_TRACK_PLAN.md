## Операционная модель (как двигаются чаты)

**Ветка и откат.** Все изменения 1.2 идут в **`release/1.2`**. PR'ы целить в `release/1.2`, **не в
`main`**. `main` заморожен на теге `v1.1.4` (точка возврата). Откат трека = не вливать `release/1.2`.
Выпуск 1.2 = merge `release/1.2`→`main` + bump `Directory.Build.props` 1.1.4→1.2.0 + тег `v1.2.0` +
релиз (публикует владелец). НИКОГДА не `reset --hard`.

**Старт любой сессии:**
1. Прочитать `research/perf-investigation/00_INDEX.md` (+ файлы по теме задачи).
2. Прочитать этот план и `docs/PROJECT_BACKLOG.md` → `NEXT TASK`.
3. Работать **только над `NEXT TASK`** (одна задача за сессию).

**Куратор-чат** (не тот, кто составил план — контекст ограничен): ставит одну задачу постановочным
PR в `release/1.2`, раздаёт исполнителю в worktree, ревьюит **реальный diff**, вливает, ставит
следующую `NEXT TASK`. Куратор инкрементирует номера `MLC-NNN` по выдаче (см. `PROJECT_BACKLOG.md`).

**Исполнитель-чат:** одна задача, ветка-worktree от `release/1.2`, локальный гейт `build.ps1`
зелёный, PR в `release/1.2`. Док-правки канона — только по заданию куратора.

**Гейт качества.** `build.ps1` локально (CI может быть красным по биллингу — это штатно). **Парсер ТЖ
тестировать на РЕАЛЬНЫХ JSON-образцах со стенда 8.5** (формат снят, см. `40_TECHLOG` §4), не выдумывать.
Стенд-приёмка живого сбора — за владельцем.

**Инварианты (не нарушать молча):** безопасный сбор ТЖ (`60_SAFETY` — целевые события, окно/авто-стоп,
лимит места, один активный, гарантированное снятие + сторож на старте, изоляция арендатора через
`p:processName`); enum аудита заморожен (новое действие = новое число); parity BE↔FE (Zod); single-host
(ADR-28).

## Этапы и задачи

Порядок: **0 → A → B → C → D**. Каждая задача указывает управляющий файл KB и критерий готовности.
Номера `MLC-228+` — провизорные; куратор выдаёт по одной (228 уже выставлен как `NEXT TASK`).

### Этап 0 — Канонизация и стенд-уточнения (фундамент)

- **MLC-228 · Канонизация трека (docs-only).** ADR «Производительность = Наблюдение + Расследование»
  и ADR «Политика безопасного сбора ТЖ» в `docs/DECISIONS.md`; перенос ключевого из KB в present-tense
  канон (`02_ARCHITECTURE` — слой ТЖ; `04_BACKEND` — сервисы сбора/парсинга; `05_FRONTEND` — экраны
  `70_UI_SPEC`); трек в `ROADMAP.md`. Готово: ADR'ы есть, канон ссылается на KB, build зелёный.
- **MLC-229 · Снять оставшиеся ⚠ на стенде 8.5 (needs stand).** Единицы `duration` в JSON (на
  заведомо медленной операции), имя/регистр поля длительности в logcfg-фильтре (`Dur`/`duration`),
  грант ACL сервисному аккаунту `NT SERVICE\MitLicenseCenter` на `conf`/`logcfg.xml`. Зафиксировать
  факты в `40_TECHLOG`/`60_SAFETY`. Готово: все `⚠ verify` сняты или подтверждены.

### Этап A — Управление ТЖ (logcfg lifecycle). Управляет: `40_TECHLOG` §1/§6, `60_SAFETY`

- **MLC-230 · Сервис logcfg.** Генерация целевого `logcfg.xml` (события сценария + `p:processName` +
  `Dur` + `format="json"`), установка в `conf`, проверка/грант ACL, **сторож на старте** (сверка
  фактического `logcfg` с ожидаемым; принудительное снятие, если нет активного дела). BE + порт.
  Готово: панель ставит/снимает целевой ТЖ под своим аккаунтом; сторож закрывает «забытый» конфиг.
- **MLC-231 · Безопасный сбор.** Окно + авто-стоп, лимит места + сторож диска (авто-стоп `DiskLimit`),
  один активный сбор, восстановление осиротевших (`Interrupted`, как `PerfRecording`). BE.
  Готово: нельзя оставить «полный ТЖ» / переполнить диск; сбор всегда снимается.

### Этап B — Парсер + анализ ТЖ. Управляет: `40_TECHLOG` §4/§5/§7

- **MLC-232 · Парсер NDJSON-ТЖ 8.5.** Построчно, плоский объект `{ts,duration,name,depth,level,
  process,<свойства>}`, значения-строки, толерантность к дублям ключей, нормализация `duration`.
  **Тесты на реальных образцах стенда.** BE. Готово: парсит типовые `rphost`-файлы без потерь.
- **MLC-233 · Дерево 1С-блокировок.** Из `TLOCK/TTIMEOUT/TDEADLOCK` — «кто кого ждёт», таймауты, дедлоки.
- **MLC-234 · Топ долгих запросов.** Из `DBMSSQL` — длительность, текст `Sql`, `Context`, план (`plansql`).
- **MLC-235 · Исключения.** Из `EXCP` — топ, частота; учесть дедлок СУБД = пара `EXCP`.
- **MLC-236 · (опц.) СУБД-блокировки.** Из `<dbmslocks/>`/`lkX` — дерево «жертва.lksrc → источник».

### Этап C — Объект «Дело» + сведение + API ✅. Управляет: `50_DATA_MODEL`

- **MLC-237 ✅ · Модель данных.** `Investigation`/`CollectionConfig`/`Finding` — `Investigation` ЗАМЕНИЛ
  прокси `TechLogCollection` (миграция данных + drop). `Finding` = версионированный JSON. `RowVersion` +
  concurrency-тест на SQLite. `PerfRecording` НЕ тронут (фолд — этап D). BE.
- **MLC-238 ✅ · Оркестрация дела.** Снятие сбора → конвейер `Collecting → Analyzing → Completed`/`Failed`:
  парсинг (B) → анализаторы по сценарию → `Finding` → удаление сырья. `CollectionConfig` наполняется на старте.
- **MLC-239 ✅ · API.** Endpoints `/api/v1/investigations`: старт (резолв инфобазы → привязка арендатора)/
  стоп/список/дело/отчёт/прогресс/удаление. Zod-парити + omit-null тесты на FE. Audit 809.

### Этап D — Frontend. Управляет: `70_UI_SPEC` (6 экранов)

> **Нумерация этапа D сдвинута `240→241..244`:** `MLC-240` занят вне трека (вход из LAN, ADR-59, `v1.1.5`).

- **MLC-241 · Режим «Наблюдение».** Переработка нынешней «Быстродействие» под 2-режимную модель
  (экран 1). Переиспользовать probe/данные (`20_CURRENT_STATE`). ← NEXT (этап C ✅).
- **MLC-242 · Мастер запуска + прогресс сбора** (экраны 2, 6): сценарий + выбор инфобазы из реестра + лимиты.
- **MLC-243 · «Дело» + список дел** (экраны 3, 5): вердикт, дерево блокировок, топ запросов, исключения, рекомендации.
- **MLC-244 · «Отчёт» + экспорт** (экран 4): документ-вид + PDF/копия.

## Зависимости (кратко)

`Этап 0` → всё. `A` → `B` (парсеру нужны собранные/стендовые образцы) → `C` (дело сводит сбор+парсинг)
→ `D` (FE по API из `C`). `MLC-229` (стенд) желательно до `B`/`A`. `E` — независим, поздний.

## Критерий завершения трека

Оператор может: на «Наблюдении» увидеть симптом → через Мастер запустить безопасное расследование по
конкретной инфобазе → получить «Дело» с деревом блокировок, топом долгих запросов и исключениями,
привязанными к арендатору → выгрузить «Отчёт» с рекомендациями. Сбор ТЖ безопасен (лимиты/авто-снятие).
Затем — bump 1.2.0, тег `v1.2.0`, релиз (владелец).
