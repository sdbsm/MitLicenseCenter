## 1. Назначение и расположение

- `logcfg.xml` лежит в каталоге **conf** платформы (у нас стенд: `C:\Program Files\1cv8\conf\`).
  Наличие файла **не обязательно**. Если файла нет — ТЖ выключен (`<log>` отсутствует), пишется
  только «журнал по умолчанию» (`<defaultlog>`: события `SYSTEM` уровня `ERROR`, 24 ч, в
  `%LOCALAPPDATA%\1C\1cv8\logs`), и минимальные дампы аварий в `%LOCALAPPDATA%\1C\1cv8\dumps`.
- Платформа перечитывает `logcfg.xml` периодически (до ~минуты).
- Корневой элемент — **`<config xmlns="http://v8.1c.ru/v8/tech-log">`**.

## 2. Структура `<config>` (порядок и кратность детей)

```xml
<config xmlns="http://v8.1c.ru/v8/tech-log">
  <log …>…</log>        <!-- 0..N: каталог + отбор событий. Рекомендация: НЕ более 20 <log> -->
  <dump … />            <!-- 0..1: дампы аварий (только Windows) -->
  <plansql/>            <!-- 0..1: включает сбор планов запросов СУБД -->
  <dbmslocks/>          <!-- 0..1: включает сбор СУБД-блокировок (свойства lkX) -->
  <mem/> <leaks>…</leaks> <ftextupd …/> <query …/> <scriptcircrefs/>
  <inputbystring …/> <sessiondatacontext/> <system …/> <defaultlog …/>
</config>
```

**Три группы элементов (ключ к пониманию модели):**
1. **Включатели формирования событий** — `<dump>`, `<leaks>`, `<mem>`, `<plansql>`, `<ftextupd>`,
   `<system>`, `<query>`, `<scriptcircrefs>`, `<inputbystring>`, `<dbmslocks>`. Если элемента нет —
   соответствующее событие/данные **не формируются вообще** (отбор по ним бесполезен). Это
   **config-level директивы**, рядом с `<log>`, **не внутри** него.
2. **Отбор уже сформированных данных** — `<event>` и `<property>` (только **внутри** `<log>`). Могут
   лишь **уменьшить** объём.
3. **Размещение файлов** — `<log>`, `<defaultlog>`, `<dump>` (у каждого свой `location`; **каталоги
   должны различаться**).

## 3. Элемент `<log>` — каталог и отбор

Атрибуты:

| Атрибут | Значения | Примечание |
|---|---|---|
| `location` | путь | каталог ТЖ; **отличается** от `<dump>`/`<defaultlog>` |
| `format` | `text` (умолч.) / `json` | `json` = последовательность JSON-объектов |
| `history` | часы | через сколько часов удалять; **`0` = НЕ удалять** (опасно: переполнение диска) |
| `compress` | `none` (умолч.) / `zip` | сжатие при ротации → файлы `.zip` |
| `placement` | `folders` (умолч.) / `plain` | `folders` = подкаталоги `<appName>_<pid>` |
| `rotation` | `period` (умолч.) / `size` | по времени или по размеру |
| `rotationperiod` | часы (умолч. 1) | период ротации |
| `rotationsize` | МБ (умолч. 100) | порог размера для `rotation="size"` |

В `<log>` вкладываются **только** `<event>` и `<property>`. **Если в `<log>` нет ни одного
`<event>` — в журнал не пишется ничего.**

## 4. 🔑 Модель отбора `<event>` / `<property>` (критично — здесь легко ошибиться)

### 4.1. `<event>` — какие СОБЫТИЯ попадают в журнал

- Внутри `<event>` — условия сравнения: **`eq` ne gt ge lt le like**. Каждое сравнивает свойство
  (атрибут `property`) со значением (`value`).
- **Условия внутри одного `<event>` объединяются по «И»; разные `<event>` — по «ИЛИ».**
  Событие пишется, если удовлетворяет всем условиям **хотя бы одного** `<event>`.
- ⇒ Чтобы получить «событие = DBMSSQL **И** база = X», оба условия должны лежать **внутри одного**
  `<event>`:

```xml
<event>
  <eq property="name" value="DBMSSQL"/>
  <eq property="p:processName" value="ut11_saratov"/>
</event>
```

  Положить `<eq property="p:processName">` **отдельным** `<event>` или **прямо в `<log>`** — НЕВЕРНО:
  отдельный `<event>` даст ИЛИ (все DBMSSQL любых баз ИЛИ все события базы X), а bare-`<eq>` под
  `<log>` структурно недопустим (в `<log>` вкладываются только `<event>`/`<property>`).

- Имена событий: `name=` (регистр не важен). Под наши сценарии (полный перечень — §5 `40_TECHLOG`):
  `TLOCK` `TTIMEOUT` `TDEADLOCK` `SDBL` `DBMSSQL` `EXCP` `EXCPCNTX` `CALL` `SCALL` `DBMSSQLCONN`
  `CONN` `SCOM` `EXCP`. Группы `SDBL/DBMSSQL/DB*` сильно растят объём; `PROC/SCOM/EXCP/CONN/ADMIN`
  редки и малы.
- `like` — маска (`%` 0+ символов, `_` 1 символ, `[...]`, `[^...]`, `\` экранирование). Регистр не
  важен; **медленнее** прочих сравнений.

### 4.2. `<property>` — какие СВОЙСТВА события пишутся в журнал

> **🛑 Главное правило: «Свойство события выводится, ТОЛЬКО если для него присутствует элемент
> `<property>`».** Без `<property>` событие пишется **лишь с базовыми полями** (`ts/duration/name/
> depth/level/process`) — без `Sql`, `Context`, `Locks`, `Descr`, `p:processName` и т. д.

- `<property name="Имя">` без вложенных `<event>` → свойство пишется для всех событий, где оно есть.
- `<property name="Имя"><event>…</event></property>` → свойство пишется только для событий,
  удовлетворяющих условию.
- **`<property name="all"/>` → пишутся ВСЕ свойства всех событий.** Именно с ним сняты структуры
  полей на стенде MLC-229 (см. `60_SAFETY` — «полный ТЖ `<property name="all"/>`»).
- `<property name="Context">` (или `all`) включает запись контекста исполнения (стек встроенного
  языка/интерфейсный контекст).

Пример «DBMSSQL длиннее 1 секунды, с текстом SQL» (из офиц. спеки — обратите внимание на единицы):

```xml
<log location="c:\logs" history="1">
  <event><eq property="name" value="dbmssql"/></event>
  <property name="sql">
    <event>
      <eq property="name" value="dbmssql"/>
      <gt property="duration" value="10000"/>   <!-- 10000 = 1 секунда (сотни мкс!) -->
    </event>
  </property>
</log>
```

## 5. ⏱ Длительность: имена свойств и ЕДИНИЦЫ (важная тонкость)

Официальные свойства события:
- **`Duration`** — длительность **в сотнях микросекунд** (100 µs). Используется в фильтрах:
  `<gt property="duration" value="10000"/>` = «дольше 1 c» (10000 × 100 µs = 1 000 000 µs = 1 c).
- **`Durationus`** — длительность **в микросекундах** (µs).
- В **JSON-выводе** базовое поле называется **`duration`** и измеряется в **микросекундах** (µs)
  (офиц. таблица JSON: «Длительность события (в микросекундах) → duration»).

⚠ Сравнение имён свойств **регистронезависимо** (`duration` = `Duration`). **`Dur` — не свойство**
(в перечне его нет) ⇒ фильтр `<… property="Dur">` ни с чем не сопоставится и отберёт 0.

## 6. Элемент `<dump>` (только Windows)

Дамп при аварийном завершении. Атрибуты:

| Атрибут | Значения |
|---|---|
| `create` | `1`/`true` создавать, `0`/`false` не создавать |
| `location` | каталог дампов (отличается от `<log>`) |
| `type` | комбинация флагов (десятич./hex `x0002`): `0` минимальный, `2` (x0002) вся память процесса, `3` (=1+2) «достаточно для большинства случаев», `512` всё адресное пространство |
| `externaldump` | `1` — формировать внешним `dumper.exe` (исключает зависание; рекомендуется для серверов без ежесуточного обслуживания) |
| `prntscrn` | `1` — скриншот при аварии клиента |

Пользователь процесса (у нас агент `.\mitpro`) должен иметь **полные права** на каталоги временных
файлов, ТЖ и дампов; **чтение** — на каталог conf. `<dump>` без `location` пишет в
`%LOCALAPPDATA%\1C\1cv8\dumps`. `<dump>` — про дампы **памяти при крахе процесса**, это НЕ то же, что
события `EXCP`/`EXCPCNTX` (штатно перехваченные/неперехваченные исключения в самом ТЖ).

## 7. `<plansql>` — планы запросов

`<plansql/>` (config-level) включает сбор планов. **Сам план попадает в свойство `planSQLText`** events
СУБД — значит нужны ещё `<property name="planSQLText"/>` и (рекомендуется) `<property name="sql"/>`,
иначе план не запишется. Замедляет выполнение запросов — только для анализа, не для штатной работы.
Формат плана MS SQL: строки `Rows,Executes,EstimateRows,EstimateIO,EstimateCPU,AvgRowSize,
TotalSubtreeCost,EstimateExecutions,StmtText` (последнее поле — до конца строки, запятые внутри
игнорировать; строки разделены переводом строки).

## 8. `<dbmslocks>` — СУБД-блокировки

`<dbmslocks/>` (config-level) включает сбор СУБД-блокировок (раз в ≤2 c доп. запрос «кто кого
заблокировал»). Блокировочные свойства событий: `lka=1` источник, `lkp=1` жертва, `lkpid` номер
запроса (у жертвы), `lkaid` список (у источника), `lksrc` номер соединения источника (у жертвы),
`lkpto`/`lkato` секунды с момента. Алгоритм дерева: `жертва.lksrc` → `источник.connectID`. Также
пространство блокировки — в `Txt` события `TLOCK`. (Подробнее — `40_TECHLOG` §5.)

## 9. Имена и ротация файлов ТЖ

- `placement="folders"` (умолч.): подкаталоги **`<appName>_<pid>`** (`rphost_1234`, `rmngr_…`,
  `ragent_…`), файлы по часам **`ГГММДДЧЧ.log`** (`yymmddhh`). Расширение `.log` при `format="json"`
  **не меняется** (меняется только содержимое).
- `placement="plain"` ИЛИ `rotation="size"`: имя включает `appName_pid`; при `rotation="size"` —
  суффикс `.counter` (`ragent_18972.log.1`, `.2`).
- `compress="zip"`: суффикс `.zip` после ротации.
- `history` чистит старые часы (и пустые каталоги). Чистятся только файлы работающих в этот момент
  приложений.
- **Следствие для парсера (этап B):** при наших настройках (`placement=folders` умолч.,
  `rotation=period` умолч., `compress=none` умолч.) файлы — `<процесс>_<pid>/ГГММДДЧЧ.log`. Но если
  оператор/иной инструмент включит `size`/`zip`, появятся `*.log.N` и `*.log.zip` — парсер должен
  это терпеть (искать `*.log*`, при `.zip` — пропускать или распаковывать).

## 10. JSON-формат строки (офиц.)

`format="json"` → одна JSON-строка на событие; безымянные txt-поля становятся именованными:
`ts` (дата-время до µs, в формате XML), `duration` (µs), `depth` (вложенность в стеке потока),
`name` (имя события); далее — свойства как ключи. Пример офиц.:

```json
{"ts":"2023-08-02T08:48:05.982000","duration":"46998","name":"SCALL","depth":"0","level":"INFO",
 "process":"1cv8c","OSThread":"15968","ClientID":"8","CallID":"28284","MName":"send"}
```

(Нативная 8.5 со стенда — **NDJSON**, объект на строку, файл с BOM, все значения строки; детали и
грабли парсера — `40_TECHLOG` §4/§7.)

## 11. Что из этого прямо влияет на наш генератор/парсер

| Факт спеки | Наша точка |
|---|---|
| `<eq p:processName>` должен лежать **внутри каждого `<event>`** (И с `name`) | изоляция арендатора (60_SAFETY №2) |
| Без `<property>` свойства **не пишутся** (`<property name="all"/>` для всех) | состав полей собранного ТЖ |
| `<plansql>` требует ещё `<property name="planSQLText"/>`+`sql` | сценарий SlowQueries |
| `<dump>` — память при крахе, нужен `location`/`create`/`type`; ≠ `EXCP` | сценарий Exceptions |
| Свойство фильтра — `Duration` (сотни µs)/`Durationus` (µs), не `Dur` | порог длительности |
| `history=0` = не удалять (опасно); ротация/сжатие меняют имена файлов | настройки + парсер |

Сверка нашей реализации с этими пунктами — `CONFORMANCE_LOGCFG.md` (записи о расхождениях).
